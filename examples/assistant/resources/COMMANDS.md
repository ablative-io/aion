# COMMANDS.md — aion authoring lifecycle, exact reference

Ground truth for every command in this file: `docs/packaging.md`,
`examples/assistant/DAVID.md`, `examples/agent-dev/RUNBOOK.md`, and
`crates/aion-cli/src/main.rs` (clap definitions) in the aion repo. Verified
against a real checkout on 2026-07-03 (aion 0.8.0 on that host's PATH). Your
own environment may differ — CHECK, never assume. Every command below is
`aion <subcommand>`; if the binary is missing, the fallback is
`cargo run -p aion-cli -- <subcommand>` run from the aion repo root.

## 1. Preflight — always run this first, every session

```bash
which gleam
which aion
aion --version
gleam --version
git -C /path/to/aion log --oneline -3
```

Expect: two real paths, `aion --version` prints `aion <semver>` (e.g.
`aion 0.8.0`), `gleam --version` prints `gleam <semver>`. Do not assume a
version — the operator's install may be older or newer than what these docs
were written against; if a later command's flags don't match this doc,
`aion <subcommand> --help` is the authority, not this file.

- **`which aion` finds nothing:** every `aion` command below becomes
  `cargo run -p aion-cli -- <subcommand>`, run from the aion repo root. A
  cold `cargo run` compiles the whole `aion-cli` crate graph first — this can
  take 10+ minutes on a clean target dir. Tell the operator before you start
  a cold build; don't let it run silently.
- **`which gleam` finds nothing:** stop. Nothing past step 2 works without
  the Gleam toolchain. Tell the operator; do not attempt to install it
  yourself unless asked.
- **`aion --version` errors or hangs:** the binary on PATH may be a
  different tool or a broken install. Try `aion --help` and read the `about`
  line — it should say "Operate Aion workflows over gRPC and run the Aion
  workflow server". If it doesn't, treat `aion` as absent and use the
  `cargo run` fallback.

## 2. Project layout you are authoring into

A packageable workflow project (per `docs/packaging.md`) is a Gleam project
plus `workflow.toml` next to `gleam.toml`:

```text
my-workflow/
├── gleam.toml
├── manifest.toml          # Gleam lockfile
├── workflow.toml          # Aion packaging descriptor
├── schemas/*.json         # generated — do not hand-author
├── src/
│   ├── my_workflow_io.gleam         # you author this: boundary types
│   ├── my_workflow_activities.gleam # you author this: activity manifest() (optional)
│   ├── my_workflow.gleam            # you author this: entry function
│   ├── my_workflow_codecs.gleam     # GENERATED — never hand-edit
│   └── my_workflow_activity_wrappers.gleam  # GENERATED, only if activities declared
└── build/                  # produced by `gleam build`
```

CHECK which of these exist before assuming a project's shape — a fresh
project may not yet have `workflow.toml` or an `_io` module at all. Use
`aion new <name>` (see `--help`) to scaffold one from a template if starting
from nothing; do not hand-write the skeleton from memory.

Every file whose first line is:

```
//// Generated by aion generate — do not edit; regenerate from the project's types module.
```

is generated. Confirmed generated set for a project with activities:
`<pkg>_codecs.gleam`, `<pkg>_activity_wrappers.gleam`, a per-tier worker
module, a remote wire-compat golden test, `test/<entry>_scaffold_test.gleam`
(write-once — kept once the author fills it in), and every `schemas/*.json`.
Grep for that marker line in a project before touching any file you didn't
just author, to be sure:

```bash
grep -rl "Generated by aion generate" src/ test/ schemas/
```

## 3. `aion generate` — derive codecs/activities from authored types

The source of truth is `src/<package>_io.gleam` (types) and, if present,
`src/<package>_activities.gleam` (`manifest()`). Nothing else feeds
generation; edit those two files, then regenerate.

```bash
aion generate .
```

Runs entirely locally, drives `gleam export package-interface` and a
manifest-extraction probe under the hood. On success prints one JSON
document to stdout, e.g.:

```json
{
  "types_module": "src/my_workflow_io.gleam",
  "codecs_module": "src/my_workflow_codecs.gleam",
  "schemas_emitted": ["input.json", "output.json"],
  "schemas_removed": [],
  "activity_modules": ["src/my_workflow_activity_wrappers.gleam", "..."],
  "test_scaffold": "test/my_workflow_scaffold_test.gleam",
  "declarations": 2,
  "workflow_toml": "synced",
  "action": "written"
}
```

`workflow_toml` tells you what happened to the `activities` list in
`workflow.toml`: `synced` (rewritten), `unchanged`, `skipped` (no activities
module). Re-run `aion generate .` any time you edit `_io.gleam` or
`_activities.gleam` — it is idempotent; running it twice with no source
change produces byte-identical output.

**Drift check (CI-shaped, use it before packaging):**

```bash
aion generate . --check
```

Regenerates everything in memory, writes nothing, exits non-zero naming the
first drifted file if disk doesn't match what generation would produce right
now. Run this before every `aion package` — a stale generated file is a
silent bug (the codec on disk doesn't match the types it's supposed to
encode).

**Failure modes:**

| Symptom | Meaning |
|---|---|
| `.aiongen.lock` error, "a concurrent generate run" | Another `aion generate` is mid-run on this exact project (or a crashed one left the lock — check for a live process before deleting it by hand). |
| Fails inside `gleam export package-interface` with a compile error | Your `_io.gleam` (or something it imports) doesn't compile. Fix the Gleam error first; generation cannot proceed on broken source. |
| "type ... is outside the supported subset" | A type in `_io.gleam` uses a shape the codec generator doesn't support (see `docs/guides/codegen.md` — READ it, don't guess the subset from this file). |
| `--check` reports a drifted file | Run `aion generate .` (no `--check`) to bring disk back in sync, then re-run `--check` to confirm clean. |
| Leftover `*.aiongen-aside` file | A prior run crashed mid-pass before restoring a renamed-aside module. Check it against the real file of the same name minus the suffix; if the real file is missing, rename the `.aiongen-aside` file back by hand and re-run. |

## 4. `gleam build` / `gleam test` / `gleam format` — and the formatting trap

```bash
gleam build
gleam test
```

Standard Gleam toolchain, run from the project root. `gleam build` must
succeed before `aion package` can read `build/dev/erlang` (packaging never
compiles for you unless you pass `--build`, see §5). `gleam test` runs
`gleeunit`; failing generated `*_wire_compat_test.gleam` or
`*_scaffold_test.gleam` means either your types changed in a
wire-incompatible way, or the scaffold test still has its placeholder body —
CHECK which before treating it as a real regression.

**Formatting — never blanket-format over generated files.**

```bash
gleam format --check src test    # safe: read-only, tells you what's unformatted
gleam format src/my_workflow.gleam src/my_workflow_io.gleam src/my_workflow_activities.gleam test/
```

`gleam format` takes explicit file/dir arguments (default `.` = everything).
Generated files (`*_codecs.gleam`, `*_activity_wrappers.gleam`, the worker
module, the wire-compat golden) are emitted by `aion generate` in whatever
byte-exact form its own generator produces — reformatting them is not known
to be safe and is not something this codebase documents as supported. The
concrete risk: `aion generate . --check` (§3) compares disk against a fresh
in-memory regeneration; if `gleam format` has touched a generated file's
whitespace, the next `--check` reports drift that has nothing to do with
your actual types.

**Rule:** name your own authored files explicitly when formatting
(`gleam format src/<entry>.gleam src/<pkg>_io.gleam src/<pkg>_activities.gleam
test/<your-hand-written-tests>.gleam`). Never run bare `gleam format src
test` on a project that already has generated files in it. If you ever do
run a blanket format by mistake, immediately run `aion generate . --check`
(§3); if it reports drift, run `aion generate .` (not `gleam format
--check`) to restore the canonical generated bytes — generation is
idempotent and authoritative over formatting for those files.

## 5. Packaging — `aion package`

**Prefer the installed binary.** Confirm it exists first (§1):

```bash
aion package <dir> --build
```

**Fallback only if `aion` is not on PATH**, and only after warning the
operator:

> A cold `cargo run -p aion-cli -- package ...` compiles the whole
> `aion-cli` crate graph from scratch — this routinely takes 10+ minutes on
> an empty `target/` directory. Say so before you start it; do not run it
> silently and go quiet for ten minutes.

```bash
cargo run -p aion-cli -- package <dir> --build
```

Flags:

- `--build` runs `gleam build` first (compiler output on stderr). Without
  it, packaging an unbuilt project fails with `project is not built: ...
  run gleam build first`.
- `--out <FILE>` overrides the output path; only legal for single-workflow
  projects (one `[[workflow]]` table in `workflow.toml`).
- `--pretty` (global flag, put before the subcommand) pretty-prints the JSON.

On success, one JSON document on stdout, exit 0:

```json
{
  "packages": [
    {
      "workflow_type": "my_workflow",
      "output": "/abs/path/my-workflow/my_workflow.aion",
      "version": "ecc4510a5c16641d9f93c95420cbca302c9f428321e41b84c9db9692a6af881a",
      "deployed_name": "my_workflow$ecc4510a5c16641d9f93c95420cbca302c9f428321e41b84c9db9692a6af881a",
      "modules": 42
    }
  ],
  "excluded": [
    { "module": "aion_flow_ffi", "package": "aion_flow", "reason": "sdk_test_only" }
  ]
}
```

Field meanings — do not guess these, they matter downstream:

- `workflow_type` — the exact string to pass to `aion start` (§6). It is
  `entry_module` from `workflow.toml`, nothing else.
- `output` — the `.aion` archive path. This is what you deploy (§6) — either
  through the console or `aion deploy <this path>`.
- `version` — SHA-256 of compiled module bytes only. Unrelated to
  `workflow.toml` edits (timeout/schemas/activities changes do NOT bump it);
  only compiled-code changes do.
- `deployed_name` — `<workflow_type>$<version>`, the immutable identity the
  server routes to. Useful for `aion versions` / `aion route` /
  `aion unload` later.
- `excluded` — modules the packager deliberately dropped (dev-dependencies,
  SDK test machinery). Not an error; informational.

Extract just the archive path for a script: `jq -r '.packages[0].output'`.

Errors print an `Error:` chain on stderr, exit 1 (bad project state) or exit
2 (bad CLI usage — wrong flags). Common ones, verbatim cause:

| Error text (substring) | Fix |
|---|---|
| `no workflow.toml found in <root>` | Wrong directory, or `workflow.toml` not yet written. |
| `project is not built: <path> does not exist; run gleam build first` | Run `gleam build`, or add `--build`. |
| `invalid workflow.toml: <field>: <reason>` | A semantic rule failed (zero timeout, empty entry module, duplicate activities, etc). The message names the field. |
| `entry module <m> not found in compiled output` | `entry_module` in `workflow.toml` doesn't match a compiled module name. Nested modules use `@` not `/` (`billing/cycle.gleam` → `billing@cycle`). |
| `dependency <package> is in gleam.toml but missing from manifest.toml; rebuild` | Lockfile stale — rebuild so Gleam regenerates `manifest.toml`. |
| `module <m> uses an engine-reserved namespace` | You (or a dependency) defined a module named `aion_flow_ffi` or under `aion@testing`; rename it. |
| `--out is only valid for single-workflow projects` | Drop `--out`, use per-workflow `output` in `workflow.toml` instead. |

## 6. Deploy + run

### Server basics (defaults — CHECK the running config, don't assume)

From `examples/agent-dev/demo-config.toml` (a representative dev config):

- Console / HTTP: `127.0.0.1:8080`
- gRPC: `127.0.0.1:50051`
- Liminal (server-push worker listener, only if `[outbox] enabled = true`):
  `127.0.0.1:50061`

These are config-file values, not hardcoded — a different `--config` or
`[server]` block changes them. Read the operator's actual config
(`aion.toml` or whatever `--config` path they used) before assuming ports;
`aion` CLI's own default `--endpoint` is `127.0.0.1:50051` if unset.

`aion server --config <path>` starts it. Expect in the boot log: the store
opening (haematite or the configured backend), gRPC/HTTP bind lines, and (if
outbox enabled) the liminal listener. Deploy through the console requires
`[deploy] enabled = true` in that config — without it the console shows
"Deploy is disabled" and packages can only be pre-loaded via
`workflow_packages` at server startup.

### Deploy

**Console (preferred, matches operator workflow):** Actions/deploy page →
upload the `.aion` file from §5's `output` field. Expect the workflow_type to
appear in the workflow/version list after upload.

**Terminal:**

```bash
aion deploy <path/to/workflow.aion>
```

Requires the server's `[deploy]` surface enabled. With no `--token` and no
`AION_TOKEN` env var, the CLI sends the development `x-aion-deploy: true` +
`x-aion-subject` headers automatically — this only works against servers
running in dev/no-auth mode. Against an authenticated server you need
`--token` (or `AION_TOKEN`) carrying the `deploy` claim; a valid subject with
no deploy grant is explicitly denied (verified in
`crates/aion-server/src/api/deploy_grpc.rs`).

### Start a run

```bash
aion start <workflow_type> --input-file path/to/input.json
# or inline:
aion start <workflow_type> --input '{"key":"value"}'
```

`<workflow_type>` is the `entry_module` / the packaging output's
`workflow_type` field (§5) — not a display name, not the `.aion` filename.
`--input-file` and `--input` conflict; pick one. On success prints the
started workflow's id/run id as JSON.

**"workflow type X is not registered"** — the exact server error string
(`crates/aion-server/src/api/handlers/error.rs`). Two causes, check both:

1. **Not deployed at all** — the package was never loaded into this server
   (wrong server, server restarted without `workflow_packages`, or the
   deploy step silently failed). Confirm with `aion versions --workflow-type
   <name>` — an empty result means truly not registered.
2. **Wrong namespace** — the workflow type is registered somewhere but you
   started against the wrong `--namespace` (default `default`; the demo
   configs also default to `default`). Re-check the operator's intended
   namespace before assuming a redeploy is needed.

Also confirm a worker is actually up and serving the activity types this
workflow schedules — a workflow that starts fine but never progresses past
its first activity, with no error, usually means no worker is connected for
that activity type (check the console's worker/cluster view, or the
worker's own stderr for "connected and registered; serving").

### Checking state

```bash
aion list                                   # all workflow executions
aion list --status running                  # filter: running|completed|failed|cancelled|timed-out|continued-as-new
aion describe <workflow_id>                 # latest run, decoded payloads
aion describe <workflow_id> --run-id <id>   # specific run
aion describe <workflow_id> --raw           # raw payload bytes instead of decoded
aion query <workflow_id> <query_name>       # e.g. a status query the workflow exposes
aion versions --workflow-type <name>        # loaded versions + routing flag
```

`aion inspect <workflow_id>` time-travels the recorded history (event-store
oplog) — useful for diagnosing "why did this run take the path it took"
without a debug log; see its `--help` for the `--from`/`--mock` what-if
fork, which is read-only and never writes the event log.

## 7. Quick sanity order for one authoring loop

1. `which gleam && which aion && aion --version` (§1)
2. Edit `<pkg>_io.gleam` / `<pkg>_activities.gleam` (never the generated files)
3. `aion generate .` then `aion generate . --check` to confirm clean (§3)
4. `gleam build && gleam test` (§4)
5. Format only your own files, never a blanket `src test` sweep (§4)
6. `aion package . --build` (or the cargo fallback, with the 10-minute
   warning) → note `workflow_type` and `output` from the JSON (§5)
7. Deploy the `output` archive (console or `aion deploy`) (§6)
8. `aion start <workflow_type> --input-file ...`, then `aion list` /
   `aion describe` to confirm it's actually running against real workers,
   not just accepted (§6)

Never report a step "done" from reading source alone — every claim above
about output shape or error text should be reproduced by actually running
the command in this session before you tell the operator it worked.
