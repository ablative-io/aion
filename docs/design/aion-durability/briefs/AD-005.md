# AD-005: Non-determinism violation detection

> **Cluster:** aion-durability
> **Depends on:** AD-004, AD-002
> **Checklist:**
> - C20 — When the next recorded event differs in family or correlation key from the issued Command, the Resolver raises NonDeterminismError and never returns a Resolution for that command.
> - C21 — A non-determinism violation causes the workflow to fail deterministically: exactly one WorkflowFailed is recorded (via the Recorder) carrying the violation classification.
> - C22 — A dedicated test drives a recorded history that deliberately diverges from the replayed command stream (wrong activity family, wrong correlation key) and asserts the typed NonDeterminismError with correct expected/found shapes.
> **Stories:**
> - S7 (Operator, Recovering and Diagnosing) — As an operator, I want a non-deterministic workflow to fail loudly with an error that names the workflow, the position, and what diverged so that I can find and fix the offending workflow code instead of debugging corrupted state.

## Purpose

Complete the violation-detection path that makes Aion safe: when replayed workflow code issues a command that does not line up with recorded history, the Resolver must catch it at the point of mismatch and fail the workflow loudly with a typed error, never silently producing a wrong Resolution or corrupting state. This is the single most safety-critical behaviour in the cluster and is held to dedicated test coverage.

## Task

Fill in the Resolver's Mismatch arm so a cursor Mismatch becomes a NonDeterminismError carrying the workflow id, the cursor's current sequence position, the expected command shape, and the found recorded-event shape. Wire the deterministic-failure consequence: a violation causes the workflow to fail deterministically, recording exactly one WorkflowFailed (via the Recorder from AD-002) classified as a non-determinism failure. Add the dedicated determinism-violation test file driving a recorded history that deliberately diverges from the replayed command stream. Test against InMemoryStore.

## Requirements

### R1: Mismatch -> typed NonDeterminismError

WHEN the HistoryCursor reports a Mismatch for an issued Command, THE Resolver SHALL raise a NonDeterminismError carrying the WorkflowId, the cursor's current sequence position, the expected command shape (family + correlation key), and the found recorded-event shape. THE Resolver SHALL NOT return any Resolution for that Command.

**Acceptance:**
- resolve(command) returns Err(NonDeterminismError{..}) when the cursor reports Mismatch.
- The error's expected field names the issued Command's family and CorrelationKey; the found field names the recorded event's family and key.
- The error's seq equals the cursor position at the point of mismatch.
- There is no code path where a Mismatch yields Ok(Resolution) (CO10) — a test asserts the error is returned, not a guessed Resolution.

### R2: Deterministic workflow failure on violation

WHEN a NonDeterminismError is raised during replay or live execution, THE SYSTEM SHALL fail the workflow deterministically by recording exactly one WorkflowFailed (via the Recorder) carrying a non-determinism failure classification, and SHALL NOT record more than one terminal failure for the same violation.

**Acceptance:**
- A function (e.g. fail_on_violation) takes the Recorder and the NonDeterminismError and records a single WorkflowFailed whose WorkflowError reflects the non-determinism classification.
- Calling it is idempotent at the call site (it is invoked once per violation; a guard or single call-site ensures one terminal event).
- A test seeds a workflow, triggers a violation, invokes the failure path, reads history, and asserts exactly one terminal WorkflowFailed event with the non-determinism classification.
- The WorkflowError variant/message used for the non-determinism case is documented so AE surfaces it consistently.

### R3: Dedicated divergence test suite

THE SYSTEM SHALL provide a dedicated test that drives a recorded history which deliberately diverges from the replayed command stream and asserts the typed NonDeterminismError with the correct expected and found shapes.

**Acceptance:**
- tests/determinism_violation.rs builds a recorded history (over InMemoryStore) and replays a command stream that diverges in two distinct ways: (a) wrong family (workflow issues a timer where history recorded an activity) and (b) wrong correlation key (workflow issues Activity(1) where history recorded Activity(0) next / a signal of the wrong name).
- Each case asserts Err(NonDeterminismError) and asserts the expected and found descriptors and the sequence position.
- The test is a real runtime test (not #[ignore], not env-gated) and passes under cargo test.
- A control case in the same file replays a matching command stream and asserts no violation, proving the detector does not false-positive.

## Boundaries

- SHALL NOT weaken detection with tolerance or heuristic matching — detection is exact family + key comparison at the cursor position (CO10).
- SHALL NOT swallow, log-and-continue, or retry past a violation.
- SHALL NOT define the WorkflowError type itself (that is aion-core AC-004) — it selects/constructs the appropriate existing variant for the non-determinism case.
