# AD-006: Determinism context: recorded-now and seeded RNG

> **Cluster:** aion-durability
> **Depends on:** AD-001
> **Checklist:**
> - C23 — DeterminismContext exposes the recorded timestamp of the event currently being applied as the sole source for workflow.now; it advances as the Resolver consumes recorded events.
> - C24 — No code path in this cluster calls the wall clock (SystemTime::now / Utc::now) to produce a workflow-visible value.
> - C25 — DeterminismContext provides a deterministic RNG seeded solely from WorkflowId + RunId as the sole source for workflow.random.
> - C26 — Two DeterminismContexts constructed for the same WorkflowId + RunId produce identical random sequences (asserted by test).
> **Stories:**
> - S9 (Workflow Author, Writing Deterministic Workflows) — As a workflow author, I want workflow.now to return the recorded event time and workflow.random to return a workflow-seeded value so that my workflow replays to exactly the same state every time without me thinking about it.

## Purpose

Supply the two non-deterministic inputs that workflow code is allowed to read — current time and randomness — in a form that is identical on every replay. workflow.now returns the recorded timestamp of the event currently being processed; workflow.random returns a value from an RNG seeded solely from the workflow identity. These are the engine-side behaviours the Gleam SDK bindings (AF) resolve to, and they are what make a workflow replay to exactly the same state.

## Task

Implement DeterminismContext (determinism.rs). It holds the recorded timestamp of the event currently being applied, advanced as the Resolver consumes recorded events, exposed as the sole source for workflow.now. It holds a deterministic RNG seeded only from WorkflowId + RunId, exposed as the sole source for workflow.random. No code path in this cluster may call the wall clock or any entropy source to produce a workflow-visible value. Test that two contexts for the same run produce identical random sequences and that now reflects the advancing recorded timestamp.

## Requirements

### R1: Recorded-now source

THE SYSTEM SHALL define DeterminismContext exposing the recorded timestamp of the event currently being applied as the sole source for workflow.now. THE timestamp SHALL advance as the Resolver consumes recorded events and SHALL NOT be derived from the wall clock.

**Acceptance:**
- determinism.rs defines DeterminismContext with a current recorded timestamp (DateTime<Utc>) and a now() accessor returning it.
- A method advances the current timestamp to a newly-applied event's recorded_at (called by the Resolver/replay driver as it consumes events).
- Before any event is applied, now() returns the WorkflowStarted timestamp (the run's first recorded time).
- A unit test applies a sequence of events with increasing timestamps and asserts now() returns each in turn.

### R2: No wall-clock for workflow-visible values

THE SYSTEM SHALL NOT call SystemTime::now, Utc::now, or any wall-clock source anywhere in the durability module set to produce a value visible to workflow code.

**Acceptance:**
- A grep over crates/aion/src/durability/ finds no SystemTime::now or Utc::now used to produce a now()/random() value (the only permitted wall-clock read, if any, is the engine-supplied recovery as_of for expired_timers, which is not workflow-visible and is documented as such).
- DeterminismContext::now derives strictly from the stored recorded timestamp.
- A unit test confirms two DeterminismContexts seeded with the same recorded-event sequence return identical now() values at each step (no clock drift).

### R3: Seeded deterministic RNG

THE SYSTEM SHALL provide, on DeterminismContext, a deterministic RNG seeded solely from WorkflowId + RunId as the sole source for workflow.random. THE RNG SHALL NOT consult any entropy source (OS RNG, time, thread id).

**Acceptance:**
- DeterminismContext constructs its RNG from a seed derived deterministically from WorkflowId and RunId (e.g. a stable hash of their bytes into the seed of a named, versioned PRNG such as ChaCha or a fixed PCG).
- A next_random (or typed draw) method advances the RNG deterministically.
- The PRNG algorithm is named and fixed (documented) so the sequence is stable across builds and platforms.
- A unit test draws N values, reconstructs a fresh context for the same WorkflowId+RunId, draws N again, and asserts the sequences are identical.

### R4: Cross-replay determinism guarantee

WHEN two DeterminismContexts are constructed for the same WorkflowId + RunId, THE SYSTEM SHALL produce identical random sequences from them. WHEN constructed for different RunIds, THE SYSTEM SHALL produce different sequences with overwhelming probability.

**Acceptance:**
- A test asserts byte-for-byte identical random sequences for two contexts of the same WorkflowId+RunId.
- A test asserts the sequences differ for two contexts of the same WorkflowId but different RunId.
- The seed derivation is total (no panic) for all WorkflowId/RunId values.
- No unwrap/expect/panic in the seed-derivation or draw paths.

## Boundaries

- SHALL NOT define the Gleam workflow.now / workflow.random functions — those are AF; this defines only the engine-side behaviour they bind to.
- SHALL NOT generate the recorded timestamps written into events — that is the engine at record time (AD-002 takes the timestamp as input); this only reads recorded timestamps back during replay.
- SHALL NOT use the system RNG or any non-seeded source.
