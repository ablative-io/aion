# AD-007: Live executor seam and resume-live handoff

> **Cluster:** aion-durability
> **Depends on:** AD-004, AD-002
> **Checklist:**
> - C27 — A LiveExecutor trait defines the engine-implemented operations AD invokes only on resume-live: run an activity, start a timer, await a signal, spawn a child workflow.
> - C28 — While the cursor can satisfy a command, the LiveExecutor is never invoked (replay performs no live side effect).
> - C29 — After ResumeLive, the handoff glue invokes the LiveExecutor and records the resulting event(s) through the Recorder.
> **Stories:**
> - S5 (Engine, Recording and Replaying Workflows) — As the engine, I want a single seam (LiveExecutor) that is called only after the resume point so that replay performs no live side effect and live execution is cleanly separated.

## Purpose

Define the single seam between durability (AD) and the engine (AE): the LiveExecutor trait that AE implements to actually run an activity, start a timer, await a signal, or spawn a child. The handoff rule is the load-bearing invariant — the LiveExecutor is invoked only after the resume point, never during replay — and after a live action its resulting event is recorded through the Recorder. This is how replay performs no side effect and live execution is cleanly separated.

## Task

Define the LiveExecutor trait (executor.rs) with one operation per world-touching command family. Implement the handoff glue: a function that, given a Command and a Resolver, returns the recorded Resolution when the cursor satisfies it, and otherwise (ResumeLive) invokes the LiveExecutor for that command and records the resulting event(s) through the Recorder, producing a live Resolution. The invariant is that the LiveExecutor is never touched while the cursor can satisfy a command. Test with a recording mock LiveExecutor over InMemoryStore that asserts zero live calls during replay and exactly the expected live calls after resume.

## Requirements

### R1: LiveExecutor trait

THE SYSTEM SHALL define a LiveExecutor trait, implemented by the engine (AE), with one async operation per world-touching family: run an activity, start a timer, await a signal, and spawn a child workflow. Each operation SHALL return the live outcome that AD will both return and record.

**Acceptance:**
- executor.rs defines trait LiveExecutor: Send + Sync with methods run_activity, start_timer, await_signal, spawn_child taking the relevant Command payload and returning the live outcome (Payload / fired / signal payload / child result or error).
- The trait is object-safe (usable as &dyn LiveExecutor) or generic — pick one and document it; the choice must let AE supply a beamr-backed implementation without AD depending on beamr.
- AD provides no production implementation of LiveExecutor (that is AE's); it provides only the trait and a test mock.
- Doc comments state the AD/AE boundary: AD calls this only on resume-live.

### R2: No live call while the cursor can satisfy a command

WHILE the HistoryCursor can satisfy a Command (the Resolver returns a recorded Resolution), THE SYSTEM SHALL NOT invoke any LiveExecutor operation.

**Acceptance:**
- The handoff function returns the recorded Resolution from the Resolver without touching the LiveExecutor when the cursor matches.
- A test replays a fully-recorded history through the handoff with a mock LiveExecutor that panics if any method is called, and the replay completes with zero panics (proving no live call) (CO11).
- The test asserts each returned Resolution equals the recorded outcome.

### R3: Resume-live handoff records the live outcome

WHEN the Resolver returns ResumeLive for a Command, THE SYSTEM SHALL invoke the corresponding LiveExecutor operation and SHALL record the resulting event(s) through the Recorder, then return the live outcome as a Resolution.

**Acceptance:**
- On ResumeLive for RunActivity, the handoff calls LiveExecutor::run_activity, records ActivityScheduled (if not already), ActivityStarted/Completed (or Failed) via the Recorder, and returns the live Resolution.
- The events recorded on the live path are appended via the Recorder (single-writer, correct expected_seq) — no direct EventStore::append.
- A test replays a partial history to its resume point, then drives one live RunActivity with a mock LiveExecutor returning a known Payload, and asserts: the LiveExecutor was called once, the new ActivityCompleted is in the store at the next sequence, and the returned Resolution carries the live Payload.
- Re-reading the store after the live step and replaying again returns the now-recorded outcome from cache with zero further live calls (proving the live result was durably recorded).

## Boundaries

- SHALL NOT implement a real (beamr-backed) LiveExecutor — that is AE; AD provides the trait and a test mock only.
- SHALL NOT execute timers/signals/children's live semantics itself — the trait delegates to AE/AT; AD only orchestrates the call-and-record handoff.
- SHALL NOT append events outside the Recorder (CO6).
