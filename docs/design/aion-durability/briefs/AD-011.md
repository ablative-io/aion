## Purpose

Lay the durability-core foundation for resuming a failed workflow from the step that failed. A Failed workflow still holds its whole event history, so resume = re-drive from the failed step while replay returns the recorded results of the steps that already succeeded. The obstacle is that the failed step has a recorded terminal ActivityFailed, so faithful replay would return that failure again, and the workflow's WorkflowFailed makes its status terminal. This brief introduces a WorkflowResumed compensating event that supersedes the terminal failure in the status projection (returning the run to Running) and reopens the failed step in replay so it re-dispatches live, and it consolidates terminal detection so every site agrees that 'terminal' means 'terminal since the last reopen point'. This refines load-bearing invariant #4 (status is a projection; one terminal event per terminal status) to 'one terminal event per lease of a run', which is signed off in docs/WORKFLOW-RESUME-DESIGN.md section 5. This brief is the foundation only; the reopen operation (AD-012), the operator surface (AO-007), and discovery enrichment (visibility) build on it.

## Task

In aion-core add a WorkflowResumed event variant carrying the reopened run id and the activity correlation keys to re-dispatch, and project it to Running in status_from_events (reusing the existing reverse-scan supersession). Thread the variant through every exhaustive match (event envelope, filter/run-chain end-timestamp projection, libsql event_kind, resolver/cursor event-kind labels, visibility, engine api/delegated folds, server stream selector) — it is serde-transparent at the wire/store layers so no proto or codec work is needed. Consolidate the scattered terminal-detection helpers (terminal_outcome_from_history, run_has_terminal_history, terminal_recorded_at, the startup is_terminal gate, and the memory/libsql list_active filters) onto one reset-aware predicate: a run is terminal iff a terminal event was recorded since its last reopen point. Scope the exactly-one-terminal guard (ensure_no_recorded_terminal and the process-exit recorder) to the current lease so a reopened run can record a fresh terminal but cannot double-record within a lease. Teach the history cursor that a WorkflowResumed naming key K is a reset point for K: superseded prior attempts are walked past and K resolves to its next recorded attempt or reports exhausted (ResumeLive) for live re-dispatch. Add a single-writer Recorder builder for WorkflowResumed. Cover every behaviour with real runtime tests over InMemoryStore. No reopen operation, server RPC, CLI, or visibility-column work in this brief.

### R2: Reset-aware terminal detection (invariant #4 refinement)

THE SYSTEM SHALL detect run terminality through one reset-aware predicate — a run is terminal iff a terminal event was recorded since its last reopen point (or run start) — used by every terminal-detection site, and SHALL scope the exactly-one-terminal guard to the current lease.

**Acceptance:**
- A single shared predicate decides 'is this run currently terminal' by scanning the run segment back to the last WorkflowResumed (or run start) and looking only at terminal events after it; terminal_outcome_from_history, run_has_terminal_history, terminal_recorded_at, the startup is_terminal gate, and the memory and libsql list_active filters all route through it (or its directly-derived form) and agree.
- list_active and startup recovery include a workflow whose latest events are [WorkflowFailed, WorkflowResumed] and exclude one whose latest event is WorkflowFailed with no later reopen.
- ensure_no_recorded_terminal and the process-exit terminal recorder reject a second terminal recorded within the same lease but permit a fresh terminal recorded after an intervening WorkflowResumed.
- Signal routing and completion that previously treated any terminal as permanent now permit a reopened (Running) run to receive signals and to complete normally.
- Real runtime tests over InMemoryStore cover: reopened run is active again; second-terminal-within-lease rejected; terminal-after-reopen accepted; signal accepted post-reopen.

### R3: Cursor reopen reset rule for replay re-dispatch

THE SYSTEM SHALL treat a WorkflowResumed event naming correlation key K as a reset point for K in the history cursor, so that replay supersedes K's pre-reset attempts and resolves K to its next recorded attempt or, if none yet, reports K exhausted so the resolver returns ResumeLive for live re-dispatch.

**Acceptance:**
- When the cursor walks events for key K and encounters a WorkflowResumed naming K, attempts recorded for K before that marker are treated as superseded and are not returned as K's resolution.
- Replay of [ActivityScheduled(K), ActivityFailed(K, terminal), WorkflowFailed, WorkflowResumed{K}] with no later attempt resolves K to ResumeLive (live re-dispatch), not ActivityFailedTerminal.
- Replay of [ActivityScheduled(K), ActivityFailed(K, terminal), WorkflowFailed, WorkflowResumed{K}, ActivityScheduled(K), ActivityCompleted(K)] resolves K to the recorded ActivityCompleted on every replay (determinism holds because the marker is recorded).
- Reopen lists only activities with a terminal failure and no later success; an in-flight (scheduled, no terminal) activity is not named in WorkflowResumed and still resolves to ResumeLive via the existing exhaustion path.
- Existing retry-chain replay (walks_retry_failures_to_eventual_activity_success) and all prior cursor behaviour are unchanged for histories without a WorkflowResumed.
- Real runtime tests over InMemoryStore cover each case above, including a negative test proving the pre-reset terminal is no longer returned.

### R4: Single-writer Recorder emit for WorkflowResumed

THE SYSTEM SHALL append WorkflowResumed only through the single-writer Recorder, using the same WriteToken and expected-sequence discipline as every other recorded event.

**Acceptance:**
- Recorder exposes a builder method that appends WorkflowResumed at the recorder's tracked head with a single engine-supplied recorded timestamp and the reopened run id + keys, returning the recorded event.
- No code path constructs or appends WorkflowResumed outside the Recorder; the store WriteToken discipline is unchanged.
- The reset-aware terminal helpers owned by the Recorder (e.g. terminal_recorded_at) account for the reopen marker per R2.
- A SequenceConflict on this append surfaces a hard DurabilityError exactly as for other events (double-writer bug), with no head re-read/retry.
- Real runtime tests assert the recorder appends the event at the correct sequence and that it is the only append path.

## Boundaries

- SHALL NOT implement the reopen operation (acquire writer for a dead workflow + append + re-drive via recovery) — that is AD-012.
- SHALL NOT add the ResumeWorkflow server RPC, the aion resume CLI, or the client op — that is AO-007.
- SHALL NOT add failed_step/failure_reason summary projection or any aion list output change — that is the visibility brief.
- SHALL NOT change the wire/proto contract or add a hand-written per-variant codec — the event is serde-transparent at the store and wire layers.
- SHALL NOT introduce automatic retry, classification, or any RetryPolicy plumbing — that is the L1+L2 follow-on.
- SHALL NOT alter behaviour for any history that contains no WorkflowResumed event — all prior projection, replay, recovery, signal, and completion behaviour is preserved exactly.
