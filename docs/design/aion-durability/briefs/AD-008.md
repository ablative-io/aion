# AD-008: Replay driver and activity-result caching

> **Cluster:** aion-durability
> **Depends on:** AD-004, AD-006, AD-007
> **Checklist:**
> - C30 — replay builds a Resolver from read_history and a DeterminismContext, and drives a freshly-created execution forward, satisfying each command from the cursor until exhaustion.
> - C31 — On replay a matched ActivityCompleted returns its recorded result Payload without dispatching the activity; a matched exhausted-retry ActivityFailed returns the recorded terminal ActivityError.
> - C32 — replay returns a resume point (the first command whose recorded event is absent) so the engine can continue live execution from there.
> - C33 — Behavioural tests over InMemoryStore assert: a fully-recorded history replays to its recorded terminal state with zero live calls; a partial history replays to the correct resume point; activity results are returned from cache.
> **Stories:**
> - S2 (Engine, Recording and Replaying Workflows) — As the engine, I want to replay a workflow from its recorded history and have every activity, timer, signal, and child call return its recorded outcome so that I reconstruct the exact prior state without repeating side effects.
> - S3 (Engine, Recording and Replaying Workflows) — As the engine, I want replay to tell me the resume point so that I know precisely where live execution should continue.
> - S4 (Engine, Recording and Replaying Workflows) — As the engine, I want activity results served from recorded history during replay so that a completed activity is never dispatched a second time.
> - S10 (Workflow Author, Writing Deterministic Workflows) — As a workflow author, I want replay to be invisible so that I write a normal deterministic function and never code differently for the replay case.

## Purpose

Assemble the parts into the replay engine: given a workflow's history, build a Resolver and a DeterminismContext, drive a freshly-created execution forward satisfying every world-touching command from recorded history, and return the resume point so the engine continues live execution from exactly there. Activity results are served from history (the cursor is the cache), so a completed activity is never dispatched twice. Replay is invisible to workflow code.

## Task

Implement the replay driver (replay.rs). It takes a WorkflowId, the read_history output, and the collaborators (Resolver over a HistoryCursor, DeterminismContext, Recorder, LiveExecutor). It drives the execution by repeatedly accepting the next Command the workflow issues and resolving it: recorded Resolutions are returned and the DeterminismContext timestamp advances; the first Exhausted is the resume point. Activity-result caching falls out of the Resolver (matched ActivityCompleted -> recorded Payload; matched exhausted ActivityFailed -> recorded terminal error). Add a behavioural test file over InMemoryStore. Note: replay does not itself run the Gleam workflow function (AE drives the BEAM process); replay provides the resolution loop AE feeds commands into and consumes the resume point from.

## Requirements

### R1: Replay driver assembly

THE SYSTEM SHALL provide a replay entry point that, given a WorkflowId and its recorded history, constructs a HistoryCursor, a Resolver, and a DeterminismContext, and drives command resolution forward until the cursor is exhausted.

**Acceptance:**
- replay.rs defines a Replay (or replay function) taking the WorkflowId, the Vec<Event> history, and the Recorder + LiveExecutor handles.
- It builds the DeterminismContext seeded from the WorkflowId+RunId in the history's WorkflowStarted and initialises its recorded-now to that start timestamp.
- It exposes a step that accepts the next Command from the (engine-driven) workflow and returns the Resolution / ResumeLive / error from the Resolver, advancing the DeterminismContext timestamp on each consumed recorded event.
- A unit test drives a sequence of commands matching a recorded history and asserts each Resolution and that the DeterminismContext now() advanced to each event's recorded timestamp.

### R2: Activity-result caching from history

WHILE replaying, WHEN a RunActivity command matches a recorded ActivityCompleted, THE SYSTEM SHALL return the recorded result Payload without invoking the LiveExecutor; WHEN it matches a recorded exhausted-retry ActivityFailed, THE SYSTEM SHALL return the recorded terminal ActivityError.

**Acceptance:**
- A matched ActivityCompleted yields Resolution::ActivityCompleted with the exact recorded Payload and zero LiveExecutor calls.
- A matched exhausted ActivityFailed yields Resolution::ActivityFailedTerminal with the exact recorded ActivityError and zero LiveExecutor calls.
- There is no separate cache structure; the result comes from the HistoryCursor (the history is the cache).
- A test asserts a recorded activity is served from cache and the mock LiveExecutor is never called for it.

### R3: Resume point reporting

THE SYSTEM SHALL return the resume point — the first Command whose recorded event is absent (the first ResumeLive) — so the engine can continue live execution from exactly there.

**Acceptance:**
- When the cursor exhausts mid-stream, the driver signals ResumeLive for that command and the engine takes over (via the AD-007 handoff) from that command onward.
- A fully-recorded terminal history (ending in WorkflowCompleted/Failed/Cancelled) drives to completion with no ResumeLive — the resume point is 'terminal', reported distinctly.
- A test asserts: a partial history reports ResumeLive at the correct command index; a complete history reports terminal with no ResumeLive.

## Boundaries

- SHALL NOT spawn or drive the BEAM workflow process — AE owns the process; replay provides the resolution loop AE feeds commands into.
- SHALL NOT implement live timer/signal/child semantics — those are AT, reached via the LiveExecutor after the resume point.
- SHALL NOT iterate the active-workflow set or run on startup — that is recovery, AD-009.
