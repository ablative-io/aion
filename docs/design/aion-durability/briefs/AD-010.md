# AD-010: End-to-end record-then-replay round-trip

> **Cluster:** aion-durability
> **Depends on:** AD-008, AD-006, AD-005
> **Checklist:**
> - C38 — An integration test records a workflow's events via the Recorder, then replays from the same store and asserts the replayed Resolutions match the recorded outcomes in order and the resume point is correct.
> - C39 — An integration test asserts workflow.now values during replay equal the recorded event timestamps in order, and that random draws match across two independent replays of the same run.
> **Stories:**
> - S2 (Engine, Recording and Replaying Workflows) — As the engine, I want to replay a workflow from its recorded history and have every activity, timer, signal, and child call return its recorded outcome so that I reconstruct the exact prior state without repeating side effects.
> - S9 (Workflow Author, Writing Deterministic Workflows) — As a workflow author, I want workflow.now to return the recorded event time and workflow.random to return a workflow-seeded value so that my workflow replays to exactly the same state every time without me thinking about it.
> - S10 (Workflow Author, Writing Deterministic Workflows) — As a workflow author, I want replay to be invisible so that I write a normal deterministic function and never code differently for the replay case.

## Purpose

Prove the whole durability loop closes: events recorded through the Recorder, when replayed from the same store, reproduce the same Resolutions in order, land on the correct resume point, and reproduce the same workflow.now timestamps and workflow.random draws. This is the integration proof that recording and replay are two halves of one mechanism and that determinism holds across independent replays.

## Task

Add an integration test (in tests/replay_behaviour.rs or a dedicated tests/round_trip.rs) that exercises the full record-then-replay loop over InMemoryStore. First, record a workflow's lifecycle (start, several activities, a timer, a signal, completion) via the Recorder using engine-supplied recorded timestamps. Then construct a fresh replay from the same store and assert the replayed Resolutions match the recorded outcomes in order and the resume point is terminal. Separately, assert determinism: workflow.now during replay equals the recorded event timestamps in order, and two independent replays of the same run produce identical random draws. No new production logic beyond what AD-001..009 deliver — this brief is the integration assertion layer.

## Requirements

### R1: Record-then-replay Resolution round-trip

THE SYSTEM SHALL provide an integration test that records a workflow's full event history via the Recorder and then replays from the same store, asserting the replayed Resolutions equal the recorded outcomes in order and the resume point is terminal.

**Acceptance:**
- The test records WorkflowStarted, then ActivityScheduled/Completed for two activities, a TimerStarted/TimerFired, a SignalReceived, and WorkflowCompleted, all via the Recorder into InMemoryStore with monotonically increasing engine-supplied recorded timestamps.
- It then builds a replay from read_history and drives the matching command stream, asserting each Resolution equals the recorded outcome (activity result Payloads, timer fired, signal payload) in order.
- It asserts the replay reaches terminal with no ResumeLive (the history is complete).
- The test is a real runtime test (not #[ignore], not env-gated) and passes under cargo test.

### R2: Determinism round-trip: now and random

THE SYSTEM SHALL provide an integration test asserting that workflow.now values during replay equal the recorded event timestamps in order, and that workflow.random draws are identical across two independent replays of the same run.

**Acceptance:**
- During the replay of R1's history, the test reads DeterminismContext::now() as each event is consumed and asserts it equals that event's recorded timestamp, in order.
- The test runs two independent replays of the same WorkflowId+RunId, drawing the same number of random values from each, and asserts the two draw sequences are byte-for-byte identical.
- A negative assertion confirms a replay of a different RunId yields a different random sequence.
- The test is a real runtime test and passes under cargo test.

## Boundaries

- SHALL NOT add new production behaviour — it asserts the integrated behaviour of AD-001 through AD-009.
- SHALL NOT depend on AT, AF, or a real beamr process — it drives the resolution loop directly with a test command stream and a mock/absent LiveExecutor (the history is complete, so no live calls occur).
- SHALL NOT use a durable backend — InMemoryStore only (durable round-trip across process restart is exercised by AS/AX).
