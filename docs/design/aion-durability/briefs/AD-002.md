# AD-002: Recorder and single-writer append path

> **Depends on:** AD-001

> - C8 — Recorder exposes recording methods for each event family: workflow lifecycle, activity (scheduled/started/completed/failed), timer (started/fired/cancelled), signal received, and child workflow (started/completed/failed).
> - C9 — When EventStore::append returns SequenceConflict, the Recorder surfaces a hard DurabilityError (double-writer bug) and does not retry by re-reading the head.

## Purpose

Implement the single writer of a workflow's event history. The Recorder owns the per-workflow sequence head and is the only code path that appends to that workflow's log, structurally enforcing the single-writer discipline that keeps sequence numbers monotonic and gap-free. This is the recording half of durability.

## Task

Build a sequence-head tracker (seq.rs) and the Recorder (recorder.rs). The tracker holds the current head sequence for one workflow and advances it only on a successful append. The Recorder is constructed for a single WorkflowId, computes expected_seq from the tracked head for every append, and exposes a recording method per event family (lifecycle, activity, timer, signal, child). A SequenceConflict from the store is a double-writer bug and is surfaced as a hard DurabilityError, never retried by re-reading the head. The Recorder can be initialised at a head implied by existing history so a recovered workflow continues at the right sequence. Test against InMemoryStore from aion-store.

## Requirements

### R1: Per-workflow sequence-head tracker

THE SYSTEM SHALL define a sequence-head tracker that holds the current head sequence for one workflow and advances it by the appended batch length only after EventStore::append succeeds. IF an append fails, THEN the head SHALL NOT advance.

**Acceptance:**
- seq.rs defines a tracker holding the current head (u64) for a single workflow.
- On a successful append of N events, the head advances by exactly N.
- On a failed append, the head is unchanged.
- The tracker can be constructed with an explicit starting head (default 0 for a fresh workflow).
- A unit test asserts the head advances on success and is unchanged on a simulated failure.

### R2: Recorder construction and sequence authority

THE SYSTEM SHALL define Recorder, constructed for a single WorkflowId over a shared EventStore, owning the sequence-head tracker. THE Recorder SHALL compute expected_seq from its tracked head for every append and SHALL NOT read the head from outside itself nor guess it.

**Acceptance:**
- recorder.rs defines Recorder holding a WorkflowId, an Arc<dyn EventStore> (or equivalent shared handle), and the seq tracker.
- Recorder::new(workflow_id, store) starts a fresh workflow at head 0; Recorder::resume_at(workflow_id, store, head) starts at a given head.
- Every append uses expected_seq equal to the current tracked head; there is no public method to set the head other than via a successful append or resume_at.
- A unit test appends two single-event batches and asserts the second used expected_seq advanced past the first.

### R3: Per-family recording methods

THE SYSTEM SHALL provide a Recorder method for each event family: workflow lifecycle (started/completed/failed/cancelled), activity (scheduled/started/completed/failed), timer (started/fired/cancelled), signal received, and child workflow (started/completed/failed). Each method SHALL construct the aion-core Event with the correct envelope (seq from the tracker, recorded timestamp, owning WorkflowId) and append it.

**Acceptance:**
- Recorder exposes one method per Event variant family listed above, each taking the variant's domain payload (e.g. ActivityId + result Payload for activity completed).
- Each method builds the aion-core Event with envelope seq = current head and the owning WorkflowId, and appends a single-event batch.
- The recorded timestamp written into the envelope is supplied by the caller (the engine) — the Recorder does not read the wall clock (defer the determinism source to the engine; see CO8 note in the design).
- A unit test records an activity scheduled then completed and reads the history back via the store, asserting two events in sequence order with matching ActivityId.

### R4: Sequence-conflict is a hard error; resume at existing head

WHEN EventStore::append returns StoreError::SequenceConflict, THE Recorder SHALL surface a hard DurabilityError and SHALL NOT retry by re-reading the head. THE SYSTEM SHALL support initialising a Recorder at a head derived from existing recorded history.

**Acceptance:**
- On SequenceConflict, the Recorder returns DurabilityError (the double-writer signal) and does not re-read or retry.
- Recorder::resume_at sets the starting head so subsequent appends continue at the correct sequence for a recovered workflow.
- A test pre-seeds InMemoryStore with K events for a workflow, constructs a Recorder via resume_at at head K, appends one event, and asserts it lands at sequence K (no conflict).
- A test forces a SequenceConflict (by appending out from under the recorder via a second writer) and asserts the Recorder returns a DurabilityError rather than retrying.

## Boundaries

- SHALL NOT read history for replay or build a cursor — that is AD-003.
- SHALL NOT decide recorded-vs-live or detect violations — that is AD-004/AD-005.
- SHALL NOT generate the recorded timestamp from the wall clock; the timestamp is supplied by the caller (the determinism source is AD-006, the engine supplies it at record time).
