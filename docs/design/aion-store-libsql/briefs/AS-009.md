# AS-009: Enforce single-writer discipline at the type level

> **Cluster:** aion-store-libsql
> **Depends on:** AS-001

## Purpose

Invariant #3 (single writer per workflow) is currently enforced by documentation and code review alone. EventStore::append is public, so any caller can bypass Recorder and append directly. This brief splits the trait into a read-only public surface and a write-only crate-internal surface, making the single-writer constraint a compile-time guarantee rather than a social contract.

## Task

Split the EventStore trait into ReadableEventStore (public, read-only methods) and WritableEventStore (append, crate-internal). Introduce a sealed capability token that only Recorder can construct, gating WritableEventStore access. Migrate all non-Recorder call sites to ReadableEventStore. Retain SequenceConflict detection as runtime defense-in-depth.

## Requirements

### R1: Split EventStore into ReadableEventStore and WritableEventStore

THE SYSTEM SHALL define a ReadableEventStore trait containing all read-only methods (read_history, query, get_latest_sequence, etc.) and a WritableEventStore trait containing append. ReadableEventStore SHALL be public. WritableEventStore SHALL be crate-internal (pub(crate)) so that only code within aion-store can implement or call it directly.

**Acceptance:**
- ReadableEventStore is a public trait in aion-store exposing all read-only EventStore methods.
- WritableEventStore is a pub(crate) trait in aion-store exposing append.
- Both traits have blanket or supertrait relationships so that a type implementing both can be used wherever either is required.
- The original EventStore trait is removed or becomes a convenience alias/supertrait of both, not directly implemented by consumers.

**Files:**
- modify: crates/aion-store/src/lib.rs

### R2: Recorder gets exclusive WritableEventStore access via a sealed capability token

THE SYSTEM SHALL define a sealed newtype (WriteToken) that only Recorder can construct. WritableEventStore::append SHALL require a WriteToken parameter (or WritableEventStore SHALL only be obtainable through a sealed accessor that Recorder holds), so that external code cannot call append even if it has an Arc<dyn EventStore>.

**Acceptance:**
- A WriteToken (or equivalent sealed capability) exists in a private module that Recorder can reach but external crates cannot construct.
- Calling append without the token is a compile error, not a runtime error.
- Recorder constructs the token internally; no public API exposes the token constructor.
- A doc comment on the token explains the single-writer invariant it enforces.

**Files:**
- modify: crates/aion-store/src/lib.rs
- modify: crates/aion/src/durability/recorder.rs

### R3: Migrate existing code to ReadableEventStore where only reads are needed

WHEN code outside the Recorder requires event store access, THE SYSTEM SHALL accept ReadableEventStore (not the full EventStore or WritableEventStore). THE SYSTEM SHALL update all function signatures, trait bounds, and Arc types that currently accept EventStore but only use read methods.

**Acceptance:**
- Engine, lifecycle transitions, query handlers, and visibility projection accept Arc<dyn ReadableEventStore> (or equivalent) where they only read.
- The conformance suite in aion-store tests both ReadableEventStore and WritableEventStore methods.
- The libSQL store implementation implements both traits.
- No non-Recorder code path holds a WritableEventStore reference or calls append.

**Files:**
- modify: crates/aion-store/src/conformance.rs
- modify: crates/aion-store-libsql/src/store.rs

### R4: SequenceConflict retained as runtime defense-in-depth

EVEN WHEN the type-level split prevents external append calls at compile time, THE SYSTEM SHALL retain SequenceConflict detection in WritableEventStore::append so that any future bypass (e.g. via unsafe or internal misuse) is caught at runtime.

**Acceptance:**
- The SequenceConflict error variant remains in StoreError and is returned when a sequence number collision is detected on append.
- The conformance suite includes a test that triggers SequenceConflict to prove the runtime guard still works.
- The error message on SequenceConflict explicitly names it as a double-writer bug indicator.
- No SequenceConflict-related logic is removed or weakened by the trait split.

**Files:**
- modify: crates/aion-store/src/lib.rs
- modify: crates/aion-store/src/conformance.rs
