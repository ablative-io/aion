# AS-001: aion-store-libsql crate scaffold, config, and error mapping

> **Cluster:** aion-store-libsql
> **Checklist:**
> - C1 — crates/aion-store-libsql exists, is a workspace member, and depends only on aion-store among Aion crates (plus external crates: libsql, serde, thiserror, chrono, async-trait or native async).
> - C2 — src/lib.rs contains only module declarations and re-exports, no logic.
> - C3 — LibSqlConfig is a Deserialize-only type whose mode enum selects Embedded { path } or EmbeddedReplica { path, primary_url, auth_token }.
> - C4 — LibSqlConfig exposes libSQL durability/WAL settings (e.g. sync interval for replica mode, journal/WAL options) with no hardcoded default values baked into the crate.
> - C5 — A boundary error-mapping module maps every libsql driver error and serde (de)serialisation failure into StoreError (Backend, Serialization), so no foreign error type leaks across the EventStore surface.
> **Stories:**
> - S3 (Engine Operator, Deploying Aion with Durable Storage) — As an engine operator, I want to tune libSQL durability and sync settings through configuration with no values silently assumed for me, so that I control the durability-versus-throughput trade-off for my deployment.
> - S8 (Store Author, Maintaining the libSQL Backend) — As a store author, I want every libSQL and serialisation error mapped into the StoreError taxonomy so that the engine handles one uniform error surface regardless of backend.
> - S9 (Store Author, Maintaining the libSQL Backend) — As a store author, I want libSQL's internals to be swappable for the Turso-native engine later without touching the engine, so that adopting the production-ready rewrite is a drop-in change when it leaves beta.

## Purpose

Create the aion-store-libsql leaf-of-the-store-tree crate, its Deserialize-only configuration, and the boundary error mapping into StoreError. This is the foundation every other brief in the cluster builds on: the crate exists, depends only on aion-store, and can express its modes and its errors.

## Task

Create the aion-store-libsql crate depending only on aion-store among Aion crates (plus libsql, serde, thiserror, chrono, and async support). Write a thin lib.rs. Define LibSqlConfig as a Deserialize-only type whose mode enum is Embedded { path } or EmbeddedReplica { path, primary_url, auth_token }, exposing libSQL durability/WAL/sync settings with no hardcoded defaults. Define the error-mapping surface that converts libsql and serde errors into StoreError. Do NOT implement EventStore here — that begins in AS-003.

## Requirements

### R2: LibSqlConfig mode enum

THE SYSTEM SHALL define LibSqlConfig as a Deserialize-only configuration type whose mode is an enum with Embedded { path } and EmbeddedReplica { path, primary_url, auth_token } variants. THE SYSTEM SHALL NOT derive Serialize on LibSqlConfig.

**Acceptance:**
- config.rs defines LibSqlConfig deriving Deserialize and Debug but NOT Serialize.
- The mode is an enum: Embedded { path: PathBuf } and EmbeddedReplica { path: PathBuf, primary_url: String, auth_token: String }.
- A unit test deserialises a TOML or JSON fixture for each variant and asserts the parsed fields.
- Embedded mode requires only a path; EmbeddedReplica additionally requires primary_url and auth_token (deserialising a replica config missing primary_url is an error).

### R3: Durability and sync settings, no assumed defaults

THE SYSTEM SHALL expose libSQL durability/WAL settings (and, for replica mode, a sync interval) on LibSqlConfig. THE SYSTEM SHALL NOT bake hardcoded default values for these tunables into the crate; values absent from config SHALL be represented as unset (Option) and left to libSQL's own behaviour or surfaced for the operator, not silently defaulted by this crate.

**Acceptance:**
- LibSqlConfig carries durability/WAL tunables (e.g. an Option for journal/WAL mode and an Option sync_interval used in replica mode) as Option fields, not bare values with crate-chosen defaults.
- No literal default magic numbers for sync interval or WAL settings appear in the crate; absent config maps to None.
- A unit test deserialises a config omitting the tunables and asserts the corresponding fields are None.
- A doc comment states that operators set these; the crate does not assume values (per CLAUDE.md no-assumed-defaults).

### R4: Boundary error mapping into StoreError

THE SYSTEM SHALL provide a single boundary that maps libsql driver errors and serde (de)serialisation failures into StoreError. THE SYSTEM SHALL NOT allow any libsql or serde error type to appear in the public EventStore-facing surface.

**Acceptance:**
- error.rs defines conversions (From impls or helper fns) mapping libsql::Error into StoreError::Backend and serde_json errors into StoreError::Serialization.
- The mapping preserves a human-readable message from the source error in the StoreError variant payload.
- No public function signature in the crate returns or exposes a libsql or serde error type; all fallible store operations surface StoreError.
- A unit test constructs a representative libsql error and a serde error and asserts each maps to the expected StoreError variant.

**Files:**
- create: crates/aion-store-libsql/src/error.rs

## Boundaries

- SHALL NOT implement the EventStore trait — store wiring is AS-003.
- SHALL NOT open a libSQL connection or create schema — that is AS-002.
- SHALL NOT depend on any Aion crate other than aion-store.
- SHALL NOT derive Serialize on LibSqlConfig — it is a Deserialize-only config type.
