# AS-003: LibSqlStore struct and EventStore impl wiring

> **Cluster:** aion-store-libsql
> **Depends on:** AS-002
> **Checklist:**
> - C11 — LibSqlStore is a Send + Sync + 'static type holding the libSQL connection, constructed from a LibSqlConfig.
> - C12 — LibSqlStore implements the aion-store EventStore trait and is usable as Arc<dyn EventStore>.
> - C13 — An open/constructor associated function (e.g. LibSqlStore::open and/or LibSqlStore::connect) builds the store from a config and ensures the schema exists before returning.
> **Stories:**
> - S1 (Engine Operator, Deploying Aion with Durable Storage) — As an engine operator, I want a zero-infrastructure durable store I get by pointing at a file path, so that embedding Aion needs no database server to survive restarts.

## Purpose

Define the LibSqlStore type and wire up its EventStore trait implementation skeleton, with a constructor that opens a connection and ensures the schema. The method bodies are filled in by AS-004 (append), AS-005 (reads/query), and AS-006 (timers); this brief establishes the type, the constructor, and the trait impl block they extend.

## Task

Create store.rs defining LibSqlStore (holding the libSQL connection from AS-002) as a Send + Sync + 'static type, with an open/connect constructor that builds it from a LibSqlConfig and ensures the schema. Add the EventStore impl block for LibSqlStore wiring in the methods that AS-004/005/006 implement. Confirm object-safety via Arc<dyn EventStore>. Method bodies may delegate to the append/read/timer modules those briefs create.

## Requirements

### R1: LibSqlStore type and constructor

THE SYSTEM SHALL define LibSqlStore as a Send + Sync + 'static type holding the libSQL connection. WHEN constructed from a LibSqlConfig, THE SYSTEM SHALL open the connection (embedded or replica per the config) and ensure the schema exists before returning the store.

**Acceptance:**
- store.rs defines struct LibSqlStore holding the connection/database handle from AS-002; the type is Send + Sync + 'static.
- An async constructor (LibSqlStore::connect(config: LibSqlConfig) or equivalent) opens the connection and calls ensure_schema before returning.
- A convenience LibSqlStore::open(path) constructs an Embedded config and delegates to the config constructor.
- A test constructs a LibSqlStore at a tempfile path and asserts construction succeeds and the schema is present.
- Connection/schema errors during construction surface as StoreError.

### R2: EventStore impl wiring and object-safety

THE SYSTEM SHALL implement the aion-store EventStore trait for LibSqlStore, with the impl block declaring every trait method, and the store SHALL be usable as Arc<dyn EventStore>.

**Acceptance:**
- store.rs contains impl EventStore for LibSqlStore declaring append, read_history, list_active, query, schedule_timer, and expired_timers, each delegating to the module that owns its logic (append.rs, read.rs, timer.rs) — those bodies are completed in AS-004/005/006 and may be minimal-but-compiling stubs returning a clearly-marked unimplemented-path StoreError here only if a later brief in this same cluster replaces them (note this split in the task).
- A compile/integration test constructs an Arc<dyn EventStore> from a LibSqlStore, proving the trait is object-safe as implemented.
- The impl forwards the connection handle to the per-method modules so AS-004/005/006 implement against a shared store surface.
- No method body in this brief silently swallows an error or returns Ok without doing its work; any not-yet-filled method returns a StoreError that the cluster's later briefs replace with the real implementation (the split is recorded so review knows AS-004/005/006 complete C14-C23).

### R3: Shared connection access for per-method modules

THE SYSTEM SHALL expose, within the crate, an accessor or shared handle by which append.rs, read.rs, and timer.rs reach the libSQL connection, so the per-method logic in later briefs operates against one store without re-opening connections.

**Acceptance:**
- store.rs exposes a crate-internal way (e.g. a method returning &Connection or a cloned handle) for the per-method modules to obtain the connection.
- Concurrency: the handle is shared safely (the store is Send + Sync); any interior lock is documented and lock poisoning is handled explicitly with no panicking unwrap.
- A test confirms two sequential operations through the store reuse the same underlying database (e.g. a write then a read observe the same data) without opening a second connection.
