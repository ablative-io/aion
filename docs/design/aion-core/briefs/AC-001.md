# AC-001: aion-core crate scaffold, domain identifiers, and payload

> **Cluster:** aion-core
> **Checklist:**
> - C1 — WorkflowId is a newtype over Uuid deriving Serialize, Deserialize, Clone, Debug, Eq, Hash, and Display.
> - C2 — ActivityId is a newtype whose value derives from the activity's scheduling sequence position, enabling deterministic replay matching.
> - C3 — TimerId is a newtype supporting both author-assigned (named) and engine-assigned (anonymous) construction.
> - C4 — RunId is a newtype distinguishing successive runs of the same logical workflow (reset / continue-as-new).
> - C5 — No public signature in aion-core accepts a bare Uuid or String where a domain identifier is meant.
> - C13 — Payload is a newtype over bytes with a content-type tag.
> - C14 — Payload constructs from and converts to serde_json::Value and round-trips losslessly.
> **Stories:**
> - S2 (Engine, Producing and Consuming Events) — As the engine, I want to read a workflow's complete ordered history so that I can replay it to reconstruct the workflow's state.
> - S8 (Workflow Author, Writing Workflows via the SDK) — As a workflow author, I want time and identifiers to be deterministic so that my workflow replays to exactly the same state every time.
> - S11 (Worker, Implementing Activities) — As a worker, I want to depend on the domain types without pulling in the store trait so that my worker binary stays small and decoupled.

## Purpose

Create the aion-core leaf crate and define the foundational value types every other type and crate references: the newtype identifiers and the type-erased Payload carrier. This is the foundation of the entire Aion type system. Payload lives here, not in AC-002, because both the error taxonomy (AC-004) and the event model (AC-002) depend on it — placing it at the root keeps the brief dependency graph acyclic.

## Task

Create the aion-core crate with a thin lib.rs re-export surface, an ids module defining WorkflowId, ActivityId, TimerId, and RunId as distinct newtypes, and a payload module defining the Payload newtype that carries opaque serialised values through events and errors. The crate must depend on no other Aion crate. Identifiers must be constructible, serialisable, and printable, and must be distinct types the compiler will not interchange. Payload must round-trip losslessly to and from serde_json::Value.

### R2: Identifier newtypes

THE SYSTEM SHALL define WorkflowId, ActivityId, TimerId, and RunId as distinct newtypes. WorkflowId and RunId SHALL wrap a Uuid. ActivityId SHALL derive from a scheduling sequence position. TimerId SHALL support both an author-assigned name and an engine-assigned value.

**Acceptance:**
- ids.rs defines WorkflowId(Uuid) and RunId(Uuid) with new() and new_v4()-style constructors.
- ActivityId is constructible from a u64 sequence position and exposes that position, so replay can match a recorded result to its scheduling call deterministically.
- TimerId has two constructors: a named one (from a String) and an anonymous one (from the engine, derived from sequence position).
- Each identifier is a separate type; the compiler rejects passing one identifier type where another is expected.
- No bare Uuid or String appears in the public surface where one of these identifiers is meant.

**Files:**
- create: crates/aion-core/src/ids.rs
- modify: crates/aion-core/src/lib.rs

**Checklist:**
- C1 — WorkflowId is a newtype over Uuid deriving Serialize, Deserialize, Clone, Debug, Eq, Hash, and Display.
- C2 — ActivityId is a newtype whose value derives from the activity's scheduling sequence position, enabling deterministic replay matching.
- C3 — TimerId is a newtype supporting both author-assigned (named) and engine-assigned (anonymous) construction.
- C4 — RunId is a newtype distinguishing successive runs of the same logical workflow (reset / continue-as-new).

### R3: Identifier traits — serde, display, equality

THE SYSTEM SHALL derive Serialize, Deserialize, Clone, Debug, PartialEq, Eq, and Hash on every identifier, and SHALL implement Display. Round-tripping an identifier through serde_json SHALL produce an equal value.

**Acceptance:**
- All four identifiers derive Serialize, Deserialize, Clone, Debug, PartialEq, Eq, Hash.
- All four implement Display rendering a stable, human-readable form.
- A unit test round-trips each identifier through serde_json and asserts equality.
- WorkflowId and RunId are usable as HashMap keys (Eq + Hash verified by a test).

### R4: Payload newtype

THE SYSTEM SHALL define Payload as a newtype over a byte buffer plus a content-type tag. THE SYSTEM SHALL provide lossless conversion between Payload and serde_json::Value. Payload is the type-erased carrier for all user data that flows through events and error details.

**Acceptance:**
- payload.rs defines Payload holding bytes and a content-type tag (an enum or stable string identifying the encoding).
- Payload::from_json(&serde_json::Value) and Payload::to_json() -> Result round-trip losslessly for all JSON value kinds.
- Payload derives Serialize, Deserialize, Clone, Debug, PartialEq.
- A unit test asserts round-trip fidelity for null, bool, number, string, array, and object.

**Files:**
- create: crates/aion-core/src/payload.rs
- modify: crates/aion-core/src/lib.rs

**Checklist:**

- C14 — Payload constructs from and converts to serde_json::Value and round-trips losslessly.

## Boundaries

- SHALL NOT define the Event enum — that lands in AC-002. Payload (the value carrier) lands here because both AC-002 and AC-004 depend on it.
- SHALL NOT define WorkflowStatus, filters, or errors — those land in AC-003 and AC-004.
- SHALL NOT define the EventStore trait — that is the aion-store crate in AC-005.
