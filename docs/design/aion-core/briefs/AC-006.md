# AC-006: InMemoryStore reference implementation

> **Cluster:** aion-core
> **Depends on:** AC-005
> **Checklist:**
> - C30 — InMemoryStore implements EventStore in full, including timer scheduling and retrieval.
> - C31 — InMemoryStore append enforces the expected-sequence guard and applies events atomically (all or none).
> **Stories:**
> - S6 (Store Author, Implementing a Persistence Backend) — As a store author, I want an in-memory reference implementation and a shared behavioural test suite so that I can verify my backend matches the contract exactly.

## Purpose

Provide the correct, non-durable reference implementation of EventStore. It is the behavioural oracle every durable backend must match and the store the engine's own tests use.

## Task

Implement InMemoryStore: a Send + Sync EventStore backed by in-memory maps behind a lock. It must implement every trait method correctly, including the atomic append with expected-sequence guard and durable-timer scheduling and retrieval. Correctness matters more than performance — this is the oracle.

## Requirements

### R1: InMemoryStore structure and event methods

THE SYSTEM SHALL implement InMemoryStore as an EventStore backed by in-memory state behind a lock, implementing append, read_history, list_active, and query.

**Acceptance:**
- memory.rs defines InMemoryStore holding per-workflow event vectors and any derived index behind a Mutex or RwLock; the struct is Send + Sync and Default-constructible.
- read_history returns the stored events in sequence order; NotFound is returned for an unknown workflow only where the contract specifies (otherwise an empty history per the documented semantics — match the trait docs from AC-005).
- list_active returns workflows whose projected status (via aion-core status projection) is Running.
- query evaluates each workflow's summary against the filter using the aion-core filter semantics.
- Lock poisoning is handled explicitly (no unwrap that can panic on a poisoned lock).

### R2: Atomic append with sequence guard

WHEN append is called with an expected sequence equal to the stored head, THE SYSTEM SHALL apply all events atomically and advance the head. WHEN the expected sequence does not match, THE SYSTEM SHALL apply no events and return SequenceConflict.

**Acceptance:**
- append checks the current head sequence for the workflow against expected_seq before applying.
- On match, all events in the batch are appended together and the head advances by the batch length.
- On mismatch, no events are applied and StoreError::SequenceConflict { expected, found } is returned.
- A test appends two batches in order (success), then attempts an append with a stale expected_seq and asserts SequenceConflict and that no partial write occurred.
- Concurrency test: two appends racing on the same expected_seq result in exactly one success and one SequenceConflict.

### R3: Durable timer implementation

THE SYSTEM SHALL implement schedule_timer and expired_timers in InMemoryStore so that scheduled timers are retrievable once due.

**Acceptance:**
- schedule_timer records a TimerEntry keyed by (workflow_id, timer_id).
- expired_timers(as_of) returns all entries with fire_at <= as_of and does not return future timers.
- Re-scheduling the same timer id replaces the prior entry (idempotent under replay).
- A test schedules timers at different instants and asserts expired_timers returns exactly the due ones for a given as_of.

## Boundaries

- SHALL NOT define the shared behavioural test suite — that is AC-007 (this brief provides the implementation it runs against).
