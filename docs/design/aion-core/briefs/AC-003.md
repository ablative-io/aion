# AC-003: Workflow status, filters, and summaries

> **Cluster:** aion-core
> **Depends on:** AC-002
> **Checklist:**
> - C16 — WorkflowStatus enum has variants Running, Completed, Failed, Cancelled, TimedOut.
> - C17 — WorkflowStatus is computed by projecting over event history (the last lifecycle event determines the status).
> - C18 — No code path sets WorkflowStatus independently of the events that justify it.
> - C19 — WorkflowFilter supports filtering by workflow type, status, time range, and parent workflow.
> - C20 — WorkflowSummary carries workflow id, type, status, start time, and end time.
> **Stories:**
> - S9 (Operator, Observing Workflows) — As an operator, I want to query workflows by type, status, and time range so that I can see what is running and what has failed.
> - S10 (Operator, Observing Workflows) — As an operator, I want a lightweight summary per workflow so that listing many workflows does not require loading every full history.

## Purpose

Define the derived, queryable view of a workflow: its status (a projection over event history) and the filter and summary types the query surface uses.

## Task

Define WorkflowStatus as an enum and a projection function that computes it from an event history. Define WorkflowFilter (the query input) and WorkflowSummary (the lightweight query output). Status must never be a free-standing mutable field — it is always derived from events.

## Requirements

### R1: WorkflowStatus and projection

THE SYSTEM SHALL define WorkflowStatus with variants Running, Completed, Failed, Cancelled, TimedOut. THE SYSTEM SHALL provide a function that computes WorkflowStatus from a slice of events by inspecting the terminal lifecycle event, defaulting to Running when none is present.

**Acceptance:**
- status.rs defines WorkflowStatus { Running, Completed, Failed, Cancelled, TimedOut } deriving Serialize, Deserialize, Clone, Copy, Debug, PartialEq, Eq.
- A function status_from_events(&[Event]) -> WorkflowStatus returns Running when no terminal event exists, and the matching terminal status otherwise.
- WorkflowCompleted maps to Completed, WorkflowFailed to Failed, WorkflowCancelled to Cancelled, and WorkflowTimedOut to TimedOut — every terminal WorkflowStatus variant has exactly one corresponding terminal event (AC-002 R2), so the projection is total.
- WorkflowStatus does not derive or implement Default and exposes no new()/constructor; status_from_events is the only function in aion-core that produces a WorkflowStatus from history, and a doc comment states that status must be obtained only by projection from event history, never assigned directly (enforces C18 / CO7).
- Unit tests cover: empty history (Running), each terminal event including WorkflowTimedOut, and a history with events after start but before any terminal event (Running); a test asserts the projection is deterministic (the same history always yields the same status).

### R2: WorkflowFilter

THE SYSTEM SHALL define WorkflowFilter supporting optional filtering by workflow type, status, time range, and parent workflow. An empty filter SHALL match all workflows.

**Acceptance:**
- filter.rs defines WorkflowFilter with optional fields: workflow_type, status, started_after, started_before, parent.
- WorkflowFilter has a builder or a Default that yields a match-all filter.
- A matches(&self, &WorkflowSummary) -> bool helper evaluates a summary against the filter (used by the in-memory store and as reference semantics for durable backends).
- Unit tests cover match-all, single-field filters, and a combined filter.

**Files:**
- create: crates/aion-core/src/filter.rs
- modify: crates/aion-core/src/lib.rs

**Checklist:**

### R3: WorkflowSummary

THE SYSTEM SHALL define WorkflowSummary as the lightweight projection of a workflow: id, type, status, start time, and optional end time.

**Acceptance:**
- WorkflowSummary carries workflow_id: WorkflowId, workflow_type: String, status: WorkflowStatus, started_at: DateTime<Utc>, ended_at: Option<DateTime<Utc>>.
- WorkflowSummary derives Serialize, Deserialize, Clone, Debug, PartialEq.
- A constructor or helper builds a summary from a workflow's event history (start event plus projected status plus terminal timestamp).
- A unit test builds a summary from a small history and asserts the fields.

## Boundaries

- SHALL NOT define the EventStore trait or query method — those are AC-005; this brief defines the types the query method uses.
- SHALL NOT cache status anywhere; status is always computed from events here.
