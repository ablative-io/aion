# AC-004: Error taxonomy — activity and workflow errors

> **Cluster:** aion-core
> **Depends on:** AC-001
> **Checklist:**
> - C21 — ActivityError carries an explicit retryable-vs-terminal classification, not a string or bool.
> - C22 — WorkflowError is defined as the terminal failure type of a workflow.
> **Stories:**
> - S4 (Engine, Producing and Consuming Events) — As the engine, I want a failed activity's error to tell me whether it is retryable so that I apply the retry policy correctly instead of guessing.

## Purpose

Define the domain error types that appear inside events. The retryable/terminal classification on activity errors is load-bearing for the engine's retry decisions.

## Task

Define ActivityError carrying an explicit retryable-vs-terminal classification, a message, and an optional details Payload. Define WorkflowError as the terminal failure type of a workflow. Both are referenced by Event variants defined in AC-002.

## Requirements

### R1: ActivityError with retryable classification

THE SYSTEM SHALL define ActivityError carrying a classification that is explicitly Retryable or Terminal, a human-readable message, and optional structured details. The classification SHALL be a dedicated type, not a bool or string.

**Acceptance:**
- error.rs defines ActivityError with a kind: ActivityErrorKind { Retryable, Terminal } (or equivalent dedicated enum), a message: String, and details: Option<Payload>.
- ActivityError exposes is_retryable() -> bool derived from the classification.
- ActivityError derives Serialize, Deserialize, Clone, Debug, PartialEq and implements std::error::Error and Display.
- A doc comment states that the engine consults is_retryable() to decide whether to apply the activity's retry policy or fail the workflow.
- Unit tests construct a retryable and a terminal error and assert is_retryable().

**Files:**
- create: crates/aion-core/src/error.rs
- modify: crates/aion-core/src/lib.rs

**Checklist:**
- C21 — ActivityError carries an explicit retryable-vs-terminal classification, not a string or bool.

**Stories:**
- S4 (Engine, Producing and Consuming Events) — As the engine, I want a failed activity's error to tell me whether it is retryable so that I apply the retry policy correctly instead of guessing.

### R2: WorkflowError

THE SYSTEM SHALL define WorkflowError as the terminal failure type of a workflow, carrying a message and optional structured details.

**Acceptance:**
- WorkflowError carries a message: String and details: Option<Payload>.
- WorkflowError derives Serialize, Deserialize, Clone, Debug, PartialEq and implements std::error::Error and Display.
- A From<ActivityError> conversion exists so an unrecovered activity failure can become the workflow's terminal error, preserving the message and details.
- A unit test exercises the From conversion.

**Files:**
- modify: crates/aion-core/src/error.rs

**Checklist:**
- C22 — WorkflowError is defined as the terminal failure type of a workflow.

**Stories:**
- S4 (Engine, Producing and Consuming Events) — As the engine, I want a failed activity's error to tell me whether it is retryable so that I apply the retry policy correctly instead of guessing.

## Boundaries

- SHALL NOT define StoreError — that belongs to the aion-store crate in AC-005.
- SHALL NOT implement retry policy logic — this defines the error type the engine's policy consults; the policy itself is engine work (cluster AE/AD).
