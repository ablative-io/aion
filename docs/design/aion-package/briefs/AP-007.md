# AP-007: Version record production and round-trip conformance suite

> **Cluster:** aion-package
> **Depends on:** AP-005, AP-006
> **Checklist:**
> - C25 — A version record is producible from a loaded Package, carrying the logical entry module, the content hash, the declared activities, and the input/output schemas, using the shared aion-core version-record type.
> - C26 — A test suite round-trips PackageBuilder output through Package::load and asserts the manifest, beam set, and namespaced names survive intact.
> - C27 — A test asserts reproducibility (two builds of identical inputs are byte-identical) and asserts integrity rejection (a package whose beams are altered after packing fails to load).
> **Stories:**
> - S1 (Engine, Loading and Registering Workflows) — As the engine, I want to load a .aion from a path or byte buffer and get a validated package so that I can register its modules without trusting loose files on disk.
> - S2 (Engine, Loading and Registering Workflows) — As the engine, I want the package to reject itself when its beams do not match its stored hash so that I never register a truncated or tampered workflow into the VM.
> - S7 (Workflow Author, Packaging and Deploying a Workflow) — As a workflow author, I want identical inputs to produce a byte-identical .aion so that my builds are reproducible and auditable.

## Purpose

Define and produce the cross-system WorkflowVersion record from a loaded Package so the engine references a workflow version through one shared type, and provide the conformance suite that proves the write and read paths fit together: a built package loads back intact, builds are reproducible, and integrity rejection actually fires. WorkflowVersion is defined in aion-package (not aion-core) because it embeds the ContentHash, an aion-package type — aion-core is a leaf and cannot reference it (D10).

## Task

Define the WorkflowVersion record type in aion-package and a function producing it from a loaded Package (logical entry module, content hash, declared activities, input/output schemas). Then build the round-trip conformance suite: write a package, load it, assert manifest/beams/namespaced-names survive; assert two builds of identical inputs are byte-identical; assert a post-pack alteration of the beams is rejected on load. Do not write the record to any store.

## Requirements

### R1: WorkflowVersion record and production from a loaded Package

THE SYSTEM SHALL define the WorkflowVersion record in aion-package and produce it from a loaded Package, carrying the logical entry module, the content hash, the declared activities, and the input/output schemas.

**Acceptance:**
- version.rs defines WorkflowVersion (logical entry module, ContentHash, declared activities, input/output schemas) deriving Serialize, Deserialize, Clone, Debug, PartialEq, and a function (e.g. Package::version_record()) returning it.
- The record carries the logical entry module, the content hash, the declared activities, and the input/output schemas drawn from the loaded Package.
- WorkflowVersion is the single canonical version type: it is defined here (it embeds the aion-package ContentHash, which aion-core cannot reference), and consumers — the engine (aion) — depend on aion-package to use it rather than re-deriving a parallel type.
- A unit test builds and loads a package, produces its version record, and asserts each field matches the manifest and content hash.

### R2: Write/read round-trip suite

THE SYSTEM SHALL provide a test suite that builds a package, loads it back, and asserts the manifest, beam set, and namespaced module names survive intact.

**Acceptance:**
- tests/roundtrip.rs builds a package (non-trivial: multiple beams, declared activities, schemas, with and without source) and loads it via Package::load_from_bytes.
- The loaded manifest equals the built manifest (with the version stamped), the loaded BeamSet equals the input BeamSet, and the optional source matches.
- The loaded deployed module names equal namespace::deployed_name(logical, content_hash) for every module.
- The test is a real runtime test, not gated out or ignored.

**Files:**
- create: crates/aion-package/tests/roundtrip.rs

**Checklist:**

### R3: Reproducibility and integrity-rejection assertions

THE SYSTEM SHALL assert that two builds of identical inputs are byte-identical and that a package whose beams are altered after packing fails to load with IntegrityMismatch.

**Acceptance:**
- A test builds identical inputs twice and asserts the output byte buffers are equal (reproducibility, CO8).
- A test takes a valid package's bytes, mutates a byte inside a beam entry (re-zipping or patching the entry), and asserts Package::load returns PackageError::IntegrityMismatch.
- A test asserts that including vs omitting source yields the same content hash / version (CO7).
- All tests run under cargo test without #[ignore].

## Boundaries

- SHALL NOT write the version record to any store — that is AC (in-memory) / AS (libSQL) / AX (Postgres).
- SHALL NOT test VM registration or execution — that is the engine cluster (AE).
- SHALL NOT define a version type in aion-core; WorkflowVersion is canonical here because it embeds the aion-package ContentHash.
