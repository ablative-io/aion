# AP-003: Content-hash versioning over the canonical beam set

> **Cluster:** aion-package
> **Depends on:** AP-001
> **Checklist:**
> - C11 — The content hash is computed over the canonical beam set only (sorted logical names plus exact bytes), never over archive bytes, compression, timestamps, or source inclusion.
> - C12 — Two BeamSets with identical modules and bytes produce identical content hashes regardless of insertion order; any change to a beam's name or bytes changes the hash.
> - C13 — The content hash has a stable, documented textual form usable as a version identifier and as a module-name component.
> **Stories:**
> - S6 (Workflow Author, Packaging and Deploying a Workflow) — As a workflow author, I want the package version to be derived from my compiled code so that I never hand-maintain a version string that could lie about what is in the package.
> - S2 (Engine, Loading and Registering Workflows) — As the engine, I want the package to reject itself when its beams do not match its stored hash so that I never register a truncated or tampered workflow into the VM.

## Purpose

Implement the content hash that IS the workflow's version. The hash is computed over the canonical beam set alone — module names and exact bytes in sorted order — so it is stable across compression, timestamps, and source inclusion, unforgeable, and usable both as a version identifier and as a module-name component. This is the heart of Aion's versioning answer.

## Task

Define a ContentHash value type and a function that computes it over a BeamSet by hashing a canonical serialisation of (sorted logical name, exact bytes) for each module. Guarantee that insertion order does not affect the result and that any change to a name or byte changes the result. Give ContentHash a stable, documented textual form suitable for embedding in a version field and in a namespaced module name. Do not hash archive bytes, compression, timestamps, or source.

## Requirements

### R1: ContentHash type and computation

THE SYSTEM SHALL define ContentHash and a function computing it over a BeamSet by hashing a canonical serialisation of each module's logical name and exact bytes in canonical (sorted-by-name) order. THE SYSTEM SHALL NOT incorporate archive bytes, compression, timestamps, or source into the hash.

**Acceptance:**
- hash.rs defines ContentHash (a newtype over the digest) and content_hash(beams: &BeamSet) -> ContentHash.
- The computation iterates modules in BeamSet canonical order, feeding each module's logical name and its exact bytes into the digest with an unambiguous framing (length-prefixed or delimited) so distinct (name, bytes) splits cannot collide.
- ContentHash derives Clone, Debug, PartialEq, Eq, Hash and is Serialize + Deserialize.
- A doc comment states the hash is computed over the canonical beam set only and is independent of compression, timestamps, and source inclusion (CO7).
- The hashing crate is a vetted, collision-resistant algorithm (e.g. SHA-256 / BLAKE3), not an ad-hoc hash.

### R2: Order-independence and sensitivity

THE SYSTEM SHALL produce identical ContentHashes for two BeamSets with identical modules and bytes regardless of insertion order, and SHALL produce a different ContentHash whenever any module's name or bytes change.

**Acceptance:**
- A unit test builds two BeamSets from the same modules in different insertion orders and asserts equal ContentHashes.
- A unit test mutates one byte of one module and asserts the ContentHash changes.
- A unit test renames one module and asserts the ContentHash changes.
- A unit test asserts that two modules whose (name, bytes) framing would be ambiguous without delimiting (e.g. name+bytes boundary shifts) produce different hashes, proving the framing is unambiguous.

### R3: Stable textual form

THE SYSTEM SHALL give ContentHash a stable, documented textual form usable as a version identifier and as a component of a deployed module name.

**Acceptance:**
- ContentHash implements Display and FromStr (or to_hex/from_hex) producing a stable lowercase textual form.
- The textual form contains only characters valid in a BEAM module name component (so AP-004 can embed it), documented as such.
- A unit test round-trips ContentHash through its textual form and asserts equality.
- A doc comment states this textual form is the version identifier stored in the manifest and embedded in namespaced module names.

## Boundaries

- SHALL NOT build or read the archive — write path is AP-005, read path is AP-006.
- SHALL NOT derive deployed module names — that is the namespacing transform in AP-004 (which consumes the textual form defined here).
- SHALL NOT stamp the hash into the manifest — AP-005 does that during packing.
