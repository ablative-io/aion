# AP-004: Content-hash module namespacing scheme

> **Cluster:** aion-package
> **Depends on:** AP-003
> **Checklist:**
> - C14 — A namespacing function maps (logical module name, content hash) to a deployed module name (e.g. order_workflow$<hash>) deterministically.
> - C15 — An inverse function recovers the logical module name from a deployed module name; the transform is a verified round-trip (bijection).
> - C16 — Two packages with different content hashes produce disjoint deployed module-name sets, so distinct versions never collide in the registry.
> **Stories:**
> - S3 (Engine, Loading and Registering Workflows) — As the engine, I want each deployed version's modules to have distinct, content-hash-namespaced names so that version N and version N+5 coexist and a months-long sleeping workflow never blocks a new deploy.
> - S4 (Engine, Loading and Registering Workflows) — As the engine, I want a loaded package to hand me the namespaced module names directly so that I apply one agreed naming scheme rather than deriving names myself.

## Purpose

Implement the namespacing scheme that makes each deployed workflow version a distinct, immutable module in beamr's registry. Deployed module names are derived from (logical module name, content hash), so version N and N+5 coexist without conflict and beamr's two-deep same-name limit never binds for workflow modules. This crate owns the transform; the engine applies it.

## Task

Define a pure, total transform from (logical module name, ContentHash) to a deployed module name (e.g. order_workflow$<hash>), and its inverse that recovers the logical name from a deployed name. Prove the round-trip is a bijection. Guarantee that two packages with different content hashes produce disjoint deployed-name sets. Do not load anything into a VM — this is naming only.

## Requirements

### R1: Forward namespacing transform

THE SYSTEM SHALL define a deterministic function mapping (logical module name, ContentHash) to a deployed module name that embeds the content hash so distinct versions of the same logical module get distinct deployed names.

**Acceptance:**
- namespace.rs defines deployed_name(logical: &str, hash: &ContentHash) -> String (or a NamespacedName type) joining the logical name and the hash textual form with the literal '$' separator (logical$hash).
- The '$' separator is mandated (CO13), exposed as a named constant, not a local literal: the engine (AE-005) must split on the identical character to recover logical names, and '$' is chosen because it cannot occur in a Gleam logical module name, keeping the boundary unambiguous.
- The transform is pure and total: same inputs always yield the same output, every valid (name, hash) pair maps successfully.
- The produced name is a valid beamr module-atom name (uses the hash textual form constrained in AP-003 R3).
- A doc comment states this is the application-level versioning layered on top of beamr's VM-level dual-version hot-loading (D4).

**Files:**

### R2: Inverse transform and bijection round-trip

THE SYSTEM SHALL define an inverse function recovering the logical module name (and content hash) from a deployed module name, and the forward/inverse pair SHALL be a verified bijection.

**Acceptance:**
- namespace.rs defines a parse function returning the logical name and ContentHash from a deployed name, yielding a typed error (not a panic) for a malformed deployed name.
- A unit test round-trips many (logical name, hash) pairs through forward-then-inverse and asserts equality (CO12).
- A unit test round-trips deployed-then-inverse-then-forward and asserts the deployed name is recovered.
- Separator choice guarantees the logical name and hash boundary is unambiguous even when the logical name contains the separator's neighbouring characters (a test covers a logical name with underscores/dollars as applicable).

**Files:**
- modify: crates/aion-package/src/namespace.rs

**Checklist:**

**Stories:**

### R3: Disjoint name sets across versions

THE SYSTEM SHALL guarantee that two packages with different content hashes produce disjoint sets of deployed module names, so distinct versions never collide in the registry.

**Acceptance:**
- A function maps a whole BeamSet plus its ContentHash to the set of deployed module names for that package.
- A unit test takes one logical beam set, derives deployed names under two different content hashes, and asserts the two name sets are disjoint.
- A unit test asserts that the same logical module under the same hash always maps to the same deployed name (idempotent), proving versions coexist rather than swap (S3).
- A doc comment states this is what sidesteps beamr's two-deep same-name version limit for workflow modules.
