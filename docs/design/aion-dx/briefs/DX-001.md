# DX-001: Fix activity dispatch to connected Python worker in hello-world flow

> **Cluster:** aion-dx

## Purpose

The hello-world workflow starts but gets stuck in Running — ActivityScheduled never appears and the connected Python worker never receives a task. This is the #1 blocking issue: until the demo works end-to-end, the system cannot be demonstrated or adopted. Investigate the dispatch path from BEAM workflow code through the NIF bridge to the ConnectedWorkerRegistry and fix the gap.

## Task

Trace the activity dispatch path end-to-end: workflow BEAM process calls run_activity NIF → ActivityDispatcher → ConnectedWorkerRegistry → worker stream. Identify where the chain breaks (likely: the workflow BEAM process isn't executing past WorkflowStarted, OR the activity NIF result isn't routing to the connected worker's task queue). Fix the gap and verify the hello-world completes from start to workflow completion.

## Requirements

### R1: Diagnose and fix activity dispatch gap

WHEN a workflow calls run_activity and a worker is connected for that activity type, THE SYSTEM SHALL dispatch the task to the worker and return the result to the workflow process. THE SYSTEM SHALL NOT leave the workflow stuck in Running with no ActivityScheduled event.

**Acceptance:**
- hello-world workflow completes end-to-end: WorkflowStarted → ActivityScheduled → ActivityCompleted → WorkflowCompleted.
- The Python worker receives the task and returns a result.
- cargo test --workspace passes.
- The fix does not require the developer to do anything beyond what the hello-world README describes.

### R2: Add server log output for activity dispatch lifecycle

THE SYSTEM SHALL log at INFO level when: a workflow schedules an activity, an activity is dispatched to a worker, a worker returns a result, and a workflow completes. THE SYSTEM SHALL log at ERROR level when dispatch fails (no worker registered, worker disconnected).

**Acceptance:**
- Server logs show 'activity scheduled', 'task dispatched to worker', 'activity completed' during hello-world run.
- If no worker is connected, an ERROR log says which activity type has no registered worker.
- Logs include workflow_id and activity_type.

**Files:**
- modify: crates/aion-server/src/worker/dispatch.rs
- modify: crates/aion-server/src/worker/bridge.rs

## Boundaries

- SHALL NOT redesign the activity dispatch architecture — find and fix the specific wiring gap.
- SHALL NOT modify the worker protocol or worker SDK — the issue is server-side.

## Verification

- cargo clippy --workspace --all-targets -- -D warnings passes.
- cargo fmt --check passes.
- cargo test --workspace passes.
- hello-world workflow completes end-to-end with the Python worker.
