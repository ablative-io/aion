# DX-004: Differentiated auth error messages and header documentation

> **Cluster:** aion-dx

## Purpose

All four auth failure modes return the identical 'namespace access denied' message. A developer cannot tell if they're missing the Authorization header, have a wrong token, are missing x-aion-subject, or have a namespace mismatch. Additionally, required headers (x-aion-subject, x-aion-namespaces) are undocumented.

## Task

Differentiate auth error responses by failure mode. Add documentation for required headers in the hello-world README and a root-level API reference.

## Requirements

### R1: Differentiated auth error messages

WHEN authentication fails, THE SYSTEM SHALL return a specific error message for each failure mode: missing Authorization header, invalid/expired bearer token, missing x-aion-subject header, namespace mismatch. THE SYSTEM SHALL NOT return the same generic message for all four cases.

**Acceptance:**
- No Authorization header → 'missing Authorization header with Bearer token'.
- Invalid token → 'invalid or expired bearer token'.
- Missing x-aion-subject → 'missing required header: x-aion-subject'.
- Namespace mismatch → 'subject not authorized for namespace [name]'.
- Each error includes a hint about what to fix.

**Files:**
- modify: crates/aion-server/src/api/http.rs
- modify: crates/aion-server/src/state.rs

### R2: Document required headers and API conventions

THE SYSTEM SHALL document all required headers (x-aion-subject, x-aion-namespaces, Authorization) in: (1) the hello-world README's curl examples with explanatory comments, and (2) a root-level docs/API.md reference. THE SYSTEM SHALL explain what each header does and what values are valid.

**Acceptance:**
- hello-world README curl examples have inline comments explaining each header.
- docs/API.md exists with a headers section listing all required/optional headers.
- A developer reading only the README can construct valid requests.

**Files:**
- create: docs/API.md
- modify: examples/hello-world/README.md

## Boundaries

- SHALL NOT change the auth enforcement logic — only the error messages.
- SHALL NOT add new headers — only document existing ones.

## Verification

- cargo clippy --workspace --all-targets -- -D warnings passes.
- cargo fmt --check passes.
- cargo test --workspace passes.
- Each auth failure mode returns a distinct, actionable error message.
