# DX-002: Stale database detection and actionable error messages

> **Cluster:** aion-dx

## Purpose

A pre-existing .db file from a previous schema version crashes the server with 'store serialization error: missing field run_id'. Developers hit this after any schema-changing update. The server should detect stale databases at startup and give clear guidance.

## Task

Add a schema version check at server startup. If the database schema is incompatible, emit a clear error message telling the developer to delete the .db file and restart. Also improve the cancel endpoint error from 'engine backend failure' to include the actual reason.

## Requirements

### R1: Detect stale database schema at startup

WHEN the server starts and the store contains events with an incompatible schema, THE SYSTEM SHALL detect this before serving requests and exit with a clear error message: 'Database schema mismatch — delete [path] and restart, or run migrations.' THE SYSTEM SHALL NOT crash with an opaque deserialization error.

**Acceptance:**
- Starting with a stale .db file produces a clear error naming the file path.
- The error exits with code 2 (config/startup error).
- Starting with a fresh or compatible .db file works normally.

**Files:**
- modify: crates/aion-server/src/state.rs
- modify: crates/aion-server/src/main.rs

### R2: Actionable error messages for cancel and other engine operations

WHEN an engine operation fails, THE SYSTEM SHALL return the actual error reason in the response body, not a generic 'engine backend failure'. Cancel, signal, query, and describe endpoints SHALL all include the typed error variant and reason.

**Acceptance:**
- Cancel of a dead workflow returns a message explaining why (e.g. 'workflow process is not live').
- No endpoint returns the bare string 'engine backend failure' without a reason.
- Error responses include an 'error_type' field with the variant name.

**Files:**
- modify: crates/aion-server/src/api/handlers.rs
- modify: crates/aion-server/src/api/http.rs
- modify: crates/aion-server/src/api/grpc.rs

## Boundaries

- SHALL NOT implement database migrations — detect and advise, don't auto-migrate.
- SHALL NOT change the error taxonomy in the engine crate.

## Verification

- cargo clippy --workspace --all-targets -- -D warnings passes.
- cargo fmt --check passes.
- cargo test --workspace passes.
- Stale database produces a clear, actionable error at startup.
