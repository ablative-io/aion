# DX-014: Fix CLI run_id resolution and list request construction

> **Cluster:** aion-dx
> **Depends on:** DX-008

## Purpose

Every CLI command except 'start' is broken. 'list' returns 'invalid argument', and describe/signal/cancel/query all fail with 'run id is missing'. The CLI passes None for run_id but the server rejects requests without one. The CLI should resolve the latest run automatically — a developer should never need to know or provide a run_id for basic operations.

## Task

Fix the CLI so that describe, signal, cancel, and query commands work with just a workflow_id by auto-resolving the latest run_id from the server. Fix the list command's request construction. If the server requires a run_id, the client should fetch it first via describe or list, then use it. If the server can be changed to accept None as 'latest run', that's even better.

## Requirements

### R1: CLI commands work with workflow_id alone

WHEN a developer runs 'aion describe <workflow_id>' without specifying a run_id, THE SYSTEM SHALL resolve the latest run and return the description. The same applies to signal, cancel, and query. THE SYSTEM SHALL accept an optional --run-id flag for cases where a specific run is needed.

**Acceptance:**
- 'aion list' returns workflow summaries without error.
- 'aion describe <workflow_id>' returns the workflow description with history.
- 'aion signal <workflow_id> <name> --payload {}' succeeds against a running workflow.
- 'aion cancel <workflow_id>' succeeds against a running workflow.
- An optional --run-id flag is accepted by describe/signal/cancel/query.

**Files:**
- modify: crates/aion-cli/src/main.rs
- modify: crates/aion-cli/src/payload.rs

## Boundaries

- SHALL NOT change the gRPC protocol definitions.
- MAY change server handler code if needed to accept None as 'use latest run'.
- SHALL NOT require the developer to look up or provide run_id for basic operations.
