# DX-022: Approval gate example with signals and timer timeout

> **Cluster:** aion-dx
> **Depends on:** DX-016

## Purpose

Human-in-the-loop workflows are a core use case for durable execution — the workflow pauses for hours or days waiting for a human decision, and the engine must keep it alive without holding resources. Developers need a concrete example showing how to use signals to receive external input (approve/reject), durable timers for deadline enforcement, and branching logic on the outcome. This pattern appears in every approval chain, review gate, and escalation flow.

## Task

Create an approval-gate example that defines a Gleam workflow using the aion_flow SDK. The workflow submits a document for approval, waits for an approval signal with a configurable timer timeout, and branches on the outcome: approved routes to a publish activity, rejected or timed-out routes to an archive activity. Include a gleam.toml with the aion_flow dependency and a README explaining the signal/timer pattern and how to run the example.

## Requirements

### R1: Signal-based approval gate with timer timeout

THE SYSTEM SHALL define a workflow using workflow.define() that calls signal.receive() to wait for an approval signal. THE SYSTEM SHALL start a durable timer using timer.start() with a configurable timeout duration. THE SYSTEM SHALL use concurrency.race() to resolve whichever completes first — the signal or the timer.

**Acceptance:**
- approval_gate.gleam imports from aion_flow and uses workflow.define().
- The workflow calls signal.receive() to wait for an 'approval_decision' signal.
- A durable timer is started with timer.start() using a configurable timeout.
- concurrency.race() is used to race the signal against the timer.
- The workflow survives engine restart and resumes waiting without re-executing completed steps.

**Files:**
- create: examples/approval-gate/src/approval_gate.gleam

### R2: Branching on approval, rejection, or timeout

WHEN the approval signal carries an 'approved' payload, THE SYSTEM SHALL execute a publish activity. WHEN the signal carries a 'rejected' payload or the timer fires first, THE SYSTEM SHALL execute an archive activity. THE SYSTEM SHALL record the outcome as the workflow result.

**Acceptance:**
- An 'approved' signal payload causes the publish activity to execute.
- A 'rejected' signal payload causes the archive activity to execute.
- Timer expiry (no signal received) causes the archive activity to execute with a timeout reason.
- The workflow result includes the decision outcome (approved, rejected, or timed_out) and the action taken.

**Files:**
- modify: examples/approval-gate/src/approval_gate.gleam

### R3: Project setup and documentation

THE SYSTEM SHALL provide a gleam.toml with the aion_flow dependency and a README explaining the signal/timer approval pattern, how to send signals to a running workflow, and step-by-step instructions for building and running the example.

**Acceptance:**
- gleam.toml lists aion_flow as a dependency and has correct project metadata.
- README.md explains the human-in-the-loop pattern and why durable timers matter for deadlines.
- README.md shows how to send an approval signal to the running workflow (via CLI or client SDK).
- README.md provides build and run instructions including how to observe the timeout path.

**Files:**
- create: examples/approval-gate/gleam.toml
- create: examples/approval-gate/README.md

## Boundaries

- SHALL NOT modify the aion_flow SDK, engine, or server code — only consume them.
- SHALL NOT add dependencies beyond aion_flow to the Gleam example.
- SHALL NOT implement a real document management system — stub activities are sufficient.
- SHALL NOT require an external service to send signals — the README shows how to send them via the Aion client or CLI.

## Verification

- gleam build succeeds in the approval-gate directory without warnings.
- The .aion package can be built from the compiled source.
- The workflow starts and blocks waiting for a signal.
- Sending an 'approved' signal causes the publish activity to execute and the workflow to complete.
- Sending a 'rejected' signal causes the archive activity to execute and the workflow to complete.
- Letting the timer expire causes the archive activity to execute with a timeout reason.
- README instructions are complete and accurate.
