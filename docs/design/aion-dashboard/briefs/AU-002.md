## Purpose

Give the dashboard a single, typed door to the server's REST surface and ensure its wire types are generated from the engine's Rust types so the UI cannot drift out of sync with the API contract.

## Task

Source the wire types (Event, WorkflowSummary, WorkflowStatus, WorkflowFilter, namespace) from the house generated-types pipeline into src/types/generated, and build a typed REST client over the server's list/query, history, and namespaces endpoints. Isolate every not-yet-pinned AW contract detail in the client so a contract change is a one-file edit. Keep local-only view models separate from generated wire types.

## Requirements

### R1: Generated wire types

THE SYSTEM SHALL source the Aion wire types from the house generated-types pipeline (Rust #[derive(TS)] on aion-core/aion-proto types → src/types/generated). THE SYSTEM SHALL NOT hand-define a type the engine owns, and SHALL NOT hand-edit generated files.

**Acceptance:**
- src/types/generated holds the generated TypeScript for Event (all variants), WorkflowSummary, WorkflowStatus, WorkflowFilter, and the namespace type.
- The generation is wired through the same mechanism apps/web uses (the just gen-types / ts-rs path), documented in the app README.
- src/types/index.ts re-exports the generated wire types and holds no hand-written copy of a generated shape.
- No generated file is hand-edited (a comment header marking it generated is present).

**Files:**
- create: apps/aion-dashboard/src/types/generated/index.ts
- create: apps/aion-dashboard/src/types/index.ts

**Checklist:**
- C6 — Wire types (Event, WorkflowSummary, WorkflowStatus, WorkflowFilter, namespace) are sourced from the generated-types pipeline (Rust #[derive(TS)] → src/types/generated/), not hand-defined, and generated files are not hand-edited.
- C9 — Local-only view models (timeline entry, filter form state) are hand-written in feature types.ts and kept distinct from generated wire types.

**Stories:**
- S13 (Frontend Maintainer, Maintaining the Dashboard Codebase) — As a frontend maintainer, I want wire types generated from the Rust engine types so that the dashboard cannot silently drift out of sync with the server's API contract.

### R2: Typed REST client over the AW surface

THE SYSTEM SHALL provide a typed REST client exposing listWorkflows/queryWorkflows (taking a WorkflowFilter plus pagination), getHistory(workflowId) returning the ordered Event list, and listNamespaces, each returning typed results and typed errors.

**Acceptance:**
- src/lib/api/client.ts exposes queryWorkflows(filter, page) → typed page of WorkflowSummary, getHistory(workflowId) → typed Event[], and listNamespaces() → typed namespace list.
- Requests carry the namespace and whatever credential the server expects (passed in, not invented here).
- Failures surface as a typed ApiError carrying status and message, mirroring apps/web's api/base pattern.
- src/lib/api/index.ts barrel re-exports the client and its types.
- A unit test with a mocked fetch asserts each method parses a typed result and maps a failure to ApiError.

**Files:**
- create: apps/aion-dashboard/src/lib/api/client.ts
- create: apps/aion-dashboard/src/lib/api/index.ts

**Checklist:**
- C7 — A typed REST client exposes listWorkflows/queryWorkflows (filter + pagination), getHistory(workflowId), and listNamespaces, each returning typed results.

**Stories:**
- S14 (Frontend Maintainer, Maintaining the Dashboard Codebase) — As a frontend maintainer, I want the server contract isolated in one api module so that when the server's endpoint or protocol detail is finalised I change it in one place.

### R3: Contract isolation for not-yet-pinned AW details

THE SYSTEM SHALL isolate every AW contract detail that is not yet finalised (endpoint paths, query/pagination parameter names, response envelope shape) in the client module so that a contract change is a single-file edit.

**Acceptance:**
- All endpoint path strings and parameter names live as named constants/config in client.ts, not scattered across feature code.
- The pagination contract (page/cursor) is expressed once in the client and consumed via the client's typed return, so features never construct raw query strings.
- An inline comment documents that these are the AW contract surface and the single place to update when AW pins the contract.
- No feature module constructs a raw URL or query string against the server.

**Files:**
- modify: apps/aion-dashboard/src/lib/api/client.ts
