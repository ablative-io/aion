# AU-008: Shared presentational components — badge, icon, states, indicator

> **Cluster:** aion-dashboard
> **Depends on:** AU-001, AU-003
> **Checklist:**
> - C35 — Shared StatusBadge, EventIcon, EmptyState, ErrorState (with retry), LoadingSkeleton, and ConnectionIndicator components exist and are reused across views.
> **Stories:**
> - S4 (On-call Operator, Monitoring Running Workflows) — As an on-call operator, I want explicit empty and error states so that I can tell the difference between 'nothing matches' and 'the query failed', instead of staring at a spinner.

## Purpose

Provide the small, reused presentational vocabulary the views depend on — status badge, event icon, the three async-state panels, and the connection indicator — so every view renders consistent loading, empty, error, and connection feedback.

## Task

Build the shared components: StatusBadge (WorkflowStatus → badge), EventIcon (event kind → icon + colour), EmptyState, ErrorState (with retry), LoadingSkeleton, and ConnectionIndicator (driven by useConnectionStatus). These are the reusable building blocks AU-004, AU-005, and AU-007 consume.

## Requirements

### R1: Status badge and event icon

THE SYSTEM SHALL provide a StatusBadge mapping each WorkflowStatus to a labelled, colour-coded badge and an EventIcon mapping each event kind to an icon and colour.

**Acceptance:**
- StatusBadge renders a distinct colour/label for Running, Completed, Failed, Cancelled, and TimedOut.
- EventIcon renders a distinct icon and colour for lifecycle, activity, timer, signal, and child-workflow kinds (lucide-react icons, matching apps/web).
- A test asserts each WorkflowStatus and each event kind maps to a distinct rendering.

**Files:**
- create: apps/aion-dashboard/src/components/StatusBadge.tsx
- create: apps/aion-dashboard/src/components/EventIcon.tsx

**Checklist:**
- C35 — Shared StatusBadge, EventIcon, EmptyState, ErrorState (with retry), LoadingSkeleton, and ConnectionIndicator components exist and are reused across views.

**Stories:**
- S4 (On-call Operator, Monitoring Running Workflows) — As an on-call operator, I want explicit empty and error states so that I can tell the difference between 'nothing matches' and 'the query failed', instead of staring at a spinner.

### R2: Empty, error, and loading state panels

THE SYSTEM SHALL provide reusable EmptyState (message + optional icon), ErrorState (cause message + retry affordance), and LoadingSkeleton (layout-reserving skeleton) components.

**Acceptance:**
- EmptyState renders a configurable message and optional icon.
- ErrorState renders a cause message and a retry button that invokes a supplied callback.
- LoadingSkeleton renders layout-reserving skeletons (configurable for list rows and timeline entries), not a bare spinner.
- A test asserts ErrorState's retry invokes the callback and the three panels render distinctly.

### R3: Connection indicator

THE SYSTEM SHALL provide a ConnectionIndicator that renders the live connection state from useConnectionStatus (connected, reconnecting, disconnected).

**Acceptance:**
- ConnectionIndicator consumes useConnectionStatus and renders a distinct visual for connected, reconnecting, and disconnected.
- It is always present in the layout (it does not disappear when connected; it shows a steady 'connected' affordance).
- A test drives the three states and asserts the matching rendering.
