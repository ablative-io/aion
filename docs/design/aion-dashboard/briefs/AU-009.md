# AU-009: Live updates — list rows, detail timeline, resync, firehose

> **Cluster:** aion-dashboard
> **Depends on:** AU-003, AU-004, AU-005
> **Checklist:**
> - C19 — A filtered WebSocket subscription updates rows live: status transitions update the corresponding row and newly-matching workflows are inserted.
> - C28 — A per-workflow WebSocket subscription appends live events to the timeline in sequence order as the execution progresses.
> - C29 — Live-appended events are de-duplicated by sequence number against events already loaded from the history fetch.
> - C30 — The timeline reflects the terminal outcome (completed/failed/cancelled/timed-out) when the closing lifecycle event arrives.
> - C31 — On socket drop the detail view resyncs (re-fetch history or replay after last-seen sequence) and the list view re-runs its query, with a reconnecting indicator visible throughout.
> - C32 — A firehose feed renders an unfiltered live event stream for observability.
> **Stories:**
> - S2 (On-call Operator, Monitoring Running Workflows) — As an on-call operator, I want the workflow list to update live so that status transitions and new workflows appear without me refreshing the page.
> - S8 (Debugging Engineer, Investigating a Single Execution) — As a debugging engineer watching a running workflow, I want new events to stream into the timeline live and the terminal outcome to appear when it completes or fails so that I can watch the execution unfold.
> - S11 (Platform Operator, Operating a Multi-Tenant Server) — As a platform operator, I want a firehose feed of all events so that I can watch overall engine activity for observability.

## Purpose

Make the dashboard live: wire the WebSocket subscriptions into the list and detail views so rows and timelines update in real time, with de-duplication, terminal-outcome reflection, drop resync, and a firehose feed.

## Task

Build the live layer on top of AU-004's list and AU-005's timeline: a shared subscription lifecycle hook, a filtered list subscription that patches rows live, a per-workflow subscription that appends events to the timeline (de-duplicated by sequence and reflecting the terminal outcome), resync on socket drop for both views, and a firehose feed for unfiltered observability.

## Requirements

### R1: Shared subscription lifecycle and live list rows

THE SYSTEM SHALL provide a shared subscription lifecycle hook and use it to drive a filtered list subscription that updates rows live: status transitions update the corresponding row and newly-matching workflows are inserted.

**Acceptance:**
- src/features/live-feed/hooks/useEventSubscription.ts encapsulates subscribe-on-mount / unsubscribe-on-unmount against the websocket manager for a given filter, exposing the resync signal.
- src/features/workflow-list/hooks/useLiveListUpdates.ts subscribes to the active filter and patches the list query cache: a status-changing event updates the matching row; a newly-matching workflow is inserted.
- A test feeds synthetic events and asserts a row's status updates and a new matching workflow appears.

### R2: Live timeline append with de-dup and terminal outcome

THE SYSTEM SHALL subscribe per-workflow on the detail view and append live events to the timeline in sequence order, de-duplicating by sequence number against history already loaded, and reflecting the terminal outcome when the closing lifecycle event arrives.

**Acceptance:**
- src/features/workflow-detail/hooks/useLiveWorkflowEvents.ts subscribes to the workflow id and merges incoming events into the timeline model in sequence order.
- Events already present from the history fetch (same sequence number) are not rendered twice.
- When WorkflowCompleted/Failed/Cancelled (or timed-out) arrives, the timeline reflects the terminal outcome and the view stops expecting further events.
- A test seeds a history then streams overlapping + new events and asserts no duplicates, correct ordering, and terminal-outcome reflection.

### R3: Resync on drop and firehose feed

WHEN the socket drops, THE SYSTEM SHALL resync the active view — the detail view re-fetches history (or replays after its last-seen sequence) and the list view re-runs its query — with a reconnecting indicator visible. THE SYSTEM SHALL also provide a firehose feed rendering an unfiltered live event stream.

**Acceptance:**
- On the manager's resync signal, the detail view re-fetches history or replays after_seq and the list view re-runs its query; the ConnectionIndicator shows reconnecting throughout.
- FirehoseFeed subscribes to the unfiltered stream and renders incoming events live (scoped by the selected namespace).
- A test simulates a drop and asserts the detail view resyncs and the list re-queries; a test asserts the firehose renders streamed events.
- The firehose feed renders explicit empty (no events yet) and disconnected states.

## Boundaries

- SHALL NOT reimplement the WebSocket manager or its reconnect — AU-003 (this brief consumes its subscribe/resync surface).
- SHALL NOT reimplement the list query or timeline projection — AU-004, AU-005 (this brief patches their caches/models live).
- SHALL NOT define the WebSocket protocol or after_seq semantics — that is cluster AW.
