# AE-004: Active-execution registry and WorkflowHandle

> **Cluster:** aion-engine
> **Depends on:** AE-001
> **Checklist:**
> - C9 — WorkflowHandle carries the beamr pid, the workflow type, the loaded version, and a cached WorkflowStatus.
> - C10 — The registry maps (WorkflowId, RunId) to a WorkflowHandle and supports insert, lookup, remove, and list of live handles.
> - C11 — The registry is concurrency-safe and lock poison is mapped to EngineError::RegistryPoisoned rather than panicking.
> - C12 — The cached status is reconciled against aion-core's event-history projection and the projection wins on disagreement; no code path sets status independently of events.
> **Stories:**
> - S5 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to start a workflow by type and get a handle so that I can later await its result or cancel it by ID.
> - S7 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to list and query workflows by type, status, and time so that I can see what is running and what has finished without loading full histories.

## Purpose

Map a running workflow to its beamr process. The registry is the lookup path for cancel/signal/query and the source for list_workflows, and it keeps a cached status that is always reconciled against the authoritative event projection.

## Task

Implement the registry module: WorkflowHandle (pid, workflow type, loaded version, cached status) and a concurrency-safe registry keyed by (WorkflowId, RunId) supporting insert/lookup/remove/list. Lock poison maps to EngineError::RegistryPoisoned. Provide a reconcile operation that takes the projected WorkflowStatus from aion-core and overwrites the cached status, with the projection winning. Scope is the registry data structure and reconciliation — lifecycle transitions that drive it are later briefs.

## Requirements

### R1: WorkflowHandle type

THE SYSTEM SHALL define WorkflowHandle carrying the beamr pid, the workflow type, the loaded version identifier, and a cached WorkflowStatus.

**Acceptance:**
- handle.rs defines WorkflowHandle with fields for the beamr pid, the workflow type (the logical entry module name), the loaded version (content hash / version-record identifier from aion-package), and a cached aion_core WorkflowStatus.
- WorkflowHandle is Clone and Debug; it stores no beamr runtime state, only the pid value.
- registry/mod.rs contains only pub mod and pub use.

**Files:**
- create: crates/aion/src/registry/handle.rs
- create: crates/aion/src/registry/mod.rs
- modify: crates/aion/src/lib.rs

### R2: Registry keyed by (WorkflowId, RunId)

THE SYSTEM SHALL implement a concurrency-safe registry mapping (WorkflowId, RunId) to a WorkflowHandle, supporting insert, lookup, remove, and list of live handles.

**Acceptance:**
- table.rs defines a Registry holding a map keyed by (WorkflowId, RunId) behind a lock (Mutex or RwLock), Default-constructible and Send + Sync.
- insert((id, run), handle), get(&id, &run) -> Option<WorkflowHandle>, remove(&id, &run), and list() -> Vec<WorkflowHandle> are provided.
- A test inserts two runs of the same WorkflowId and asserts both are retrievable and a stale run does not shadow a newer one.
- list() returns a snapshot vector, not a guard, so callers do not hold the lock.

**Files:**
- create: crates/aion/src/registry/table.rs
- modify: crates/aion/src/registry/mod.rs

**Checklist:**

### R3: Explicit lock-poison handling

IF the registry lock is poisoned, THEN THE SYSTEM SHALL return EngineError::RegistryPoisoned. THE SYSTEM SHALL NOT call .unwrap() or .expect() on a lock guard.

**Acceptance:**
- Every lock acquisition maps a PoisonError to EngineError::RegistryPoisoned (registry methods return Result accordingly).
- grep of registry/ shows no .unwrap()/.expect() on lock results.
- A test simulates a poisoned lock (e.g. by panicking while holding the guard in a spawned thread) and asserts the next access returns RegistryPoisoned rather than panicking.

**Files:**
- modify: crates/aion/src/registry/table.rs

**Checklist:**

### R4: Status reconciliation against the event projection

WHEN reconciling a handle's status, THE SYSTEM SHALL compute the WorkflowStatus by projecting over the workflow's event history using aion-core and SHALL overwrite the cached status with the projection. THE SYSTEM SHALL NOT set a handle's status from any source other than that projection.

**Acceptance:**
- table.rs (or handle.rs) provides reconcile(&id, &run, events: &[Event]) that recomputes status via aion-core's projection and updates the cached status on the handle.
- No registry API mutates status with a caller-supplied status value; the only writer is the projection path.
- A test builds an event history ending in WorkflowCompleted, reconciles, and asserts the cached status is Completed; a history ending in WorkflowCancelled reconciles to Cancelled.
- A test asserts that if the cached status disagrees with the projection, reconcile makes the projection win.

## Boundaries

- SHALL NOT drive lifecycle transitions (start/cancel/complete) — the registry is updated by AE-009/AE-010.
- SHALL NOT define WorkflowStatus or its projection — those are aion-core; this brief consumes them.
- SHALL NOT query the store or run replay — reconcile takes the events it is given; loading them is AD's append/read path used by lifecycle.
