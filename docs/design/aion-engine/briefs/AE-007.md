# AE-007: In-VM activity dispatch and outcome propagation

> **Cluster:** aion-engine
> **Depends on:** AE-002, AE-003, AE-006
> **Checklist:**
> - C21 — An in-VM activity is dispatched as a child BEAM process linked to the workflow process, running on the dirty scheduler when the underlying NIF is flagged dirty.
> - C22 — The activity's outcome — a result Payload or an exit signal carrying an ActivityError — is surfaced back to the workflow process through the link/mailbox.
> - C23 — Activity dispatch performs no event recording and makes no retry decision itself; those are delegated to the AD append path and AT retry machinery.
> **Stories:**
> - S11 (Reliability Engineer, Relying on Supervision and Fault Isolation) — As a reliability engineer, I want an activity crash to reach the workflow process as a trapped exit rather than killing it outright so that the workflow can apply its retry policy instead of dying.

## Purpose

Dispatch a Tier-2 in-VM activity as a linked child BEAM process and surface its outcome to the workflow process. This is the in-VM execution path; recording the activity events and deciding retry are seams owned by AD and AT, called from elsewhere.

## Task

Implement the activity module: dispatch an in-VM activity as a child process linked to the workflow process, choosing the dirty scheduler when the activity's underlying NIF was registered dirty — the dirty flag is looked up from the NIF registration surface (AE-003), which records per (module, function) whether it was registered via register_dirty. Then surface the activity's outcome — a result Payload or an exit signal carrying an ActivityError — back to the workflow process through the link/mailbox. This brief performs no event recording and no retry decision; it provides the dispatch and outcome plumbing those seams build on.

## Requirements

### R1: Dispatch as a linked child process

WHEN an in-VM activity is dispatched, THE SYSTEM SHALL spawn it as a child BEAM process linked to the workflow process via the RuntimeHandle, running on the dirty scheduler when the activity's underlying NIF is flagged dirty.

**Acceptance:**
- dispatch.rs exposes dispatch_activity(parent_pid, deployed_module, function, input: Payload) -> Result<Pid, EngineError> that calls RuntimeHandle::spawn_activity to spawn-and-link.
- The dirty flag is resolved by querying the NIF registration surface (AE-003) for (deployed_module, function): NIFs registered via register_dirty are recorded as dirty there, and dispatch consults that record — it does not guess or carry the flag separately. RuntimeHandle exposes an is_dirty(module, function) -> bool (or equivalent) lookup over the registered NIFs.
- When the looked-up flag is dirty, the child runs on beamr's dirty scheduler pool; otherwise on the normal scheduler.
- activity/mod.rs contains only pub mod and pub use.
- A test dispatches a trivial in-VM activity from a workflow process and asserts the child is spawned and linked to the parent.

### R2: Surface the outcome to the workflow process

WHEN an activity child completes, THE SYSTEM SHALL surface its result Payload to the workflow process; WHEN it exits abnormally, THE SYSTEM SHALL surface an exit signal carrying an ActivityError to the workflow process through the link/mailbox.

**Acceptance:**
- On success, the activity's result is delivered to the workflow process as a result Payload (via the mailbox / linked completion path).
- On failure, the workflow process receives a trapped-exit message whose reason carries an aion_core ActivityError (preserving the retryable/terminal classification).
- A test dispatches an activity that returns a value and asserts the workflow process receives the result Payload; a second test dispatches one that fails and asserts the workflow process receives an ActivityError-bearing exit.

### R3: No recording, no retry decision here

THE SYSTEM SHALL NOT append any Activity* event and SHALL NOT make any retry decision in the dispatch path; recording is delegated to the AD append path and the retry decision to AT's policy machinery.

**Acceptance:**
- dispatch.rs contains no call that appends ActivityScheduled/Started/Completed/Failed events and no RetryPolicy evaluation.
- Module docs state that recording is AD's append path and retry is AT's machinery, and that dispatch only spawns, links, and propagates the outcome.
- grep of activity/ shows no store append call and no retry-loop construct.

## Boundaries

- SHALL NOT record Activity* events — that is the AD append path.
- SHALL NOT evaluate or apply RetryPolicy — that is AT's activity machinery, which consumes the ActivityError this brief surfaces.
- SHALL NOT dispatch remote (Tier-3) activities — those ride aion-server's worker protocol (AW); this is the in-VM path only.
- SHALL NOT set trap-exit or build the supervision tree — that is AE-006, on which this brief depends.
