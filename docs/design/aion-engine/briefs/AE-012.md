# AE-012: Engine API: start_workflow, cancel, result, list_workflows, shutdown

> **Cluster:** aion-engine
> **Depends on:** AE-008, AE-009, AE-011
> **Checklist:**
> - C30 — Engine exposes start_workflow, cancel, result, list_workflows, and shutdown.
> - C31 — list_workflows returns live handles from the registry and falls through to the store query for terminal workflows, returning WorkflowSummary values.
> - C32 — shutdown stops accepting new starts, lets in-flight store appends finish, drains and stops the runtime scheduler, and returns once beamr has stopped.
> **Stories:**
> - S3 (Engine Operator, Embedding and Running the Engine) — As an engine operator, I want a graceful shutdown that finishes in-flight appends and stops the runtime cleanly so that stopping the engine never abandons writes or leaks scheduler threads.
> - S5 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to start a workflow by type and get a handle so that I can later await its result or cancel it by ID.
> - S6 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to cancel a running workflow and have its in-flight activities torn down automatically so that cancellation leaves no orphaned work.
> - S7 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to list and query workflows by type, status, and time so that I can see what is running and what has finished without loading full histories.
> - S8 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to await a workflow's result and receive either its output payload or its terminal error so that I can react to the outcome.

## Purpose

Expose the runtime embedding API consumers actually call. Engine wraps the lifecycle transitions, the registry, the store, and the runtime into start_workflow, cancel, result, list_workflows, and a graceful shutdown.

## Task

Implement engine/engine.rs: the Engine type returned by build(), exposing start_workflow (delegating to the start transition and returning a WorkflowHandle), cancel (delegating to the cancel transition), result (awaiting completion/failure and returning the Payload or WorkflowError), list_workflows (live handles from the registry plus terminal workflows from the store query), and shutdown (graceful drain). Scope is these methods; signal/query/subscribe are AE-013.

## Requirements

### R1: start_workflow and cancel

THE SYSTEM SHALL expose start_workflow(workflow_type, input: Payload) -> Result<WorkflowHandle, EngineError> delegating to the start transition, and cancel(&id, &run, reason) -> Result<(), EngineError> delegating to the cancel transition.

**Acceptance:**
- engine.rs defines Engine holding the RuntimeHandle, the Arc<dyn EventStore>, the registry, the supervision tree, and the loaded-workflow records.
- start_workflow calls the AE-008 start operation and returns the WorkflowHandle.
- cancel calls the AE-009 cancel transition and returns Ok or EngineError::WorkflowNotFound.
- A test starts then cancels a workflow against InMemoryStore and asserts the resulting history is WorkflowStarted followed by WorkflowCancelled.

### R2: result()

WHEN result(&id, &run) is called, THE SYSTEM SHALL await the workflow's terminal transition and return the result Payload on completion or the WorkflowError on failure. IF the workflow id/run is unknown and has no terminal history, THEN THE SYSTEM SHALL return EngineError::WorkflowNotFound.

**Acceptance:**
- result awaits the completion-notification channel resolved by the AE-009 complete/fail transitions and returns Result<Payload, WorkflowError> wrapped in EngineError where appropriate.
- For an already-terminal workflow, result reads the terminal event from the store and returns its Payload/WorkflowError without blocking forever.
- An unknown id/run with no history returns EngineError::WorkflowNotFound.
- A test starts a workflow that completes and asserts result returns the recorded Payload; a test for a failing workflow asserts the WorkflowError.

### R3: list_workflows

WHEN list_workflows(filter: WorkflowFilter) is called, THE SYSTEM SHALL return WorkflowSummary values for live workflows from the registry and for terminal workflows from the store query, deduplicated and matching the filter.

**Acceptance:**
- list_workflows builds summaries from live registry handles and merges them with the store's query(filter) results, with no duplicate entry for a workflow present in both.
- The returned summaries match the WorkflowFilter (type/status/time-range/parent) semantics from aion-core.
- A test with one running and one completed workflow asserts both appear once, with correct statuses, and that a status filter narrows the result.

### R4: Graceful shutdown

WHEN shutdown() is called, THE SYSTEM SHALL stop accepting new starts, allow in-flight store appends to finish, drain and stop the runtime scheduler, and return once beamr has stopped. THE SYSTEM SHALL NOT drop the runtime to stop it.

**Acceptance:**
- shutdown sets a flag so subsequent start_workflow calls return an EngineError (shutting-down) rather than spawning.
- shutdown waits for outstanding store appends initiated by lifecycle transitions to complete before stopping the runtime.
- shutdown calls RuntimeHandle::shutdown to drain and stop the scheduler and returns Ok once beamr has stopped.
- A workflow mid-execution at shutdown is left with its durable history intact (Suspended), not force-completed.
- A test builds an Engine, starts and completes a workflow, calls shutdown, and asserts it returns Ok and a subsequent start_workflow is rejected.

## Boundaries

- SHALL NOT implement signal/query/subscribe — those are AE-013.
- SHALL NOT implement HTTP/gRPC/WebSocket — Engine is transport-agnostic (CO6); networking is AW.
- SHALL NOT reimplement lifecycle transitions — Engine delegates to AE-008/AE-009.
- SHALL NOT force-complete or discard in-flight workflows on shutdown — they remain replayable via their durable history (D11).
