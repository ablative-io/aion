# AE-008: Workflow start lifecycle transition

> **Cluster:** aion-engine
> **Depends on:** AE-004, AE-005, AE-006, AD-002
> **Checklist:**
> - C24 — start_workflow assigns a WorkflowId and RunId, appends WorkflowStarted via the store, spawns the workflow process for the loaded type, registers the handle, and returns a WorkflowHandle.
> **Stories:**
> - S5 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to start a workflow by type and get a handle so that I can later await its result or cancel it by ID.

## Purpose

Implement the start of a workflow's life: assign its identifiers, create its single-writer Recorder, record WorkflowStarted through it, spawn its process under the right supervisor, register the handle (with its completion notifier), and return it. This is the entry point start_workflow drives.

## Task

Implement lifecycle/start.rs: a start operation that takes a loaded workflow type and an input Payload, assigns a WorkflowId and RunId, creates the workflow's AD Recorder (AD-002) — the single writer for its history — appends WorkflowStarted through that Recorder (the Recorder owns the expected-sequence head; the engine never appends to the store directly, per AD CO6), resolves the deployed entry module/function via the loader record, spawns the workflow process under its per-type supervisor with trap-exit set, registers the WorkflowHandle (including a completion notifier and its Recorder), and returns it. Scope is start only — cancel/complete/fail are AE-009 and suspend/resume are AE-010.

## Requirements

### R1: Assign identifiers and append WorkflowStarted through the Recorder

WHEN starting a workflow, THE SYSTEM SHALL assign a fresh WorkflowId and RunId, create the workflow's single-writer Recorder (AD-002), and append WorkflowStarted (workflow type, input Payload, timestamp) through that Recorder, which owns the expected-sequence head for the new history.

**Acceptance:**
- start.rs assigns a new WorkflowId and RunId for the execution.
- It creates the AD Recorder for the new workflow (initial sequence head 0) and appends WorkflowStarted (workflow type + input Payload) through it; the engine does not call EventStore::append directly and does not compute expected_seq itself — the Recorder is the single writer and the sequence-head authority (AD CO6/CO7).
- A Recorder/append error surfaces as EngineError::Store (or EngineError::Durability) with no process spawned.
- A test starts a workflow against InMemoryStore and asserts the first event in history is WorkflowStarted at sequence 0 with the given type and input.

### R2: Resolve deployed entry module and spawn under the supervisor

THE SYSTEM SHALL resolve the workflow type to its deployed entry module and function via the loader record and SHALL spawn the workflow process under its per-type supervisor with trap-exit set. IF the workflow type is not loaded, THEN THE SYSTEM SHALL return EngineError::WorkflowNotFound and append nothing.

**Acceptance:**
- start.rs looks up the LoadedWorkflow record (from AE-005) to get the deployed entry module/function and version.
- It ensures the per-type supervisor (AE-006) and spawns the workflow process under it via RuntimeHandle::spawn_workflow, with trap-exit set per the supervision policy (AE-006).
- An unloaded workflow type returns EngineError::WorkflowNotFound and produces no WorkflowStarted append (the type check precedes the append).
- A test starts an unknown type and asserts WorkflowNotFound with an empty store.

### R3: Register the handle (with completion notifier) and return it

WHEN the workflow process is spawned, THE SYSTEM SHALL register a WorkflowHandle keyed by (WorkflowId, RunId) with status Running and a freshly-created completion notifier, and SHALL return the handle to the caller.

**Acceptance:**
- On successful spawn, a WorkflowHandle (pid, type, version, status Running, residency Resident, the workflow's Recorder, and a completion notifier) is inserted into the registry under (WorkflowId, RunId).
- The completion notifier is created here at start and stored on the handle; it is the channel a later result() awaiter blocks on and that the terminal transitions (AE-009) fulfil. A oneshot is insufficient (multiple awaiters); use a multi-consumer notifier (e.g. a watch/broadcast of the eventual TerminalOutcome) so any number of result() callers can observe completion.
- start returns the WorkflowHandle to the caller.
- A test starts a workflow and asserts the registry contains a Running, Resident handle with a completion notifier for the new (WorkflowId, RunId) and that the returned handle matches.
- IF the registry lock is poisoned, the operation returns EngineError::RegistryPoisoned.

## Boundaries

- SHALL NOT implement cancel/complete/fail — those are AE-009.
- SHALL NOT implement suspend/resume — those are AE-010.
- SHALL NOT implement the replay path used to re-create handles (and Recorders) on restart — that is AD (recovery, AD-009).
- SHALL NOT define the Engine public method — start_workflow on Engine is the engine surface; this brief provides the lifecycle operation it calls.
- SHALL NOT append to the EventStore directly — all appends go through the Recorder (AD CO6).
