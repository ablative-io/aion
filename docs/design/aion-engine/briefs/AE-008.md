# AE-008: Workflow start lifecycle transition

> **Cluster:** aion-engine
> **Depends on:** AE-004, AE-005, AE-006
> **Checklist:**
> - C24 — start_workflow assigns a WorkflowId and RunId, appends WorkflowStarted via the store, spawns the workflow process for the loaded type, registers the handle, and returns a WorkflowHandle.
> **Stories:**
> - S5 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to start a workflow by type and get a handle so that I can later await its result or cancel it by ID.

## Purpose

Implement the start of a workflow's life: assign its identifiers, record WorkflowStarted, spawn its process under the right supervisor, register the handle, and return it. This is the entry point start_workflow drives.

## Task

Implement lifecycle/start.rs: a start operation that takes a loaded workflow type and an input Payload, assigns a WorkflowId and RunId, appends WorkflowStarted through the store (AD append path), resolves the deployed entry module/function via the loader record, spawns the workflow process under its per-type supervisor with trap-exit set, registers the WorkflowHandle, and returns it. Scope is start only — cancel/complete/fail are AE-010 and suspend/resume are AE-011.

## Requirements

### R1: Assign identifiers and append WorkflowStarted

WHEN starting a workflow, THE SYSTEM SHALL assign a fresh WorkflowId and RunId and SHALL append a WorkflowStarted event (workflow type, input Payload, timestamp) through the store's append path with the expected sequence for a new history.

**Acceptance:**
- start.rs assigns a new WorkflowId and RunId for the execution.
- It constructs a WorkflowStarted event carrying the workflow type and the input Payload and appends it with expected_seq for an empty history (sequence 0 / the documented initial expectation).
- A store append error surfaces as EngineError::Store with no process spawned.
- A test starts a workflow against InMemoryStore and asserts the first event in history is WorkflowStarted with the given type and input.

### R2: Resolve deployed entry module and spawn under the supervisor

THE SYSTEM SHALL resolve the workflow type to its deployed entry module and function via the loader record and SHALL spawn the workflow process under its per-type supervisor with trap-exit set. IF the workflow type is not loaded, THEN THE SYSTEM SHALL return EngineError::WorkflowNotFound and append nothing.

**Acceptance:**
- start.rs looks up the LoadedWorkflow record (from AE-005) to get the deployed entry module/function and version.
- It ensures the per-type supervisor (AE-006) and spawns the workflow process under it via RuntimeHandle::spawn_workflow, with trap-exit set per the supervision policy (AE-006).
- An unloaded workflow type returns EngineError::WorkflowNotFound and produces no WorkflowStarted append (the type check precedes the append).
- A test starts an unknown type and asserts WorkflowNotFound with an empty store.

### R3: Register the handle and return it

WHEN the workflow process is spawned, THE SYSTEM SHALL register a WorkflowHandle keyed by (WorkflowId, RunId) with status Running and SHALL return the handle to the caller.

**Acceptance:**
- On successful spawn, a WorkflowHandle (pid, type, version, status Running) is inserted into the registry under (WorkflowId, RunId).
- start returns the WorkflowHandle to the caller.
- A test starts a workflow and asserts the registry contains a Running handle for the new (WorkflowId, RunId) and that the returned handle matches.
- IF the registry lock is poisoned, the operation returns EngineError::RegistryPoisoned.

## Boundaries

- SHALL NOT implement cancel/complete/fail — those are AE-010.
- SHALL NOT implement suspend/resume — those are AE-011.
- SHALL NOT implement the replay path used to re-create handles on restart — that is AD (build wires it in AE-010-engine).
- SHALL NOT define the Engine public method — start_workflow on Engine is AE-013; this brief provides the lifecycle operation it calls.
