# AE-009: Cancel, complete, and fail lifecycle transitions

> **Cluster:** aion-engine
> **Depends on:** AE-008, AD-002
> **Checklist:**
> - C25 — Completion appends WorkflowCompleted with the result Payload, deregisters the handle, and unblocks result() awaiters; failure appends WorkflowFailed with the WorkflowError analogously.
> - C26 — cancel kills the workflow process (propagating to activity children), appends WorkflowCancelled with the reason, and deregisters the handle.
> **Stories:**
> - S6 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to cancel a running workflow and have its in-flight activities torn down automatically so that cancellation leaves no orphaned work.
> - S8 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to await a workflow's result and receive either its output payload or its terminal error so that I can react to the outcome.

## Purpose

Implement the terminal transitions of a workflow's life: completion records the result and unblocks awaiters, failure records the terminal error, and cancellation kills the process (propagating to activity children) and records the reason. All recording goes through the workflow's single-writer Recorder. Together with start, this is the full terminal lifecycle.

## Task

Implement lifecycle/terminate.rs: complete (append WorkflowCompleted through the Recorder, fulfil the completion notifier, deregister), fail (append WorkflowFailed through the Recorder, fulfil the notifier with the error, deregister), and cancel (kill the workflow process so links tear down activity children, append WorkflowCancelled through the Recorder, fulfil the notifier, deregister). Recording is via the handle's Recorder (AD-002) — never EventStore::append directly. Scope is the terminal transitions; suspend/resume are AE-010 and the Engine public methods are the engine surface.

## Requirements

### R1: Complete transition

WHEN a workflow function returns Ok, THE SYSTEM SHALL append WorkflowCompleted carrying the result Payload, deregister the handle, and unblock any awaiter of the workflow's result.

**Acceptance:**
- terminate.rs provides complete(id, run, result: Payload) that appends WorkflowCompleted through the handle's Recorder (which supplies the expected sequence); no direct EventStore::append.
- Before the handle is removed, the completion notifier created at start (AE-008, stored on the WorkflowHandle) is fulfilled with the terminal outcome (the result Payload) so already-waiting result() callers wake; then the handle is removed from the registry.
- Ownership of the result channel is fixed: it is created at start (AE-008) on the handle, fulfilled here, and consumed by Engine::result. Because the handle is deregistered on terminal, a result() call arriving AFTER completion finds no handle and instead reads the terminal WorkflowCompleted/WorkflowFailed from history (read_history) — so result() is correct whether called before or after completion.
- A test completes a workflow and asserts the last event is WorkflowCompleted with the result, a pending awaiter is woken with the result, and the registry no longer holds the handle.

### R2: Fail transition

WHEN a workflow function returns Err, THE SYSTEM SHALL append WorkflowFailed carrying the WorkflowError, deregister the handle, and resolve any awaiter with that error.

**Acceptance:**
- terminate.rs provides fail(id, run, error: WorkflowError) that appends WorkflowFailed through the handle's Recorder; no direct EventStore::append.
- The completion notifier is fulfilled with the WorkflowError, then the handle is removed from the registry.
- Any pending result() awaiter is resolved with the WorkflowError; a result() call after deregistration reads the terminal WorkflowFailed from history.
- A test fails a workflow and asserts the last event is WorkflowFailed with the error and the handle is deregistered.

### R3: Cancel transition with link propagation

WHEN cancel is requested for a workflow, THE SYSTEM SHALL kill the workflow process so its linked activity children are torn down by link propagation, append WorkflowCancelled carrying the reason, and deregister the handle. IF the workflow is not in the registry, THEN THE SYSTEM SHALL return EngineError::WorkflowNotFound.

**Acceptance:**
- terminate.rs provides cancel(id, run, reason) that looks up the handle, calls RuntimeHandle::cancel_pid on the workflow pid, appends WorkflowCancelled through the handle's Recorder (no direct EventStore::append), fulfils the completion notifier, and removes the handle.
- Because activity children are linked (AE-006/AE-007), killing the workflow process terminates them with no per-child teardown loop.
- Cancelling an unknown (id, run) returns EngineError::WorkflowNotFound.
- A test starts a workflow with a linked activity child, cancels it, and asserts WorkflowCancelled is appended, the child is no longer alive, and the handle is deregistered.

## Boundaries

- SHALL NOT implement suspend/resume — those are AE-010.
- SHALL NOT decide retry or re-run a failed activity — failure here is workflow-terminal; activity retry is AT.
- SHALL NOT define the Engine public cancel/result methods — those are the engine surface; this brief provides the transitions they call.
- SHALL NOT append to the EventStore directly — all terminal events go through the Recorder (AD CO6).
- SHALL NOT compute WorkflowStatus independently — terminal status follows from the appended event via the aion-core projection (CO10).
