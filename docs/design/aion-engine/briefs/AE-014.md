# AE-014: End-to-end engine integration tests over InMemoryStore

> **Cluster:** aion-engine
> **Depends on:** AE-012, AE-007
> **Checklist:**
> - C34 — An integration test builds an Engine over InMemoryStore, starts a test workflow, and asserts WorkflowStarted is appended and the handle is registered and reported by list_workflows.
> - C35 — An integration test asserts cancelling a workflow kills its linked activity child and appends WorkflowCancelled, and that a completed workflow's result() returns the recorded Payload.
> **Stories:**
> - S1 (Engine Operator, Embedding and Running the Engine) — As an engine operator, I want to build an Engine from a store, loaded workflows, and my NIFs with one builder so that embedding durable workflows in my Rust app is a single, clear assembly step.
> - S6 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to cancel a running workflow and have its in-flight activities torn down automatically so that cancellation leaves no orphaned work.
> - S8 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to await a workflow's result and receive either its output payload or its terminal error so that I can react to the outcome.

## Purpose

Prove the engine works end to end against InMemoryStore: start records the right event and registers a discoverable handle, cancel tears down activity children and records the cancellation, and completion yields the recorded result. These are the cross-cutting tests that exercise the whole lifecycle through the public Engine API.

## Task

Build integration tests in crates/aion/tests that drive a real Engine over InMemoryStore using a minimal test workflow (a tiny compiled-beam fixture or a test-only module registered through the runtime) that dispatches an in-VM activity. Assert the start/register/list path, the cancel/teardown/append path, and the complete/result path. Tests run a real runtime at runtime — never gated out or #[ignore]'d.

## Requirements

### R1: Start, append, register, and list

THE SYSTEM SHALL provide an integration test that builds an Engine over InMemoryStore, starts a test workflow, and asserts WorkflowStarted is appended, the handle is registered, and list_workflows reports it.

**Acceptance:**
- tests/engine_lifecycle.rs builds an Engine over a fresh InMemoryStore with one loaded test workflow.
- After start_workflow, the store history's first event is WorkflowStarted with the workflow type and input.
- The registry contains a Running handle for the new (WorkflowId, RunId) and list_workflows returns a matching WorkflowSummary.
- The test runs under cargo test (real runtime), not gated out or ignored.

**Files:**
- create: crates/aion/tests/engine_lifecycle.rs

### R2: Cancel tears down activity child and appends cancellation

THE SYSTEM SHALL provide an integration test that starts a workflow which dispatches an in-VM activity child, cancels the workflow, and asserts the activity child is no longer alive and WorkflowCancelled is appended.

**Acceptance:**
- The test workflow dispatches an in-VM activity (AE-007) as a linked child that stays alive long enough to observe.
- After cancel, the activity child process is no longer alive (link propagation) and the store history ends with WorkflowCancelled carrying the reason.
- The registry no longer holds the handle after cancel.
- The test is a real runtime test, not gated out or ignored.

**Files:**
- create: crates/aion/tests/engine_cancel.rs

**Checklist:**
- C35 — An integration test asserts cancelling a workflow kills its linked activity child and appends WorkflowCancelled, and that a completed workflow's result() returns the recorded Payload.

**Stories:**
- S6 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to cancel a running workflow and have its in-flight activities torn down automatically so that cancellation leaves no orphaned work.

### R3: Completion yields the recorded result

THE SYSTEM SHALL provide an integration test that starts a workflow which completes and asserts result() returns the recorded result Payload and the final history event is WorkflowCompleted.

**Acceptance:**
- The test workflow runs to completion returning a known value.
- result() returns the recorded result Payload equal to the workflow's output.
- The store history's final event is WorkflowCompleted carrying that Payload, and list_workflows reports the workflow as Completed.
- The test is a real runtime test, not gated out or ignored.

## Boundaries

- SHALL NOT require a database — all tests use InMemoryStore (CO13).
- SHALL NOT use #[ignore] or feature-gate the tests out; any runtime requirement is satisfied by the embedded beamr runtime the test builds.
- SHALL NOT test AT/AD mechanics (signal routing, replay, durable timers) beyond the AE lifecycle surface — those are exercised in AT/AD.
