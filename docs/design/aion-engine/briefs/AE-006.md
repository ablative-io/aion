# AE-006: Three-level supervision tree and trap-exit policy

> **Cluster:** aion-engine
> **Depends on:** AE-002
> **Checklist:**
> - C17 — An engine supervisor is the root and one workflow supervisor is created per registered workflow type.
> - C18 — Each workflow execution runs as a process under its type's supervisor, and activity invocations run as linked child processes of the workflow process.
> - C19 — Workflow processes trap exits so an activity child crash arrives as a message, while activity processes do not trap exits.
> - C20 — Cancelling a workflow kills its linked activity children via link propagation with no manual teardown.
> **Stories:**
> - S11 (Reliability Engineer, Relying on Supervision and Fault Isolation) — As a reliability engineer, I want an activity crash to reach the workflow process as a trapped exit rather than killing it outright so that the workflow can apply its retry policy instead of dying.
> - S12 (Reliability Engineer, Relying on Supervision and Fault Isolation) — As a reliability engineer, I want workflows supervised per type under an engine root so that crashes are isolated and the supervision shape is explicit and bounded.
> - S6 (Workflow Caller, Starting, Cancelling, and Inspecting Workflows) — As a workflow caller, I want to cancel a running workflow and have its in-flight activities torn down automatically so that cancellation leaves no orphaned work.

## Purpose

Stand up the engine -> per-type -> workflow -> activity supervision tree on beamr's link/monitor/trap-exit primitives. This is the structural backbone that makes an activity crash a message the workflow can act on and makes cancellation propagate cleanly to children.

## Task

Implement the supervision module: construction of the engine supervisor root and per-workflow-type supervisors, and the trap-exit/crash-propagation policy that workflow processes run under. Workflow processes trap exits; activity children do not. Cancelling a workflow kills its linked activity children via link propagation. Scope is the supervision shape and policy; activity spawning mechanics are AE-007 and lifecycle is AE-009/010.

## Requirements

### R1: Engine supervisor and per-type supervisors

THE SYSTEM SHALL construct an engine supervisor as the root and SHALL create exactly one workflow supervisor per registered workflow type, under the engine supervisor.

**Acceptance:**
- tree.rs constructs an engine supervisor (the root) and provides ensure_type_supervisor(workflow_type) that creates the per-type supervisor on first use and returns the existing one thereafter.
- The number of workflow supervisors equals the number of distinct registered workflow types (per D6), never one-per-execution.
- supervision/mod.rs contains only pub mod and pub use.
- A test registers two workflow types, ensures their supervisors, and asserts exactly two per-type supervisors exist under the one engine supervisor.

**Files:**
- create: crates/aion/src/supervision/tree.rs
- create: crates/aion/src/supervision/mod.rs
- modify: crates/aion/src/lib.rs

### R2: Workflow process under its type supervisor; activities as linked children

WHEN a workflow execution is placed in the tree, THE SYSTEM SHALL run it as a process under its workflow type's supervisor, and SHALL run activity invocations as child processes linked to the workflow process.

**Acceptance:**
- tree.rs exposes placing a workflow process under the correct per-type supervisor and recording the parent->children link relationship for its activities.
- An activity child is linked to its workflow process (the link is established at spawn, via RuntimeHandle::spawn_activity from AE-002).
- A test asserts that a workflow process's activity children are linked to it and sit one level below it in the tree.

**Files:**

### R3: Trap-exit policy: workflows trap, activities do not

THE SYSTEM SHALL set the trap-exit flag on workflow processes so an activity child's exit signal arrives as a message. THE SYSTEM SHALL NOT set trap-exit on activity processes.

**Acceptance:**
- policy.rs sets trap-exit on a workflow process at spawn (through the RuntimeHandle / beamr primitive surface).
- Activity processes are spawned without trap-exit.
- WHEN an activity child exits abnormally, the workflow process receives the exit as a trapped-exit message rather than crashing.
- A test spawns a workflow process (trap-exit) with a linked activity child that exits abnormally and asserts the workflow process observes a trapped-exit message and stays alive.

### R4: Cancellation propagates to activity children

WHEN a workflow process is killed, THE SYSTEM SHALL rely on link propagation to terminate its linked activity children with no manual per-child teardown.

**Acceptance:**
- Because activity children are linked and do not trap exits, killing the workflow process propagates an exit along the links that terminates them.
- policy.rs documents that cancellation teardown is link propagation, not an explicit loop over children.
- A test spawns a workflow process with two linked activity children, kills the workflow process, and asserts both children are no longer alive.

## Boundaries

- SHALL NOT implement activity result/error propagation back to the workflow — that is AE-007 (this brief establishes the link/trap-exit shape it relies on).
- SHALL NOT implement workflow restart-on-crash replay — triggering replay is AD; this brief notifies the supervisor and re-registers, the reconstruction is AD's.
- SHALL NOT implement supervisor restart strategies beyond the structural tree and trap-exit policy needed for cancellation and crash observation.
