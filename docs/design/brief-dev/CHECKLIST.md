# Brief-Dev — Checklist

## Stage Contracts and Codecs

- [ ] **C1** — schemas/scout_report.json, dev_report.json, and review_report.json exist in the package and are byte-identical to their docs/design-system/schemas/ canon files.
- [ ] **C2** — scripts/check-schema-drift.sh fails loudly when any package stage-contract schema diverges from canon, and runs as part of the test gate.
- [ ] **C3** — src/aion_stacked_dev_io.gleam is generated by aion codegen from the package schemas, carries the do-not-edit header, and aion codegen --check passes clean in CI.
- [ ] **C4** — Brief document types (authored fields + optional scout/dev/review per requirement + optional execution block) are defined in Gleam with hand-written codecs that round-trip the seeded RM-011 brief files byte-faithfully.
- [ ] **C5** — Resolved-context types (ADR texts for design_anchor, C#/S# texts, roadmap provenance quote) are defined with codecs; workflow code never sees an unresolved reference.

## Prompt Projection

- [ ] **C6** — prompts.gleam exposes pure functions scout_prompt, dev_prompt, review_prompt, and resume_feedback taking only the enriched brief document and resolved context.
- [ ] **C7** — Each projection renders anchored ADRs as 'id: title — decision' lines and verbatim quotes as quoted words with speaker attribution, never paraphrased.
- [ ] **C8** — Unit tests pin each projection's output for a fixture brief and assert character budgets: scout ≤ 6000, dev ≤ 9000, review ≤ 12000.
- [ ] **C9** — The dev projection renders scout blocks inline per requirement; the review projection renders scout and dev blocks, the dev attestation, and the measured check results.

## brief_dev Pipeline

- [ ] **C10** — src/brief_dev.gleam exists as the inner-child entry module and src/onatopp_dev.gleam is deleted along with its workflow.toml entry and schemas (no parallel family).
- [ ] **C11** — The scout stage runs the scout activity (read-only norn, session '<branch>-scout', scout-report output schema) before any dev work.
- [ ] **C12** — The dev stage preserves the concurrent warm_build + dev fan-out via the tagged StartupTask envelope and the dev session id remains '<branch>'.
- [ ] **C13** — The verify-fix loop preserves onatopp_dev semantics: scoped_checks then dev_resume with diagnostics, bounded by required verify_fix_cap with required round_backoff_ms sleeps.
- [ ] **C14** — The review stage runs the dev_review activity (norn reviewer, session '<branch>-review', review-report output schema) after verify convergence, never sharing the dev session.
- [ ] **C15** — When the review report's fixes are non-empty, the pipeline re-runs scoped_checks; a harden pass that breaks verification fails the run with a typed error naming the review stage.
- [ ] **C16** — Any requirement left with alignment 'drifted' in the review report fails the run with a typed error listing the drifted R# ids and their issues.
- [ ] **C17** — brief_dev re-registers its status query at every stage transition with phases: scouting, developing, verifying, fixing, reviewing, hardening, converged.
- [ ] **C18** — BriefDevResult carries the scout, dev, and review reports plus verify_rounds and build_warm; BriefDevError covers scout failure, dev block, verify exhaustion, review drift, and stage failure.

## Enrichment

- [ ] **C19** — enrich.gleam merges a stage report into the brief document purely and append-only: replacing that stage's blocks wholesale, never merging field-by-field, never touching authored fields.
- [ ] **C20** — The enrich_brief activity deep-compares the authored subset of the on-disk worktree brief against the document it was handed and fails terminally on any divergence before writing.
- [ ] **C21** — Enrichment is written into the worktree brief file after scout, after dev convergence, and after review; the execution block (gate, attestation, review_verdict, branch, session, status, landed fields) is written before land.
- [ ] **C22** — Enrichment merge round-trips through validate.py: an enriched fixture brief validates against brief.schema.json.

## stacked_dev Integration

- [ ] **C23** — StackedDevInput carries the v2 brief document and resolved context in place of the four document strings; the input schema mirrors it with additionalProperties false and every cap/deadline still required.
- [ ] **C24** — stacked_dev spawns brief_dev (not onatopp_dev) and lifts its typed errors into the outer union; the gate child, review loop, signal contract, and land sequence are unchanged.
- [ ] **C25** — The landed merge contains the enriched brief file: implementation and execution record arrive on main in the same commit lineage.

## dispatch Workflow

- [ ] **C26** — src/dispatch.gleam exists as a third entry module with workflow.toml entry and input/output schemas; its input is an ordered wave of brief documents with resolved context plus required repo_root, base_ref, reviewers, caps, and deadlines.
- [ ] **C27** — A dispatcher activity assembles the wave outside workflow code: reads roadmap.json and the cluster documents, resolves all references, orders by depends_on, and refuses briefs whose depends_on have not landed or whose coverage check fails.
- [ ] **C28** — dispatch runs stacked_dev children serially in input order, records a per-brief outcome (landed with branch and commit, or failed with the typed error), and continues or halts on failure according to a required halt_on_failure input.
- [ ] **C29** — dispatch re-registers a status query per child transition answering current brief id, position, and per-brief outcomes so far.

## Worker

- [ ] **C30** — Worker handlers for scout, dev_review, and enrich_brief mirror locals invocation-for-invocation; SERVED_ACTIVITIES lists every activity name the family dispatches.
- [ ] **C31** — Norn scout and review invocations use --print, --session-id or --resume, --output-schema with the generated stage schemas, and --output-format json; output parsing accepts the bare shape and the real {"output": ...} envelope only.
- [ ] **C32** — wire_compat.rs pins every new payload (brief document, resolved context, stage reports, brief_dev and dispatch IO) byte-compatible both directions against the Gleam codecs.

## Tests and Template

- [ ] **C33** — The hermetic suite covers: scout failure, dev block, verify exhaustion, review drift unfixed (run fails), review fixes with passing re-verify, harden regression caught by re-verify, and the full happy path with an enriched brief asserted on the landed branch.
- [ ] **C34** — CLI shims emit captured real norn envelopes for scout and review stages; shim argv logs prove session id discipline ('<branch>-scout' / '<branch>' / '<branch>-review') and that diagnostics and review notes travel intact.
- [ ] **C35** — A hermetic dispatch test runs a two-brief wave with a depends_on edge and asserts ordering, per-brief outcomes, and halt_on_failure both ways.
- [ ] **C36** — The live e2e extends to the dispatch arc: deploy all archives, dispatch a one-brief wave, drive the review signal, assert the enriched brief exists in the merged result.
- [ ] **C37** — The dev-pipeline template mirrors every family change with {{name}} placeholders and its rendered-worker gates pass (fmt, clippy -D warnings, tests).
- [ ] **C38** — cargo fmt --check, cargo clippy --workspace --all-targets -- -D warnings, the full workspace suite, the package gleam test suite, and check-schema-drift.sh all pass clean at the cluster's landing commit.
