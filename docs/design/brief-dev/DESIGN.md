# Brief-Dev Workflow Family: the v2 pipeline as durable workflows

> **Cluster:** brief-dev

## Intention

Design-system v2 briefs become executable: an all-norn pipeline (scout → dev → verify-fix → adversarial review → harden) runs as a durable aion workflow, enriching the brief in place at every stage, composed under the live-proven stacked_dev outer arc (provision → dev → gate → human review → land), with a dispatcher above that selects briefed work from the roadmap ledger and fans it out. When this is done, 'do this roadmap item' means authoring documents and approving a DM — the workflow does the rest, every stage durable across crashes, every claim recorded next to what was actually measured. aion develops aion; the dogfood becomes the dev process.

## Problem

The queued work (RM-003, RM-011..014, and everything after) is currently executed by hand or through the rhai-scripted onatopp-dev-norn workflow, which dies with the process running it, trusts dev attestations as gates (its verify loop runs check+clippy but never tests or fmt), entangles review with hardening in a single graded-homework pass, and stores its execution record in workflow output instead of with the brief. Meanwhile the stacked_dev family is live-proven end to end but its inner child (onatopp_dev) is a scout-less, review-less dev loop fed unstructured document strings. Neither side speaks the v2 design system: stage outputs are ad-hoc shapes, briefs go in but never come back enriched, and nothing above a single brief exists — no dependency-ordered multi-brief dispatch.

## Solution

Evolve the stacked-dev family in place (no parallel family — ADR-002, ADR-008 proposed). The inner child onatopp_dev becomes brief_dev: it takes a v2 brief document plus pre-resolved context (ADR texts for the brief's design_anchor, C#/S# texts — resolution happens at dispatch time, in activities, never in deterministic workflow code), runs scout (read-only norn, deterministic session '<branch>-scout'), then dev (norn, session '<branch>', warm build concurrent via the existing tagged StartupTask fan-out), then the verify-fix loop (scoped_checks + dev_resume, cap and backoff as required inputs per ADR-001), then adversarial review+harden (norn reviewer, separate session '<branch>-review', prompt built per PROMPTING.md projection with the dev's attestation AND the measured check results), then re-verifies after any hardening. Each stage's structured report (scout-report/dev-report/review-report schemas — the codegen-subset stage contracts) is validated and appended into the brief document via the enrich_brief activity, which writes the brief file in the worktree append-only: authored fields are never touched, so the enriched record rides the same branch as the code and lands with the merge (ADR-009 proposed). The outer stacked_dev keeps its live-proven arc and contracts (request_review argv, land-from-repo-root, single review_verdict signal per ADR-006); its input reshapes to carry the v2 brief document and resolved context, and before land it calls enrich_brief once more to write the execution block (gate results, attestation, verdict, branch). A new top-level dispatch workflow takes a wave (list of brief documents + resolved context + concurrency limit, all assembled by dispatcher activities that read roadmap.json/decisions.json and refuse briefs whose depends_on have not landed), spawns stacked_dev children in dependency order — serially in the first delivery; bounded fan-out follows once parent-close (RM-001/ADR-004) lands — and returns a per-brief outcome report. Worker: the existing stacked-dev worker crate gains handlers for the new/changed activities, mirroring locals invocation-for-invocation, with wire_compat pinning every payload byte-compatible both directions. Stage-contract codecs are generated by aion codegen from schemas copied into the package's schemas/ dir, with a drift gate asserting byte-equality against the design-system canon; workflow IO and error unions stay hand-written (codegen v1 rejects unions) under wire-compat pins.

## Principles

- **P1** — Attestation is never the gate: agents attest, activities measure, both are recorded, divergence is review signal.
- **P2** — Projection, not forwarding: every stage prompt is a budgeted pure function of the enriched brief (scout ~5K chars, dev ~7K, review ~9K), unit-tested like a codec.
- **P3** — Expected negative outcomes are data, not errors: failed checks, failed gates, forfeited warm builds, and review-found drift are recorded results; only can't-execute conditions fail activities, and those fail terminally.
- **P4** — Test fixtures are captured real envelopes, never invented shapes: contract drift breaks a test, not a live run.
- **P5** — Deterministic workflow code never touches the filesystem or builds prompts from unresolved references: reading ledgers, resolving references, and writing enrichment are activities.
- **P6** — Verbatim intent rides the prompt: when the brief's roadmap item or anchored ADRs carry quotes, the agent sees the quoted words with their speaker, never a paraphrase.

## Decisions

- ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
- ADR-002 — No backwards compatibility during the build — Replace, don't add alongside. No compat shims, no zombie code, no #[deprecated] markers. Breaking changes are made cleanly and consumers move forward. Rejected: incremental deprecation cycles — they double the surface under test for an audience of zero.
- ADR-003 — No default timeouts anywhere in the engine — The engine imposes no activity time bound of its own. Activity waits are unbounded, terminated only by completion, worker loss, server shutdown, or a workflow-level timeout the author explicitly chose. Rejected: a bigger default — agentic activities legitimately run for over an hour, and any number we picked would be ADR-001 violated.
- ADR-004 — Parent-close policy: required per spawn, RequestCancel | Terminate | Abandon — Temporal-style per-spawn parent-close policy — RequestCancel (graceful cascade), Terminate (immediate kill), Abandon (child hands off and keeps running past the parent's terminal; today's behaviour made explicit) — as a REQUIRED argument on child.spawn / spawn_and_wait. Rejected: defaulting to any of the three (ADR-001).
- ADR-005 — Failed runs are terminal; recovery resumes Running runs only — A failed run is a terminal, immutable record. Retry means a fresh `aion start` with a new run identity; recovery after restart resumes Running runs only. Rejected: in-place retry of failed runs — it would rewrite history that event-sourcing exists to preserve, and 'which attempt was this event from' becomes unanswerable.
- ADR-006 — Multi-reviewer verdicts: votes aggregate in Meridian, one signal decides — The workflow keeps a single review_verdict signal as THE decision. Reviewers vote via `meridian review complete --verdict`; the Meridian coordinator applies the quorum policy and fires the one aion signal. Rejected: per-reviewer signals into the workflow — quorum policy is a Meridian concern, and every workflow re-implementing vote-counting is the wrong layer. Integration seam: Meridian needs the branch→workflow-id mapping at review-request time.
- ADR-007 — Design system v2: JSON ledgers above clusters, enrichment in place — Two project ledgers (roadmap.json, decisions.json) above the clusters; stage contracts as first-class schemas inside the aion codegen subset; the brief is one living document — the pipeline appends scout/dev/review per requirement and an execution block per brief, in place, never touching authored fields. Rejected: a sibling runs/ ledger — the brief as a single spec-plus-record document was the original intent, and aion's event history already provides the append-only audit trail.
- ADR-008 — brief_dev replaces onatopp_dev inside the stacked-dev family — Evolve in place: onatopp_dev.gleam is deleted and brief_dev.gleam takes its slot as stacked_dev's inner child; the outer arc keeps its live-proven contracts. Rejected: a parallel brief-dev family — two families serving one purpose is the zombie-code pattern ADR-002 prohibits, and the outer arc's provision/gate/review/land contracts took a full dogfood night to prove against real CLIs; duplicating them duplicates that risk.
- ADR-009 — Enrichment rides the worktree branch and lands with the merge — The enrich_brief activity writes the brief file inside the run's worktree; the enriched brief is committed by land alongside the implementation and arrives on main in the same merge. Rejected: main-tree writes from a running workflow (races concurrent runs and pollutes main with in-flight state) and a separate execution store (re-creates the spec/record split ADR-007 closed; aion's event history already serves as the append-only store).

## Goals

- The family executes one of aion's own briefs end to end with real norn against the aion repo: dispatched from roadmap.json, landed on main with the enriched brief riding the merge.
- Every stage payload validates against the design-system stage-contract schemas, and the stage-contract codecs are generated by aion codegen from package schemas that are byte-identical to the design-system canon (drift gate in CI).
- An enriched brief on main carries: per-R# scout/dev/review blocks, the execution block with measured gate results AND the dev attestation, and untouched authored fields (verified by diffing authored fields pre/post run).
- The hermetic in-process suite covers every stage outcome (scout failure, dev block, verify exhaustion, review drift fixed and unfixed, harden regression caught by re-verify, gate fail, review timeout/reject/changes, land) with exclusive-PATH CLI shims emitting captured real envelopes.
- The dispatch workflow executes a multi-brief wave in dependency order, refuses briefs with unlanded depends_on, and returns a per-brief outcome report; a kill -9 of the server at any point during a wave resumes without re-executing completed activities.
- The dev-pipeline template ships the new family so aion new scaffolds the v2 pipeline.

## Non-Goals

- Parallel fan-out of briefs within a wave in the first delivery. — Lands serially first; bounded concurrency follows once parent-close (RM-001) gives the dispatcher cancellation semantics over in-flight children.
- Multi-reviewer quorum aggregation. — ADR-006 places quorum in the Meridian coordinator; this family keeps the single review_verdict signal it already has.
- Generating codecs for workflow IO and error unions. — codegen v1 rejects unions and $ref; hand-written codecs with wire-compat pins remain the proven pattern for those shapes.
- Replacing the rhai onatopp-dev-norn workflow in Meridian's repo. — Meridian migrates on its own schedule after the re-pin (RM-015); this cluster only retires aion's inner onatopp_dev module.
- Cross-repo dispatch (running this family against repos other than the one the worker sits in). — Worker placement/affinity is RM-006; the first consumer is aion itself.

## Constraints

- **CN1** — Deterministic workflow code performs no filesystem reads, no ledger parsing, and no reference resolution: dispatcher activities resolve roadmap/decisions/checklist/stories references and assemble the brief context before any workflow logic sees it (P5).
- **CN2** — Every cap, backoff, deadline, and concurrency limit is a required input field, schema-enforced with additionalProperties: false; the engine and the workflows impose no bound of their own (ADR-001, ADR-003).
- **CN3** — enrich_brief is append-only: it refuses to alter any authored field (deep-compares the authored subset before writing) and fails terminally on divergence rather than silently overwriting.
- **CN4** — Stage sessions are deterministic and disjoint: '<branch>' for dev (existing), '<branch>-scout' for scout, '<branch>-review' for review. The reviewer NEVER shares the dev session — fresh eyes are the point of the stage (REVIEW.md).
- **CN5** — Activity failure taxonomy: can't-execute is terminal failure; expected negative outcomes are successful results carrying data (P3). No retries of broken environments.
- **CN6** — The family deploys as a unit (shared content hash, child type = entry module name); all archives rebuilt and redeployed together on any change.
- **CN7** — Package stage-contract schemas are byte-identical to docs/design-system/schemas/ canon, enforced by a drift gate that runs with the test suite.
- **CN8** — The outer arc's live-proven CLI contracts are immutable in this cluster: request_review argv (branch positional first, --as Meridian), land = git commit in worktree then yg merge from repo_root, single review_verdict signal (ADR-006).
- **CN9** — Hermetic tests run under gleeunit's sequential runner (exclusive PATH is VM-global); e2e suites never run concurrently with gleam builds in the same checkout.
