# BD-004: Append-only enrichment

> **Cluster:** brief-dev
> **Depends on:** BD-001
> **Design anchor:**
> - ADR-007 — Design system v2: JSON ledgers above clusters, enrichment in place — Two project ledgers (roadmap.json, decisions.json) above the clusters; stage contracts as first-class schemas inside the aion codegen subset; the brief is one living document — the pipeline appends scout/dev/review per requirement and an execution block per brief, in place, never touching authored fields. Rejected: a sibling runs/ ledger — the brief as a single spec-plus-record document was the original intent, and aion's event history already provides the append-only audit trail.
> - ADR-009 — Enrichment rides the worktree branch and lands with the merge — The enrich_brief activity writes the brief file inside the run's worktree; the enriched brief is committed by land alongside the implementation and arrives on main in the same merge. Rejected: main-tree writes from a running workflow (races concurrent runs and pollutes main with in-flight state) and a separate execution store (re-creates the spec/record split ADR-007 closed; aion's event history already serves as the append-only store).
> **Checklist:**
> - C19 — enrich.gleam merges a stage report into the brief document purely and append-only: replacing that stage's blocks wholesale, never merging field-by-field, never touching authored fields.
> - C20 — The enrich_brief activity deep-compares the authored subset of the on-disk worktree brief against the document it was handed and fails terminally on any divergence before writing.
> - C21 — Enrichment is written into the worktree brief file after scout, after dev convergence, and after review; the execution block (gate, attestation, review_verdict, branch, session, status, landed fields) is written before land.
> - C22 — Enrichment merge round-trips through validate.py: an enriched fixture brief validates against brief.schema.json.
> **Stories:**
> - S12 (Future Maintainer, Reading landed work months later) — As a future maintainer, I want the landed brief to carry what was asked, what the scout found, what the dev did and why, and what review proved so that provenance is one file away from the code.
> - S13 (Future Maintainer, Reading landed work months later) — As a future maintainer, I want gate results and agent attestations recorded separately in the execution block so that I can tell measured truth from believed claims.

## Purpose

Deliver the enrichment mechanism that makes the brief one living document (ADR-007): a pure append-only merge of stage reports and the execution block into the brief document, and the enrich_brief activity that writes the merged document into the run's worktree brief file (ADR-009) after deep-comparing the authored subset (CN3). Every downstream stage prompt and the landed provenance record depend on this merge never touching an authored field, so the guarantee is delivered as a pure function with its own tests, separate from the activity that performs the file IO.

## Task

Build the enrichment layer in two strata. First, src/stacked_dev/enrich.gleam: pure functions that merge one scout/dev/review report or one execution block into a brief document, replacing the targeted blocks wholesale and copying every authored field through verbatim. Second, the enrich_brief activity (definition in activities.gleam, payload types in types.gleam, codecs in codecs_flow.gleam, local implementation in locals.gleam): it locates the brief file inside the run's worktree at docs/design/<cluster>/briefs/<id>.json (cluster and id read from the handed document — the design-system layout is a format constraint, like the briefs/ directory validate.py keys on), deep-compares the on-disk authored subset against the handed document, fails terminally on any divergence before writing (CN3), and otherwise writes the merged document back and returns it. The brief document, stage report, and execution block types and their codecs come from BD-001 — this brief only consumes them. SPLIT NOTE on C21: this brief delivers the merge functions and the activity for all four write points (after scout, after dev convergence, after review, and the execution block before land); the call sites that invoke the activity live in brief_dev.gleam (BD-003) and in the outer arc of stacked_dev.gleam (BD-005), and the workflow.toml activities declarations ride with those call-site briefs. The acceptance here proves the mechanism supports each write point; the wiring proof lands with BD-003/BD-005 and the hermetic scenarios with BD-007. Out of scope: prompts, the brief_dev pipeline, worker handlers and wire pins for enrich payloads (BD-005), and all hermetic pipeline scenarios (BD-007).

## Requirements

### R1: Pure append-only merge functions in enrich.gleam

Create src/stacked_dev/enrich.gleam exposing pure functions merge_scout(document, report), merge_dev(document, report), merge_review(document, report), and merge_execution(document, block), each returning Result(BriefDocument, EnrichError) where EnrichError is defined in this module. WHEN a stage report is merged, THE SYSTEM SHALL, for each enrichment entry in the report, set the matching requirement's stage block (scout/dev/review) to exactly that entry's fields, replacing any existing block for that stage wholesale, and SHALL NOT merge field-by-field with a previously present block, SHALL NOT modify any other stage's blocks, and SHALL NOT modify any authored field (id, cluster, title, depends_on, blocked_by, checklist, stories, design_anchor, purpose, task, boundaries, verification, or any requirement's id/title/spec/acceptance/files/checklist/stories). WHEN an execution block is merged, THE SYSTEM SHALL set the document's execution field to exactly the handed block, replacing any existing one wholesale. IF a report enrichment entry carries an R# id that matches no requirement in the document, THEN THE SYSTEM SHALL return an EnrichError naming that id and SHALL NOT return a partially merged document. The module SHALL NOT perform any IO, FFI call, or CLI invocation — it is a pure data transformation.

**Acceptance:**
- Merging a scout report with enrichment entries for R1 and R2 onto a fixture brief with requirements R1 and R2 produces a document where requirements[0].scout and requirements[1].scout equal the corresponding entries' files/context/approach/notes, and the encoded authored subset of the result is byte-identical to the encoded authored subset of the input document.
- Merging a second scout report whose R1 entry has notes "" onto a document whose R1.scout.notes is "watch the codec ordering" yields R1.scout.notes == "" — the block was replaced wholesale, not field-merged.
- Merging a dev report onto a document that already carries scout blocks leaves every requirement's scout block byte-identical and sets only the dev blocks.
- merge_execution on a document with no execution field returns a document whose execution field equals the handed block; calling it again with a different block replaces it entirely.
- Merging a scout report containing an entry with id "R9" onto a two-requirement (R1, R2) document returns Error and the error value carries the string "R9".
- enrich.gleam contains no @external declarations and imports neither stacked_dev/cli nor any file-IO module.

### R5: The enrich_brief local implementation: deep-compare, merge, write back

Implement locals.enrich_brief(input: EnrichInput) -> Result(BriefDocument, error.ActivityError) in src/stacked_dev/locals.gleam. WHEN invoked, THE SYSTEM SHALL derive the brief path as input.workspace.path <> "/docs/design/" <> document.cluster <> "/briefs/" <> document.id <> ".json", read and decode that file with BD-001's brief document codec, deep-compare the on-disk document's authored subset (every authored field, including each requirement's authored fields but excluding scout/dev/review blocks and the execution block) against the handed document's authored subset, and on equality apply the matching enrich.gleam merge function, write the merged document's encoding back to the same path, and return the merged document. IF any authored field diverges between disk and the handed document, THEN THE SYSTEM SHALL fail with error.terminal naming the brief path and the first divergent field, and SHALL NOT write the file (CN3 — divergence is never silently overwritten). IF the file is missing, unreadable, or fails to decode, THEN THE SYSTEM SHALL fail with error.terminal carrying the path and reason — a broken worktree is a can't-execute condition (CN5), never a retry or a skip. WHEN the enrichment is ExecutionEnrichment, THE SYSTEM SHALL write the handed block exactly as given — the gate results and the attestation arrive as separate fields and are recorded as separate fields (P1). THE SYSTEM SHALL NOT write any path outside the workspace's worktree, and SHALL NOT touch the main repository tree (ADR-009).

**Acceptance:**
- With workspace.path "/w" and a handed document with cluster "brief-dev" and id "BD-004", the activity reads and writes exactly "/w/docs/design/brief-dev/briefs/BD-004.json".
- When the on-disk brief equals the handed document's authored subset, after the call the file decodes as the merged document, the activity's return value equals that document, and the file's authored subset encodes byte-identically to its pre-call encoding.
- When the on-disk brief's task field differs from the handed document's, the activity returns a Terminal failure whose message contains "task", and the file's bytes are unchanged.
- When the brief file is absent, the activity returns a Terminal failure whose message contains the derived path.
- An ExecutionEnrichment write leaves the on-disk execution.gate and execution.attestation fields equal to the handed block's values, with both objects present and distinct.
