# BD-005: stacked_dev integration and worker parity

> **Cluster:** brief-dev
> **Depends on:** BD-003, BD-004
> **Design anchor:**
> - ADR-006 — Multi-reviewer verdicts: votes aggregate in Meridian, one signal decides — The workflow keeps a single review_verdict signal as THE decision. Reviewers vote via `meridian review complete --verdict`; the Meridian coordinator applies the quorum policy and fires the one aion signal. Rejected: per-reviewer signals into the workflow — quorum policy is a Meridian concern, and every workflow re-implementing vote-counting is the wrong layer. Integration seam: Meridian needs the branch→workflow-id mapping at review-request time.
> - ADR-008 — brief_dev replaces onatopp_dev inside the stacked-dev family — Evolve in place: onatopp_dev.gleam is deleted and brief_dev.gleam takes its slot as stacked_dev's inner child; the outer arc keeps its live-proven contracts. Rejected: a parallel brief-dev family — two families serving one purpose is the zombie-code pattern ADR-002 prohibits, and the outer arc's provision/gate/review/land contracts took a full dogfood night to prove against real CLIs; duplicating them duplicates that risk.
> **Checklist:**
> - C23 — StackedDevInput carries the v2 brief document and resolved context in place of the four document strings; the input schema mirrors it with additionalProperties false and every cap/deadline still required.
> - C24 — stacked_dev spawns brief_dev (not onatopp_dev) and lifts its typed errors into the outer union; the gate child, review loop, signal contract, and land sequence are unchanged.
> - C25 — The landed merge contains the enriched brief file: implementation and execution record arrive on main in the same commit lineage.
> - C30 — Worker handlers for scout, dev_review, and enrich_brief mirror locals invocation-for-invocation; SERVED_ACTIVITIES lists every activity name the family dispatches.
> - C31 — Norn scout and review invocations use --print, --session-id or --resume, --output-schema with the generated stage schemas, and --output-format json; output parsing accepts the bare shape and the real {"output": ...} envelope only.
> - C32 — wire_compat.rs pins every payload that exists at its landing byte-compatible both directions against the Gleam codecs; the dispatch IO pins complete with the dispatch brief.
> **Stories:**
> - S2 (Tom, Approving and steering work others dispatch) — As Tom, I want each brief's review gate to DM its assigned reviewers — usually AI leads, me for design-level calls — with the workflow parked until the verdict signal so that the decision point is explicit without me being the bottleneck.
> - S18 (AI Lead, Reviewing code at the review gate) — As an AI lead reviewing code at the review gate, I want the review request and the enriched brief on the branch to carry everything a rigorous review needs — per-criterion acceptance evidence, measured gate results, attestation divergence — so that I can let nothing through without re-deriving context.
> - S19 (AI Lead, Reviewing code at the review gate) — As an AI lead, I want the review_verdict signal to be sender-agnostic so that my verdict, cast through the Meridian coordinator, decides the run exactly as a human's would.

## Purpose

Integrate the v2 pipeline into the live-proven outer arc and restore deployed parity: StackedDevInput reshapes to carry the v2 brief document and resolved context (ADR-008 — evolve in place, no parallel family), stacked_dev spawns brief_dev as its inner child and writes the execution block into the worktree brief before land so the enriched record and the implementation arrive on main in the same commit lineage, and the standalone worker gains handlers for scout, dev_review, and enrich_brief mirroring the locals invocation-for-invocation with every new payload pinned byte-compatible. The outer arc's live-proven CLI contracts are immutable here (CN8): request_review argv with the branch positional first and --as Meridian, land as git commit in the worktree then yg merge from repo_root, and the single review_verdict signal (ADR-006 — quorum is Meridian's, one signal decides).

## Task

Reshape and rewire, replacing rather than adding alongside (ADR-002 via ADR-008). Gleam side: StackedDevInput drops the brief/design/checklist/stories document strings in favour of brief_document (BD-001's BriefDocument) and resolved_context (BD-001's ResolvedContext); schemas/input.json mirrors the reshape with additionalProperties false and every cap/deadline still required; codecs_workflows.gleam re-encodes the input; stacked_dev.gleam spawns brief_dev (BD-003's child) instead of onatopp_dev, lifts each BriefDevError variant into a typed StackedDevError variant, and — between the Approve verdict and the land activity — runs BD-004's enrich_brief with the ExecutionEnrichment so the landed merge carries the brief's own provenance (C25, ADR-009). The phase flow (provision → child → gate child → review loop → land), the gate child, the review loop, the signal contract, and the land sequence are unchanged (C24, CN8). Worker side: handlers for scout, dev_review, and enrich_brief mirror the locals BD-003 and BD-004 landed, invocation for invocation; the dev/dev_resume handlers track BD-003's reshaped DevInput (the projected prompt replaces the four document fields — derive the serde shape from the landed Gleam codec, never from memory); SERVED_ACTIVITIES grows to the eleven names the family now dispatches; wire_compat.rs pins every new payload byte-compatible both directions; handlers_shims.rs proves the new handlers against fake CLIs. Also mechanically adapt the existing hermetic suite (base_input, child registration) so the package test gate stays green at this brief's landing commit — new scenarios remain BD-007's. SPLIT NOTE on C32: this brief pins the brief document, resolved context, stage reports, enrich payloads, and brief_dev/stacked_dev IO; the dispatch IO payloads do not exist until BD-006, which extends wire_compat.rs with their pins when it creates them. NOTE on coverage tooling: examples/stacked-dev/schemas/input.json and examples/stacked-dev/worker/src/main.rs are required by this work but are absent from design.json's structure array — per CLAUDE.md, when a structure annotation disagrees with a brief, the brief is authoritative. Out of scope: brief_dev.gleam internals and the scout/dev_review activity definitions (BD-003), prompts.gleam (BD-002), enrich.gleam and the enrich_brief local (BD-004), the dispatch workflow (BD-006), and all new hermetic scenarios, e2e, and template work (BD-007).
