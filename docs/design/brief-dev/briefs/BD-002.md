# BD-002: Stage prompt projection

> **Cluster:** brief-dev
> **Depends on:** BD-001
> **Design anchor:**
> - ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
> - ADR-007 — Design system v2: JSON ledgers above clusters, enrichment in place — Two project ledgers (roadmap.json, decisions.json) above the clusters; stage contracts as first-class schemas inside the aion codegen subset; the brief is one living document — the pipeline appends scout/dev/review per requirement and an execution block per brief, in place, never touching authored fields. Rejected: a sibling runs/ ledger — the brief as a single spec-plus-record document was the original intent, and aion's event history already provides the append-only audit trail.
> **Checklist:**
> - C6 — prompts.gleam exposes pure functions scout_prompt, dev_prompt, review_prompt, and resume_feedback over the brief document, resolved context, and explicit stage inputs (prior stage reports, the attestation, measured diagnostics) — no filesystem access, no ambient state.
> - C7 — Each projection renders anchored ADRs as 'id: title — decision' lines and verbatim quotes as quoted words with speaker attribution, never paraphrased.
> - C8 — Unit tests pin each projection's output for a fixture brief and assert character budgets: scout ≤ 6000, dev ≤ 9000, review ≤ 12000.
> - C9 — The dev projection renders scout blocks inline per requirement; the review projection renders scout and dev blocks, the dev attestation, and the measured check results.
> **Stories:**
> - S8 (Norn Agent, Executing a pipeline stage) — As a norn agent, I want my prompt to carry the resolved requirement texts, anchored decisions, and the humans' verbatim words so that I act on what was actually asked without re-deriving context.

## Purpose

Delivers the projection layer (P2): every stage prompt becomes a budgeted pure function of the brief document and its pre-resolved context, unit-tested like a codec. This is how resolved requirement texts, anchored decisions, and the humans' verbatim words reach a norn agent that has never seen the ledgers (S8, P6), and how prompt size stays bounded by filtering instead of forwarding — the budgets are review criteria for the functions themselves. BD-003's pipeline calls these functions from deterministic workflow code; nothing here performs IO or touches an unresolved reference (P5).

## Task

Create src/stacked_dev/prompts.gleam: pure projection functions over the BD-001 types (BriefDocument, ResolvedContext from stacked_dev/types; ScoutReport/DevReport from the generated aion_stacked_dev_io; CheckResult from stacked_dev/types) plus their unit tests. Signatures — a declared deviation from C6's literal 'taking only the enriched brief document and resolved context': scout_prompt(document, context); dev_prompt(document, context, scout: ScoutReport); review_prompt(document, context, scout: ScoutReport, dev: DevReport, check: CheckResult); resume_feedback(document, diagnostics: String). Rationale: the in-document merge of stage reports belongs to BD-004's enrich.gleam (this brief must not reach into it), the measured check results C9 demands are not part of the document at all, and fix-round diagnostics arrive per round — so the stage reports and diagnostics ride as explicit value arguments, which keeps the functions pure and total while honouring PROMPTING.md's 'projection of the enriched brief' (the scout/dev blocks rendered are exactly the blocks BD-004 will append). Rendering discipline per PROMPTING.md: static per-stage instructions (~500 chars, imperative — scout: read-only, 2-5 files per R# with line ranges, conventions, concrete approach, gotchas; dev: implement every R#, declare deviations, the workflow runs the real gate afterwards so do not burn the session on the full suite; review: verify the actual diff, never the report, one verdict per acceptance criterion with evidence, fix everything found); decision context as 'id: title — decision' lines from the context's resolved ADRs in design_anchor order; verbatim quotes as quoted words with speaker attribution (adr.decided_by, provenance.requested_by), never paraphrased; design context (intention, constraint lines, the design_path as a reference — never design prose); requirements with their resolved C#/S# texts inline per R#; the brief's boundaries in EVERY stage prompt, no stage exempt. Budgets are the checklist's numbers (scout 6000 / dev 9000 / review 12000 chars — PROMPTING.md's ~5K/7K/9K are the v1 reference points; the checklist governs), pinned against the seeded fixture docs/design/brief-dev/briefs/BD-001.json. Because no captured real norn envelope exists yet for the v2 stage contracts, the fixture's ScoutReport/DevReport/CheckResult values are constructed schema-valid values built in a shared test-support module (P4's captured-real-envelope rule binds BD-007's CLI shims, not these pure-function fixtures; noted here so the reviewer does not flag it as an undeclared invention). The test files and test/support/prompt_fixture.gleam are not in the design.json structure array (which pins only prompts.gleam for this work); per CLAUDE.md the brief is authoritative. Out of scope: any workflow or activity change (BD-003), enrichment (BD-004), worker (BD-005), dispatch (BD-006), hermetic pipeline suite (BD-007).

## Requirements

### R1: Projection module with shared rendering and per-stage instructions

Create src/stacked_dev/prompts.gleam with public rendering helpers the four projections share, plus the static per-stage instruction constants. WHEN rendering decision context, THE SYSTEM SHALL emit one line per resolved ADR in the order given, formatted exactly '<id>: <title> — <decision>', and SHALL NOT include ADR context or consequences prose (budget discipline). WHEN a resolved ADR's quote is non-empty, THE SYSTEM SHALL emit a following attribution line formatted exactly '<decided_by>: "<quote>"' with the quote text verbatim; WHEN the quote is empty, THE SYSTEM SHALL NOT emit an attribution line. The same quoted-words-with-speaker rule applies to the roadmap provenance quote using provenance.requested_by. THE SYSTEM SHALL NOT paraphrase, truncate, or re-punctuate any quote (P6). WHEN rendering a requirement, THE SYSTEM SHALL inline each of the requirement's C#/S# references with its resolved text from the context as '<id> — <text>' and SHALL NOT render the full checklist or stories documents. Every projection output SHALL include the brief's boundaries verbatim and the context's design_path as a file reference; THE SYSTEM SHALL NOT inline design.json prose beyond the context's intention string and constraint lines. The module SHALL NOT import aion/workflow, aion/activity, aion/query, aion/signal, stacked_dev/cli, stacked_dev/locals, or stacked_dev/activities, and SHALL NOT perform any IO — pure functions of their arguments only (P2, P5).

**Acceptance:**
- For ResolvedAdr(id: "ADR-003", title: "No default timeouts anywhere in the engine", decision: <the ledger's decision text>, quote: "we shouldn't have a default timeout, the agent steps can take well over an hour", decided_by: "Tom"), the rendered decision block's first line starts with 'ADR-003: No default timeouts anywhere in the engine — The engine imposes no activity time bound of its own.'
- The same input renders the attribution line 'Tom: "we shouldn't have a default timeout, the agent steps can take well over an hour"' immediately after the decision line.
- A ResolvedAdr with quote == "" renders exactly one line (no attribution line, no empty quotes).
- Each of the three instruction constants is non-empty and at most 900 characters; the scout instructions contain 'read-only', the dev instructions state the workflow runs the real gate afterwards, and the review instructions direct verifying the actual diff rather than the dev report.

### R3: Dev projection inlining scout blocks per requirement

WHEN dev_prompt(document, context, scout) is called, THE SYSTEM SHALL render everything the scout projection renders for orientation (instructions are the dev set, decision lines, quotes, design context, boundaries, design_path) and additionally, per requirement, the scout enrichment whose id matches that R# — files, context findings, approach, and notes rendered inline under the requirement (C9). IF the scout report has no enrichment for an R#, THEN THE SYSTEM SHALL render an explicit 'scout: none recorded' marker for that requirement and SHALL NOT omit the requirement. The dev instructions SHALL direct the agent to declare any deviation from the scouted approach in the dev report's deviation field and SHALL state that the workflow measures the real gate afterwards; THE SYSTEM SHALL NOT instruct the dev to run the full workspace suite (PROMPTING.md dev stage).

### R4: Review projection with attestation and measured results

WHEN review_prompt(document, context, scout, dev, check) is called, THE SYSTEM SHALL render, per requirement: the requirement (id, title, spec, every acceptance criterion verbatim, files, resolved C#/S# texts), its scout block, and its dev block (status, files_changed, how, deviation, per-C#/S# claims) inline; plus the dev report's attestation under a label naming it the dev's claim, the measured CheckResult (verdict, checked_scope, and diagnostics when failed) under a label naming it measured, and the brief's verification steps. THE SYSTEM SHALL render attestation and measured results as two distinct labelled sections so the reviewer can read their divergence as signal (S10, P1), and SHALL NOT present the attestation as a gate outcome. The review instructions SHALL direct the reviewer to verify the actual diff (never the dev report), return one verdict per acceptance criterion with file:line or test-name evidence, and fix everything found (the harden pass) — there are no minor issues. THE SYSTEM SHALL NOT instruct the reviewer to resume or reuse the dev session (CN4 lives in BD-003; the prompt must not undermine it).

**Acceptance:**
- string.length(review_prompt(fixture_document, fixture_context, fixture_scout, fixture_dev, fixture_check)) <= 12000.
- The review prompt contains every acceptance criterion string of the fixture document's R1 verbatim.
- The review prompt contains both a section labelled as the dev's attestation and a separately labelled measured-checks section, and with a fixture attestation of tests_pass: True alongside a failing CheckResult both renderings appear (divergence visible).
- The review prompt contains the fixture dev report's deviation sentinel string (e.g. 'FIXTURE-DEV-DEVIATION-R1') inside the R1 section, and every string of the fixture document's verification array.

### R5: Resume feedback projection

THE SYSTEM SHALL NOT re-render the requirements or scout blocks (the resumed session already holds them — resuming the same session is the point, S9 context economy) and SHALL NOT truncate or reformat the diagnostics text.
