# BD-003: The brief_dev pipeline workflow

> **Cluster:** brief-dev
> **Depends on:** BD-001, BD-002
> **Design anchor:**
> - ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
> - ADR-003 — No default timeouts anywhere in the engine — The engine imposes no activity time bound of its own. Activity waits are unbounded, terminated only by completion, worker loss, server shutdown, or a workflow-level timeout the author explicitly chose. Rejected: a bigger default — agentic activities legitimately run for over an hour, and any number we picked would be ADR-001 violated.
> - ADR-008 — brief_dev replaces onatopp_dev inside the stacked-dev family — Evolve in place: onatopp_dev.gleam is deleted and brief_dev.gleam takes its slot as stacked_dev's inner child; the outer arc keeps its live-proven contracts. Rejected: a parallel brief-dev family — two families serving one purpose is the zombie-code pattern ADR-002 prohibits, and the outer arc's provision/gate/review/land contracts took a full dogfood night to prove against real CLIs; duplicating them duplicates that risk.
> **Checklist:**
> - C10 — src/brief_dev.gleam exists as the inner-child entry module and src/onatopp_dev.gleam is deleted along with its workflow.toml entry and schemas (no parallel family).
> - C11 — The scout stage runs the scout activity (read-only norn, session '<branch>-scout', scout-report output schema) before any dev work.
> - C12 — The dev stage preserves the concurrent warm_build + dev fan-out via the tagged StartupTask envelope and the dev session id remains '<branch>'.
> - C13 — The verify-fix loop preserves onatopp_dev semantics: scoped_checks then dev_resume with diagnostics, bounded by required verify_fix_cap with required round_backoff_ms sleeps.
> - C14 — The review stage runs the dev_review activity (norn reviewer, session '<branch>-review', review-report output schema) after verify convergence, never sharing the dev session.
> - C15 — When the review report's fixes are non-empty, the pipeline re-runs scoped_checks; a harden pass that breaks verification fails the run with a typed error naming the review stage.
> - C16 — Any requirement left with alignment 'drifted' in the review report fails the run with a typed error listing the drifted R# ids and their issues.
> - C17 — brief_dev re-registers its status query at every stage transition with phases: scouting, developing, verifying, fixing, reviewing, hardening, converged.
> - C18 — BriefDevResult carries the scout, dev, and review reports plus verify_rounds and build_warm; BriefDevError covers scout failure, dev block, verify exhaustion, review drift, harden regression, and stage failure.
> **Stories:**
> - S6 (Claude, Authoring briefs and operating the pipeline) — As Claude, I want the live status query to name the current stage and round so that I can watch a run without parsing event histories.
> - S9 (Norn Agent, Executing a pipeline stage) — As a norn agent resuming after a crash or a fix round, I want my same session resumed by deterministic id so that I keep my context instead of rediscovering the codebase.
> - S10 (Reviewer Agent, Adversarial verification of a dev round) — As a reviewer agent, I want the dev's attestation alongside the measured check results so that I can treat their divergence as a signal about where to dig.
> - S11 (Reviewer Agent, Adversarial verification of a dev round) — As a reviewer agent, I want a fresh session separate from the dev's so that I verify the diff with my own eyes instead of inheriting the dev's framing.

## Purpose

Replaces the scout-less, review-less onatopp_dev inner child with brief_dev: the v2 all-norn pipeline (scout → concurrent warm-build+dev → verify-fix loop → adversarial review → harden re-verify) as a durable workflow over the BD-001 types and BD-002 projections (ADR-008 — evolve in place, no parallel family). Every stage runs in its own deterministic norn session ('<branch>-scout' / '<branch>' / '<branch>-review', CN4), every negative outcome is either recorded data or a typed error per the C18 taxonomy, and every cap and backoff is a required input the engine never bounds (ADR-001, ADR-003, CN2). This is the workflow that makes attestation-never-the-gate executable: the reviewer sees the dev's claims next to measured check results and verifies the diff itself (P1, S10).

## Task

Replace src/onatopp_dev.gleam with src/brief_dev.gleam and rebuild the inner-child contract around the brief document. Stage flow: scout activity (read-only norn, session '<branch>-scout', scout-report output schema) → concurrent warm_build + dev preserved via the existing tagged StartupTask/StartupResult envelope through workflow.all (dev session stays '<branch>') → the onatopp verify-fix loop preserved verbatim in shape (scoped_checks, then on CheckFail a durable round_backoff_ms sleep and dev_resume with projected feedback, bounded by required verify_fix_cap) → dev_review activity (norn reviewer, session '<branch>-review', review-report output schema, NEVER the dev session) → if any review enrichment is left 'drifted', fail typed listing the drifted R# ids and their issues → if review fixes are non-empty, re-run scoped_checks and fail typed on regression → converge. Activity contract reshapes that follow from the design ('all reused as-is except dev/dev_resume prompts now come from prompts.gleam projections' and stage outputs are the stage-contract reports): DevInput becomes (workspace, prompt) — the projected prompt is built in workflow code from BD-002's pure functions, so locals.dev drops its internal prompt assembly; the dev and dev_resume activities now return the dev-report shape (StartupResult's Developed carries a DevReport; fix rounds return a FULL replacement report per PROMPTING.md), with files_touched for scoped_checks derived in workflow code as the deduplicated files_changed paths of the latest report's enrichments, and the dev session id remaining deterministic from the branch. The DevResult type and its codec are KEPT — the outer arc's request_review/land payloads still use them until BD-005 reshapes the outer arc. Deletions: src/onatopp_dev.gleam, its workflow.toml entry, schemas/onatopp_input.json + onatopp_output.json, the stale onatopp-dev.aion archive, the Onatopp* types and codecs. New manifest entry brief_dev with schemas/brief_dev_input.json + brief_dev_output.json written inside the codegen v1 subset (no $ref/$defs/oneOf — codegen reads every schemas/*.json, so these must generate; the input schema inlines the brief-document and resolved-context shapes with additionalProperties false throughout; the generated BriefDevInput* types are an unused byproduct, the workflow keeps the hand-written codecs); timeout_seconds 86400 is carried over from the retired onatopp entry — an author-chosen budget from the live family, not a new invented default (ADR-001 note). Error taxonomy note: C18 names five categories (scout failure, dev block, verify exhaustion, review drift, stage failure); C15 additionally mandates a typed error for a harden pass that breaks verification, so BriefDevError carries a sixth variant HardenRegressed — a deliberate, declared addition, with startup fan-out failure folded into the stage-failure variant. Enrichment call sites (writing scout/dev/review blocks into the worktree brief file, C19-C22) are NOT here — BD-004 inserts enrich_brief calls into this module after it lands; brief_dev returns the three reports in its result so the parent and BD-004 have everything they need. Any build error outside stacked_dev.gleam is this brief's defect.

### R5: The brief_dev workflow module

Create src/brief_dev.gleam (deleting src/onatopp_dev.gleam — brief_dev takes its slot, ADR-008) with pub const workflow_type = "brief_dev", pub const status_query_name = "brief_dev_status", definition() binding the R2 codecs to execute, and run() mirroring the existing raw-input entry convention. Flow: WHEN execution starts, THE SYSTEM SHALL register status ("scouting", 0) and run the scout activity with prompts.scout_prompt(document, context); IF scout fails, THEN THE SYSTEM SHALL fail with ScoutFailed (C11). WHEN scout succeeds, THE SYSTEM SHALL register ("developing", 0) and dispatch workflow.all([activities.warm_build(workspace), activities.dev(DevInput(workspace, prompts.dev_prompt(document, context, scout_report)))]) — the tagged StartupTask fan-out preserved, results in input order, a shape violation failing as BriefDevStageFailed(stage: "startup") (C12). IF the dev report contains any enrichment with status blocked, THEN THE SYSTEM SHALL fail with DevBlocked listing those R# ids (C18). WHILE verifying, THE SYSTEM SHALL register ("verifying", round) and run scoped_checks over the deduplicated files_changed paths of the latest dev report; on CheckPass proceed to review; on CheckFail, IF round >= verify_fix_cap THEN fail with VerifyFixExhausted(rounds, diagnostics), ELSE register ("fixing", round), durably sleep round_backoff_ms, run dev_resume(ResumeInput(session_id: workspace.branch, feedback: prompts.resume_feedback(document, diagnostics))) for a full replacement DevReport, and recurse with round + 1 (C13 — onatopp semantics preserved, both bounds are required inputs, no default anywhere, ADR-001/ADR-003). WHEN verification converges, THE SYSTEM SHALL register ("reviewing", round) and run dev_review with prompts.review_prompt(document, context, scout_report, latest_dev_report, converged_check_result) — the reviewer sees the attestation AND the measured result (S10) in its own session (C14). IF any review enrichment has alignment drifted, THEN THE SYSTEM SHALL fail with ReviewDrifted carrying every drifted enrichment's id and issues (C16). IF any review enrichment's fixes list is non-empty, THEN THE SYSTEM SHALL register ("hardening", round) and re-run scoped_checks with the same files_touched; IF that re-run fails, THEN THE SYSTEM SHALL fail with HardenRegressed(diagnostics) (C15). WHEN converged, THE SYSTEM SHALL register ("converged", round) and return BriefDevResult(scout, final dev report, review report, verify_rounds, build_warm). The status handler is re-registered at every transition with exactly the phases scouting, developing, verifying, fixing, reviewing, hardening, converged (C17, S6). THE SYSTEM SHALL NOT read the filesystem, write the brief file, or call any enrichment (CN1/P5 — BD-004 inserts enrich_brief call sites later), SHALL NOT impose any cap, backoff, or deadline beyond the two required inputs, and SHALL NOT dispatch any activity other than scout, warm_build, dev, scoped_checks, dev_resume, and dev_review. LOC note: the module is estimated above 300 lines; its behavioural tests live in BD-007's hermetic suite by design, and its codec tests in R7 — no test code in this module.
