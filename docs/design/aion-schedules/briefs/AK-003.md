# AK-003: ScheduleEvaluator with timer-driven triggers and policy enforcement

> **Cluster:** aion-schedules
> **Depends on:** AK-001, AK-002
> **Checklist:** C5, C6, C7, C10
> **Stories:** S1, S2, S4, S7

## Purpose

Implement the ScheduleEvaluator that drives schedule execution: arms durable timers for each active schedule's next fire time, evaluates overlap policy when a timer fires, starts workflow executions according to policy, handles catch-up on recovery, and re-arms timers for the next trigger. This is the scheduling heartbeat of the engine.

## Task

Create aion::schedule::evaluator module with a ScheduleEvaluator that manages active schedules. On creation/resume, it computes next_fire_time and arms a durable timer. When the timer fires, it: (1) checks overlap policy against the current execution (if any), (2) starts a workflow if policy allows, (3) records ScheduleTriggered, (4) computes and arms the next fire time. On recovery, it evaluates catch-up policy for any schedules whose next_fire_time is in the past. Schedule state is projected from events.

## Requirements

### R1: Schedule state projection from events

THE SYSTEM SHALL maintain per-schedule state projected from schedule events: schedule_id, config, is_paused, is_deleted, last_triggered_at, next_trigger_at, current_execution.

**Acceptance:**
- After ScheduleCreated, state reflects the initial config with is_paused=false.
- After ScheduleUpdated, config is updated.
- After SchedulePaused, is_paused=true; after ScheduleResumed, is_paused=false.
- After ScheduleDeleted, is_deleted=true.
- After ScheduleTriggered, last_triggered_at is updated.

**Files:**
- create: crates/aion/src/schedule/state.rs
- modify: crates/aion/src/schedule/mod.rs

**Checklist:**

**Stories:**
- S7

### R2: Timer-driven trigger evaluation

THE SYSTEM SHALL arm a durable timer for each active schedule. When the timer fires, the evaluator SHALL evaluate the trigger, optionally start a workflow, record ScheduleTriggered, and re-arm. THE SYSTEM SHALL NOT use polling.

**Acceptance:**
- A schedule with a short interval fires within the expected window.
- After each trigger, the timer is re-armed for the next fire time.
- Paused and deleted schedules do not have armed timers.

**Files:**
- create: crates/aion/src/schedule/evaluator.rs

**Checklist:**
- C5

**Stories:**
- S1

### R3: Overlap policy enforcement

THE SYSTEM SHALL enforce overlap policy when a trigger fires and a previous execution is still running. Skip: do not start. BufferOne: queue one pending. CancelPrevious: cancel running, start new. AllowAll: start regardless.

**Acceptance:**
- Each of the four overlap policies has a dedicated test.
- Skip skips and re-arms; BufferOne queues at most one; CancelPrevious cancels then starts; AllowAll always starts.

**Files:**
- create: crates/aion/src/schedule/policy.rs
- modify: crates/aion/src/schedule/evaluator.rs

**Checklist:**
- C6

**Stories:**
- S2

### R4: Catch-up policy enforcement on recovery

THE SYSTEM SHALL evaluate catch-up policy when next_trigger_at is in the past. All: fire every missed trigger. One: fire once immediately. Skip: advance to next future time.

**Acceptance:**
- Each of the three catch-up policies has a dedicated test.
- Catch-up fires are recorded as ScheduleTriggered events.

**Files:**
- modify: crates/aion/src/schedule/evaluator.rs
- modify: crates/aion/src/schedule/policy.rs

**Checklist:**
- C7

**Stories:**
- S4

### R5: Recovery reconstructs state and re-arms timers

THE SYSTEM SHALL reconstruct all schedule state from events on recovery and re-arm timers for active schedules, applying catch-up policy for past-due triggers.

**Acceptance:**
- After recovery, active schedules have armed timers.
- Paused and deleted schedules are not re-armed.
- Catch-up policy is applied before re-arming.
