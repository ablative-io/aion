## Purpose

Implement the other half of the two-use split: the declaration surface for a side-effectful native operation. It is the spine of the cluster's determinism guarantee — a NIF with a side effect is typed as an activity from the first line, runs on the dirty scheduler, returns an ActivityError on failure, and can only be invoked through the engine's activity contract, never inline.

## Task

In declare.rs, implement activity_nif: the same shim-generation ergonomics as deterministic_nif but for a body returning Result<T, ActivityError>, producing a Nif tagged Determinism::SideEffectful with the dirty-scheduler flag set. Ensure the ActivityError's retryable/terminal classification survives the boundary back to the engine, and that there is no untyped declaration path — the type system alone makes a side-effectful NIF impossible to express as an inline deterministic helper. Expose beamr's existing request_suspend / wake_with_result for the case where a side-effectful NIF must suspend rather than block.

## Requirements

### R1: activity_nif declaration

THE SYSTEM SHALL provide activity_nif, taking a Rust fn/closure with FromTerm arguments and a body returning Result<T, ActivityError> where T: IntoTerm-able (directly or via Payload), producing a Nif tagged Determinism::SideEffectful with the dirty-scheduler flag set.

**Acceptance:**
- declare.rs provides activity_nif yielding a Nif with determinism = SideEffectful and is_dirty() == true by default (a doc comment explains side effects may block, so the dirty scheduler is the correct default).
- The generated shim performs the same arity check and per-argument FromTerm decode as the deterministic path, then calls the body, which returns Result<T, ActivityError>.
- On Ok(value) the shim encodes value; on Err(activity_error) the shim encodes the error so the engine can record an ActivityFailed event — it never returns a bare string error.
- Panic containment from AN-004 applies here too: a panic in the body becomes a Terminal ActivityError-shaped failure, not an unwind across the FFI boundary.
- A unit test declares a side-effectful NIF (e.g. one that reads an env-provided value or a temp file), invokes it, and asserts the SideEffectful tag, the dirty flag, and the Ok encoding.

### R2: No untyped path; the split is type-enforced

THE SYSTEM SHALL ensure deterministic_nif can only produce Determinism::Pure and activity_nif can only produce Determinism::SideEffectful, with no public declaration path that produces an arbitrary tag, so a side-effectful operation cannot be expressed as an inline deterministic helper.

**Acceptance:**
- There is no public constructor or macro that sets the Determinism tag to a caller-chosen value; the tag is fixed by which builder is used.
- deterministic_nif's body type cannot return an ActivityError-classified failure as its success path, and activity_nif's body type requires the Result<T, ActivityError> shape — the signatures keep the two uses distinct.
- A doc comment on both builders states the determinism invariant (DESIGN-OVERVIEW 'The Core Concept: Determinism') and that a side-effectful NIF is recorded through the activity contract and returned from history on replay, never re-run.
- A compile-fail test (trybuild or equivalent) or a clearly documented type-level argument demonstrates that a side-effectful body cannot be passed to deterministic_nif.

### R3: ActivityError preservation and suspend/wake exposure

THE SYSTEM SHALL encode a body's ActivityError so its retryable/terminal classification survives back to the engine, and SHALL expose beamr's existing request_suspend / wake_with_result mechanism for a side-effectful NIF that must suspend rather than block, without introducing a new async model.

**Acceptance:**
- An ActivityError returned by the body is encoded such that the engine can reconstruct its classification (retryable vs terminal) and message — a test round-trips a retryable and a terminal ActivityError through the encoding and asserts the classification is recoverable.
- The crate exposes a documented helper or pattern over beamr's request_suspend / wake_with_result so a long-running side-effectful NIF can suspend the calling process and be woken with a result term (mirroring the beamr-meridian async step pattern), with a doc comment pointing at that precedent.
- The suspend/wake path is documented as the only async mechanism — the crate does not define its own futures/executor model.
- A unit or integration test exercises the suspend-then-wake path delivering a result term to a (test-driven) process, or, where a full scheduler is unavailable in unit context, asserts the handle/closure wiring that delivers the woken result.

## Boundaries

- SHALL NOT record activity events or decide retries — AE dispatches the activity body and AD records, AT decides retry per the RetryPolicy consulting the ActivityError classification. This brief produces the body and preserves the classification; it does not record or retry.
- SHALL NOT implement remote (Tier-3) worker dispatch — that is AR.
- SHALL NOT implement NifSet / registration — that is AN-006.
- SHALL NOT introduce a new async runtime; only beamr's request_suspend / wake_with_result is used.
