# AN-006: NifSet registration value and end-to-end illustrative NIF set

> **Cluster:** aion-nif
> **Depends on:** AN-004, AN-005
> **Checklist:**
> - C27 — NifSet collects declared Nifs under a module name and produces the value EngineBuilder::register_nifs consumes.
> - C28 — NifSet rejects a duplicate (module, function, arity) at build time with a typed NifDeclError and never silently overwrites.
> - C29 — aion-nif does not call beamr's BifRegistryImpl::register itself; the descriptor set is registered by the engine's runtime module.
> - C31 — A minimal illustrative NIF set (at least one Pure helper and one SideEffectful activity) exercises the full declaration-to-descriptor path and asserts the determinism tags and dispatch shapes.
> **Stories:**
> - S7 (Engine Operator, Registering NIFs with the Engine) — As an engine operator, I want to hand a single NIF set to register_nifs so that wiring native helpers into the engine is one call, not per-function boilerplate.
> - S8 (Engine Operator, Registering NIFs with the Engine) — As an engine operator, I want a duplicate NIF declaration to be rejected at build time so that a registration clash is caught before the engine runs rather than silently overwriting a function.
> - S10 (Reviewer, Auditing the Native Boundary) — As a reviewer, I want the deterministic-vs-side-effectful distinction enforced by the type system so that I can trust the determinism invariant without reading every call site.

## Purpose

Produce the value the engine consumes (NifSet for EngineBuilder::register_nifs) and prove the whole crate end-to-end with a minimal illustrative NIF set — one deterministic helper and one side-effectful activity — exercised through the full declaration-to-descriptor path.

## Task

In registry.rs, define the NifSet builder: it collects declared Nifs under a module name into the value EngineBuilder::register_nifs consumes, rejecting a duplicate (module, function, arity) at build time with a typed NifDeclError, and never calling beamr's BifRegistryImpl::register itself (the engine's runtime module does that). Then build a minimal illustrative NIF set — at least one Pure helper and one SideEffectful activity — and a test that builds the set and asserts the descriptors' identities, determinism tags, dirty flags, and dispatch shapes.

## Requirements

### R1: NifSet builder

THE SYSTEM SHALL define NifSet, collecting declared Nifs under a module name and producing the value EngineBuilder::register_nifs (AE) consumes, exposing the per-NIF identity, NativeFn, dirty flag, and Determinism tag the engine needs to register and bind them.

**Acceptance:**
- registry.rs defines NifSet with a builder API (e.g. new(module_name), add(nif), build() -> Result<NifSet, NifDeclError>) and an accessor yielding the contained Nif descriptors.
- The produced value carries, per NIF, the module/function/arity, the NativeFn, the dirty flag, and the Determinism tag, so AE's runtime registration can register each by MFA with the dirty flag and bind it inline (Pure) or as an activity body (SideEffectful).
- A doc comment states NifSet is the argument shape for EngineBuilder::register_nifs and that aion-nif does not itself perform beamr registration.
- A unit test builds a NifSet from a couple of declared NIFs and reads back their descriptors.

**Files:**
- create: crates/aion-nif/src/registry.rs
- modify: crates/aion-nif/src/lib.rs

**Checklist:**
- C27 — NifSet collects declared Nifs under a module name and produces the value EngineBuilder::register_nifs consumes.
- C29 — aion-nif does not call beamr's BifRegistryImpl::register itself; the descriptor set is registered by the engine's runtime module.

**Stories:**
- S7 (Engine Operator, Registering NIFs with the Engine) — As an engine operator, I want to hand a single NIF set to register_nifs so that wiring native helpers into the engine is one call, not per-function boilerplate.

### R2: Duplicate rejection at build time

THE SYSTEM SHALL reject a duplicate (module, function, arity) triple when building a NifSet, returning a typed NifDeclError::Duplicate, and SHALL never silently overwrite an existing declaration.

**Acceptance:**
- Adding two NIFs with the same module, function name, and arity causes build() to return NifDeclError::Duplicate carrying the conflicting identity.
- Two NIFs with the same name but different arity are allowed (distinct MFA), matching beamr's arity-keyed registry.
- A unit test asserts the duplicate is rejected and that a same-name-different-arity pair builds successfully.

**Files:**
- modify: crates/aion-nif/src/registry.rs

**Checklist:**
- C28 — NifSet rejects a duplicate (module, function, arity) at build time with a typed NifDeclError and never silently overwrites.

**Stories:**
- S8 (Engine Operator, Registering NIFs with the Engine) — As an engine operator, I want a duplicate NIF declaration to be rejected at build time so that a registration clash is caught before the engine runs rather than silently overwriting a function.

### R3: End-to-end illustrative NIF set

THE SYSTEM SHALL provide a minimal illustrative NIF set with at least one Determinism::Pure helper and one Determinism::SideEffectful activity, exercising the full declaration-to-descriptor path and asserting determinism tags and dispatch shapes.

**Acceptance:**
- A test (or a small examples module) declares a Pure helper (e.g. a JSON-field extract or string transform over the term/Payload bridge) and a SideEffectful activity (e.g. reading a temp file) and collects them into a NifSet.
- The test asserts the Pure NIF carries Determinism::Pure and is not dirty, and the SideEffectful NIF carries Determinism::SideEffectful and is dirty.
- The test invokes both generated NativeFns through a real ProcessContext: the Pure one returns its transformed value, the SideEffectful one returns its Ok value and (in a failure case) an encoded ActivityError preserving classification.
- The test is a real runtime test, not gated out or ignored, and demonstrates the value is the shape EngineBuilder::register_nifs expects (constructed, not necessarily registered, since registration is AE).

**Files:**

## Boundaries

- SHALL NOT call beamr's BifRegistryImpl::register — registration is AE's runtime module (the sole beamr boundary).
- SHALL NOT ship domain-specific NIFs (run_step_norn, commit, ...) — those live in the Meridian tree using this crate; the illustrative set is minimal and generic.
- SHALL NOT implement engine wiring, event recording, or retry — those are AE/AD/AT.
