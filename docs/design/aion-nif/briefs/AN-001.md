## Purpose

Create the aion-nif leaf-of-the-helper-tree crate, its typed error taxonomy, and the single isolated module that will hold any unsafe needed at the beamr term-construction boundary. This establishes the dependency boundaries and the audited unsafe seam before any conversion or declaration logic lands.

## Task

Create the aion-nif crate as a workspace member depending only on beamr and aion-core. Add a thin lib.rs of declarations and re-exports. Define TermError and NifDeclError as thiserror enums. Create the raw module as the sole place permitted to use unsafe: it wraps beamr's process-heap term construction (cons cells, tuples, binaries) behind safe functions, preferring beamr's existing safe ProcessContext allocation APIs (alloc_cons, alloc_tuple) and using unsafe only where a return shape has no safe beamr path, each unsafe block documented with a // SAFETY: comment.

### R2: TermError taxonomy

THE SYSTEM SHALL define TermError as a typed thiserror enum describing term-conversion failures, including a variant that records the failing argument index for argument-decode failures, rather than surfacing a bare badarg.

**Acceptance:**
- error.rs defines TermError with variants covering type mismatch (expected vs found kind), argument-position decode failure carrying the zero-based index, atom-resolution failure, and heap-allocation failure.
- TermError implements std::error::Error and Display and derives Debug.
- A doc comment explains that TermError replaces beamr's bare badarg so authors and the engine get a typed, descriptive failure.
- A unit test constructs each variant and asserts its Display text.

### R3: NifDeclError taxonomy

THE SYSTEM SHALL define NifDeclError as a typed thiserror enum describing NIF declaration and registration-set failures, including a duplicate (module, function, arity) declaration.

**Acceptance:**

- NifDeclError implements std::error::Error and Display and derives Debug.

### R4: Isolated FFI seam module

THE SYSTEM SHALL create the raw module as the only module permitted to use unsafe. It SHALL expose safe wrapper functions for process-heap term construction (cons cell, tuple, binary), preferring beamr's existing safe ProcessContext allocation APIs and using unsafe only where a return shape has no safe beamr path. Any unsafe SHALL be confined to this module behind those wrappers, each block carrying a // SAFETY: comment.

**Acceptance:**
- raw.rs exposes safe functions to build a binary term, a tuple term, and a cons-list term from Rust values, returning TermError on allocation failure.
- Where beamr's ProcessContext provides a safe allocation path (alloc_cons, alloc_tuple), raw uses it and contains no unsafe for that shape.
- If any unsafe is required for a return shape with no safe beamr path, it appears only in raw.rs behind a single module-level documented #![allow(unsafe_code)] seam, and every unsafe block carries a // SAFETY: comment stating the invariant; no unsafe appears in any other module.
- raw.rs does NOT use Box::leak: leaking the process heap per call is explicitly rejected (a doc comment records why, referencing the beamr-meridian limitation it supersedes).
- A unit test builds a tuple and a small cons list through the safe wrappers and reads the values back.
- cargo clippy --workspace --all-targets -- -D warnings passes, confirming no allow other than the documented raw.rs seam.
