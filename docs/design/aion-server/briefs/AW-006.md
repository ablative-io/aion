> **Cluster:** aion-server
> **Depends on:** AW-003, AW-004, AW-005
> **Checklist:**
> - C16 — aion-server is a workspace member (binary) depending on aion, aion-proto, and a store backend, with a thin main.rs where anyhow is the only error type at the top level.
> - C17 — Config types (store DSN, listen ports, TLS, auth, dashboard asset path, namespace mode, worker heartbeat window, WebSocket buffer bound) derive Deserialize only — never Serialize — and no operational value is a hardcoded default.
> - C18 — ServerError is a thiserror taxonomy used throughout the server library modules; no unwrap/expect in library code and lock/stream poison is handled explicitly.
> - C19 — Shared server state holds the engine handle and the NamespaceResolver and is constructed once at startup.
> **Stories:**

## Purpose

Create the standalone server binary that wraps a live Engine and exposes it over the network. Establish the crate skeleton: a thin main, Deserialize-only config (credential policy), a thiserror error taxonomy, and the shared state holding the engine handle and namespace resolver. Everything in AW-007..AW-013 builds on this.

## Task

Create the aion-server crate as a workspace binary depending on aion, aion-proto, a store backend, tonic, axum, and tokio. Write a thin main.rs where anyhow is the only top-level error type. Define Deserialize-only config types (store DSN, listen ports, TLS, auth, dashboard asset path, namespace mode, worker heartbeat window, WebSocket buffer bound) with no hardcoded operational defaults. Define ServerError as a thiserror taxonomy used by the library modules. Define shared server state holding the engine handle and the NamespaceResolver, constructed once at startup.

## Requirements

### R1: Crate scaffold and thin main

THE SYSTEM SHALL create the aion-server crate as a workspace binary depending on aion, aion-proto, and a store backend (plus tonic, axum, tokio), with a thin main.rs in which anyhow is the only error type and a lib.rs that is declarations and re-exports only.

**Acceptance:**
- crates/aion-server exists, is registered in the workspace, and produces a binary target.
- Cargo.toml declares aion, aion-proto, and a store backend among workspace deps and no other Aion crate beyond those needed.
- main.rs is thin: it loads config, builds state, starts the transports, and returns anyhow::Result; anyhow appears nowhere else in the crate (CO8).
- lib.rs contains only pub mod declarations and re-exports.
- The crate sets unsafe_code = "deny"; cargo check -p aion-server passes.

### R2: Deserialize-only config types

THE SYSTEM SHALL define configuration types covering store DSN, listen ports, TLS material, auth, dashboard asset path, namespace mode, worker heartbeat window, and WebSocket buffer bound. Types that carry secrets SHALL derive Deserialize but NOT Serialize, and no operational value SHALL be a hardcoded default.

**Acceptance:**
- config.rs defines the config types loaded from file/env.
- Secret-carrying config (store DSN, TLS key/cert material, auth token) derives Deserialize only — never Serialize (D4, CO7); a compile-time or review check confirms no Serialize derive on those types.
- No operational value (port, heartbeat window, buffer bound, timeout) has a baked-in default in the crate; absence is a configuration error, and engine-owned defaults are deferred to the engine (CO11).
- Loading invalid or incomplete config returns a typed ServerError, not a panic.

**Files:**
- create: crates/aion-server/src/config.rs
- modify: crates/aion-server/src/lib.rs

**Checklist:**

### R3: ServerError taxonomy

THE SYSTEM SHALL define ServerError as a thiserror taxonomy used throughout the server's library modules, with no anyhow in library code, no unwrap/expect, and explicit lock/stream poison handling.

**Acceptance:**
- error.rs defines ServerError with thiserror, covering config, transport-bind, namespace, engine-call, and stream failures.
- ServerError carries or maps to a WireError where a failure crosses the wire (per D3); internal-only failures stay ServerError.
- No library module uses anyhow, .unwrap(), or .expect(); lock poison maps to a typed ServerError variant (CO8).
- ServerError implements std::error::Error and Display and is Send + Sync.

**Files:**
- create: crates/aion-server/src/error.rs
- modify: crates/aion-server/src/lib.rs

**Checklist:**
- C18 — ServerError is a thiserror taxonomy used throughout the server library modules; no unwrap/expect in library code and lock/stream poison is handled explicitly.

**Stories:**

### R4: Shared server state

THE SYSTEM SHALL define a shared server state holding the engine handle and the NamespaceResolver, constructed once at startup and shared across transports.

**Acceptance:**
- state.rs defines a state type holding an Arc<Engine> (or per-namespace engine access via the resolver) and the NamespaceResolver, plus the loaded config it needs.
- State is constructed once in main from config and shared (cloneable Arc) into the gRPC, HTTP, WebSocket, and worker transports.
- State construction validates that the engine built successfully and the store backend connected, returning ServerError on failure.
- A unit test constructs state against an InMemoryStore-backed engine (per aion-engine CO13) with no database.
