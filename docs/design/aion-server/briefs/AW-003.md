# AW-003: Workflow-management wire service (start/signal/query/cancel/list/describe)

> **Cluster:** aion-server
> **Depends on:** AW-002

> - C9 — Each request carries a namespace; inputs and results are carried as aion-core Payload, and ListWorkflows carries a WorkflowFilter and returns WorkflowSummary values.

## Purpose

Define the gRPC service and request/response messages for the core workflow-management surface every caller uses. This is the wire form of the engine's start/signal/query/cancel/list/describe API — the contract client SDKs generate stubs from.

## Task

Add workflow.proto declaring a service with StartWorkflow, Signal, Query, Cancel, ListWorkflows, and DescribeWorkflow RPCs and their request/response messages. Each request carries a namespace; inputs and results are aion-core Payload; ListWorkflows carries a WorkflowFilter and returns WorkflowSummary values; DescribeWorkflow returns a summary (and optionally history). Provide the serde request/response types and assert lossless proto<->serde round-trips.

## Requirements

### R1: Workflow service definition

THE SYSTEM SHALL define a gRPC service in workflow.proto with StartWorkflow, Signal, Query, Cancel, ListWorkflows, and DescribeWorkflow RPCs, each with a request and a response message.

**Acceptance:**
- workflow.proto defines a service exposing all six RPCs.
- StartWorkflow request carries namespace, workflow type, and input Payload; response carries the assigned WorkflowId and RunId.
- Signal request carries namespace, target WorkflowId, signal name, and payload Payload; Cancel request carries namespace, WorkflowId, and reason.
- Query request carries namespace, WorkflowId, and query name; response carries a result Payload or a WireError.
- tonic-build generates the service stub and the crate compiles.

**Files:**
- create: crates/aion-proto/proto/workflow.proto
- modify: crates/aion-proto/src/lib.rs

**Checklist:**

### R2: Namespace, Payload, Filter, and Summary on the wire

THE SYSTEM SHALL carry a namespace on every request, carry inputs/results as aion-core Payload, and carry ListWorkflows as a WorkflowFilter returning WorkflowSummary values.

**Acceptance:**
- Every request message in workflow.proto has a namespace field.
- Inputs and results use the common Payload message (from AW-002), not bespoke byte fields.
- ListWorkflows request carries a WorkflowFilter (via the AW-002 envelope/conversion) and its response carries a repeated WorkflowSummary.
- DescribeWorkflow response carries a WorkflowSummary (history inclusion is optional and, if present, carried as repeated Event envelopes).

### R3: Serde types and round-trip

THE SYSTEM SHALL provide serde request/response types for the workflow-management messages that derive Serialize + Deserialize and round-trip losslessly to and from their proto form.

**Acceptance:**
- Each workflow-management request/response has a serde-deriving Rust type (the generated prost type or a hand-paired serde type per the crate's chosen pattern).
- A unit test round-trips at least StartWorkflow, ListWorkflows, and Query request+response through serde_json and through proto encode/decode, asserting equality.
- All wire data types here derive both Serialize and Deserialize (CO7) — none is Deserialize-only (these are data, not credentials).
