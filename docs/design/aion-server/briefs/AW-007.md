# AW-007: Namespace isolation — resolver and adapter-boundary guard

> **Cluster:** aion-server
> **Depends on:** AW-006
> **Checklist:**
> - C20 — A NamespaceResolver authorises a caller for a namespace and scopes an operation to it, abstracting whether namespaces map to separate engine/store instances or one shared engine with a namespace key.
> - C21 — Namespace authorisation and scoping happen at the adapter boundary before any Engine call; no cross-namespace operation (start/signal/query/cancel/list/subscribe/worker dispatch) reaches the engine.
> **Stories:**
> - S12 (Engine Operator, Deploying and running the standalone server) — As an operator running many tenants, I want each namespace isolated so that a caller or worker in one namespace can never touch another's workflows, events, or tasks.

## Purpose

Implement multi-tenancy. A namespace isolates one tenant's workflows, events, and workers from another's. Enforcement happens at the adapter boundary before any Engine call, so no cross-namespace operation ever reaches the engine. This is a server-mode-only concern and the gate every handler passes through.

## Task

Implement the NamespaceResolver that authorises a caller for a namespace and scopes an operation to it, abstracting whether namespaces map to separate engine/store instances or one shared engine with a namespace key (a deploy-time config choice). Implement a guard applied at the adapter boundary that authorises and scopes before any Engine call, rejecting cross-namespace access with a NamespaceDenied WireError.

## Requirements

### R1: NamespaceResolver

THE SYSTEM SHALL implement a NamespaceResolver that authorises a caller for a requested namespace and yields a namespace-scoped engine access, abstracting the deploy-time mapping (separate engine/store per namespace vs one shared engine keyed by namespace).

**Acceptance:**
- resolver.rs defines NamespaceResolver with a method that takes the caller's credentials/identity and a requested namespace and returns either a namespace-scoped engine handle or a NamespaceDenied error.
- The resolver hides whether the deployment uses per-namespace engines or one shared engine with a namespace dimension; callers see only a scoped handle (per D8).
- The namespace mode is read from config (AW-006), not hardcoded.
- A unit test exercises both an authorised and an unauthorised namespace request against a test resolver.

### R2: Adapter-boundary guard

THE SYSTEM SHALL enforce namespace authorisation and scoping at the adapter boundary before any Engine call, so that no cross-namespace operation (start/signal/query/cancel/list/subscribe/worker dispatch) reaches the engine.

**Acceptance:**
- guard.rs provides a single entry the API handlers, the stream layer, and the worker endpoint call first, which authorises the namespace and returns the scoped handle (or NamespaceDenied) before any Engine method is invoked.
- list/query/subscribe are scoped to the namespace; signal/query/cancel verify the target workflow belongs to the caller's namespace; a request that names a workflow in another namespace is rejected with NamespaceDenied and never reaches the engine.
- A unit test asserts that a request scoped to namespace A cannot signal/query/cancel/list/subscribe a workflow in namespace B, and that the engine is not called on a denied request (verified against a recording fake engine handle).
- The guard is the only path to a scoped engine handle; there is no way to obtain one that bypasses authorisation (CO9).

## Boundaries

- SHALL NOT implement the request handlers themselves — AW-008/AW-009 call the guard.
- SHALL NOT implement worker dispatch partitioning beyond exposing the namespace scope the worker endpoint (AW-011) consumes.
- SHALL NOT add workflow execution logic; the resolver only scopes and authorises access to the engine (CO6).
