# AW-014: Round-robin worker dispatch with closed-channel fallback

> **Cluster:** aion-server
> **Checklist:**
> - C34 — The connected-worker registry tracks a rotation index per (namespace, activity_type) as in-memory state (not persisted); the index advances on each dispatch and wraps at the worker count, remaining valid after registrations and deregistrations.
> - C35 — The dispatch method starts iteration from the rotation index, wrapping around the sorted worker list; a closed-channel send deregisters the worker and tries the next in rotation order; the existing error for no-workers and all-closed-channels is preserved.
> **Stories:**
> - S14 (Engine Operator, Deploying and running the standalone server) — As an operator scaling workers horizontally, I want dispatched activities round-robin'd across connected workers so that adding workers actually distributes load instead of every activity going to the lowest-ID worker.

## Purpose

The live activity-dispatch path does not distribute load and does not survive a worker whose stream just closed. The engine seam is WorkerActivityDispatcher in crates/aion-server/src/worker/bridge.rs (constructed once in state.rs and handed to the engine): it selects a single worker via registry.select_worker, which returns the lowest-id worker every time with no rotation, so every activity for a given (namespace, activity_type) lands on the same worker and adding workers does not spread load. Worse, send_activity_task fails the whole dispatch the moment that one worker's channel is full or closed — there is no try-next. A second, richer dispatcher exists in crates/aion-server/src/worker/dispatch.rs (the ActivityDispatcher push struct: rotation via workers_for, deregister-and-try-next on a closed channel, wait-for-worker) but it is constructed nowhere in production — only inside its own unit tests (verified: every ActivityDispatcher::new and ScheduledActivity construction sits below the #[cfg(test)] line, and nothing else in the workspace references them). This brief makes the live path round-robin across connected workers using a per-(namespace, activity_type) rotation index on the registry, fall back across closed channels to the next worker in rotation order, and it retires the dead duplicate dispatcher so a single dispatch path remains. The rotation index is in-memory state that resets on server restart — acceptable because dispatch is best-effort load distribution, not a durability guarantee.

## Task

Lock the registry's rotation primitive and route the live dispatcher through it. In crates/aion-server/src/worker/registry.rs, workers_for already returns the matching workers ordered by worker id and rotated by a per-(namespace, activity_type) index that advances (wrapping) and stays valid across registrations and deregistrations; cover it with tests and remove the non-rotating registry.select_worker (its only caller is the live dispatch, which switches to the rotated list). In crates/aion-server/src/worker/bridge.rs, change WorkerActivityDispatcher so that dispatch obtains the rotation-ordered candidate list for the activity's key and attempts delivery to each candidate in order: the first whose channel accepts the task wins (then the existing await-result model runs unchanged — register the pending completion, await the result on its receiver), a candidate whose channel is full or closed is deregistered and the next is tried, and only when every candidate is exhausted does it return the existing all-closed error; the existing no-worker wait and the drain check before each attempt are preserved. Then retire the duplicate dispatcher in crates/aion-server/src/worker/dispatch.rs: delete the ActivityDispatcher push struct, ScheduledActivity, and the dispatch-only log helper, keeping the completion machinery (ActivityCompletion, ActivityCompletionOutcome, ActivityCompletionSink, handle_activity_result, lost_worker_error) that the live path and worker_grpc.rs use, update the worker/mod.rs re-exports, and move the round-robin and closed-channel test scenarios onto the live WorkerActivityDispatcher. No new event type, no wire protocol change, no change to the activity completion contract or the await-result dispatch model.

## Requirements

### R1: Registry exposes a deterministic round-robin candidate list

THE SYSTEM SHALL maintain a rotation index per (namespace, activity_type) key inside the RegistryState, alongside the existing by_activity map. workers_for SHALL return the matching workers ordered by worker id and then rotated so the start offset advances (wrapping at the worker count) on each call for that key, making the rotation index the sole deterministic source of ordering. The rotation index SHALL be in-memory state only — NOT persisted and resetting to 0 on server restart — and SHALL remain valid (wrapped to the current worker count) after a worker registers or deregisters, so no out-of-bounds indexing occurs. THE SYSTEM SHALL remove the non-rotating registry.select_worker convenience method: its only caller is the live dispatch path, which switches to the rotated candidate list in R2, so a single-worker min-by-id selection no longer exists to bypass rotation.

**Acceptance:**
- RegistryState carries a rotation index keyed by (namespace, activity_type), initialised empty; the index is not written to any persistent store and is not exposed on the wire.
- workers_for orders candidates by worker id and then rotates: a test registering 3 workers and calling workers_for 3 times asserts each call returns a different starting worker and the same membership.
- Deregistering the worker at the current rotation position and then calling workers_for succeeds without panic and returns the remaining workers (the index stays in bounds).
- registry.select_worker is gone — grep for 'select_worker' in registry.rs returns no definition, and no caller remains in the crate.

### R2: Live dispatch round-robins and falls back across closed channels

WHEN WorkerActivityDispatcher dispatches a scheduled activity, THE SYSTEM SHALL obtain the registry's rotation-ordered candidate list for the activity's (namespace, activity_type) key and attempt delivery to each candidate in that order. The first candidate whose channel accepts the task wins, and the existing await-result model is preserved unchanged: the pending completion is registered for the chosen worker and the dispatching call awaits the result on its receiver exactly as today. IF a candidate's channel is full or closed, THE SYSTEM SHALL deregister that worker, clean up any per-attempt tracking for it, and advance to the next candidate in rotation order without skipping any remaining candidate. IF every candidate's channel is closed, THE SYSTEM SHALL return the existing worker_dispatch error 'all matching worker streams closed before task could be delivered' with the pending completion cleaned up. IF no worker is registered for the key, THE SYSTEM SHALL retain the existing wait-for-worker behaviour and no-worker error path. The drain_state.ensure_accepting check SHALL be evaluated before each delivery attempt in the fallback loop.

**Acceptance:**
- Dispatching 3 activities of the same type through WorkerActivityDispatcher with 3 registered live workers results in each worker's receiver getting exactly one task — successive dispatches begin at different workers (round-robin reaches the live path, not just the registry helper in isolation).
- Dispatching 6 activities across 3 live workers distributes approximately evenly — a test asserts each worker received 2.
- A dispatch where the rotated-to worker's channel is closed deregisters that worker and delivers to the next candidate in rotation order; a test asserts the registry count drops and the surviving worker's receiver has the task.
- A dispatch where every candidate's channel is closed returns Err(ServerError::WorkerDispatch { .. }) with the existing message text, and the pending completion and per-worker tracking are cleaned up (no leaked pending entry, no leaked heartbeat tracking).
- The drain check still runs before each attempt — a test that flips the drain state to draining mid-iteration confirms the dispatch fails with the drain error, not a channel error.
- send_activity_task no longer fails the whole dispatch on the first closed channel: the single-pick-and-fail behaviour is replaced by the rotation-ordered fallback loop.

The round-robin and closed-channel test scenarios SHALL be relocated onto the live WorkerActivityDispatcher so coverage is preserved, not lost. ## Boundaries

- SHALL NOT persist, serialise, or expose the rotation index — it is volatile in-memory state that resets on server restart.
- SHALL NOT add a load-aware dispatch strategy (channel depth, latency, CPU) or any configuration knob for the dispatch strategy — round-robin with closed-channel fallback is the whole scope; load-aware dispatch is a future concern.
- SHALL NOT change the wire protocol, add a new proto message, or modify the worker registration handshake.
- SHALL NOT introduce a new Event variant or modify the activity completion contract or the await-result dispatch model (register pending completion, await the result on its receiver).
- SHALL NOT remove the shared completion machinery in dispatch.rs (ActivityCompletion, ActivityCompletionOutcome, ActivityCompletionSink, handle_activity_result, lost_worker_error) — only the unused ActivityDispatcher push struct, ScheduledActivity, and the dispatch-only log helper are deleted.
- SHALL NOT leave a second dispatch path behind — the live WorkerActivityDispatcher is the single dispatcher after this brief.
