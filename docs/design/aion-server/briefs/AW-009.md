# AW-009: gRPC and HTTP transports over the shared handler layer

> **Cluster:** aion-server
> **Depends on:** AW-008
> **Checklist:**
> - C23 — A tonic gRPC service and an axum HTTP/JSON facade both delegate to the shared handler layer; the transport is a thin skin.
> - C24 — Handler outcomes (including failures) are serialised as the wire response or a mapped WireError; engine internals never cross the wire.
> **Stories:**
> - S5 (Workflow Caller, Driving workflows over the network) — As a workflow caller, I want to start, signal, query, cancel, list, and describe workflows over gRPC or HTTP so that I can drive the engine from any service without embedding it.

## Purpose

Expose the workflow-management handler layer over the two request/response transports: a tonic gRPC service (the primary transport, what client SDKs use) and an axum HTTP/JSON facade (for browsers and curl). Both are thin skins over the same handlers, so behaviour cannot diverge.

## Task

Implement a tonic service that implements the aion-proto workflow service and delegates each RPC to the AW-008 handler layer, and an axum HTTP/JSON facade that routes each operation to the same handlers. Wire both into main's startup using the configured listen ports and TLS. Both transports decode to the same handler input and encode the handler output (or mapped WireError); neither adds logic.

## Requirements

### R1: tonic gRPC service

THE SYSTEM SHALL implement the aion-proto workflow service as a tonic service that decodes each RPC, delegates to the matching AW-008 handler, and encodes the response or a WireError-bearing status.

**Acceptance:**
- grpc.rs implements the generated workflow service trait; each RPC method decodes the request, calls the shared handler, and encodes the response.
- A handler WireError is encoded into the gRPC response (a status carrying the stable WireError code/message), not a stringified internal error (D3).
- The service holds the shared state (engine handle + resolver) and adds no logic beyond decode/delegate/encode (CO6).
- An integration-style test (in-process tonic) drives StartWorkflow and ListWorkflows against an InMemoryStore-backed engine.

### R2: axum HTTP/JSON facade

THE SYSTEM SHALL implement an axum HTTP/JSON facade that routes each workflow-management operation to the same AW-008 handlers, decoding JSON requests and encoding JSON responses or a WireError.

**Acceptance:**
- http.rs defines axum routes for start/signal/query/cancel/list/describe, each delegating to the shared handler.
- Request bodies decode from JSON into the same handler input the gRPC path uses; responses encode the same handler output as JSON.
- A handler WireError is returned with an appropriate HTTP status and the stable WireError code/message in the body (D3).
- A test exercises the HTTP path for at least start and list, asserting parity with the gRPC path's behaviour (same handler outcome).

### R3: Startup wiring on configured ports/TLS

THE SYSTEM SHALL start both transports from main using the configured listen ports and TLS material, with no hardcoded port or TLS default.

**Acceptance:**
- main starts the gRPC and HTTP listeners on ports from config (AW-006); absence of a required port/TLS value is a ServerError, not a default (CO11).
- TLS material is read from the Deserialize-only config (D4) and never re-serialised.
- A graceful-shutdown path stops both listeners and calls Engine::shutdown (per aion-engine D11) so in-flight appends finish.
- Tests that need a bound socket gate at runtime via a *_TEST_URL env var with a logged skip, never #[ignore] (CO2).

## Boundaries

- SHALL NOT add logic beyond decode/delegate/encode; all behaviour lives in the AW-008 handlers (CO6, D5).
- SHALL NOT implement WebSocket streaming or the worker endpoint — AW-010, AW-011.
- SHALL NOT define new wire messages — those are owned by aion-proto (AW-003).
