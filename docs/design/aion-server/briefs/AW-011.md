# AW-011: Remote-worker endpoint — registry and task dispatch

> **Cluster:** aion-server
> **Depends on:** AW-007, AW-005

> - C30 — A returned ActivityResult is fed into the engine's activity contract so the engine records ActivityCompleted/ActivityFailed and resumes; the server does not record activity events or decide retries itself.
> **Stories:**
> - S9 (Remote Worker, Executing Tier-3 activities out of process) — As a remote worker, I want to connect once, advertise the activity types I implement, and have tasks pushed to me so that I run heavy work without polling.

## Purpose

Implement the server side of the remote-worker protocol — the rendezvous between the engine's Tier-3 activity dispatch and the pool of connected out-of-process workers. A worker connects, advertises its activity types, receives pushed tasks, and reports results that flow back into the engine's activity contract.

## Task

Implement the bidirectional gRPC worker endpoint: accept a worker's stream, record its RegisterWorker advertisement in a connected-worker registry keyed by namespace and activity type, push a scheduled Tier-3 activity to a matching worker as an ActivityTask, and feed the returned ActivityResult into the engine's activity contract so the engine records ActivityCompleted/ActivityFailed and resumes. The server records no activity events and decides no retries itself.

## Requirements

### R1: Connected-worker registry

THE SYSTEM SHALL accept a worker's bidirectional gRPC stream, validate its namespace via the guard, and record its RegisterWorker advertisement in a connected-worker registry keyed by namespace and activity type.

**Acceptance:**
- worker/mod.rs and registry.rs define a registry mapping (namespace, activity type) -> the set of connected workers offering it, with the stream handle to push tasks.
- On connect, the worker's namespace is authorised via the AW-007 guard before registration (CO9, D8); registration in namespace A never makes the worker visible to namespace B.
- On disconnect, the worker is removed from the registry (no stale entries).
- Registry access handles lock poison explicitly as a typed ServerError (CO8); a test registers and deregisters workers across two namespaces and asserts isolation.

### R2: Push dispatch to a matching worker

THE SYSTEM SHALL match a scheduled Tier-3 activity to a registered worker for that activity type in that namespace and push it as an ActivityTask down the worker's stream; it SHALL NOT use polling.

**Acceptance:**
- dispatch.rs takes a scheduled-activity need (activity type, input Payload, WorkflowId/ActivityId, namespace) and selects a registered worker for that (namespace, activity type), pushing an ActivityTask down its stream (D7).
- If no worker is registered for the (namespace, activity type), the dispatch surfaces this to the engine as an activity that cannot be placed (so the engine applies its retry/timeout policy); the server does not silently drop it.
- Dispatch is push, not poll: there is no worker-initiated 'fetch task' RPC (per D7).
- A test dispatches a task to a fake worker stream and asserts the ActivityTask is pushed with correct correlation.

### R3: Feed results into the engine's activity contract

THE SYSTEM SHALL feed a returned ActivityResult into the engine's activity contract so the engine records ActivityCompleted/ActivityFailed and resumes; the server SHALL NOT record activity events or decide retries itself.

**Acceptance:**
- On receiving an ActivityResult, dispatch hands it (success Payload or ActivityError) to the engine's activity-completion path keyed by the WorkflowId/ActivityId correlation (D9).
- The server appends no Event and makes no retry decision; recording ActivityCompleted/ActivityFailed and the retry-or-fail decision are the engine's (aion-engine D7, AT policy).
- The ActivityError's retryable/terminal classification (from aion-core, carried in AW-005's ActivityResult) is preserved into the engine path unchanged.
- A test returns both a success and an error ActivityResult and asserts the engine's activity path is invoked with the correct outcome (against a recording fake engine handle / InMemoryStore-backed engine).

## Boundaries

- SHALL NOT implement heartbeat/lost-worker handling — that is AW-012.
- SHALL NOT build any worker client — that is cluster AR.
- SHALL NOT record activity events or decide retries; those stay in the engine (D9, CO6).
