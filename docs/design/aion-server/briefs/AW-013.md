# AW-013: Dashboard hosting — serve cluster AU's static assets

> **Cluster:** aion-server
> **Depends on:** AW-009, AW-010
> **Checklist:**
> - C32 — The server serves cluster AU's built static asset bundle from a configured (or embedded) path.
> - C33 — The dashboard is backed only by the same public gRPC/HTTP handlers and WebSocket feed every other client uses; there is no dashboard-specific API.
> **Stories:**
> - S8 (Dashboard / Observer, Watching workflow execution in real time) — As a dashboard, I want to be served by the server and backed by the same public API every client uses so that there is no privileged side channel to maintain.

## Purpose

Host the monitoring dashboard. The dashboard UI is built by cluster AU; this brief serves its static asset bundle from the server and backs it with the same public API and WebSocket event feed every other client uses. There is no dashboard-specific API — the dashboard is just another first-class client.

## Task

Serve cluster AU's built static asset bundle from a configured (or embedded) path via the axum HTTP transport, alongside the public gRPC/HTTP handlers (AW-009) and the WebSocket event feed (AW-010). Add no dashboard-specific API surface: the dashboard consumes exactly the public contract.

## Requirements

### R1: Serve the static asset bundle

THE SYSTEM SHALL serve cluster AU's built static dashboard asset bundle from a configured asset path (or an embedded bundle), mounted on the HTTP transport.

**Acceptance:**
- dashboard/assets.rs serves the static bundle (index + assets) from the configured dashboard asset path (AW-006 config), or from an embedded bundle when configured to embed.
- The asset path/embedding mode comes from config, with no hardcoded default path (CO11).
- Requests for unknown asset routes fall through correctly (SPA fallback to index where appropriate) without shadowing the API routes.
- A test asserts the bundle root is served and an API route is not shadowed by the asset handler.

**Files:**
- create: crates/aion-server/src/dashboard/mod.rs
- create: crates/aion-server/src/dashboard/assets.rs
- modify: crates/aion-server/src/lib.rs
- modify: crates/aion-server/src/api/http.rs
- modify: crates/aion-server/src/main.rs

**Checklist:**
- C32 — The server serves cluster AU's built static asset bundle from a configured (or embedded) path.

**Stories:**
- S8 (Dashboard / Observer, Watching workflow execution in real time) — As a dashboard, I want to be served by the server and backed by the same public API every client uses so that there is no privileged side channel to maintain.

### R2: No dashboard-specific API

THE SYSTEM SHALL back the dashboard solely with the same public gRPC/HTTP handlers (AW-009) and WebSocket event feed (AW-010) every other client uses; it SHALL NOT add a dashboard-specific API or privileged side channel.

**Acceptance:**
- No API route, RPC, or WebSocket subscription exists that is reachable only by the dashboard; the dashboard uses the public contract (D10).
- The dashboard's data and live feed come from the workflow-management handlers and the WebSocket stream, scoped by namespace via the same guard (CO9).
- An inspection/review confirms no handler branches on a 'dashboard' caller to expose extra data.
- A test asserts the dashboard's data routes are the same public routes a client SDK would call.

**Files:**
- modify: crates/aion-server/src/dashboard/assets.rs
- modify: crates/aion-server/src/api/http.rs

**Checklist:**
- C33 — The dashboard is backed only by the same public gRPC/HTTP handlers and WebSocket feed every other client uses; there is no dashboard-specific API.

**Stories:**
- S8 (Dashboard / Observer, Watching workflow execution in real time) — As a dashboard, I want to be served by the server and backed by the same public API every client uses so that there is no privileged side channel to maintain.

## Boundaries

- SHALL NOT build the dashboard UI — that is cluster AU; this brief only serves its built assets.
- SHALL NOT add a privileged dashboard API or bypass namespace scoping (D10, CO9).
- SHALL NOT implement the event feed itself — it reuses AW-010.
