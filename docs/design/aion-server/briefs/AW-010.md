# AW-010: WebSocket event streaming over the engine broadcast

> **Cluster:** aion-server
> **Depends on:** AW-007, AW-004
> **Checklist:**
> - C25 — A SubscriptionRequest is validated for namespace scope and mapped onto the engine's EventFilter, then engine.subscribe is called to obtain the stream tail.
> - C26 — A per-connection task forwards events from the engine stream to the socket until the client closes or the subscribed workflow terminates, then drops the subscription with no leak.
> - C27 — Each connection has a bounded outbound buffer; a consumer that cannot keep up is closed with a typed lag WireError, and the engine broadcast is never back-pressured by a subscriber.
> **Stories:**
> - S6 (Dashboard / Observer, Watching workflow execution in real time) — As an observer, I want to open a WebSocket and subscribe to one workflow, a filtered class, or the whole firehose so that I see events as they happen.
> - S7 (Dashboard / Observer, Watching workflow execution in real time) — As an observer on a slow connection, I want to be dropped with a clear lag signal so that my slowness never stalls the engine or other subscribers, and I know to resubscribe.

## Purpose

Bridge the engine's event broadcast to remote clients over WebSocket. A client subscribes to a slice of the firehose (one workflow, a filtered class, or everything) and receives events as they happen — the first-class real-time feature that backs the dashboard. A slow consumer must never stall the engine.

## Task

Implement the WebSocket endpoint: accept a connection and a SubscriptionRequest, validate the namespace scope via the AW-007 guard, map the request onto the engine's EventFilter and call engine.subscribe, then run a per-connection forward loop pushing serialised Event envelopes to the socket. Bound each connection's outbound buffer; drop a consumer that cannot keep up with a typed lag WireError; clean up the subscription on close. The engine broadcast is never back-pressured by a subscriber.

## Requirements

### R1: Subscription request to EventFilter mapping

THE SYSTEM SHALL accept a SubscriptionRequest, validate its namespace scope via the guard, and map it onto the engine's EventFilter (per-workflow, filtered, or firehose) before calling engine.subscribe.

**Acceptance:**
- subscribe.rs maps each SubscriptionRequest variant (AW-004) onto the corresponding EventFilter the engine exposes (per-workflow id, filtered by type/status/namespace, firehose).
- The namespace is authorised via the AW-007 guard before subscribe; a firehose or filtered subscription is scoped to the caller's namespace so it cannot observe another namespace's events (CO9, D8).
- engine.subscribe is called with the mapped EventFilter to obtain the stream tail; the server does not implement the broadcast itself (non-goal).
- A unit test maps each SubscriptionRequest variant to the expected EventFilter and asserts a cross-namespace firehose is denied.

### R2: Per-connection forward loop and cleanup

THE SYSTEM SHALL run a per-connection task forwarding events from the engine stream to the socket as serialised Event envelopes, until the client closes or the subscribed workflow terminates, then drop the subscription with no leak.

**Acceptance:**
- socket.rs runs a forward loop that reads from the engine subscription stream and writes serialised Event envelopes (AW-004) to the WebSocket.
- On client close, socket error, or terminal event for a per-workflow subscription, the loop ends and the subscription stream is dropped (no task or subscription leak).
- A test drives a per-workflow subscription against an InMemoryStore-backed engine, asserts events arrive, and asserts the subscription is dropped on close.
- The forward loop holds no lock across an await that could poison or deadlock; any poison maps to a typed ServerError (CO8).

### R3: Bounded buffers and lag handling

THE SYSTEM SHALL give each connection a bounded outbound buffer (size from config), close a consumer that cannot keep up with a typed lag WireError, and never apply backpressure from a subscriber to the engine broadcast.

**Acceptance:**
- Each connection's outbound buffer bound comes from config (AW-006), not a hardcoded default (CO11).
- When a consumer lags past the bound (or the engine's broadcast tail reports lag), the connection is closed with a Lagged WireError so the client can resubscribe and re-read history (D6).
- The engine broadcast is never back-pressured: a slow connection's buffer filling closes that connection only and does not block the engine or other subscribers (CO10).
- A test simulates a slow consumer and asserts it is dropped with Lagged while a second, fast consumer continues to receive events uninterrupted.
