> **Depends on:** AW-002

> - C14 — ActivityTask carries the activity type, input Payload, and the WorkflowId/ActivityId correlation; ActivityResult carries a result Payload or an ActivityError; Heartbeat carries liveness and optional progress.
> - C15 — RegisterWorker carries the worker's namespace and the activity types it implements.
> **Stories:**

> - S9 (Remote Worker, Executing Tier-3 activities out of process) — As a remote worker, I want to connect once, advertise the activity types I implement, and have tasks pushed to me so that I run heavy work without polling.

## Purpose

Define the wire contract for out-of-process Tier-3 activity workers — the protocol every worker SDK (cluster AR) generates stubs from. This is the bidirectional-streaming gRPC service: workers register, the server pushes tasks, workers report results and heartbeats.

## Task

Add worker.proto declaring a bidirectional-streaming gRPC service with RegisterWorker, ActivityTask, ActivityResult, and Heartbeat messages. ActivityTask carries the activity type, input Payload, and the WorkflowId/ActivityId correlation; ActivityResult carries a result Payload or an ActivityError; Heartbeat carries liveness and optional progress; RegisterWorker carries the worker's namespace and the activity types it implements. Provide serde types and assert lossless round-trip. This brief defines the protocol; the server-side endpoint is AW-011/AW-012.

## Requirements

### R1: Bidirectional worker service

THE SYSTEM SHALL define a gRPC service in worker.proto using bidirectional streaming, where the server pushes ActivityTask messages down the stream and the worker pushes ActivityResult and Heartbeat messages up it, opened by a RegisterWorker handshake.

**Acceptance:**
- worker.proto defines a bidirectional-streaming RPC carrying a server->worker message (ActivityTask) and a worker->server message (RegisterWorker, ActivityResult, Heartbeat) as the two stream directions.
- The transport is gRPC bidi streaming, not unary polling (per D7); the proto reflects push dispatch.
- tonic-build generates the streaming service and the crate compiles.

**Files:**
- create: crates/aion-proto/proto/worker.proto
- modify: crates/aion-proto/src/lib.rs

**Checklist:**
- C13 — worker.proto defines a bidirectional-streaming service with RegisterWorker, ActivityTask, ActivityResult, and Heartbeat messages.

### R2: Task, result, and heartbeat payloads

THE SYSTEM SHALL define ActivityTask carrying activity type, input Payload, and WorkflowId/ActivityId correlation; ActivityResult carrying a result Payload or an ActivityError; and Heartbeat carrying liveness and optional progress.

**Acceptance:**
- ActivityTask carries the activity type string, the input Payload, the owning WorkflowId, and the ActivityId for correlation.
- ActivityResult carries either a success result Payload or an ActivityError (using aion-core's ActivityError shape via the AW-002 conversion, preserving the retryable/terminal classification).
- Heartbeat carries the WorkflowId/ActivityId it concerns and an optional progress Payload.
- A unit test round-trips ActivityTask, ActivityResult (both success and error), and Heartbeat through serde and proto encode/decode.

**Files:**
- modify: crates/aion-proto/proto/worker.proto
- modify: crates/aion-proto/src/convert.rs

### R3: Worker registration advertisement

THE SYSTEM SHALL define RegisterWorker carrying the worker's namespace and the list of activity types it implements.

**Acceptance:**
- RegisterWorker carries a namespace and a repeated activity-type field.
- RegisterWorker has a serde type deriving Serialize + Deserialize and round-trips losslessly.
- The proto documents that registration scopes the worker to one namespace (server-side partitioning per D8 is AW-011).
