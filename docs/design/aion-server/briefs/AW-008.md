# AW-008: Workflow-management handler layer over the Engine

> **Cluster:** aion-server
> **Depends on:** AW-007
> **Checklist:**
> - C22 — A single handler layer translates each workflow-management request into exactly one (or a small fixed composition of) Engine call(s); it performs no retry, scheduling, sequencing, or other durable decision.
> **Stories:**
> - S5 (Workflow Caller, Driving workflows over the network) — As a workflow caller, I want to start, signal, query, cancel, list, and describe workflows over gRPC or HTTP so that I can drive the engine from any service without embedding it.

## Purpose

Implement the single handler layer that translates each workflow-management wire request into Engine calls. This is the pure-adapter core: deserialise, scope the namespace, call exactly one Engine method, serialise the result. No orchestration, no durable decision — that all stays in the engine.

## Task

Implement a transport-independent handler layer with one handler per workflow-management operation (start/signal/query/cancel/list/describe). Each handler takes the deserialised wire request, obtains a namespace-scoped engine handle via the AW-007 guard, calls exactly one (or a small fixed composition of) Engine method(s), and returns the wire response or a mapped WireError. The handlers are shared by both the gRPC and HTTP transports (AW-009).

## Requirements

### R1: Pure-adapter handlers

THE SYSTEM SHALL implement one handler per workflow-management operation that deserialises the request, scopes the namespace via the guard, calls the matching Engine method, and serialises the result, performing no retry, scheduling, sequencing, or other durable decision.

**Acceptance:**
- handlers.rs defines start/signal/query/cancel/list/describe handlers, each transport-independent (taking the decoded request + the scoped engine handle).
- start -> Engine::start_workflow, signal -> Engine::signal, query -> Engine::query, cancel -> Engine::cancel, list -> Engine::list_workflows (via WorkflowFilter), describe -> a WorkflowSummary (and optional history from the store).
- No handler retries, schedules, sequences, or composes multiple workflow operations with server-side decision logic (per D5); each is a thin translation.
- An inspection/review confirms no handler contains a loop, retry, or branch that constitutes a durable decision (CO6).

### R2: Namespace scoping and error mapping in every handler

THE SYSTEM SHALL route every handler through the AW-007 guard before calling the Engine, and SHALL map every Engine/store failure onto a WireError, never exposing engine internals.

**Acceptance:**
- Every handler calls the namespace guard first and aborts with NamespaceDenied if authorisation fails, before any Engine call (CO9).
- Engine and store errors are mapped to WireError via the AW-002 mapping; no Debug/Display of an internal error crosses the handler boundary (D3).
- A unit test drives each handler against an InMemoryStore-backed engine (aion-engine CO13) asserting the correct Engine method is invoked and the result/error is mapped correctly.
- A test asserts a handler returns NamespaceDenied without calling the engine when the guard denies.

## Boundaries

- SHALL NOT bind a transport (tonic/axum) — that is AW-009; these handlers are transport-independent.
- SHALL NOT implement WebSocket subscription or worker dispatch — AW-010, AW-011.
- SHALL NOT add orchestration or durable decisions (CO6, D5).
