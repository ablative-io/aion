# NB-001: NifContext infrastructure, WorkflowHandle interior mutability, and NIF entry registration expansion

> **Cluster:** aion-nif-bridge
> **Checklist:**
> - C1 — NifContext resolves a calling BEAM PID to its WorkflowHandle via the Registry.
> - C2 — NifContext provides synchronous access to the workflow's Recorder from a dirty NIF thread via tokio Handle::block_on.
> - C3 — WorkflowHandle wraps its Recorder in Arc<Mutex<Recorder>> for interior-mutable access from the NIF bridge.
> - C4 — engine_nifs.rs registers NIF entries for all 18 aion_flow_ffi functions.
> - C5 — Every NIF checks recorded history via the Resolver before executing a side effect.
> **Stories:**
> - S10 (Platform Operator, Running and monitoring Aion workflows in production) — As a platform operator, I want completed workflows to show a full event history so that I can audit what happened, when, and in what order.
> - S12 (Platform Operator, Running and monitoring Aion workflows in production) — As a platform operator, I want crashed workflows to be recoverable by restarting the engine so that I have Temporal-class durability guarantees.

## Purpose

Establish the foundation every NIF bridge brief depends on: a NifContext that resolves the calling BEAM process to its workflow's durability context (Recorder + Resolver), interior-mutable Recorder access on WorkflowHandle, and expanded NIF entry registration for all 18 aion_flow_ffi functions. Without this, no NIF can reach the durability layer and no side effect is recorded or replayed.

## Task

Create nif_context.rs with the NifContext struct that resolves a calling PID to a WorkflowHandle via the Registry, provides synchronous Recorder access from a dirty NIF thread via tokio Handle::block_on (D2), and carries a Resolver for replay checks (CO1). Expand engine_nifs.rs to register NIF entries for all 18 aion_flow_ffi functions (stubs for NB-002 through NB-008 that return {error, not_yet_implemented}). Confirm WorkflowHandle's existing Arc<Mutex<Recorder>> wrapping (D7) is sufficient for the NIF bridge's interior-mutable access pattern and add any missing accessor methods.

## Requirements

### R1: NifContext PID-to-handle resolution

WHEN a dirty NIF is invoked, THE SYSTEM SHALL construct a NifContext from the calling process's PID by looking up the WorkflowHandle in the active-execution Registry (D1). IF no handle exists for the PID, THEN THE SYSTEM SHALL return an error term to the BEAM process without panicking.

**Acceptance:**
- nif_context.rs defines NifContext with a constructor that takes a PID, a reference to the Registry, and a tokio Handle.
- The constructor resolves the PID to a WorkflowHandle by scanning the Registry's active handles; returns Result<NifContext, NifContextError> where the error case is UnknownProcess.
- NifContext exposes workflow_id(), run_id(), and recorder() accessors derived from the resolved handle.
- A unit test constructs a Registry with a registered handle, resolves the PID to a NifContext, and asserts the workflow_id matches; a second test uses an unregistered PID and asserts UnknownProcess.

### R2: Synchronous Recorder access from dirty NIF thread

WHEN the NifContext needs to record or resolve events, THE SYSTEM SHALL use tokio Handle::block_on to run async Recorder methods synchronously from the dirty NIF thread (D2). THE SYSTEM SHALL NOT spawn async tasks that outlive the NIF call.

**Acceptance:**
- NifContext carries a tokio::runtime::Handle and exposes a block_on_recorder() or equivalent method that locks the Arc<Mutex<Recorder>> and executes an async closure synchronously via Handle::block_on.
- The method returns the closure's Result, propagating DurabilityError as NifContextError::Durability.
- A test verifies that a Recorder method (e.g. current_head()) can be called synchronously through the NifContext's block_on path without deadlock.
- No async task is spawned that outlives the NIF call scope.

**Acceptance:**

- run_activity/3 retains its existing dirty registration; all other NIFs are registered dirty (they may block on service calls via block_on).
- A test asserts engine_nif_entries().len() == 18 and that each MFA is unique.

### R5: Replay-check contract for every NIF

THE SYSTEM SHALL provide on NifContext a resolve_command() method (or equivalent) that wraps the Resolver's resolve() call so every NIF can check recorded history before executing a side effect (CO1). WHEN the Resolver returns a recorded result, THE SYSTEM SHALL return it to the NIF without executing the live path.

**Acceptance:**
- NifContext exposes a method that accepts a Command and returns ResolveOutcome (Recorded or ResumeLive).
- The method delegates to a Resolver constructed from the workflow's recorded history via the HistoryCursor.
- NifContext construction builds or receives the Resolver from the workflow's durability state so each NIF call can check history.
- A test constructs a NifContext over a history containing a recorded activity event, calls resolve_command with a matching Command, and asserts the Recorded resolution is returned.

### R6: NifContextError taxonomy

THE SYSTEM SHALL define NifContextError as a thiserror enum with variants UnknownProcess, RecorderPoisoned, Durability wrapping DurabilityError, and TermEncoding. THE SYSTEM SHALL NOT use anyhow.

**Acceptance:**
- nif_context.rs defines NifContextError with at least UnknownProcess { pid: u64 }, RecorderPoisoned, Durability(DurabilityError), and TermEncoding { reason: String }.
- The error type derives thiserror::Error and Debug.
- From<DurabilityError> is implemented for the ? operator.
- A helper method converts NifContextError to a BEAM error term for return from any NIF.

## Boundaries

- SHALL NOT implement the production logic for any specific NIF (activity, timer, signal, etc.) — those are NB-002 through NB-008.
- SHALL NOT change the WorkflowHandle's existing Arc<Mutex<Recorder>> wrapping — it is already the correct D7 implementation.
- SHALL NOT import beamr outside the runtime module (CO8).
- SHALL NOT add any #[allow], #[expect], or #[ignore] directives (CO6).
- SHALL NOT use unsafe code (CO5).
- SHALL NOT add networking or transport dependencies to the aion crate (CO4).
