## Purpose

Close the workflow lifecycle gap: when a BEAM workflow process exits (normal return or crash), the engine must detect the exit, record the terminal event (WorkflowCompleted or WorkflowFailed), update the registry, and fulfil the completion notifier. Without this, workflows stay in Running status forever after their process dies — the zombie workflow problem (C36). This is the final piece needed for end-to-end workflow lifecycle: start, run activities, complete, describe full history.

## Task

Create lifecycle/completion.rs with process exit detection using beamr's process monitoring (D5). Wire the monitor on workflow spawn in lifecycle/start.rs. On normal exit, extract the return value and record WorkflowCompleted through the Recorder (CO2). On abnormal exit, extract the exit reason and record WorkflowFailed. After recording, update the WorkflowHandle's cached status in the Registry (C35) and fulfil the CompletionNotifier. The monitor callback runs outside the workflow's dirty NIF thread — it accesses the Recorder through the handle's Arc<Mutex<Recorder>>.

## Requirements

### R1: Process monitor installed on workflow spawn

WHEN a workflow process is spawned, THE SYSTEM SHALL install a beamr process monitor on the PID so that process exit is detected automatically (D5). THE SYSTEM SHALL NOT poll for process liveness.

**Acceptance:**
- lifecycle/start.rs is modified to call RuntimeHandle::monitor_process (or equivalent) on the spawned PID immediately after successful spawn and registry insertion.
- The monitor callback is a function or closure that receives the exit reason and has access to the WorkflowHandle (via Registry lookup or captured reference).
- No polling loop or periodic liveness check exists.
- A test spawns a workflow process, asserts the monitor is installed, kills the process, and asserts the monitor callback fires.

### R2: Normal exit records WorkflowCompleted

WHEN a workflow process exits normally (return value), THE SYSTEM SHALL extract the return value as the result Payload and record WorkflowCompleted through the Recorder (CO2). THE SYSTEM SHALL NOT record WorkflowCompleted for abnormal exits.

**Acceptance:**
- completion.rs implements a handle_process_exit function (or equivalent) that distinguishes normal from abnormal exit.
- On normal exit, the return value is decoded into a Payload and WorkflowCompleted is recorded through the handle's Arc<Mutex<Recorder>> via tokio Handle::block_on.
- The WorkflowCompleted event's result payload matches the process return value.
- A test spawns a workflow process that returns a value, waits for exit, and asserts WorkflowCompleted is the last event in the store with the correct result.

### R3: Abnormal exit records WorkflowFailed

WHEN a workflow process exits abnormally (crash, error, killed), THE SYSTEM SHALL extract the exit reason as a WorkflowError and record WorkflowFailed through the Recorder (CO2). THE SYSTEM SHALL NOT record WorkflowFailed for normal exits.

**Acceptance:**
- On abnormal exit, the exit reason is decoded into a WorkflowError and WorkflowFailed is recorded through the handle's Recorder.
- The WorkflowFailed event's error preserves the crash reason (class, term, stacktrace where available).
- A test spawns a workflow process that crashes, waits for exit, and asserts WorkflowFailed is the last event in the store with the crash reason.

### R4: Registry status updated after terminal event

AFTER recording the terminal event, THE SYSTEM SHALL update the WorkflowHandle's cached_status in the Registry to reflect the terminal status (Completed or Failed) (C35). THE SYSTEM SHALL set the residency to Suspended since the process is no longer alive.

**Acceptance:**
- After recording WorkflowCompleted, the handle's cached_status is updated to WorkflowStatus::Completed and residency to Residency::Suspended.
- After recording WorkflowFailed, the handle's cached_status is updated to WorkflowStatus::Failed and residency to Residency::Suspended.
- The registry update happens after the durable event is recorded (crash between record and update is safe because status is a projection that can be recomputed).
- A test completes a workflow via normal exit and asserts the registry shows Completed + Suspended; a test crashes a workflow and asserts Failed + Suspended.

### R5: CompletionNotifier fulfilled on terminal event

AFTER recording the terminal event, THE SYSTEM SHALL fulfil the WorkflowHandle's CompletionNotifier with the terminal outcome so that any pending Engine::result() awaiter is unblocked.

**Acceptance:**
- After recording WorkflowCompleted, the CompletionNotifier is fulfilled with TerminalOutcome::Completed(result_payload).
- After recording WorkflowFailed, the CompletionNotifier is fulfilled with TerminalOutcome::Failed(error).
- A test subscribes to the completion notifier before exit, verifies the subscription blocks, then triggers exit and asserts the subscriber wakes with the correct outcome.
- A test subscribes after exit and asserts the outcome is immediately available (no hang).

### R7: Monitor callback does not run on workflow's dirty NIF thread

THE SYSTEM SHALL run the process exit monitor callback outside the workflow's dirty NIF thread. The callback accesses the Recorder through the handle's Arc<Mutex<Recorder>> and the tokio runtime through a captured Handle. THE SYSTEM SHALL NOT deadlock if the NIF thread and the monitor callback contend for the Recorder mutex.

**Acceptance:**
- The monitor callback runs on beamr's scheduler thread or a dedicated monitor thread, not on the workflow's dirty NIF thread.
- The callback acquires the Recorder mutex, records the terminal event, and releases it.
- A test verifies that a workflow whose NIF thread is blocking (e.g. on a sleep) can still have its exit detected and recorded when the process is killed externally.
- No deadlock occurs when the NIF thread holds the Recorder and the monitor fires concurrently.

## Boundaries

- SHALL NOT poll for process liveness — use beamr's process monitoring (D5).
- SHALL NOT bypass the Recorder to call EventStore::append directly (CO2).
- SHALL NOT compute WorkflowStatus independently — status is a projection of the terminal event appended to history.
- SHALL NOT import beamr outside the runtime module (CO8) — RuntimeHandle wraps monitor installation.
- SHALL NOT handle workflow restart or recovery here — that is the recovery module (AD).
- SHALL NOT handle continue-as-new transitions — those have their own lifecycle path.
