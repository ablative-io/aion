# NB-006: Query NIFs (register_query, reply_query, dispatch_query)

> **Cluster:** aion-nif-bridge
> **Depends on:** NB-001
> **Checklist:**
> - C22 — register_query/3 NIF registers a query handler for the workflow.
> - C23 — reply_query/2 NIF sends a query response.
> - C24 — dispatch_query/2 NIF dispatches a query to a workflow and returns the response.
> **Stories:**
> - S6 (Workflow Author, Writing durable workflows in Gleam using the aion_flow SDK) — As a workflow author, I want to register query handlers so that external callers can inspect my workflow's current state without modifying it.

## Purpose

Implement the query NIFs that workflow code calls to register query handlers, send query responses, and dispatch queries to other workflows. Queries are read-only inspections of workflow state — they do not modify history and are not replayed (they are evaluated against current in-memory state). The NIFs delegate to AT's QueryService (D4) for handler registration, response delivery, and cross-workflow dispatch.

## Task

Create nif_query.rs with production NIF implementations for register_query/3, reply_query/2, and dispatch_query/2. register_query registers a handler function with the QueryService for the current workflow. reply_query sends a response to a pending query. dispatch_query sends a query to another workflow and blocks until the response arrives. Replace the stubs in engine_nifs.rs.

## Requirements

### R1: register_query/3 registers a handler with QueryService

WHEN register_query/3 is called, THE SYSTEM SHALL register a query handler for the given query name on the current workflow through AT's QueryService (D4). THE SYSTEM SHALL associate the handler with the workflow's PID so incoming queries are routed to the correct process.

**Acceptance:**
- nif_query.rs implements register_query_impl that takes a query name, handler reference, and config, constructs a NifContext, and registers the handler with crate::query::QueryService.
- The registration associates the query name with the workflow's PID (from NifContext) and the handler callback.
- Duplicate registration for the same query name on the same workflow replaces the previous handler.
- A test registers a query handler and asserts the QueryService knows about it for the workflow.

### R2: reply_query/2 sends a response to a pending query

WHEN reply_query/2 is called, THE SYSTEM SHALL deliver the response payload to the pending query caller through AT's QueryService (D4). IF no pending query exists for the given query ID, THEN THE SYSTEM SHALL return an error term.

**Acceptance:**
- nif_query.rs implements reply_query_impl that takes a query ID and response payload, constructs a NifContext, and delivers the response through crate::query::QueryService.
- On success, the NIF returns {ok, replied} to the BEAM process.
- On failure (no pending query), the NIF returns {error, reason} without panicking.
- A test dispatches a query, calls reply_query from the handler side, and asserts the response is delivered.

### R3: dispatch_query/2 sends a query and awaits the response

WHEN dispatch_query/2 is called, THE SYSTEM SHALL dispatch a query to the target workflow through AT's QueryService and block the dirty NIF thread until the response arrives (D4). IF the target workflow has no handler for the query name, THEN THE SYSTEM SHALL return an error term.

**Acceptance:**
- nif_query.rs implements dispatch_query_impl that takes a target workflow identifier and query name+payload, constructs a NifContext, and dispatches through crate::query::QueryService.
- The NIF blocks via Handle::block_on until the query response is delivered.
- On success, the NIF returns {ok, response_payload} to the BEAM process.
- On failure (unknown workflow, unregistered handler, timeout), the NIF returns {error, reason}.
- A test registers a handler on one workflow, dispatches a query from another, and asserts the response round-trips correctly.

### R4: Queries are not recorded or replayed

THE SYSTEM SHALL NOT record query-related events in the durable event history. Queries are read-only state inspections evaluated against current in-memory state; they do not participate in the replay contract.

**Acceptance:**
- No query NIF calls Recorder::record_* methods.
- No query NIF calls Resolver::resolve or resolve_or_execute_live.
- No Query* event type appears in the store after query interactions.
- A test verifies that register_query, reply_query, and dispatch_query produce no events in the store.

## Boundaries

- SHALL NOT record query events — queries are read-only and do not participate in replay.
- SHALL NOT reimplement query routing, handler registration, or response delivery — those are AT's QueryService (D4).
- SHALL NOT modify the QueryService type.
- SHALL NOT import beamr outside the runtime module (CO8).
- SHALL NOT use unsafe code (CO5).
