## Purpose

Expose AD-012's engine reopen operation across the full wire surface so an operator can actually invoke it: a ReopenWorkflow gRPC RPC, the matching HTTP endpoint, a client SDK method, and an `aion reopen <id>` CLI subcommand — mirroring the existing cancel operation exactly. AD-012 builds Engine::reopen_workflow but gives it no caller; engine-only is half a feature. This brief is the operator's hand on the reopen lever. It is wire-and-CLI only: it adds no engine behaviour, and it carries whatever the engine reopens unchanged (so when AD-013 widens reopen to Cancelled, this surface needs no edit).

## Task

Mirror the cancel operation end-to-end for reopen. (1) Proto: add a Reopen RPC to the workflow gRPC service in crates/aion-proto/proto/workflow.proto with ReopenRequest (namespace, workflow_id, optional run_id — reusing common.proto WorkflowId/RunId) and a ReopenResponse that returns the reopened run id and its projected Running status (not an empty ack — the caller learns the run is live); add a dedicated WireErrorCode INVALID_STATE in common.proto for the AD-012 precondition failures. (2) gRPC handler: WorkflowGrpcService::reopen in aion-server/src/api/grpc.rs, decoding the request, calling the shared handler, mapping EngineError (including the new EngineError::InvalidState from AD-012) to a tonic Status via the grpc_code path. (3) HTTP handler reopen_workflow in aion-server/src/api/http/workflows.rs + route POST /workflows/reopen in api/http/router.rs. (4) Shared handler reopen() in aion-server/src/api/handlers/workflows.rs that resolves the run id (latest if omitted) and calls Engine::reopen_workflow, surfacing the engine's terminal-Failed precondition rather than re-checking status. (5) Extend wire_from_engine in aion-server/src/error.rs to map EngineError::InvalidState to the new INVALID_STATE wire code (distinct from NotFound and Backend). (6) Client SDK Client::reopen in aion-client/src/ops.rs + GrpcWorkflowTransport::reopen + encode/decode helpers in transport/convert.rs. (7) CLI: an `aion reopen <workflow-id> [--run-id]` subcommand in aion-cli/src/main.rs, dispatched to a reopen_workflow() function that prints the now-Running run or the typed failure. (8) Several touched files are already at or over the 500-code-line limit (grpc.rs ~914, client ops.rs ~924, cli main.rs ~868, handlers/workflows.rs ~536) — split by responsibility (per-operation modules) so no file crosses the limit because of this brief. Cover the happy path and every error path (not-found, not-terminal, non-Failed terminal, already-Running) at the wire and CLI layers with real tests.

## Requirements

### R1: Wire contract: Reopen RPC, messages, and a typed precondition code

THE SYSTEM SHALL add a Reopen RPC to the workflow gRPC service with request and response messages mirroring Cancel, and SHALL extend the wire error taxonomy so the AD-012 reopen precondition failures cross the wire as a distinct, typed code separable from not-found and backend errors.

**Acceptance:**
- crates/aion-proto/proto/workflow.proto declares `rpc Reopen(ReopenRequest) returns (ReopenResponse);` alongside Cancel; ReopenRequest carries namespace, workflow_id (common.WorkflowId), and an optional run_id (common.RunId; absent means the latest run).
- ReopenResponse is NOT empty: it returns the reopened run_id and the projected status (Running) so the caller learns the run is live again — distinct from CancelResponse which is an empty ack.
- A dedicated WireErrorCode INVALID_STATE is added in crates/aion-proto/proto/common.proto for the reopen precondition failures (run not terminal, terminal for a non-Failed reason, or already Running); it is distinct from NOT_FOUND and BACKEND.
- The proto regenerates cleanly via the existing tonic-build path and the new messages round-trip through the encode/decode helpers added beside the cancel helpers.

### R2: Server handlers: gRPC, HTTP, and the shared reopen handler

THE SYSTEM SHALL implement the Reopen RPC on the gRPC service, the matching HTTP endpoint and route, and one shared handler that resolves the run, calls Engine::reopen_workflow, and maps every EngineError to the correct wire and HTTP code — mirroring cancel, but surfacing the engine's terminal-Failed precondition rather than re-checking status in the handler.

**Acceptance:**
- WorkflowGrpcService::reopen in crates/aion-server/src/api/grpc.rs decodes the request, calls the shared handler, and maps the result or error to a tonic Status through the existing status_from_wire_error / grpc_code path (INVALID_STATE maps to FailedPrecondition).
- reopen_workflow in crates/aion-server/src/api/http/workflows.rs returns the reopened run + status as JSON, and is registered as POST /workflows/reopen in crates/aion-server/src/api/http/router.rs; INVALID_STATE maps to HTTP 409 Conflict via the HttpWireError wrapper, NotFound to 404.
- The shared handler reopen() in crates/aion-server/src/api/handlers/workflows.rs resolves the run id via resolve_run_id (latest if omitted) and calls engine.reopen_workflow; UNLIKE cancel's terminal guard it does NOT pre-check status — the terminal-Failed precondition is the engine's (AD-012) and the handler only surfaces its typed error.
- crates/aion-server/src/error.rs wire_from_engine maps EngineError::InvalidState -> WireErrorCode::InvalidState with error_type "InvalidState" and the reason message; EngineError::WorkflowNotFound still maps to NotFound.
- Encode/decode helpers for ReopenRequest/ReopenResponse are added beside the cancel helpers.

## Boundaries

- SHALL NOT change Engine::reopen_workflow or any AD-012 engine behaviour — this brief only exposes it across the wire and CLI.
- SHALL NOT add reopen for Cancelled or TimedOut runs — the engine reopens only Failed in this cut (AD-012); when AD-013 widens it, this surface carries the wider engine behaviour unchanged with no edit here.
- SHALL NOT add the failed_step / failure_reason summary columns or any aion list output change — that is the visibility brief.
- SHALL NOT introduce automatic retry, classification, or any RetryPolicy plumbing — that is the L1+L2 follow-on.
- SHALL NOT alter the existing start/cancel/signal/query wire surface except to share helpers and module structure (the per-operation splits the 500-line limit forces).
- SHALL NOT make the server re-implement the terminal-Failed precondition — the decision is the engine's; the server and CLI only carry and surface its typed result.
