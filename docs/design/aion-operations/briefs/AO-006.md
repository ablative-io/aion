## Purpose

Enforce tenant isolation by validating JWT tokens against a JWKS endpoint and scoping all operations to the token's namespace claim. Without this, any client that can reach the port can operate in any namespace — a security gap for multi-tenant deployments.

## Task

Implement a Tower middleware that extracts and validates Bearer tokens from HTTP Authorization headers and gRPC metadata. Validate against a cached JWKS endpoint. Extract the namespace claim and enforce it on all downstream operations. Feature-gate behind cfg(feature='auth') so the base binary stays lean. Health and metrics endpoints are exempt from auth.

## Requirements

### R1: Token extraction and validation middleware

WHEN auth.enabled=true, THE SYSTEM SHALL extract the Bearer token from the HTTP Authorization header (or gRPC authorization metadata) on every request except /health/* and /metrics. THE SYSTEM SHALL validate the token's signature against the configured JWKS endpoint. IF the token is missing, expired, or signature-invalid, THE SYSTEM SHALL reject the request with HTTP 401 / gRPC UNAUTHENTICATED before the request reaches any handler.

**Acceptance:**
- A request without Authorization header returns 401.
- A request with an expired token returns 401.
- A request with an invalid signature returns 401.
- A request with a valid token passes through to the handler.
- /health/live and /health/ready are accessible without a token.
- /metrics is accessible without a token.
- For long-lived worker streams, token validity is re-checked on each heartbeat; an expired token triggers a re-auth-required message and stream closure after one heartbeat-interval grace period (derived from existing heartbeat config, not hardcoded).

### R2: Namespace claim enforcement

THE SYSTEM SHALL extract the namespace claim from the validated token and scope all operations to that namespace. A token with namespace=A SHALL NOT be able to start, signal, query, cancel, list, or stream workflows in namespace B. IF the request body includes a namespace field that differs from the token's namespace claim, THE SYSTEM SHALL reject the request with 403/PERMISSION_DENIED (not silently override). THE SYSTEM SHALL pass the authenticated namespace to the existing NamespaceGuard mechanism.

**Acceptance:**
- A token with namespace='tenant-a' can start workflows in tenant-a.
- The same token cannot list workflows in tenant-b (returns 403/PERMISSION_DENIED).
- Worker connections are scoped to the token's namespace for task dispatch.
- The namespace from the token overrides any namespace field in the request body.

### R3: JWKS caching with configurable refresh

THE SYSTEM SHALL fetch and cache JWKS keys from the configured URL at startup and refresh them on a configurable interval. WHEN a token references an unknown key ID (kid), THE SYSTEM SHALL attempt one immediate refresh before rejecting. Key rotation SHALL NOT require server restart.

**Acceptance:**
- JWKS keys are fetched at startup from auth.jwks_url.
- A token signed with a rotated key validates after automatic refresh.
- The refresh interval is configurable (not hardcoded).
- A JWKS endpoint that is temporarily unavailable uses cached keys (does not reject all requests).
- An unknown kid triggers one immediate refresh attempt before 401.

### R4: Feature gate and conditional compilation

THE SYSTEM SHALL compile the auth module only when cfg(feature='auth') is active. THE SYSTEM SHALL NOT add jsonwebtoken, reqwest, or JWKS-related dependencies to the default feature set. IF auth.enabled=true in config but the auth feature is not compiled in, THE SYSTEM SHALL emit a startup error and exit with code 2 (message: 'auth.enabled=true but binary compiled without auth feature'). WHEN the feature is inactive and auth.enabled is false or absent, the server behaves normally without authentication.

**Acceptance:**
- cargo build (without --features auth) succeeds and produces a binary without auth dependencies.
- cargo build --features auth adds the auth module and its dependencies.
- With the auth feature active and auth.enabled=false in config, no authentication is enforced.
- With the auth feature inactive, setting auth.enabled=true in config is silently ignored (no panic).

## Boundaries

- SHALL NOT implement token issuance — the server validates, not issues.
- SHALL NOT implement OAuth2 flows (authorization code, PKCE, etc.) — tokens arrive pre-issued.
- SHALL NOT add auth dependencies to the aion engine crate (CO4).
- SHALL NOT implement per-endpoint granular permissions beyond namespace scoping — all operations within a namespace are permitted.
- SHALL NOT implement user/role management — that is the identity provider's domain.
