# AO-002: Graceful shutdown with activity drain on SIGTERM

> **Cluster:** aion-operations
> **Depends on:** AO-003
> **Checklist:**
> - C5 — On SIGTERM or SIGINT, the server stops accepting new connections and new workflow starts within 100ms of signal receipt.
> - C6 — Connected workers receive a drain signal indicating no new tasks will be dispatched; workers finish in-flight tasks.
> - C7 — The server awaits completion of all in-flight activities up to the configured drain timeout before shutting down the engine.
> - C8 — Activities that exceed the drain timeout are surfaced as retryable failures via the existing LostWorkerReport path.
> - C9 — Clean drain exits with code 0; timeout-exceeded drain exits with code 1.
> - C10 — No event writes are lost during shutdown — the engine flushes all pending appends before the store connection closes.
> **Stories:**
> - S4 (Platform Operator, Deploying and operating Aion in production) — As a platform operator, I want graceful shutdown on SIGTERM so that rolling deploys do not lose in-flight work or require manual intervention.

## Purpose

Ensure rolling deploys lose zero in-flight work by implementing coordinated shutdown: stop accepting traffic, drain connected workers, await activity completion, then shutdown the engine cleanly.

## Task

Implement a shutdown coordinator in aion-server that listens for SIGTERM/SIGINT, stops the listener, signals workers to drain, awaits in-flight activity completion up to a configurable timeout, then shuts down the engine. Activities that exceed the drain timeout surface as retryable failures via the existing LostWorkerReport path.

## Requirements

### R1: Signal handler triggers drain sequence

WHEN the process receives SIGTERM or SIGINT, THE SYSTEM SHALL initiate the drain sequence within 100ms: stop the TCP listener from accepting new connections and reject new workflow start requests with a 503/UNAVAILABLE status. THE SYSTEM SHALL NOT exit immediately on first signal receipt. IF a second SIGTERM or SIGINT arrives during drain, THE SYSTEM SHALL exit immediately with code 130 without waiting for in-flight activities.

**Acceptance:**
- SIGTERM triggers drain (verified by log output and behaviour).
- New HTTP connections after drain starts receive connection refused or 503.
- New gRPC calls after drain starts receive UNAVAILABLE status.
- The process remains alive during drain to serve in-flight requests.

### R2: Worker drain signal and activity await

WHEN drain begins, THE SYSTEM SHALL send a DrainRequest message (a new message type in aion-proto's worker protocol) to all connected workers indicating no new tasks will be dispatched. THE SYSTEM SHALL await completion of all in-flight activities up to the configured drain timeout. THE SYSTEM SHALL NOT dispatch new activity tasks after drain begins.

**Acceptance:**
- Connected workers receive a drain message on their task stream.
- No new ActivityTask messages are sent after drain starts.
- In-flight activities that complete during drain are recorded normally.
- The drain coordinator tracks the count of in-flight activities and resolves when it reaches zero.

### R3: Drain timeout surfaces retryable failures and exits

IF in-flight activities do not complete within the configured drain timeout, THEN THE SYSTEM SHALL surface remaining activities as retryable failures via LostWorkerReport, flush all pending event writes, shut down the engine, and exit with code 1. WHEN all activities complete before the timeout, THE SYSTEM SHALL shut down the engine and exit with code 0.

**Acceptance:**
- Activities exceeding the timeout are reported as retryable failures.
- On restart, those activities are re-dispatched by the engine's retry policy.
- No pending event writes are lost — the store connection closes only after flush.
- Exit code is 0 for clean drain, 1 for timeout.
- The drain timeout is configurable (not hardcoded).

## Boundaries

- SHALL NOT implement distributed drain coordination — this is single-node shutdown only.
- SHALL NOT add a drain timeout default — it must come from configuration (AO-003).
- SHALL NOT modify the Engine's shutdown() method — the server coordinates around it.
- SHALL NOT force-kill worker connections — workers are notified and given time to finish.
