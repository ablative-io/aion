## Purpose

Enable operators to configure aion-server from a TOML file, environment variables, and CLI flags without rebuilding the binary. Provide sensible development defaults so local testing requires zero configuration.

## Task

Implement a configuration module in aion-server that loads aion.toml, overlays AION_ prefixed environment variables, and accepts CLI flag overrides. Validate the merged config at startup with clear error messages for invalid values. Wire the config into ServerState initialization replacing hardcoded values.

## Requirements

### R1: TOML configuration file parsing

WHEN aion-server starts, THE SYSTEM SHALL search for aion.toml in the working directory (or a path specified by --config CLI flag), parse it into a typed ServerConfig struct covering all sections (server, store, runtime, drain, auth, metrics, namespaces), and validate all values. The auth section SHALL include enabled, jwks_url, and jwks_refresh_seconds fields. IF the file is absent and no --config flag is given, THE SYSTEM SHALL use default values suitable for local development.

**Acceptance:**
- A valid aion.toml is parsed into ServerConfig without error.
- An invalid TOML file produces a clear error naming the problematic field and exits with code 2.
- A missing file (without --config) silently uses defaults.
- A missing file with --config pointing to a nonexistent path exits with code 2.
- ServerConfig has typed fields for every configurable value (no stringly-typed config).

### R2: Environment variable overlay

THE SYSTEM SHALL overlay AION_ prefixed environment variables onto the parsed TOML config. Nested keys use underscore separation (AION_STORE_URL maps to store.url). THE SYSTEM SHALL support env var overrides for config keys up to 2 levels of nesting (section.key); deeper keys (e.g. store.connection_pool.max_size) require the TOML file. Environment values override TOML values for the same key. THE SYSTEM SHALL NOT require environment variables — they are optional overrides.

**Acceptance:**
- AION_STORE_URL=postgres://... overrides store.url from the TOML file.
- AION_SERVER_LISTEN_ADDRESS overrides server.listen_address.
- AION_RUNTIME_SCHEDULER_THREADS overrides runtime.scheduler_threads as a parsed integer.
- An unparseable environment value (e.g. non-integer for a numeric field) exits with code 2 naming the variable.
- Unset environment variables leave TOML values unchanged.

### R3: CLI flag precedence and development defaults

THE SYSTEM SHALL accept CLI flags (--listen-address, --store-url, --scheduler-threads, --drain-timeout, --config) that override both environment and file values. Precedence: CLI > env > file > defaults. THE SYSTEM SHALL provide development defaults (in-memory store, localhost listen, 1 scheduler thread) that produce a working server with no config at all.

**Acceptance:**
- --store-url overrides both AION_STORE_URL and store.url from TOML.
- Running aion-server with no flags, no env, and no config file starts a development server on localhost with an in-memory store.
- The development server is fully functional for local workflow testing.
- --config /path/to/custom.toml loads from that specific file.

### R4: Wire config into ServerState initialization

THE SYSTEM SHALL replace all hardcoded values in ServerState construction with values from the merged ServerConfig. The EngineBuilder receives scheduler_threads, store backend, and other engine-level config from ServerConfig. The HTTP/gRPC listeners bind to configured addresses.

**Acceptance:**
- state.rs reads from ServerConfig, not from hardcoded constants.
- Changing store.url in config changes the backing store without code changes.
- Changing server.listen_address changes the bind address.
- scheduler_threads from config flows through to EngineBuilder.

## Boundaries

- SHALL NOT validate store connectivity at config time — that is the readiness probe's job (AO-004).
- SHALL NOT implement hot-reload of configuration — config is loaded once at startup.
- SHALL NOT add configuration for auth settings beyond the field definitions — auth validation is AO-006.
- SHALL NOT use a configuration crate that pulls in heavy dependencies (no figment, no config-rs) — TOML parsing via toml + serde is sufficient.
