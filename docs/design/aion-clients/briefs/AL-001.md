## Purpose

Pin down, language-neutrally, exactly what each of the seven client operations means before any SDK is written. This contract is the oracle every SDK is measured against; without it, four SDKs drift into four subtly different and subtly wrong clients. It also defines the shared conformance scenario set that becomes the executable agreement gate.

## Task

Author CLIENT-CONTRACT.md: a prose specification of connect, start, signal, query, cancel, list, describe, and subscribe — their inputs, outputs, and errors — plus the precise semantics of the hard cases (start idempotency, run targeting, query timeout, cancellation-as-request, stream resumption) and the shared error taxonomy. Then author conformance/aion-clients/scenarios.json: a machine-readable scenario set encoding the happy path and the key edge cases, in a form each SDK's harness can drive. This brief writes documents and a data file only — no SDK code. It consumes the aion-server API and aion-proto types defined in AW; it does not redefine the wire format.

## Requirements

### R1: Operation catalogue

THE SYSTEM SHALL document the seven operations (connect, start, signal, query, cancel, list, describe, subscribe), each with its inputs, its return value, and the errors it can raise. THE SYSTEM SHALL NOT redefine the aion-proto wire format; it SHALL reference the AW-owned request/response types as the transport.

**Acceptance:**
- CLIENT-CONTRACT.md has one section per operation, each stating inputs, output, and possible errors.
- connect documents endpoint, auth credential, and TLS configuration as its inputs and a reusable client as its output.
- list documents the filter dimensions (type, status, time range, parent) and pagination; describe documents returning a summary plus status and optional history.
- Each section names the aion-proto request/response type it maps to (by name) rather than restating its fields.
- A reviewer can enumerate the full client surface from this document alone.

### R2: Hard-case semantics

THE SYSTEM SHALL specify start idempotency, run targeting, query timeout, cancellation semantics, and subscribe resumption precisely enough that two independent implementations behave identically. THE SYSTEM SHALL state that cancellation is a recorded cooperative request, not an immediate kill, and SHALL NOT describe query as fire-and-forget.

**Acceptance:**
- Start idempotency: a retried start with the same idempotency key and identical request returns the original handle; same key with a different request raises AlreadyExists. Stated unambiguously with both outcomes.
- Run targeting: signal, query, cancel, and describe target the latest run by default and a specific RunId when supplied.
- Query: a synchronous round-trip bounded by a caller deadline; QueryTimeout when the deadline elapses, QueryFailed when the workflow's handler errors.
- Cancellation: returns once the cancel request is recorded; the workflow stops cooperatively, so a returned cancel does not assert the workflow has already stopped.
- Subscribe resumption: on transient disconnect the client resumes from the last delivered event's per-workflow sequence number, delivering no gaps and no duplicates; on terminal failure the stream surfaces an error rather than ending silently.

### R3: Shared error taxonomy

THE SYSTEM SHALL define the closed set of distinguishable client failures and the server condition each maps from, so every SDK renders the same set idiomatically.

**Acceptance:**
- The taxonomy lists exactly: NotFound, AlreadyExists, QueryFailed, QueryTimeout, Cancelled, Unavailable, Unauthenticated, NamespaceDenied, InvalidArgument, Server.
- Each variant states the server/transport condition it maps from (e.g. AlreadyExists <- idempotency-key conflict; Unavailable <- transport/connection failure; Server <- unexpected server error carrying the server's detail).
- The document states that no SDK may collapse distinct variants into one opaque error and no SDK may swallow an error.
- Each variant is annotated with whether it is retryable in principle (e.g. Unavailable yes, InvalidArgument no).

### R4: Conformance scenario set

THE SYSTEM SHALL define a machine-readable scenario set encoding the happy path and the key edge cases in a form each SDK's harness can drive against a real aion-server.

**Acceptance:**
- scenarios.json encodes a happy-path sequence: start -> signal -> query -> list -> describe -> cancel -> subscribe, with the expected observable result of each step.
- scenarios.json encodes edge cases: idempotency-conflict (start with reused key), query-timeout, not-found (operate on an unknown workflow), and disconnect-resume (subscribe across a forced reconnect).
- Each scenario step names the operation, its inputs, and the expected output or error variant from the taxonomy in R3.
- The structure is language-neutral data (no language-specific assertions) so all four harnesses can consume it.
- scenarios.json is valid JSON and each referenced error variant exists in the R3 taxonomy.
