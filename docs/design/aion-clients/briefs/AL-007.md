# AL-007: Cross-language conformance harnesses against a live aion-server

> **Cluster:** aion-clients
> **Depends on:** AL-001, AL-002, AL-003, AL-004, AL-005, AL-006
> **Blocked by:** AW: a runnable aion-server fixture must be available to test against
> **Checklist:**
> - C27 — Each SDK has a conformance harness that runs the shared scenarios against a real aion-server fixture and asserts the observable results match the contract.
> - C28 — The four SDKs produce identical observable behaviour for every shared scenario (same effect, same returned values, same error variant).
> **Stories:**
> - S10 (Polyglot Platform Team, Consuming Aion from Multiple Languages) — As a platform team, I want the Python, TypeScript, Gleam, and Rust clients to behave identically against the same server so that teams in different languages can reason about Aion the same way.

## Purpose

Turn the AL-001 contract into an executable agreement gate. Each SDK gets a harness that drives the shared scenario set against a real aion-server and asserts the observable results match the contract, and the four are compared so identical behaviour is proven, not assumed. This is what keeps the four clients from drifting.

## Task

Implement a conformance harness in each ecosystem (Rust, Python, TypeScript, Gleam) that loads conformance/aion-clients/scenarios.json, runs each scenario through that SDK against a live aion-server fixture, and asserts each step's observable result (returned value or error variant) matches the scenario's expectation. Provide a runner doc describing how to bring up the server fixture and run all four harnesses. Assert cross-SDK equivalence on the shared scenarios. These tests connect to a real server via a runtime env-gate (read a server-URL env var; emit a skip log and pass trivially when unset) — no #[ignore], no feature-gated skips.

## Requirements

### R1: Scenario loader and assertion model

THE SYSTEM SHALL provide, in each ecosystem, a loader that reads scenarios.json and an assertion model that compares an executed step's observable result against the scenario's expected value or error variant. THE SYSTEM SHALL NOT hardcode scenarios in any single language — the data file is the source of truth.

**Acceptance:**
- Each harness reads conformance/aion-clients/scenarios.json (not a per-language copy) and iterates its scenarios and steps.
- For each step the harness records the observable result: the returned value (normalised to JSON for comparison) or the error variant name from the shared taxonomy.
- An assertion compares the recorded result to the scenario's expected result and fails the test with the scenario id and step on mismatch.
- Adding a scenario to scenarios.json causes all four harnesses to exercise it with no per-language code change.

### R2: Happy-path and edge-case execution against a live server

WHILE a server-URL environment variable is set, THE SYSTEM SHALL run the full happy path (start -> signal -> query -> list -> describe -> cancel -> subscribe) and the edge cases (idempotency-conflict, query-timeout, not-found, disconnect-resume) against the live aion-server and assert the contract's expected results. WHILE the variable is unset, THE SYSTEM SHALL emit a skip log and pass trivially.

**Acceptance:**
- Each harness reads a documented env var (e.g. AION_SERVER_TEST_URL); when set it connects and runs all scenarios, when unset it logs a skip line and returns success without failing.
- The happy path asserts: start yields a handle; signal is accepted; query returns the expected reply; list includes the workflow; describe shows the expected status; cancel is accepted; subscribe yields the expected lifecycle events.
- idempotency-conflict asserts a reused key with a different request yields the AlreadyExists variant; query-timeout asserts QueryTimeout; not-found asserts NotFound.
- disconnect-resume forces a reconnect and asserts the event stream is gap-free and duplicate-free across it.
- No test uses #[ignore], a feature-gated skip, or an _-prefixed unused binding to suppress execution — the env-gate is the only gating mechanism.

### R3: Cross-SDK equivalence and runner documentation

THE SYSTEM SHALL document how to run all four harnesses against one server fixture and SHALL establish that the four SDKs produce identical observable results for every shared scenario. THE SYSTEM SHALL NOT declare conformance passed for a scenario unless all four SDKs agree on its result.

**Acceptance:**
- conformance/aion-clients/README.md documents bringing up the server fixture, setting the env var, and the command to run each of the four harnesses.
- The README states the equivalence requirement: a scenario is conformant only when all four SDKs produce the same observable result for it.
- Each harness emits its per-scenario observable results in a normalised form (JSON value or error-variant name) so the four runs can be compared step-by-step.
- The README names how a divergence is reported (which SDK diverged on which scenario step).

## Boundaries

- SHALL NOT modify SDK behaviour to make a scenario pass — a divergence is fixed in the offending SDK's own brief follow-up, not papered over here.
- SHALL NOT define new operations or error variants beyond the AL-001 contract.
- SHALL NOT stand up the aion-server itself (AW) — it consumes a server fixture.
