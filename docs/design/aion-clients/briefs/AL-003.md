## Purpose

Complete the Rust SDK's ergonomics: a WorkflowHandle so a caller operates on what they started without re-plumbing IDs, serde-typed payload helpers over the raw Payload path, a subscribe Stream that resumes transparently across disconnects, and the embedded constructor that binds the same surface to an in-process engine. This is what makes the Rust client pleasant rather than merely functional.

## Task

On top of the AL-002 core, add: WorkflowHandle (bundling WorkflowId + RunId with signal/query/cancel/describe/subscribe methods, plus a bare-ID constructor) and have start return one; serde-based typed payload helpers with a raw-Payload escape hatch on every payload-bearing operation; the subscribe Stream with sequence-based resumption; and an embedded constructor behind an off-by-default feature flag that targets an in-process aion engine through the same handle/payload machinery. Honour the start idempotency key from the AL-001 contract.

## Requirements

### R1: WorkflowHandle and idempotent start

THE SYSTEM SHALL define WorkflowHandle bundling WorkflowId + RunId and exposing signal, query, cancel, describe, and subscribe as methods, and start SHALL return a WorkflowHandle. THE SYSTEM SHALL accept an optional idempotency key on start: a retried identical start returns the original handle and a conflicting reuse returns ClientError::AlreadyExists. THE SYSTEM SHALL also provide a bare-ID constructor for WorkflowHandle.

**Acceptance:**
- handle.rs defines WorkflowHandle { workflow_id, run_id } with methods signal/query/cancel/describe/subscribe delegating to the Client.
- Client::start returns WorkflowHandle (replacing the AL-002 bare-tuple return) and accepts StartOptions carrying an optional idempotency_key.
- A retried start with the same idempotency key and identical request returns a handle equal to the original; a reused key with a different request returns ClientError::AlreadyExists.
- WorkflowHandle::from_ids(workflow_id, run) constructs a handle for a caller holding only an ID.
- A unit test against the stub transport asserts the idempotent-retry and the conflict cases.

### R2: Typed payload helpers with raw escape hatch

THE SYSTEM SHALL provide typed convenience methods that serialise serde::Serialize inputs to a JSON Payload and deserialise results to serde::DeserializeOwned. THE SYSTEM SHALL keep a raw-Payload method on every payload-bearing operation. THE SYSTEM SHALL NOT remove the raw path.

**Acceptance:**
- payload.rs provides to_payload<T: Serialize>(value) -> Result<Payload, ClientError> producing a JSON content-type Payload and from_payload<T: DeserializeOwned>(payload) -> Result<T, ClientError>.
- start_typed/signal_typed/query_typed convenience methods accept/return typed values; the raw start/signal/query from AL-002 remain available unchanged.
- A serialisation failure returns ClientError::InvalidArgument; a result-deserialisation failure returns a distinct, non-swallowed error (InvalidArgument or Server with detail, documented which).
- A unit test round-trips a typed struct through to_payload/from_payload and asserts equality.
- query_typed::<R>(...) deserialises the reply Payload to R.

### R3: Subscribe stream with resumption

THE SYSTEM SHALL implement subscribe returning a Stream of decoded Event values. WHILE the subscription is open, IF the transport disconnects transiently, THEN THE SYSTEM SHALL reconnect and resume from the last delivered event's sequence number, delivering no gaps and no duplicates. IF the subscription fails terminally, THEN THE SYSTEM SHALL yield an Err item rather than ending the stream silently.

**Acceptance:**
- stream.rs defines an event stream implementing futures::Stream<Item = Result<Event, ClientError>> over the WebSocket transport.
- The stream tracks the last delivered per-workflow sequence number and re-subscribes from sequence+1 after a transient disconnect.
- A test with a transport stub that drops mid-stream asserts the resumed stream contains every event exactly once across the reconnect (no gap, no duplicate).
- A terminal failure (e.g. Unauthenticated, or a non-retryable transport error) yields a final Err item and then the stream ends; it never ends with Ok-then-silent-close on failure.
- WorkflowHandle::subscribe and a filter/firehose subscribe entry point both return this stream type.

### R4: Embedded engine binding behind a feature flag

THE SYSTEM SHALL provide an embedded constructor that binds the same Client/WorkflowHandle surface to an in-process aion engine, gated behind an off-by-default feature. THE SYSTEM SHALL NOT pull aion into the dependency graph unless that feature is enabled.

**Acceptance:**
- Cargo.toml defines a feature (e.g. embedded) that adds aion as an optional dependency; the feature is off by default.
- transport.rs (or a submodule) provides an embedded transport that calls the in-process engine API and is selected by Client::embedded(engine) under #[cfg(feature = "embedded")].
- The embedded path reuses the same WorkflowHandle, payload helpers, error taxonomy, and Stream type as the network path — no duplicate operation surface.
- cargo tree -p aion-client (default features) shows no aion dependency; cargo tree -p aion-client --features embedded shows aion.
- cargo check -p aion-client --features embedded passes (against the engine's in-process API or a documented stub if AE is not yet landed, noted in the task).
