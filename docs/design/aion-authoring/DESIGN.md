# Aion Authoring — the typed module is the source, plus the loop, the lens, and the canvas

> **Cluster:** aion-authoring

## Intention

Authoring a durable workflow on Aion should feel like writing one typed Gleam function and a file of typed activity signatures — and nothing else. Everything that is mirrored today (worker handlers, registration lists, manifest entries, codecs, JSON schemas, wire-compat goldens, test skeletons) is generated from that typed source; everything that is observed (a run's state, its determinism, its shape) is projected from the event history the engine already records. When this cluster is done, the gap between 'I have a typed workflow' and 'I have a durable, deployed, debuggable workflow' is a save and a glance, not a day of hand-mirroring four files that must agree byte-for-byte. The authoring language stays Gleam — the type system is the contract and Aion's sharpest edge — and the declarative and visual surfaces become projections of that typed source, never a second place to author from. This is the experience an author reaches for, the loop they iterate in, the lens they debug through, and the canvas a non-coding stakeholder reads, all anchored to one source of truth.

## Problem

Shipping one activity today means writing it in five to seven places that must agree byte-for-byte: the activity.new wrapper and its name string, the local/in-VM body, the orchestration call site (with the step name hard-coded again), the encode/decode codec pair, the worker handler that mirrors the body invocation-for-invocation, the worker's registration list, and the workflow.toml activities entry — plus, for typed workers, a hand-derived wire-compat golden. A six-activity saga pays that surface six times. The roadmap already names the pain (RM-021, 'declare activities once, generate the rest'; RM-003, CLI input ergonomics; RM-004, watch mode; RM-009, declarative DSL + visual builder) but the generator work is unbuilt: aion codegen generates workflow I/O codecs only, aion new scaffolds whole projects but not per-activity plumbing, and the server-side authoring loop (compile + type-check + package + hot-load, designed in DESIGN-OVERVIEW as aion-toolchain) has no crate. Meanwhile the engine already records every event and broadcasts it over WebSocket, and deterministic replay already reconstructs exact state — so the data a time-travel debugger needs exists, with no lens over it. The recurring temptation is to answer all of this with a separate declarative DSL or a visual builder as the authoring surface; that would forfeit the compile-time type safety that is the whole reason to author in Gleam. The pain is real and the assets to fix it are already in the codebase, mostly unexploited.

## Solution

One cluster, organised as a layered authoring stack over a single source of truth — the typed Gleam workflow-plus-activities module (ADR-014). (1) Declare-once codegen: the activity declaration (a typed signature plus a tier annotation — in-VM, remote-Python, remote-Rust) is the only thing an author writes per activity; `aion generate` derives the activity.new wrapper, the codec pairs from the value types, the worker handler stub, the registration list, the workflow.toml activities entry, the wire-compat golden, and a test skeleton. This extends the existing aion codegen machinery (which already turns I/O schemas into codecs) from 'workflow I/O' to 'every activity', with an `--check` drift gate so a hand-edit is a build failure, not silent rot. (2) The instant loop: `aion dev` watches the package, rebuilds, repackages, and hot-reloads on save (RM-004), and serves a local dev UI with production parity — trigger a run, watch its events stream live, mock an activity, replay a failed run — over the same engine, store, and event stream production uses. (3) Server-as-compiler: build the designed-but-absent aion-toolchain crate and the aion-server authoring endpoints so an author submits Gleam source, the server shells out to the gleam binary, type-checks, returns errors inline, packages, and hot-loads — an authoring REPL against a running engine with no local toolchain, an asset only an engine that owns a hot-loading VM can offer. (4) The lens: a time-travel debugger in the dashboard reads the existing event history and replay — scrub a run event-by-event, see workflow-visible state and the recorded now()/random() at each step, and when a NonDeterminismError fires, point at the exact divergent command the resolver already computed (expected vs found). (5) The canvas: because a workflow is a typed function over a small known vocabulary (run / spawn / receive / all / race / sleep), render it to a graph automatically, overlay a live run's progress on that same graph, and allow bounded structural round-trip — a projection, not a second source (the answer to RM-009's 'on top of the typed SDK, not instead of it'). (6) Agentic-first authoring and the determinism gate: `aion new agent` scaffolds a durable agent loop (scout -> act -> verify -> signal-gated review) parameterised by prompts + schemas + gate, generalising the dogfooded stacked-dev shape; and `aion check --deterministic` is a static lint that flags any non-deterministic call reachable from workflow code, turning the determinism boundary into a CI gate that complements the type-level structural block. Each layer is independently valuable and shippable; together they are the best-in-class authoring experience.

## Principles

- **P1** — Single source of truth: the typed Gleam workflow-plus-activities module is the only authored artifact; every mirrored surface (worker, registration, manifest, codecs, schemas, wire goldens, tests) is generated from it and every observed surface (run state, determinism, shape) is projected from the recorded event history — nothing load-bearing is authored twice (ADR-014).
- **P2** — No new authoring DSL: the authoring language stays Gleam, because the type system is the contract; a declarative or visual surface is a generated projection of the typed source with bounded round-trip, never a parallel place to author from (ADR-014, ADR-001).
- **P3** — Generated code is honest and drift-gated: every generated file is reproducible and `aion generate --check` regenerates it byte-identical or fails, so a hand-edit to generated output is a build failure, not a silent divergence (mirrors the existing aion codegen --check; ADR-002 forbids parallel sources).
- **P4** — No defaults in generated configuration: codegen never invents a retry policy, timeout, or backoff; it emits the author's explicit choice or leaves the required hole required (ADR-001).
- **P5** — Parity in the loop: the local development experience runs the real engine, the real store, and the real event stream; what an author sees locally is what runs in production, with no mock-only path that diverges from production semantics.
- **P6** — The oplog is the debugger: introspection — time-travel, the determinism diff, the live canvas overlay — reads the event history and deterministic replay the engine already produces; it adds a lens, never a second source of truth (mirrors ADR-007's rejection of a sibling record store).
- **P7** — Determinism is provable, not merely conventional: a static check flags any wall-clock or entropy call reachable from workflow code, turning the determinism boundary into a CI gate that complements the SDK's type-level structural block (invariant 2).

## Decisions

- ADR-014 — Authoring: the typed Gleam module is the single source of truth; no separate DSL — The typed Gleam workflow-plus-activities module is the single source of truth. The activity signature already carries the contract (its input and output types, codecs, and tier); the engine generates the rest — worker handler stubs, registration, the manifest activities list, codecs, schemas, wire-compat goldens, and test skeletons — and projects the rest — the diagram and the time-travel state. No standalone textual DSL becomes a parallel authoring surface, because it would forfeit the compile-time type safety that is Aion's core differentiator (the aion-flow guarantee that an invalid composition fails gleam build). A visual canvas is a generated projection of the typed source with a live execution overlay and bounded structural round-trip, not a second source. Rejected: a separate DSL or manifest as the source of truth — it re-creates the spec-vs-implementation drift the typed module already eliminates and trades a strong static guarantee for a weaker one.
- ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
- ADR-002 — No backwards compatibility during the build — Replace, don't add alongside. No compat shims, no zombie code, no #[deprecated] markers. Breaking changes are made cleanly and consumers move forward. Rejected: incremental deprecation cycles — they double the surface under test for an audience of zero.
- ADR-007 — Design system v2: JSON ledgers above clusters, enrichment in place — Two project ledgers (roadmap.json, decisions.json) above the clusters; stage contracts as first-class schemas inside the aion codegen subset; the brief is one living document — the pipeline appends scout/dev/review per requirement and an execution block per brief, in place, never touching authored fields. Rejected: a sibling runs/ ledger — the brief as a single spec-plus-record document was the original intent, and aion's event history already provides the append-only audit trail.
- ADR-012 — Workflow code stays thin: large activity results ride as opaque payloads — Workflow code stays thin. Large activity results ride between stages as opaque sealed payloads the workflow never opens; the workflow decodes only a small facts projection — the few fields it needs to route control flow (pass/fail, changed files, blocked, drift). Decoding and prompt rendering happen in the consuming activity on the worker, which has a full heap and runs the work once rather than on every replay. Rejected: decoding and rendering full reports in workflow code — it puts the determinism boundary's heavy lifting in the wrong place and was the regression that exposed the beamr crash.

## Goals

- A new activity is declared once (a typed signature plus a tier annotation) and `aion generate` emits the activity.new wrapper, the codec pairs, the worker handler stub, the registration entry, the workflow.toml activities entry, and the wire-compat golden — verified by deleting every generated file for an example and regenerating it byte-identical, and by `aion generate --check` failing on a hand-edited generated file.
- `aion dev` watches a package, rebuilds/repackages/hot-reloads on save, and serves a local UI that triggers a run, streams its events live, mocks an activity, and replays a failed run — verified by an end-to-end test that edits a workflow and sees the new version serve a run with no restart.
- The server compiles and type-checks submitted Gleam source, returns the gleam type error inline on failure, and packages and hot-loads on success — verified by submitting a type-erroneous workflow, getting the error over the wire, then submitting a corrected one that runs.
- A run's history is navigable event-by-event in the dashboard, showing workflow-visible state and the recorded now()/random() values at each step, and a NonDeterminismError surfaces the exact divergent command (expected vs found) — verified against a recorded run and an injected determinism fault.
- A workflow renders to a graph automatically and a live run overlays its progress on that graph — verified by rendering an example and diffing the node/edge set against the workflow's known primitive structure.
- `aion new agent` scaffolds a durable agent loop (scout -> act -> verify -> signal-gated review) parameterised by prompts + schemas + gate, and `aion check --deterministic` flags a workflow that reaches a wall-clock/entropy call — verified by the scaffold compiling and running end-to-end and by a positive and negative determinism fixture.
- No #[allow]/#[expect], no unwrap/expect in library code, clippy -D warnings clean, fmt clean, every file under 500 LOC, and no generated default values (ADR-001).

## Non-Goals

- A standalone textual workflow DSL as the authoring surface. — ADR-014: the typed Gleam module is the source of truth; a separate DSL forfeits the compile-time type safety that is Aion's differentiator and re-creates the spec-vs-implementation drift the typed module already eliminates.
- A visual builder that is the source of truth or round-trips arbitrary code. — The canvas is a generated projection of the typed source with bounded structural round-trip; unbounded diagram-to-code synthesis would make the canvas a second authoritative artifact, which P2 forbids.
- Generating the side-effecting bodies of activities or workflows. — Codegen emits the plumbing that must agree byte-for-byte (wrappers, codecs, registration, manifest, handler stubs); the side-effecting body is the author's, and the determinism boundary is unchanged (invariant 2).
- Embedding the Gleam compiler inside aion or aion-server. — aion-toolchain shells out to the gleam binary and stays optional (the DESIGN-OVERVIEW boundary); without --gleam-path the server deploys pre-built .aion files only, so aion and aion-server carry no compiler dependency.
- A new test framework. — Generated test scaffolds target the existing aion/testing harness (simulated clock, activity mocks, replay assertions); the cluster generates skeletons, not a runtime.
- A separate persistence or oplog for debugging. — Time-travel and the determinism diff read the existing event store and deterministic replay; introducing a parallel debug log would re-open the spec/record split ADR-007 closed (P6).
- Inline multi-language workflow authoring. — In-VM (Tier 1/2) authoring stays BEAM-target Gleam per the DESIGN-OVERVIEW language-reach note; other languages remain Tier-3 remote workers, which this cluster generates stubs for but does not turn into an inline authoring surface.

## Inventory

- `crates/aion-cli/src/codegen.rs` — `aion codegen [--check]` exists but generates workflow I/O codecs only (input.json / output.json -> <package>_io.gleam). It wraps aion_package::codegen_project. No activity, worker, manifest, or wire-compat codegen.
- `crates/aion-cli/src/new/` — `aion new <name> [--template ...] [--worker python|rust]` scaffolds a whole project (HelloWorld, ApprovalFlow, Saga, DevPipeline) with gleam.toml, workflow.toml, src, schemas, and an optional worker. It does NOT generate per-activity plumbing; workflow.toml and codecs are hand-written. No `aion new agent`.
- `crates/aion-package/src/` — The .aion format, content-hash versioning, and the codegen entry (codegen_project) the CLI wraps for I/O codecs. No activity-surface codegen, no structure extraction, no test-scaffold generation.
- `crates/aion-server/src/` — A pure network adapter over the embedded engine (HTTP/gRPC/WebSocket). No --gleam-path configuration, no compile/type-check/package endpoints. The event stream is already broadcast over WebSocket.
- `crates/aion-toolchain (designed, absent)` — Server-side authoring (compile -> type-check -> package -> hot-load) is specified in DESIGN-OVERVIEW.md and COMPONENT-ARCHITECTURE.md but no crate exists and aion-server has no authoring endpoints.
- `crates/aion/src/durability/` — Deterministic replay reconstructs exact state; the resolver computes expected-vs-found on a NonDeterminismError; now() returns the recorded timestamp and random() a seeded value. The data a time-travel debugger and determinism diff need exists; there is no per-event state-projection surface.
- `gleam/aion_flow/src/aion/testing.gleam` — The test harness exists: a simulated clock (advance), an activity mock registry, and replay-determinism assertions. No generator emits skeletons against it.
- `crates/aion-dashboard/src/` — React monitoring UI. A per-run event timeline is RM-007 (idea, not built). No time-travel scrubber and no code/diagram canvas.

## Constraints

- **CN1** — Generated code is regenerable and drift-gated: `aion generate --check` reproduces every generated file byte-identical or fails; no generated file is hand-maintained alongside the source (ADR-002 forbids parallel sources; mirrors the existing aion codegen --check).
- **CN2** — No defaults in generated configuration (ADR-001): codegen never emits a retry policy, timeout, backoff, or any value the author did not choose; a required value stays a required hole, never a silent default.
- **CN3** — The determinism boundary is unchanged (invariant 2): generation emits activity plumbing and handler stubs, never side-effecting bodies; generated workflow code still routes every side effect through workflow.run, and generated code introduces no wall-clock or entropy path.
- **CN4** — Production parity: the dev loop and dev UI run the real engine, the real store, and the real event stream; there is no mock-only execution path whose semantics diverge from production, and activity mocking is opt-in per run, not a separate engine.
- **CN5** — No second source of truth for introspection (ADR-007): time-travel, the determinism diff, and the live canvas overlay read the existing event history and deterministic replay; no parallel debug log or state store is introduced.
- **CN6** — The visual canvas is a projection (ADR-014): it is generated from the typed source; any structural round-trip is bounded and regenerates Gleam that still type-checks; the canvas never becomes the authoritative artifact.
- **CN7** — aion-toolchain shells out to the gleam binary and stays optional: aion and aion-server carry no compiler dependency; without --gleam-path the server deploys pre-built .aion files only (the DESIGN-OVERVIEW boundary).
- **CN8** — Thin-workflow respected (ADR-012): generated activities thread sealed payloads plus thin facts; codegen does not reintroduce heavy decode or prompt rendering into the deterministic workflow process.
- **CN9** — Library-code purity: clippy --workspace --all-targets -- -D warnings and cargo fmt --check pass clean; gleam format --check clean for SDK changes; no unwrap/expect/allow/expect-attr in library code; every touched file stays under 500 LOC excluding tests.
