## Purpose

The differentiated showcase (RM-027). The event store is already a complete oplog, deterministic replay already reconstructs exact state, and the resolver already computes expected-vs-found on a non-determinism fault — so a Golem/Temporal-class time-travel debugger is nearly free on the data side. This brief adds the lens, reading the existing history and replay, never a second store (ADR-007, P6).

## Task

Add a debugging lens over what the engine already records. In: an engine surface that exposes, per event, the workflow-visible state projection and the recorded now()/random() values (computed from existing history and replay), and the divergent command on a NonDeterminismError; and an `aion inspect` CLI surface that steps a run event-by-event and offers a what-if re-run from a chosen event via the existing aion/testing replay path (the dashboard time-travel view that renders this surface is RM-007). Out: the rendered dashboard UI, any new persistence or debug log, and any change to the event schema.

## Requirements

### R1: Per-event state projection and determinism diff from history

WHEN inspecting a run, THE SYSTEM SHALL expose, per event, the workflow-visible state projection and the recorded now() and random() values, computed from the existing event history and deterministic replay. WHEN a NonDeterminismError exists for the run, THE SYSTEM SHALL expose the divergent command as expected-vs-found at the sequence. THE SYSTEM SHALL NOT maintain a parallel mutable state store.

**Acceptance:**
- A per-event workflow-visible state projection is returned for a recorded run, computed from history and replay with no second store (C20).
- The recorded now() and random() values for each step are exposed alongside the state.
- An injected determinism fault surfaces the exact divergent command (expected vs found at the sequence) the resolver computes.

### R2: aion inspect: stepping and what-if re-run (the data surface the dashboard consumes)

WHEN an operator runs `aion inspect <run>`, THE SYSTEM SHALL let them step the run event-by-event over the state projection and run a what-if re-run from a chosen event with a mocked outcome via the existing aion/testing replay path. THE SYSTEM SHALL NOT build the rendered dashboard scrubber UI (deferred to RM-007).

**Acceptance:**
- `aion inspect <run>` steps a recorded run event-by-event, printing the projected state at each step.
- A what-if re-run forks from a chosen event with a mocked outcome and reports the resulting path, using the existing replay harness.
- It reads the existing event history; no debug-only log is introduced (C16). The dashboard time-travel view that renders this surface is RM-007.

## Boundaries

- SHALL NOT introduce a parallel debug log or state store — time-travel reads existing history and replay (ADR-007).
- SHALL NOT change the Event schema or the recorded payloads.
- SHALL NOT overlap with RM-007's per-run event timeline list view beyond consuming the same history.
- SHALL NOT build the rendered dashboard time-travel UI — RM-007 owns it; this brief delivers the engine projection and the `aion inspect` data surface.
