# WA-007: Determinism linter, generated test scaffolds, and input skeletons

> **Cluster:** aion-authoring
> **Depends on:** WA-001
> **Design anchor:**
> - ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
> **Checklist:**
> - C28 — `aion check --deterministic` statically flags any wall-clock or entropy call reachable from workflow code and passes a clean workflow, proven by a positive and a negative fixture.
> - C29 — `aion generate` emits an aion/testing skeleton per workflow: each activity pre-mocked, a clock advance per timer, and a replay-determinism assertion.
> - C30 — `aion input <workflow_type>` emits a valid input skeleton derived from the workflow's input type, so an input document is never hand-written from scratch.
> **Stories:**
> - S13 (Human Developer, Authoring a durable agent and proving determinism) — As a developer, I want a static check that flags any non-deterministic call reachable from workflow code, so that I can prove a workflow is replay-safe in CI rather than discovering a desync in production.
> - S14 (Human Developer, Authoring a durable agent and proving determinism) — As a developer, I want a generated test skeleton and an input skeleton for each workflow, so that testing and triggering a workflow start from a valid scaffold instead of a blank file.

## Purpose

The cross-cutting moves that separate good from best-in-class (RM-029). A static determinism gate turns the determinism boundary into a CI check that complements the SDK's type-level structural block — a provable-determinism guarantee no competitor offers. Alongside it, generated test skeletons and input skeletons mean testing and triggering a workflow start from a scaffold, not a blank file.

## Task

Add the quality and ergonomics scaffolds that ride on the codegen. In: `aion check --deterministic`, a static lint that flags any wall-clock or entropy call reachable from workflow code; generated aion/testing skeletons per workflow (each activity mocked, a clock advance per timer, a replay-determinism assertion); and `aion input <workflow_type>`, which emits a valid input skeleton from the workflow's input type. Out: RM-003's polymorphic --input and client-side validation (separate dispatch-time work), and any new test framework (the scaffolds target the existing aion/testing harness).

## Requirements

### R1: aion check --deterministic static gate

WHEN `aion check --deterministic` runs over a workflow module, THE SYSTEM SHALL flag any wall-clock or entropy call reachable from workflow code and SHALL pass a workflow that reaches none.

**Acceptance:**
- A positive fixture (a workflow that reaches a wall-clock or entropy call) fails the check.
- A negative fixture (a clean workflow) passes the check.
- The check is usable as a CI gate, exiting non-zero on a violation (C28).

**Files:**
- create: crates/aion-cli/src/check_deterministic.rs

**Checklist:**
- C28 — `aion check --deterministic` statically flags any wall-clock or entropy call reachable from workflow code and passes a clean workflow, proven by a positive and a negative fixture.

**Stories:**
- S13 (Human Developer, Authoring a durable agent and proving determinism) — As a developer, I want a static check that flags any non-deterministic call reachable from workflow code, so that I can prove a workflow is replay-safe in CI rather than discovering a desync in production.

### R2: Generated test scaffolds against aion/testing

WHEN `aion generate` runs, THE SYSTEM SHALL emit an aion/testing skeleton per workflow: each activity pre-mocked, a clock advance per timer, and a replay-determinism assertion. THE SYSTEM SHALL NOT introduce a new test framework.

**Acceptance:**
- The generated skeleton mocks each of the workflow's activities and advances the simulated clock once per timer.
- The generated skeleton includes a replay-determinism assertion using the existing aion/testing harness (C29).
- No new test framework is introduced — the skeleton targets aion/testing.

**Files:**
- create: crates/aion-package/src/codegen/test_scaffold.rs

**Checklist:**
- C29 — `aion generate` emits an aion/testing skeleton per workflow: each activity pre-mocked, a clock advance per timer, and a replay-determinism assertion.

**Stories:**
- S14 (Human Developer, Authoring a durable agent and proving determinism) — As a developer, I want a generated test skeleton and an input skeleton for each workflow, so that testing and triggering a workflow start from a valid scaffold instead of a blank file.

### R3: aion input skeleton from the workflow input type

WHEN `aion input <workflow_type>` runs, THE SYSTEM SHALL emit a valid input skeleton derived from the workflow's input type.

**Acceptance:**
- `aion input <workflow_type>` emits an input skeleton derived from the workflow's input type (C30).
- The emitted skeleton round-trips through the workflow's input codec without a decode error.
- The skeleton is generated from the type, not hand-written.

**Files:**
- create: crates/aion-cli/src/input.rs

**Checklist:**
- C30 — `aion input <workflow_type>` emits a valid input skeleton derived from the workflow's input type, so an input document is never hand-written from scratch.

**Stories:**

## Boundaries

- SHALL NOT implement RM-003's polymorphic --input or client-side input validation — that is separate dispatch-time work; this delivers only the type-derived skeleton.
- SHALL NOT introduce a new test framework — generated scaffolds target the existing aion/testing harness.
- SHALL NOT add default values to generated skeletons (ADR-001).
