## Purpose

The highest-ROI experience lever the durable-execution field has converged on (RM-025): a save-to-running inner loop and a local dev UI with production parity. Aion gets the hard pieces nearly free — hot code loading is the reload, content-hash versioning makes concurrent versions safe, and the existing WebSocket event firehose is the live view.

## Task

Give authors a tight inner loop and a faithful local UI. In: `aion dev` that watches a package and rebuilds, repackages, and hot-loads the new content-hash version on save (subsuming the RM-004 watch slice); and local dev UI endpoints that trigger a run, stream its events live over the existing WebSocket stream, mock a named activity per-run, and replay a failed run, all over the real engine, store, and event stream. Out: codegen (WA-001), server-side compilation of submitted source (WA-003), and any mock-only execution path whose semantics diverge from production.

## Requirements

### R1: Watch, rebuild, repackage, hot-load on save

WHEN a watched package source file changes, THE SYSTEM SHALL rebuild, repackage, and hot-load the new content-hash version without restarting the engine. WHILE a prior run is in flight, THE SYSTEM SHALL leave it on its pinned version (content-hash immutability).

**Acceptance:**
- Editing a workflow under `aion dev` triggers a rebuild, repackage, and hot-load with no engine restart.
- A workflow already running keeps executing on the version it started with.
- An end-to-end test edits a workflow and observes the new content-hash version serve a fresh run with no restart.

### R2: Local dev UI with production parity

WHEN a developer uses the dev UI, THE SYSTEM SHALL let them trigger a run, stream that run's events live over the existing WebSocket stream, mock a named activity's result on an opt-in basis for a given run, and replay a failed run. THE SYSTEM SHALL run the real engine, store, and event stream, and SHALL NOT provide a mock-only execution path whose semantics diverge from production.

**Acceptance:**
- Triggering a run from the dev UI streams that run's events live over the existing WebSocket event stream.
- Opt-in activity mocking returns a canned typed result for one run without changing the engine.
- Replaying a failed run re-drives it through the real engine and store; there is no separate mock-only engine (CN4).

## Boundaries

- SHALL NOT add activity codegen (WA-001) or server-side source compilation (WA-003).
- SHALL NOT create a mock-only engine or a dev-only execution path that diverges from production semantics.
- SHALL NOT introduce default poll intervals or timeouts in the watch loop (ADR-001) — values are configurable or deferred.
