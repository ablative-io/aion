# WA-005: Bidirectional visual canvas: code-to-diagram projection with live overlay

> **Cluster:** aion-authoring
> **Blocked by:** ADR-014 (proposed — awaiting ratification)
> **Design anchor:**
> - ADR-014 — Authoring: the typed Gleam module is the single source of truth; no separate DSL — The typed Gleam workflow-plus-activities module is the single source of truth. The activity signature already carries the contract (its input and output types, codecs, and tier); the engine generates the rest — worker handler stubs, registration, the manifest activities list, codecs, schemas, wire-compat goldens, and test skeletons — and projects the rest — the diagram and the time-travel state. No standalone textual DSL becomes a parallel authoring surface, because it would forfeit the compile-time type safety that is Aion's core differentiator (the aion-flow guarantee that an invalid composition fails gleam build). A visual canvas is a generated projection of the typed source with a live execution overlay and bounded structural round-trip, not a second source. Rejected: a separate DSL or manifest as the source of truth — it re-creates the spec-vs-implementation drift the typed module already eliminates and trades a strong static guarantee for a weaker one.
> **Checklist:**
> - C21 — A workflow's primitive structure (run / spawn / receive / all / race / sleep and control flow) is extracted from the package as a graph model automatically.
> - C22 — The graph model identifies each node by its correlation key (activity sequence, signal name, timer id, child ordinal) so a consumer can map a run's recorded events onto it.
> - C23 — Extracting an example's structure and diffing the node/edge set against the workflow's known structure matches, proving the graph model is derived from the source rather than hand-drawn.
> - C24 — A bounded structural delta regenerates Gleam that still type-checks; the graph model is never the authoritative artifact (CN6).
> **Stories:**
> - S11 (Stakeholder, Reading and shaping a workflow without writing Gleam) — As a non-coding stakeholder, I want a workflow's structure exposed as a graph model that stays in sync with the code and carries per-node run identity, so that a viewer (the dashboard, RM-007) can render and overlay it for me to follow and discuss without reading the source.

## Purpose

Reframes RM-009 ('declarative DSL + visual builder') away from a separate DSL and toward a generated, always-in-sync graph model (RM-009 as designed here). Because a workflow is a typed function over a small known vocabulary (run / spawn / receive / all / race / sleep), the graph model can be generated from the source, carry per-node run identity for a consumer to overlay, and be regenerated from bounded structural deltas — a projection of the typed source, never a second authoring surface (ADR-014). The rendered canvas UI and the live overlay are the dashboard family (RM-007).

## Task

Make the graph model a faithful projection of the typed source. In: extract a workflow's primitive structure from the package and emit it as a graph model automatically; identify each node by its correlation key so a consumer can overlay a run's events; and regenerate Gleam from bounded structural deltas such that it still type-checks. The rendered canvas UI and the live overlay are the dashboard family (RM-007). Out: unbounded diagram-to-code synthesis, the rendered UI, and any path that makes the model the authoritative artifact.

## Requirements

### R1: Structure extraction and automatic graph rendering

WHEN rendering a workflow, THE SYSTEM SHALL extract its primitive structure (run / spawn / receive / all / race / sleep and control flow) from the package and render a graph automatically. The rendered node and edge set SHALL match the workflow's actual structure.

**Acceptance:**
- Structure extraction yields the node and edge set for a workflow from the package.
- Rendering examples/order-saga produces a graph whose node/edge set matches the workflow's known structure, proven by a diff test (C23).
- The diagram is generated from the source, not hand-drawn.

**Files:**
- create: crates/aion-package/src/structure.rs

**Checklist:**
- C21 — A workflow's primitive structure (run / spawn / receive / all / race / sleep and control flow) is extracted from the package as a graph model automatically.
- C23 — Extracting an example's structure and diffing the node/edge set against the workflow's known structure matches, proving the graph model is derived from the source rather than hand-drawn.

**Stories:**
- S11 (Stakeholder, Reading and shaping a workflow without writing Gleam) — As a non-coding stakeholder, I want a workflow's structure exposed as a graph model that stays in sync with the code and carries per-node run identity, so that a viewer (the dashboard, RM-007) can render and overlay it for me to follow and discuss without reading the source.

### R2: Per-node run identity and bounded Gleam regeneration

THE SYSTEM SHALL identify each graph-model node by its correlation key (activity sequence position, signal name, timer id, child spawn ordinal) so a consumer can map a run's recorded events onto the graph. WHEN a bounded structural delta is applied to the model, THE SYSTEM SHALL regenerate Gleam that still type-checks. THE SYSTEM SHALL NOT build the rendered canvas UI or overlay (deferred to RM-007), and SHALL NOT make the model the authoritative artifact.

**Acceptance:**
- Each node in the extracted graph model carries the correlation key needed to map a run's events onto it (C22).
- A bounded structural delta regenerates Gleam that compiles under gleam build (C24).
- The model is a projection — the typed source remains the source of truth (CN6); the rendered canvas that overlays a live run is RM-007.

## Boundaries

- SHALL NOT support unbounded diagram-to-code synthesis — only bounded structural round-trip that regenerates type-checking Gleam.
- SHALL NOT make the canvas the authoritative artifact; the typed source is the source of truth (ADR-014).
- SHALL NOT change the event stream or the package format.
- SHALL NOT build the rendered canvas UI or the live overlay — RM-007 owns them; this brief delivers the graph model and the Gleam regeneration.
