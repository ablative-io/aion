# WA-005: Bidirectional visual canvas: code-to-diagram projection with live overlay

> **Cluster:** aion-authoring
> **Blocked by:** ADR-014 (proposed — awaiting ratification)
> **Design anchor:**
> - ADR-014 — Authoring: the typed Gleam module is the single source of truth; no separate DSL — The typed Gleam workflow-plus-activities module is the single source of truth. The activity signature already carries the contract (its input and output types, codecs, and tier); the engine generates the rest — worker handler stubs, registration, the manifest activities list, codecs, schemas, wire-compat goldens, and test skeletons — and projects the rest — the diagram and the time-travel state. No standalone textual DSL becomes a parallel authoring surface, because it would forfeit the compile-time type safety that is Aion's core differentiator (the aion-flow guarantee that an invalid composition fails gleam build). A visual canvas is a generated projection of the typed source with a live execution overlay and bounded structural round-trip, not a second source. Rejected: a separate DSL or manifest as the source of truth — it re-creates the spec-vs-implementation drift the typed module already eliminates and trades a strong static guarantee for a weaker one.
> **Checklist:**
> - C21 — A workflow's primitive structure (run / spawn / receive / all / race / sleep and control flow) is extracted from the package and rendered to a graph automatically.
> - C22 — A live run overlays its progress on the same graph, lighting the active node as the engine emits events.
> - C23 — Rendering an example and diffing the node/edge set against the workflow's known structure matches, proving the diagram is generated from the source rather than hand-drawn.
> - C24 — A bounded structural edit on the canvas regenerates Gleam that still type-checks; the canvas is never the authoritative artifact (CN6).
> **Stories:**
> - S11 (Stakeholder, Reading and shaping a workflow without writing Gleam) — As a non-coding stakeholder, I want to see a workflow as a diagram that stays in sync with the code and lights up as a run executes, so that I can follow and discuss the process without reading the source.

## Purpose

Reframes RM-009 ('declarative DSL + visual builder') away from a separate DSL and toward a generated, always-in-sync canvas (RM-009 as designed here). Because a workflow is a typed function over a small known vocabulary (run / spawn / receive / all / race / sleep), the diagram can be generated from the source, overlaid with a live run, and edited within bounds — a projection of the typed source, never a second authoring surface (ADR-014).

## Task

Make the diagram a faithful projection of the typed source. In: extract a workflow's primitive structure from the package and render it to a graph automatically; overlay a live run's progress on the same graph; and regenerate Gleam from bounded structural edits such that it still type-checks. Out: unbounded diagram-to-code synthesis, and any path that makes the canvas the authoritative artifact.

## Requirements

### R1: Structure extraction and automatic graph rendering

WHEN rendering a workflow, THE SYSTEM SHALL extract its primitive structure (run / spawn / receive / all / race / sleep and control flow) from the package and render a graph automatically. The rendered node and edge set SHALL match the workflow's actual structure.

**Acceptance:**
- Structure extraction yields the node and edge set for a workflow from the package.
- Rendering examples/order-saga produces a graph whose node/edge set matches the workflow's known structure, proven by a diff test (C23).
- The diagram is generated from the source, not hand-drawn.

**Files:**
- create: crates/aion-package/src/structure.rs

**Checklist:**
- C21 — A workflow's primitive structure (run / spawn / receive / all / race / sleep and control flow) is extracted from the package and rendered to a graph automatically.
- C23 — Rendering an example and diffing the node/edge set against the workflow's known structure matches, proving the diagram is generated from the source rather than hand-drawn.

**Stories:**
- S11 (Stakeholder, Reading and shaping a workflow without writing Gleam) — As a non-coding stakeholder, I want to see a workflow as a diagram that stays in sync with the code and lights up as a run executes, so that I can follow and discuss the process without reading the source.

### R2: Live execution overlay and bounded round-trip

WHILE a run executes, THE SYSTEM SHALL overlay its progress on the same graph, lighting the active node as the engine emits events. WHEN a bounded structural edit is made on the canvas, THE SYSTEM SHALL regenerate Gleam that still type-checks. THE SYSTEM SHALL NOT make the canvas the authoritative artifact.

**Acceptance:**
- A live run lights the active node on the graph as events arrive over the existing event stream.
- A bounded structural edit on the canvas regenerates Gleam that compiles under gleam build (C24).
- The canvas is a projection — the typed source remains the source of truth (CN6).

## Boundaries

- SHALL NOT support unbounded diagram-to-code synthesis — only bounded structural round-trip that regenerates type-checking Gleam.
- SHALL NOT make the canvas the authoritative artifact; the typed source is the source of truth (ADR-014).
- SHALL NOT change the event stream or the package format.
