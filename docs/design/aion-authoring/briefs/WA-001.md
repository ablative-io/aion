# WA-001: Declare-once codegen: generate the activity plumbing from the typed declaration

> **Cluster:** aion-authoring
> **Design anchor:**
> - ADR-014 — Authoring: the typed Gleam module is the single source of truth; no separate DSL — The typed Gleam workflow-plus-activities module is the single source of truth. The activity signature already carries the contract (its input and output types, codecs, and tier); the engine generates the rest — worker handler stubs, registration, the manifest activities list, codecs, schemas, wire-compat goldens, and test skeletons — and projects the rest — the diagram and the time-travel state. No standalone textual DSL becomes a parallel authoring surface, because it would forfeit the compile-time type safety that is Aion's core differentiator (the aion-flow guarantee that an invalid composition fails gleam build). A visual canvas is a generated projection of the typed source with a live execution overlay and bounded structural round-trip, not a second source. Rejected: a separate DSL or manifest as the source of truth — it re-creates the spec-vs-implementation drift the typed module already eliminates and trades a strong static guarantee for a weaker one.
> - ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
> - ADR-002 — No backwards compatibility during the build — Replace, don't add alongside. No compat shims, no zombie code, no #[deprecated] markers. Breaking changes are made cleanly and consumers move forward. Rejected: incremental deprecation cycles — they double the surface under test for an audience of zero.
> - ADR-007 — Design system v2: JSON ledgers above clusters, enrichment in place — Two project ledgers (roadmap.json, decisions.json) above the clusters; stage contracts as first-class schemas inside the aion codegen subset; the brief is one living document — the pipeline appends scout/dev/review per requirement and an execution block per brief, in place, never touching authored fields. Rejected: a sibling runs/ ledger — the brief as a single spec-plus-record document was the original intent, and aion's event history already provides the append-only audit trail.
> - ADR-012 — Workflow code stays thin: large activity results ride as opaque payloads — Workflow code stays thin. Large activity results ride between stages as opaque sealed payloads the workflow never opens; the workflow decodes only a small facts projection — the few fields it needs to route control flow (pass/fail, changed files, blocked, drift). Decoding and prompt rendering happen in the consuming activity on the worker, which has a full heap and runs the work once rather than on every replay. Rejected: decoding and rendering full reports in workflow code — it puts the determinism boundary's heavy lifting in the wrong place and was the regression that exposed the beamr crash.
> **Checklist:**
> - C1 — An activity is declared once as a typed signature plus a tier annotation (InVm | RemotePython | RemoteRust) in the activities module, and that declaration is the only per-activity artifact an author writes.
> - C2 — `aion generate` derives, from the typed declaration, the activity.new wrapper, the value-type codec pairs, the worker handler stub, the worker registration entry, and the workflow.toml activities entry.
> - C3 — `aion generate` derives the wire-compat golden for each remote activity from the value-type codecs, so the worker/SDK wire shapes are pinned without a hand-derived literal.
> - C4 — `aion generate --check` regenerates every generated file in memory and exits non-zero if any on-disk generated file differs, so a hand-edit to generated output is a build failure.
> - C5 — Generated activity configuration carries no invented defaults: a retry policy, timeout, or backoff appears in generated code only when the author declared it, otherwise the required hole remains required (ADR-001).
> - C6 — Deleting every generated file for a worked example (e.g. order-saga) and running `aion generate` reproduces them byte-identical, proven by a round-trip test.
> **Stories:**
> - S1 (Human Developer, Authoring a new workflow family) — As a developer, I want to declare an activity once as a typed signature and a tier, so that I do not hand-mirror its name, codecs, worker handler, registration, and manifest entry across five to seven files that must agree byte-for-byte.
> - S2 (Human Developer, Authoring a new workflow family) — As a developer, I want generated plumbing to be regenerable and drift-checked, so that a stale hand-edit is caught by the build instead of corrupting a wire shape silently.
> - S3 (Human Developer, Authoring a new workflow family) — As a developer, I want generated activity configuration to contain only the policies I chose, so that I am never silently bound by a retry or timeout default I did not set.

## Purpose

The keystone of the authoring cluster (RM-021). Today an activity is hand-mirrored across five to seven files that must agree byte-for-byte; this brief makes the typed activity declaration the single source of truth (ADR-014) and generates the rest. It extends the existing aion codegen machinery (which already turns I/O schemas into codecs, ADR-007 precedent) from workflow I/O to the full activity surface, with a drift gate so generated output can never silently rot.

## Task

Make the typed activity declaration the only per-activity artifact an author writes, and generate the plumbing from it. In: an activity declaration form (a typed signature plus a tier annotation InVm | RemotePython | RemoteRust) in the aion_flow activities surface; an `aion generate` command that derives the activity.new wrapper, the value-type codec pairs, the worker handler stub for the tier, the worker registration entry, the workflow.toml activities entry, and the wire-compat golden; an `aion generate --check` drift gate; a round-trip guarantee (delete all generated files, regenerate byte-identical); and an authoring guide. Out: the dev loop (WA-002), server-side compile (WA-003), the determinism linter and test/input scaffolds (WA-007), and generating the side-effecting bodies of activities (the author writes those; the determinism boundary is unchanged).

## Requirements

### R1: Typed activity declaration form with a tier annotation

WHEN an author declares an activity, THE SYSTEM SHALL accept a typed signature plus a tier annotation (InVm | RemotePython | RemoteRust) in the activities surface that the generator reads to drive codegen. THE SYSTEM SHALL NOT require the author to also hand-write the activity.new wrapper, the codecs, the worker handler, the registration entry, or the manifest entry for that activity.

**Acceptance:**
- An activity declaration form exists in gleam/aion_flow/src/aion/activity.gleam that carries the activity's input type, output type, and tier.
- A declared activity type-checks under gleam build with no hand-written activity.new wrapper present.
- The declaration is the only per-activity artifact the author writes — verified by an example activity whose wrapper/codecs/worker/registration/manifest are all absent from hand-written source.

**Files:**

**Checklist:**

**Stories:**

### R2: aion generate derives the activity plumbing

WHEN `aion generate` runs against a package, THE SYSTEM SHALL derive from each typed declaration the activity.new wrapper, the value-type codec pairs, the worker handler stub for the declared tier, the worker registration entry, the workflow.toml activities entry, and the wire-compat golden. THE SYSTEM SHALL NOT emit any retry policy, timeout, or backoff value the author did not declare (ADR-001).

**Acceptance:**
- Running `aion generate` on examples/order-saga emits the activity.new wrappers, codec pairs, worker handler stubs, registration entries, and workflow.toml activities entries for every declared activity.
- Each remote activity gets a wire-compat golden derived from its value-type codecs, with no hand-derived literal.
- No generated activity configuration contains a retry/timeout/backoff value absent from the declaration — verified by generating an activity with no policy and asserting the generated config carries none.

### R3: Drift gate, round-trip guarantee, and the authoring guide

WHEN `aion generate --check` runs, THE SYSTEM SHALL regenerate every generated file in memory and exit non-zero if any on-disk generated file differs. WHEN every generated file for a package is deleted and `aion generate` is re-run, THE SYSTEM SHALL reproduce them byte-identical.

**Acceptance:**
- `aion generate --check` exits non-zero on a hand-edited generated file and exits zero when the tree is clean.
- Deleting every generated file for examples/order-saga and running `aion generate` reproduces them byte-identical, proven by a round-trip test.
- docs/authoring.md walks the declare -> generate -> run model end to end using the saga.

## Boundaries

- SHALL NOT generate the side-effecting body of any activity or workflow — only the plumbing that must agree byte-for-byte.
- SHALL NOT build the dev loop (WA-002), the server compile loop (WA-003), or the determinism/test/input scaffolds (WA-007).
- SHALL NOT introduce any default retry policy, timeout, or backoff (ADR-001).
