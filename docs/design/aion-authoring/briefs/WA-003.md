# WA-003: Server-as-compiler: the aion-toolchain authoring loop

> **Cluster:** aion-authoring
> **Design anchor:**
> - ADR-002 — No backwards compatibility during the build — Replace, don't add alongside. No compat shims, no zombie code, no #[deprecated] markers. Breaking changes are made cleanly and consumers move forward. Rejected: incremental deprecation cycles — they double the surface under test for an audience of zero.
> **Checklist:**

> - C13 — aion-server exposes authoring endpoints, gated on --gleam-path, that accept Gleam source and return the gleam type error inline on a compile failure.
> - C14 — On a successful compile the server packages via aion-toolchain and hot-loads the new version, making authoring a live loop against the running engine.
> - C15 — Without --gleam-path the server deploys pre-built .aion files only and carries no compiler dependency (CN7).
> **Stories:**
> - S7 (AI Agent, Authoring a workflow through the running engine) — As an authoring agent, I want to submit Gleam source to the server and get type errors back inline, so that I can correct a workflow against the real type-checker without a local toolchain.
> - S8 (AI Agent, Authoring a workflow through the running engine) — As an authoring agent, I want a successful submission to package and hot-load automatically, so that authoring is one live loop against the running engine.

## Purpose

Build the designed-but-absent server-side authoring loop (RM-026): submit Gleam source, the server type-checks it, returns errors inline, packages, and hot-loads — an authoring REPL against a running engine with no local toolchain. An asset only an engine that owns a hot-loading VM can offer, and the foundation for a hosted playground.

## Task

Make authoring a live loop against the running engine, optionally. In: a new aion-toolchain crate that compiles and type-checks Gleam workflow source by shelling out to the gleam binary and packages a .aion on success; and aion-server authoring endpoints, gated on --gleam-path, that accept source, return the gleam type error inline on failure, and package and hot-load on success. Out: embedding the Gleam compiler (the toolchain always shells out); making the toolchain mandatory (without --gleam-path the server deploys pre-built .aion only).

## Requirements

### R1: aion-toolchain compiles, type-checks, and packages

WHEN given Gleam workflow source, THE SYSTEM SHALL compile and type-check it by shelling out to the gleam binary and package a .aion on success. THE SYSTEM SHALL NOT embed a Gleam compiler.

**Acceptance:**
- aion-toolchain compiles a valid workflow source to a loadable .aion by invoking the external gleam binary.
- A type-erroneous source yields the gleam type error rather than a panic or a partial package.
- The crate embeds no compiler — verified by its dependency set carrying no Gleam compiler crate.

### R2: Server authoring endpoints, gated and optional

WHEN Gleam source is submitted to the authoring endpoint and --gleam-path is configured, THE SYSTEM SHALL return the gleam type error inline on a compile failure and package and hot-load on success. WHEN --gleam-path is absent, THE SYSTEM SHALL deploy pre-built .aion files only and carry no compiler dependency.

**Acceptance:**
- Submitting a type-erroneous workflow returns the gleam error inline over the wire.
- Submitting a corrected workflow packages it via aion-toolchain and hot-loads it so it runs.
- Without --gleam-path the authoring endpoints are absent and the server still deploys pre-built .aion files (CN7).

## Boundaries

- SHALL NOT embed the Gleam compiler in aion or aion-server — aion-toolchain shells out to the gleam binary.
- SHALL NOT make server-side authoring mandatory — it is gated on --gleam-path and optional.
- SHALL NOT add codegen (WA-001) or the dev loop (WA-002).

## Verification

- Inline type-error reporting proven by submitting an erroneous then a corrected workflow.
- Optionality proven: without --gleam-path the server has no compiler dependency and still deploys .aion.
- clippy -D warnings and fmt clean.
