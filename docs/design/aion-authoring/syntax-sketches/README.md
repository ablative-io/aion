# Workflow DSL — syntax flavour sketches

**Purpose:** choose the *one* authored surface for the workflow DSL. Every file in this
folder renders the **same workflow** (same graph, same semantics) in a different syntax
flavour, so the comparison is about reading feel, not features. We will NOT ship multiple
authoring flavours — that fragments tutorials, error messages, and the canvas round-trip.
One authored surface + one canonical serialization (generated JSON) + optional views.

| File | Flavour | Verdict (short) |
|---|---|---|
| `A-javascript-flavour.md` | JS-*lookalike* bespoke DSL | **Superseded by F.** Reads perfectly, but for AI authors a lookalike gives plausible-but-wrong priors — every drift is a syntax error in a language that exists nowhere in training data. |
| `B-yaml-flavour.md` | YAML step list | Rejected for authoring. Fine graph serialization, but control flow becomes strings-in-quotes and rebinding needs a `switch` contortion (the GitHub-Actions wound). |
| `C-json-canonical.md` | JSON | **Not an authoring surface — it's the canonical serialization.** Generated by the build, edited by the canvas, diffed in git — and the direct structured-output lane for AI authors (see F). |
| `D-nix-flavour.md` | Nix-ish expression language | Elegant and semantically honest (pure let-bindings = the dataflow graph), but juxtaposition application + niche familiarity work against the adoption goal. Rejected. |
| `E-pipe-flavour.md` | Gleam/Elixir pipes | Beautiful for linear spines, breaks on the diamond (revision needs `findings` AND `approval`). Rejected as the organizing principle; possible later as sugar *inside* the primary flavour. |
| `F-typescript-subset.md` | Real TypeScript, subset-enforced, statically extracted, never executed | **Strong runner-up / later code-first view.** Correct priors for AI authors + tsc guardrails, but Tom's read: it's still a *program*, and the pull is toward a document. Kept compatible via the canonical model. |
| `G-step-document.md` | **Step document — YAML shape, designed field grammar, `about:` prose per step** | **Primary candidate.** Fixes flavour B's three wounds (expression grammar in designated fields, `when:`+`as:` rebinding, calls-look-like-calls); reads as a runbook; safest AI-authoring surface (schema-validated data); steps narrate themselves in the ops console. |

TOML was considered and dropped without a sketch: same wounds as YAML (expressions in
strings, branch contortions) plus hostility to nested lists — fan-out bodies, timeout
handlers, and compensation blocks become `[[steps.fan_out.body]]` gymnastics. TOML is
built for flat config; a step tree is unnatural TOML, and LLMs drift most when the
structure fights the format.

## The shared example workflow

`research_report` — chosen because it exercises every pattern that has ever stressed a
workflow syntax:

- **Inputs:** `brief: Brief` (structured, from a JSON file) and `corpus: Dir` (a directory
  of source documents, content-addressed snapshot — see "File and directory inputs" below).
- **Fan-out with retry:** one `investigate` agent per planned question, `retry 3 backoff 30s`.
- **Durable human gate:** `wait for review` (a signal), with a 3-day timeout that ends the
  workflow early returning the unpublished draft.
- **A diamond dependency:** the revision step needs BOTH `findings` (from step 2) and
  `approval.notes` (from step 4). Linear syntaxes break here.
- **Rebinding:** `draft` is conditionally replaced by the revised draft.
- **Compensation:** `publish` failing must `delete_assets` (undo step 6) before failing.

Action contracts (typed once, next to their bodies — Gleam in-VM by default, or scaffolded
Rust/Python workers):

```
plan          : Brief -> List(Question)
investigate   : (Question, Dir) -> Finding
synthesize    : (List(Finding), String) -> Report
upload_assets : Report -> AssetBundle
publish       : (Report, AssetBundle) -> String        // returns the URL
delete_assets : AssetBundle -> Nil
```

Boundary types (declared in the workflow file's `types` block in these sketches; the
in-VM Gleam variants can equally live beside the action bodies and be extracted via
`gleam export package-interface` — verified working 2026-07-02):

```
Brief     { topic: String, audience: String, depth: Int }
Question  { text: String, angle: String }
Finding   { question: String, summary: String, sources: List(String) }
Report    { title: String, body: String, findings: List(Finding) }
Approval  { ok: Bool, notes: String }
Published { report: Report, url: String }
```

## The authoring process (flavour-independent)

The end-to-end flow is the same whichever syntax wins. Top-down (sketch-first):

```
$ aion new research-report            # scaffolds workflow.aion + actions.gleam + gleam.toml
$ $EDITOR workflow.aion               # sketch inputs, types, steps — name actions that don't exist yet

$ aion check
  workflow.aion ok — graph parses, wiring typechecks
  warning: plan(Brief) -> List(Question) — not implemented anywhere
  warning: investigate(Question, Dir) -> Finding — not implemented anywhere
  ...

$ aion scaffold plan investigate synthesize          # in-VM Gleam stubs (default tier)
  generated actions.gleam: 3 typed stubs, bodies are todo

$ aion scaffold upload_assets publish delete_assets --rust --worker publish-worker
  generated worker/publish-worker/: typed structs + handler skeletons + registration

# ...fill in the bodies (the only code you write)...

$ aion check                          # green: every named action implemented, wires typecheck
$ aion deploy                         # parse -> graph -> typecheck -> emit Gleam -> gleam build -> package
$ aion start research_report --input brief=./brief.json --input corpus=./sources/
```

Bottom-up (code-first) is the same registry approached from the other side: write typed
action functions first, `aion check` extracts their contracts, the workflow file wires
names that already resolve.

## File and directory inputs (kills the cat-JSON-into-the-workflow inelegance)

Two kinds of input, one rule — **the workflow never does I/O; the boundary does**:

1. **Structured inputs** (`brief: Brief` ← `--input brief=./brief.json`): the CLI reads
   the file, validates it against the input schema, and embeds the *parsed data* in the
   start request. Small, typed, recorded in history like any input.
2. **Bulk inputs** (`corpus: Dir`, also `File` ← `--input corpus=./sources/`): the CLI
   snapshots the file/tree into the haematite content-addressed store and passes the
   workflow a **durable handle** (a hash). Properties:
   - replay-safe: the snapshot is immutable — a replay sees exactly the bytes the first
     execution saw, even if the directory on disk has changed;
   - cross-node safe: any worker on any node fetches content by hash;
   - size-safe: history records a hash, not megabytes.
   Actions receive the handle and read through it (`corpus.read("notes/day1.md")`,
   `corpus.list()`); the workflow itself only passes it around.

JSON Schema for every boundary type is **emitted by the build** (an artifact for agents,
HTTP clients, docs) — never authored. Shapes are written exactly once.

## Comparison summary

| Dimension | A: JS | B: YAML | C: JSON | D: Nix | E: Pipes |
|---|---|---|---|---|---|
| Lines for the same workflow | ~60 | ~105 | ~150+ | ~65 | ~55 (but see caveat) |
| Familiar to mainstream devs | +++ | ++ (as config, not logic) | ++ | – | + (Elixir/FP crowd) |
| Conditions & expressions | real syntax, checkable | strings in quotes | strings in strings | real syntax | real syntax |
| Diamond dependency (revision step) | natural | `switch` contortion | contortion | natural (`if` expr) | breaks the pipe |
| Rebinding (`draft` revised) | `let` | painful | painful | shadow-binding wart | n/a (no names) |
| Canvas round-trip | good (closed DSL parses to graph) | trivial | trivial (it IS the format) | good | poor (sub-chains hide steps) |
| Uncanny-valley risk | the one real cost | none | none | low | low |
| Verdict | **primary candidate** | serialization-adjacent, not authoring | **canonical serialization** | rejected | maybe later, as sugar |

## Current lean (revised twice on 2026-07-02; pending Tom's read of sketch G)

**Author in the step document (sketch G): real YAML structure, one tiny expression
grammar in designated fields, `when:`+`as:` conditional rebinding, `about:` prose per
step. Serialize canonically to JSON; render the canvas from the same graph; expose the
JSON Schema as a structured-output lane for AI authors. Sketch F (TypeScript subset)
stays compatible as a later code-first view via the canonical model.**

How the lean moved: A (JS-lookalike) fell to the AI-authorship constraint — bespoke
imperative code gives LLMs plausible-but-wrong priors. F (real TS, statically extracted)
answered that, but the customer's aesthetic is consistently the declarative document
("define the types and schemas, then lay out the simple control-flow sequence, almost
like markdown"). G keeps B's document cleanliness while fixing its three wounds — and
for a *data* format the AI-drift risk mostly evaporates (schema validation / constrained
decoding), leaving only the small expression grammar to specify.

Open questions: the expression micro-grammar's exact boundary; `on failure:` shape for
multi-step compensation; whether `types:` defaults inline or to a sibling file in the
scaffold; step names with spaces vs snake_case as the canonical form.
