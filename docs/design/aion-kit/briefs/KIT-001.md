# KIT-001: Opaque payload and the aion_kit package skeleton

> **Cluster:** aion-kit
> **Design anchor:**
> - ADR-011 — The standard library carries cross-cutting primitives only; runtime harnesses stay out — aion_kit (the worker standard library) carries only primitives that apply across the board — data transformation/wrangling, rendering/templating, and the opaque payload — plus further cross-cutting primitives as they prove general. Runtime harnesses, the norn agent driver chief among them, stay as worker-side code that consumers (Meridian) own and integrate; they may live in a worker for convenience but never ship as standard-library surface. Rejected: bundling the norn/agent harness into the standard library — it couples a general toolkit to one runtime and ships a Meridian concern to every consumer.
> - ADR-012 — Workflow code stays thin: large activity results ride as opaque payloads — Workflow code stays thin. Large activity results ride between stages as opaque sealed payloads the workflow never opens; the workflow decodes only a small facts projection — the few fields it needs to route control flow (pass/fail, changed files, blocked, drift). Decoding and prompt rendering happen in the consuming activity on the worker, which has a full heap and runs the work once rather than on every replay. Rejected: decoding and rendering full reports in workflow code — it puts the determinism boundary's heavy lifting in the wrong place and was the regression that exposed the beamr crash.
> - ADR-001 — No arbitrary limits, no assumed defaults — Configurable values come from the builder/author or are deferred to the layer that owns them (e.g. beamr's own defaults). No caps, rate limits, or hardcoded defaults invented at the aion layer. Values are discussed before implementation. Rejected: convenience defaults — they are decisions made for the user without telling them.
> - ADR-002 — No backwards compatibility during the build — Replace, don't add alongside. No compat shims, no zombie code, no #[deprecated] markers. Breaking changes are made cleanly and consumers move forward. Rejected: incremental deprecation cycles — they double the surface under test for an audience of zero.
> **Checklist:**
> - C1 — gleam/aion_kit/gleam.toml exists naming the package aion_kit, target erlang, with dependencies gleam_stdlib, gleam_json, and aion_flow, and dev-dependency gleeunit, matching the gleam/aion_flow manifest shape (licences array, github repository).
> - C2 — gleam/aion_kit/src/aion_kit.gleam is a declaration-only package-root module (a PackageRoot marker type and module docs only, no functions), matching the aion_flow/aion_client root convention.
> - C3 — cd gleam/aion_kit && gleam build and gleam test both exit 0 against the aion_flow dependency, and gleam format --check reports no changes.
> - C4 — aion_flow exposes a Sealed pass-through type in src/aion/payload.gleam carrying a content-type tag and the raw payload bytes as a String, with no accessor that returns a decoded structured view of the bytes.
> - C5 — A workflow can hold a Sealed and forward it as the next activity's input without decoding: a test seals a value, reads only its raw bytes, and decodes the value only with an explicit decoder on the far side — the Sealed type itself never materialises the decoded structure.
> - C6 — aion_kit/payload.seal turns a typed value into a Sealed via a caller-supplied json encoder and an explicit content-type, inventing no default content-type.
> - C7 — aion_kit/payload.raw returns the Sealed's opaque bytes (and content_type accessor returns the tag) so a consuming activity can decode them with its own typed decoder.
> - C8 — aion_kit/payload.peek decodes ONLY a caller-supplied thin facts view from a Sealed, ignoring every field the view does not name, and returns a typed decode error (never a panic) when the bytes do not satisfy the thin decoder; a test peeks a small facts view out of a sealed report far larger than the view.
> - C17 — No file in gleam/aion_kit/src nor gleam/aion_flow/src/aion/payload.gleam imports aion/workflow, aion/activity, aion/query, aion/signal, references the norn agent driver, or spawns any process/port — grep for those imports and for 'norn' over the package finds no matches (CN1, ADR-011).
> - C18 — No public function in aion_kit or the Sealed accessors reads a wall clock, draws entropy, or performs IO; library code contains no panic, assert, or let-assert — decode and lookup failures are Result/Option values (CN2, CN7).
> **Stories:**
> - S3 (Activity Author, Writing worker-side activity bodies that build prompts and wrangle reports) — As an activity author, I want to seal a large typed result into an opaque payload and to unseal one I receive with my own decoder so that I do the heavy decode once, on the worker, rather than forcing the workflow to open it.
> - S5 (Workflow Author, Writing deterministic workflow code that threads activity results) — As a workflow author, I want to hold an activity result as an opaque payload and pass it straight to the next activity without decoding it so that my deterministic workflow heap stays small and replay never re-decodes a large report.
> - S6 (Workflow Author, Writing deterministic workflow code that threads activity results) — As a workflow author, I want to peek only the few facts I need to route control flow (pass/fail, blocked, changed files, drift) out of an opaque payload so that I can branch on a result without materialising its whole structure.
> - S7 (Workflow Author, Writing deterministic workflow code that threads activity results) — As a workflow author, I want every toolkit primitive to be pure and deterministic so that I can call the same projection and JSON helpers from workflow code or activity code without risking a replay divergence.
> - S8 (Future Maintainer, Reading and extending the toolkit months later) — As a future maintainer, I want the toolkit scoped to cross-cutting primitives with the runtime harnesses kept out so that I can tell at a glance what belongs in the standard library and what is a consumer's own concern.
> - S10 (Meridian Consumer, Integrating the kit into their own workers) — As a Meridian engineer building a worker, I want to depend on aion_kit and use its templating, JSON wrangling, and payload helpers without it dragging in the norn agent driver so that the toolkit couples to no agent runtime and I integrate my own harness on top.
> - S11 (Meridian Consumer, Integrating the kit into their own workers) — As a Meridian engineer, I want the opaque payload type to live in the authoring SDK my workflows already import so that threading a sealed result between activities needs no extra dependency in my deterministic workflow code.

## Purpose

Establishes the cluster's foundation and its keystone: the aion_kit Gleam package skeleton (manifest + declaration-only root) and the opaque sealed-payload mechanism that lets a deterministic workflow carry an activity result it never opens (ADR-012). The TYPE a workflow holds — Sealed — is added to the authoring SDK aion_flow (workflow code imports it); the HELPERS that produce and read it — seal, raw, peek — land in aion_kit/payload (worker-side primitives). The split is the design's central boundary call (see solution): hold-and-forward is workflow vocabulary, open-and-decode is activity vocabulary, and the line between them is the determinism boundary. This is the only intra-cluster dependency: KIT-002 (template) and KIT-003 (json) build on this skeleton. brief_dev is NOT reshaped here — that is RM-023; this brief ships and proves the mechanism with its own tests.

## Task

Create the gleam/aion_kit package: gleam.toml (depending on aion_flow, gleam_stdlib, gleam_json; dev-dep gleeunit; matching the gleam/aion_flow manifest shape exactly — erlang target, licences array, github repository) and the declaration-only root module src/aion_kit.gleam (a PackageRoot marker, matching aion_flow.gleam and aion_client.gleam). Add the Sealed pass-through type to aion_flow at src/aion/payload.gleam: an opaque type wrapping a content-type tag (String) and the raw payload bytes (String — one binary, not a decoded structure), with a constructor and the two accessors content_type and raw_bytes, and NO accessor that returns a decoded view. Create src/aion_kit/payload.gleam with the helpers over aion_flow's Sealed: seal(value, encoder, content_type) building a Sealed from a typed value via a caller-supplied json encoder and explicit content-type (no default content-type — ADR-001); raw(sealed) returning the opaque bytes and content_type(sealed) returning the tag for a consuming activity to decode with its own decoder; peek(sealed, decoder) decoding ONLY a caller-supplied thin facts view (the ADR-012 facts projection over a Sealed), ignoring unnamed fields, returning Result(view, codec.DecodeError) — typed failure, never a panic. Write the tests: gleam/aion_flow/test/payload_test.gleam (Sealed construction, accessors, and the hold-without-decode property — seal a value, forward only raw bytes, decode on the far side) and gleam/aion_kit/test/payload_test.gleam (seal/raw/peek round-trip, and peeking a small facts view from a sealed report far larger than the view). In scope: the package skeleton, the Sealed type, the payload helpers, their tests, and the purity/boundary guarantees C17/C18 for the files this brief introduces. Out of scope: template (KIT-002), json wrangling (KIT-003), and any reshape of brief_dev or its prompts.gleam/enrich.gleam (RM-023).

### R3: seal, raw, and content_type helpers

WHEN seal(value, encoder, content_type) is called, THE SYSTEM SHALL encode value to a JSON string via the caller-supplied encoder and wrap it in a Sealed tagged with the caller-supplied content_type. ### R4: peek: a thin facts view from a Sealed

Add peek to gleam/aion_kit/src/aion_kit/payload.gleam. WHEN peek(sealed, decoder) is called, THE SYSTEM SHALL decode ONLY the caller-supplied thin facts view from the Sealed's bytes and SHALL ignore every field the view's decoder does not name (the ADR-012 facts projection over a Sealed: the workflow reads the few fields it needs to route control flow — pass/fail, blocked, changed files, drift — without materialising the whole report). IF the bytes do not satisfy the thin decoder, THEN THE SYSTEM SHALL return a typed codec.DecodeError and SHALL NOT panic. peek SHALL NOT materialise or return any field outside the supplied view, and SHALL be pure (no IO, no clock, no entropy) so a workflow may call it across the determinism boundary (P3, invariant 2).

**Acceptance:**
- Given a Sealed whose bytes are a JSON object with ten fields, peek(sealed, decoder) where decoder names only two of them returns Ok of a two-field view and the other eight fields are absent from the result.
- peek(sealed, decoder) on bytes that are not valid JSON, or that lack a field the decoder requires, returns Error(codec.DecodeError(...)) — verified by asserting the Error branch, with no panic.
- A test seals a report value far larger than the facts view (e.g. a record with a long string field and a list) and peeks a small facts view (e.g. a single boolean and one string) successfully (C8, S6).
- grep -E 'panic|let assert' gleam/aion_kit/src/aion_kit/payload.gleam produces no matches.
- cd gleam/aion_kit && gleam test runs the payload suite and the peek-ignores-the-rest case passes.

## Boundaries

- SHALL NOT create aion_kit/template.gleam or any text-composition primitive — that is KIT-002.
- SHALL NOT create aion_kit/json.gleam or any project/merge/pluck helper — that is KIT-003.
- SHALL NOT modify, reshape, or rewire examples/stacked-dev (brief_dev, prompts.gleam, enrich.gleam, activities, locals) — consuming the toolkit is RM-023.
- SHALL NOT re-export, wrap, or alias aion_client's Payload or brief_dev's bespoke helpers — aion_kit is the replacement, not a shim (ADR-002).
- SHALL NOT add a default content-type, a size or depth cap, or any hardcoded bound to Sealed, seal, raw, or peek (ADR-001).
- SHALL NOT add an accessor or function that returns a decoded structured view of a Sealed's bytes (CN3) — only raw, content_type, and the caller-decoder peek read the contents.
- SHALL NOT add a dependency on a worker crate, the norn driver, or any agent/process-spawning runtime (ADR-011).
