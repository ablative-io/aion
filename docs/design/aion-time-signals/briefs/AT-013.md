# AT-013: Fix signal delivery failure on beamr 0.3.15

> **Cluster:** aion-time-signals
> **Depends on:** AT-003

## Purpose

The signal_records_history_and_delivers_mailbox_marker integration test fails on beamr 0.3.15 because enqueue_atom_message returns false when delivering a signal to a workflow process. Process scheduling semantics changed in beamr 0.3.15 and the signal delivery path does not handle the case where the target process is not yet ready to receive. This brief fixes the delivery path to be robust against process scheduling timing and ensures signal integration tests pass reliably.

## Task

Investigate why enqueue_atom_message returns false on beamr 0.3.15 (likely the target process is not yet scheduled/ready). Fix the signal delivery path to handle this case -- either by adding a process-ready check before delivery, retry with backoff, or adapting to the new scheduling semantics. Ensure deliver_signal_received propagates or retries on delivery failure rather than silently dropping the signal. Verify the signal integration test passes reliably.

## Requirements

### R1: Investigate and fix signal delivery path for beamr 0.3.15 compatibility

WHEN deliver_signal_received attempts to deliver a signal to a workflow process via enqueue_atom_message and the call returns false, THE SYSTEM SHALL handle the failure rather than silently dropping the signal. The fix SHALL be compatible with beamr 0.3.15 process scheduling semantics.

**Acceptance:**
- The root cause of enqueue_atom_message returning false on beamr 0.3.15 is identified and documented in a code comment.
- The signal delivery path in the runtime handle is updated to handle the false return from enqueue_atom_message.
- The fix does not introduce busy-waiting or unbounded retries.
- Signal delivery succeeds for processes that are alive but not yet scheduled (the common 0.3.15 case).

**Files:**
- modify: crates/aion/src/runtime/handle.rs

### R2: deliver_signal_received handles enqueue failure with retry or typed error

WHEN enqueue_atom_message returns false, THE SYSTEM SHALL either retry delivery with bounded backoff or propagate a typed SignalError explaining that the target process could not accept the message. THE SYSTEM SHALL NOT silently drop the signal.

**Acceptance:**
- If retry strategy is chosen: delivery retries with bounded exponential backoff (configurable max attempts), and returns a typed error after exhausting retries.
- If error propagation strategy is chosen: a SignalError::DeliveryFailed (or equivalent) variant is returned with the process ID and reason.
- The signal router logs a warning on delivery failure with the workflow ID, signal name, and process ID.
- The Recorder still records the SignalReceived event in history regardless of delivery outcome (the event is durable; delivery is best-effort to the live process).

**Files:**
- modify: crates/aion/src/signal/router.rs
- modify: crates/aion/src/runtime/handle.rs

### R3: Signal integration test passes reliably on beamr 0.3.15

THE SYSTEM SHALL ensure signal_records_history_and_delivers_mailbox_marker and all other signal integration tests pass reliably on beamr 0.3.15 without timing-dependent flakiness.

**Acceptance:**
- signal_records_history_and_delivers_mailbox_marker passes on beamr 0.3.15.
- The test does not rely on immediate process scheduling -- it waits for the process to be ready or uses the retry mechanism from R1/R2.
- Running the signal test suite 10 times in a row produces zero failures.
- No #[ignore] or timing-based skip logic is added to the test.

**Files:**
- modify: crates/aion/tests/engine_signal.rs

### R4: Process-ready check before signal delivery

BEFORE delivering a signal via enqueue_atom_message, THE SYSTEM SHALL verify that the target process is alive and schedulable. IF the process is not ready, THE SYSTEM SHALL wait (bounded) for it to become ready rather than delivering to a not-yet-scheduled process.

**Acceptance:**
- A process-ready check (using beamr's process status API or equivalent) gates signal delivery.
- If the process is alive but not yet scheduled, the delivery path waits with bounded timeout for it to become schedulable.
- If the process is dead or the wait times out, the delivery returns a typed error (not a panic or silent drop).
- The ready-check does not introduce a TOCTOU race that is worse than the current failure mode -- the subsequent enqueue failure is still handled by R2.

**Files:**
- modify: crates/aion/src/runtime/handle.rs
