# AT-002: Durable timer service: schedule, wheel arm, fire-once delivery

> **Cluster:** aion-time-signals
> **Depends on:** AT-001
> **Checklist:**
> - C4 — Scheduling a durable timer records it via EventStore::schedule_timer with the timer's TimerId and fire_at, and records a TimerStarted event in the workflow history.
> - C5 — Scheduling a durable timer arms a beamr timer-wheel entry (via the engine seam) so the timer fires live during normal execution without a store read.
> - C6 — When a wheel entry fires, the service records a TimerFired event and delivers TimerFired to the owning workflow process's mailbox.
> - C7 — A timer is fired exactly once: a timer already recorded as TimerFired (by the wheel) is not fired again by recovery, and vice versa.
> **Stories:**
> - S1 (Workflow Author, Writing Durable, Interactive, Concurrent Workflows) — As a workflow author, I want my workflow to sleep for an arbitrary duration and resume after a VM restart so that long delays are durable, not lost on a crash.
> - S7 (Engine, Driving Live Interaction and Recording It) — As the engine, I want every timer firing, signal arrival, and child outcome recorded as an aion-core Event so that replay can return the same observation without re-waiting or re-spawning.
> - S8 (Engine, Driving Live Interaction and Recording It) — As the engine, I want a timer fired exactly once whether it fires on the live wheel or via recovery so that replay sees one TimerFired, never two and never none.

## Purpose

Build the live tier of the two-tier timer: scheduling a durable timer records it in the event store and arms a beamr timer-wheel entry, and a wheel firing records TimerFired and delivers it to the owning workflow process. This is the microsecond live path; AT-003 adds the recovery path. The exactly-once rule that reconciles the two tiers is established here.

## Task

Implement a TimerService that, given the EngineHandle seam and an EventStore, schedules a durable timer (record via schedule_timer + a TimerStarted event, then arm the wheel via the seam) and handles a wheel firing (record TimerFired, deliver TimerFired to the owning workflow process mailbox). Enforce fire-once: a timer already recorded as fired or cancelled is not fired again. Do not implement recovery polling here (AT-003) or the named/sleep API surface (AT-004) — this is the core service those build on.

## Requirements

### R1: Schedule a durable timer

WHEN a durable timer is scheduled for a workflow, THE SYSTEM SHALL record it via EventStore::schedule_timer with its TimerId and fire_at AND append a TimerStarted event to the workflow history. THE SYSTEM SHALL NOT consider a timer scheduled until both records succeed.

**Acceptance:**
- timer_service.rs defines TimerService holding an EngineHandle and an Arc<dyn EventStore>.
- schedule records the timer via EventStore::schedule_timer(workflow_id, timer_id, fire_at) and appends a TimerStarted { timer_id, fire_at } event (with the expected-sequence guard from the event model).
- If either the schedule_timer call or the append fails, the operation returns a thiserror error and the caller can observe the failure (no partial silent success).
- A test schedules a timer against InMemoryStore and asserts both the timer entry (visible via expired_timers at/after fire_at) and the TimerStarted event are present.

### R2: Arm the timer wheel for live firing

WHEN a durable timer is scheduled and its owning workflow process is resident, THE SYSTEM SHALL arm a beamr timer-wheel entry (via the engine seam) for fire_at so the timer fires live without a store read.

**Acceptance:**
- After recording, schedule resolves the owning workflow via the seam and, if Resident, arms a wheel entry (seam arm method) for the workflow process + TimerId + fire_at.
- If the workflow is NonResident at schedule time, the timer is still recorded (R1) and is left to recovery (AT-003) — schedule does not fail on non-residency.
- A test using the fake EngineHandle asserts that scheduling a timer for a Resident workflow arms exactly one wheel entry with the correct fire_at.
- A test asserts that scheduling for a NonResident workflow records the timer but arms no wheel entry.

### R3: Fire on wheel tick: record and deliver

WHEN a wheel entry fires for a timer, THE SYSTEM SHALL append a TimerFired event and deliver a TimerFired message to the owning workflow process's mailbox via the engine seam.

**Acceptance:**
- A fire handler, invoked when the wheel entry elapses, appends TimerFired { timer_id } to the workflow history and then delivers a TimerFired message to the workflow process mailbox via the seam.
- Delivery uses the resolved process handle; if the process became NonResident between arm and fire, the TimerFired event is still recorded so recovery/replay can deliver it (no silent drop, CO7).
- A test drives a fake wheel fire and asserts a TimerFired event was appended and a TimerFired mailbox message was delivered.

### R4: Fire-once guard

THE SYSTEM SHALL fire a timer at most once. IF a timer already has a recorded TimerFired or TimerCancelled event, THEN the fire handler SHALL NOT record a second TimerFired and SHALL NOT deliver a duplicate TimerFired message.

**Acceptance:**
- Before recording TimerFired, the fire handler checks whether the timer is already terminal (TimerFired or TimerCancelled recorded) for that workflow; if so it is a no-op.
- The check is concurrency-safe so a wheel fire and a recovery fire (AT-003) for the same timer cannot both record TimerFired (e.g. via the append expected-sequence guard plus a re-read, or an internal dedup keyed by (workflow_id, timer_id)).
- A test fires the same timer twice (simulating wheel + recovery) and asserts exactly one TimerFired event exists and exactly one mailbox message was delivered.
- A test fires a timer that was already cancelled and asserts no TimerFired is recorded.

## Boundaries

- SHALL NOT implement startup or periodic recovery polling — that is AT-003.
- SHALL NOT implement start_timer/cancel_timer/sleep public surface — that is AT-004 (this brief provides the TimerService they call).
- SHALL NOT define TimerStarted/TimerFired/TimerCancelled event shapes — those are aion-core; this brief records them.
- SHALL NOT hardcode any interval or timeout — none are introduced here (CO10).
