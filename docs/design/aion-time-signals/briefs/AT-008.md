# AT-008: Child workflow spawning: linked process, await and fire-and-forget

> **Cluster:** aion-time-signals
> **Depends on:** AT-001

> **Stories:**

> - S10 (Engine, Driving Live Interaction and Recording It) — As the engine, I want cancellation to terminate losing or remaining child processes and record it so that no orphaned children leak and replay reconstructs which children were cancelled.

## Purpose

Let a parent workflow compose larger processes from smaller ones. Spawning a child asks AE to start a new execution — its own WorkflowId/RunId and event history — as a process linked to the parent, recorded in the parent's history. The parent can await the result or fire and forget, and cancelling the parent propagates to the child.

## Task

Implement child-workflow spawning on the engine seam. spawn requests AE to start a linked child execution and records ChildWorkflowStarted. spawn_and_wait blocks the parent on its mailbox until the child result arrives, recording ChildWorkflowCompleted or ChildWorkflowFailed. A fire-and-forget spawn records the start and detaches via monitor. Cancelling the parent propagates over the link to the child. This brief owns the spawn-request / result-collection / recording mechanics; AE owns the child's lifecycle.

## Requirements

### R1: Spawn a linked child with its own history

WHEN a parent spawns a child workflow, THE SYSTEM SHALL request AE (via the seam) to start a new workflow execution with a distinct WorkflowId/RunId and its own event history as a process LINKED to the parent, AND SHALL append ChildWorkflowStarted to the parent's history.

**Acceptance:**
- spawn.rs defines a spawn(parent_id, child_type, input) that calls the seam child-spawn request (linked to the parent process) and receives the child's WorkflowId + process handle.
- ChildWorkflowStarted { child_id, workflow_type, input } is appended to the PARENT's history.
- The child is started with a distinct WorkflowId and RunId (the child has its own history; the parent only records the start/outcome).
- A test (fake seam) spawns a child, asserts a child-spawn request was made linked to the parent, and asserts ChildWorkflowStarted is in the parent history with the child's id.

### R2: spawn_and_wait collects the child result

WHEN spawn_and_wait is used, THE SYSTEM SHALL block the parent on its mailbox until the child's result arrives, appending ChildWorkflowCompleted on success and ChildWorkflowFailed on child failure.

**Acceptance:**
- spawn_and_wait spawns (R1) then blocks in selective receive on the parent mailbox for the child's result message (matched to the child id).
- On child success, ChildWorkflowCompleted { child_id, result } is appended to the parent history and the result is returned to the parent.
- On child failure, ChildWorkflowFailed { child_id, error } is appended and the failure is surfaced to the parent (as a typed error).
- A test drives a fake child-completion message and asserts ChildWorkflowCompleted is recorded and the result returned; a second test drives a failure and asserts ChildWorkflowFailed.

### R3: Fire-and-forget spawn detaches

WHEN a child is spawned fire-and-forget, THE SYSTEM SHALL append ChildWorkflowStarted and detach (monitor rather than blocking await), not waiting for the child result.

**Acceptance:**
- A fire-and-forget spawn variant records ChildWorkflowStarted and returns immediately without blocking on a result.
- The parent monitors (not blocking-links-and-awaits) the child so the parent is not blocked and a child exit arrives as a monitor message rather than blocking the parent.
- A test spawns fire-and-forget, asserts ChildWorkflowStarted is recorded, and asserts the call returns without waiting for a result message.

### R4: Parent cancellation propagates to the child

WHEN the parent workflow process is cancelled, THE SYSTEM SHALL propagate cancellation over the link to the child process.

**Acceptance:**
- Because spawn (R1) links the child to the parent, parent cancellation/termination propagates an exit signal to the linked child via the link (no separate teardown bookkeeping needed for the await case).
- The mechanism is the link established at spawn; a doc comment states parent cancellation kills linked children.
- A test cancels the parent (fake seam termination) and asserts the linked child receives a propagated exit (observable via the fake).

## Boundaries

- SHALL NOT own the child's lifecycle/supervision/replay — AE owns the child execution; this brief records the parent-side start/outcome and collects results.
- SHALL NOT implement all/race/map — those are AT-010/011/012 (they reuse this spawn mechanism for workflow children).
- SHALL NOT define the Gleam workflow.spawn/spawn_and_wait surface — that is AF.
- SHALL NOT record events in the CHILD's history — the child records its own; the parent records ChildWorkflowStarted/Completed/Failed only.
