# AT-001: Engine seam and time/signal/query/child/concurrency module scaffolding

> **Cluster:** aion-time-signals
> **Checklist:**
> - C1 — An engine-facing handle/trait (consumed, not owned, by this cluster) exposes: resolve a WorkflowId to a live process handle, deliver a message to a workflow process mailbox, request AE to spawn a child workflow, and arm/disarm a beamr timer-wheel entry.
> - C2 — The engine seam is an explicit trait (or handle type) so this cluster never manages workflow process lifecycle, supervision, or module loading directly.
> - C3 — The time/signal/query/child/concurrency modules live under the aion crate with mod.rs files containing only pub mod declarations and pub use re-exports.
> **Stories:**
> - S7 (Engine, Driving Live Interaction and Recording It) — As the engine, I want every timer firing, signal arrival, and child outcome recorded as an aion-core Event so that replay can return the same observation without re-waiting or re-spawning.

## Purpose

Establish the narrow engine-facing seam this cluster consumes (residency resolution, mailbox delivery, child-spawn request, timer-wheel arm/disarm) and the module skeleton for timers, signals, queries, child workflows, and concurrency. Every later brief builds against this seam; defining it first keeps the AE boundary explicit and lets the whole cluster be tested with a fake engine handle.

## Task

Define an engine-facing trait (the seam) inside the aion crate that exposes exactly four capabilities: resolve a WorkflowId to a live process handle (or report non-resident/terminal/unknown), deliver a message to a workflow process mailbox, request AE to spawn a child workflow execution as a linked process, and arm/disarm a beamr timer-wheel entry for a given fire time. Provide the module declarations (mod.rs files containing only pub mod + pub use) for the time, signal, query, child, and concurrency modules. Provide an in-test fake implementation of the seam so subsequent briefs can test without AE. This brief defines no timer/signal/query/child behaviour itself.

## Requirements

### R1: EngineHandle seam trait

THE SYSTEM SHALL define an EngineHandle trait exposing residency resolution, mailbox delivery, child-spawn request, and timer-wheel arm/disarm. THE SYSTEM SHALL NOT expose workflow lifecycle, supervision, or module-loading operations on this trait.

**Acceptance:**
- engine_seam.rs defines a trait (e.g. EngineHandle) with methods to: resolve a WorkflowId to a process-handle result that distinguishes Resident(handle) / NonResident / Terminal / Unknown; deliver a message to a workflow process mailbox by handle; request a child workflow spawn (input: parent WorkflowId + child workflow_type + input Payload + RunId, returns the child's process handle + WorkflowId) linked to the parent; arm a timer-wheel entry (workflow process + TimerId + fire_at) and disarm one (workflow process + TimerId).
- The trait is Send + Sync so services holding it are usable across scheduler threads.
- The trait carries no method that starts, supervises, or tears down a top-level workflow process — only the four seam capabilities (the residency-resolution and child-spawn methods are requests into AE, not lifecycle ownership here).
- Doc comments state that AE implements this trait and that this cluster never manages process lifecycle directly (CO6).

### R3: Fake engine handle for tests

THE SYSTEM SHALL provide a test-only fake implementation of EngineHandle so this cluster's tests can exercise delivery, residency, child spawn, and wheel arm/disarm without AE.

**Acceptance:**
- A fake EngineHandle (behind cfg(test) or a test-support module) records delivered mailbox messages, configurable residency answers per WorkflowId, requested child spawns, and armed/disarmed wheel entries, so tests can assert on them.
- The fake lets a test set a WorkflowId to Resident / NonResident / Terminal / Unknown and observe how a service reacts.
- The fake lets a test drive a wheel-entry fire (invoke the callback/path a real wheel would) so timer-firing logic is testable without real time passing.
- A unit test constructs the fake, marks a workflow Resident, delivers a message, and asserts the message was captured.
