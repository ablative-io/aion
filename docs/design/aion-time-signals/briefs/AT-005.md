# AT-005: Signal router: record-then-deliver to the workflow mailbox

> **Cluster:** aion-time-signals
> **Depends on:** AT-001
> **Checklist:**
> - C14 — Delivering a signal records a SignalReceived event (signal name + Payload) before/atomically with delivery, so a crash after delivery but before consumption is recoverable.
> - C15 — The signal is delivered to the target workflow process's mailbox so the workflow's selective receive picks it up without polling.
> - C16 — Signal delivery resolves a WorkflowId to a live process via the engine seam; the router does not manage process lifecycle.
> **Stories:**
> - S3 (Workflow Author, Writing Durable, Interactive, Concurrent Workflows) — As a workflow author, I want to wait for a named signal and have it delivered the instant it arrives so that my workflow reacts to external events without polling.
> - S7 (Engine, Driving Live Interaction and Recording It) — As the engine, I want every timer firing, signal arrival, and child outcome recorded as an aion-core Event so that replay can return the same observation without re-waiting or re-spawning.

## Purpose

Build the live signal path: an external signal is recorded durably as SignalReceived and delivered to the target workflow process's mailbox so the workflow's selective receive picks it up in microseconds. The record-before-deliver ordering is what makes a signal survive a crash between delivery and consumption.

## Task

Implement a SignalRouter that, given an EngineHandle and an EventStore, records SignalReceived (signal name + Payload) and delivers the signal to the target workflow process's mailbox. Record before (or atomically with) delivery so a crash after delivery but before consumption is recoverable. Resolve the WorkflowId to a live process via the seam. Non-resident and terminal/unknown handling lands in AT-006; this brief covers the resident happy path and the record-ordering guarantee.

## Requirements

### R1: Record SignalReceived before delivery

WHEN a signal is routed to a workflow, THE SYSTEM SHALL append a SignalReceived event (signal name + Payload) BEFORE delivering the signal to the mailbox. THE SYSTEM SHALL NOT deliver a signal that has not been recorded.

**Acceptance:**
- router.rs defines SignalRouter holding an EngineHandle and an Arc<dyn EventStore>.
- signal(workflow_id, name, payload) appends SignalReceived { name, payload } (with the expected-sequence guard) and only then delivers to the mailbox.
- If the append fails, no delivery occurs and the error is returned (no silent delivery without a record, CO7).
- A test routes a signal to a Resident workflow (fake seam) and asserts the SignalReceived event is appended before the mailbox message is recorded as delivered (ordering observable via the fake).

### R2: Deliver to the workflow mailbox

WHEN a signal has been recorded and the target workflow is resident, THE SYSTEM SHALL deliver the signal to the target workflow process's mailbox via the engine seam so selective receive picks it up without polling.

**Acceptance:**
- After recording, the router delivers a signal message (name + payload) to the resolved process handle's mailbox via the seam deliver method.
- The delivered message carries the signal name so the workflow's selective receive can match by name.
- A test routes a signal and asserts a mailbox message with the correct name + payload was delivered to the right process handle.

**Files:**

### R3: Resolve workflow via the seam, not lifecycle ownership

THE SYSTEM SHALL resolve a WorkflowId to a live process via the engine seam. THE SYSTEM SHALL NOT start, suspend, resume, or supervise a workflow process from the router.

**Acceptance:**
- The router calls the seam resolve method to obtain the process handle; it contains no process spawn/supervision/lifecycle code (CO6).
- A test marks a WorkflowId Resident in the fake seam and asserts the router uses the returned handle for delivery.
- No router code path constructs or tears down a workflow process.

## Boundaries

- SHALL NOT handle non-resident delivery or terminal/unknown targets — that is AT-006.
- SHALL NOT define replay of SignalReceived — AD returns the recorded signal on restore; this brief only records and delivers live.
- SHALL NOT define the network endpoint that receives an external signal — that is AW; this is the in-engine delivery it calls.
- SHALL NOT define the Gleam workflow.receive surface — that is AF.
