# AT-003: Timer recovery: startup sweep and periodic recovery tick

> **Cluster:** aion-time-signals
> **Depends on:** AT-002
> **Checklist:**
> - C8 — On engine startup the timer service polls EventStore::expired_timers(now) and, for each entry whose TimerFired has not been recorded, records TimerFired and delivers it to the owning workflow process.
> - C9 — A periodic recovery tick (interval engine-configured, not hardcoded) re-polls expired_timers as a safety net for timers whose owning process was not resident on the wheel.
> - C10 — Recovery is idempotent: re-running it does not double-fire a timer already recorded as fired or cancelled.
> **Stories:**
> - S8 (Engine, Driving Live Interaction and Recording It) — As the engine, I want a timer fired exactly once whether it fires on the live wheel or via recovery so that replay sees one TimerFired, never two and never none.
> - S9 (Engine, Driving Live Interaction and Recording It) — As the engine, I want to recover elapsed timers on startup and on a periodic tick so that a workflow that slept across a restart wakes correctly.

## Purpose

Build the recovery tier of the two-tier timer. A timer whose owning process slept across a VM restart was never armed on the live wheel; recovery polls the event store for elapsed timers and fires them. This is what makes a month-long sleep durable. It reuses AT-002's fire-once-guarded fire path so the live and recovery tiers never double-fire.

## Task

Implement a recovery component that, on engine startup and on a periodic tick, polls EventStore::expired_timers(now) and fires each due timer that has not already been recorded as fired or cancelled, by delegating to the AT-002 fire path. The recovery-tick interval is supplied by engine configuration, never hardcoded. Recovery must be idempotent.

## Requirements

### R1: Startup recovery sweep

WHEN the engine starts, THE SYSTEM SHALL poll EventStore::expired_timers(now) once and, for each entry whose timer is not already terminal, fire it via the AT-002 fire path (record TimerFired + deliver).

**Acceptance:**
- recovery.rs defines a recover_on_startup(now) routine that calls expired_timers(now), filters out timers already recorded as TimerFired or TimerCancelled, and fires the remainder through the TimerService fire path.
- Firing reuses AT-002's fire-once-guarded path so a startup sweep and any live wheel entry cannot double-fire.
- A test schedules a timer with a past fire_at against InMemoryStore (simulating a timer elapsed while down), runs the startup sweep, and asserts TimerFired is recorded and delivered.
- A test with a timer whose fire_at is in the future asserts the startup sweep does not fire it.

### R2: Periodic recovery tick

WHILE the engine is running, THE SYSTEM SHALL periodically re-poll expired_timers at an engine-configured interval and fire any newly-due, non-terminal timers, as a safety net for timers whose owning process was not resident on the wheel.

**Acceptance:**
- A recovery loop runs on an interval supplied by engine configuration (a Duration parameter / config field), NOT a hardcoded constant (CO10).
- Each tick polls expired_timers(now) and fires due, non-terminal timers via the same fire path as R1.
- The interval value is injected (constructor/config), and a test drives the tick logic directly with an injected clock rather than waiting real wall-clock time (CO9).
- A test advances the injected clock past a timer's fire_at, triggers a tick, and asserts the timer fires exactly once.

### R3: Idempotent recovery

THE SYSTEM SHALL make recovery idempotent. WHEN recovery runs again over a timer already recorded as fired or cancelled, THE SYSTEM SHALL NOT fire it again and SHALL NOT deliver a duplicate message.

**Acceptance:**
- Running the startup sweep twice in a row over the same store fires each due timer exactly once total (the second run is a no-op for already-fired timers).
- A timer recorded as TimerCancelled is never fired by recovery.
- A test runs recovery twice and asserts exactly one TimerFired per due timer across both runs.
- A test cancels a timer (TimerCancelled present), then runs recovery, and asserts no TimerFired is recorded.

## Boundaries

- SHALL NOT reimplement the fire path — it reuses AT-002's fire-once-guarded TimerService fire.
- SHALL NOT hardcode the recovery-tick interval — it is engine-configured (CO10).
- SHALL NOT spawn or own the OS/runtime task that drives the periodic tick at production cadence — the brief provides the tick logic and an injected interval; wiring the live driver into the engine runtime is AE's startup responsibility. (Note this split.)
- SHALL NOT use wall-clock sleeps in tests — inject the clock (CO9).
