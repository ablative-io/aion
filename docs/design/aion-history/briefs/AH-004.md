# AH-004: History chain navigation across continuation runs

> **Cluster:** aion-history
> **Depends on:** AH-001, AH-003
> **Checklist:** C8
> **Stories:** S3

## Purpose

Enable operators to navigate the full chain of continuations for a workflow. When a workflow continues-as-new multiple times, each run has its own event history. Querying by WorkflowId should return all runs in continuation order so operators can trace the complete execution history for debugging, compliance, and auditing.

## Task

Add a read_run_chain method to the EventStore trait that, given a WorkflowId, returns the list of RunIds in continuation order (oldest to newest). Each entry carries the RunId, parent_run_id (None for the first run), and projected status. Add an Engine::describe_workflow method that returns the full chain with summary information. Implement in InMemoryStore and libSQL store.

## Requirements

### R1: EventStore::read_run_chain returns all runs in continuation order

THE SYSTEM SHALL add a read_run_chain method to the EventStore trait that returns a Vec<RunSummary> for a given WorkflowId, ordered from oldest to newest run. Each RunSummary carries run_id, parent_run_id (Option<RunId>), status (WorkflowStatus), started_at (DateTime), and closed_at (Option<DateTime>).

**Acceptance:**
- EventStore trait has a read_run_chain method.
- RunSummary type exists in aion-store with the specified fields.
- For a workflow with 3 continuations, read_run_chain returns 3 RunSummary entries in order.
- The first run has parent_run_id = None; subsequent runs have parent_run_id = previous run's RunId.
- A workflow with no continuations returns a single RunSummary.

**Files:**
- modify: crates/aion-store/src/store.rs

**Checklist:**
- C8

**Stories:**
- S3

### R2: InMemoryStore read_run_chain implementation

THE SYSTEM SHALL implement read_run_chain for InMemoryStore by scanning all histories for the given WorkflowId, extracting RunId and parent_run_id from WorkflowStarted events, and ordering by continuation chain.

**Acceptance:**
- InMemoryStore::read_run_chain returns correct results for single-run and multi-run workflows.
- Ordering is oldest-to-newest based on the parent_run_id chain.
- An unknown WorkflowId returns an empty vec.

## Boundaries

- SHALL NOT add server/HTTP/gRPC endpoints for chain navigation — those belong in the aion-server cluster.
- SHALL NOT modify the existing read_history method — it continues to return events for a single WorkflowId (all runs combined).
- SHALL NOT add dashboard UI for chain navigation — that belongs in the dashboard cluster.
