# AH-003: Engine continue_as_new lifecycle method with pending-work guard and atomic transition

> **Cluster:** aion-history
> **Depends on:** AH-001, AH-002
> **Checklist:** C4, C5, C7, C9
> **Stories:** S1, S2, S5

## Purpose

Implement the engine-level continue_as_new method that orchestrates the full transition: guard against pending work, record the terminal WorkflowContinuedAsNew event on the old run, deregister the old run, and start a new run under the same WorkflowId with a fresh RunId. The new run's WorkflowStarted carries parent_run_id for chain navigation. This is the core lifecycle method that makes unbounded-history workflows viable.

## Task

Create aion::lifecycle::continue_as_new module with a continue_as_new function following the same pattern as terminate::cancel/complete/fail. The function takes a ContinueAsNewRequest (input payload, optional workflow_type override) and: (1) checks for pending in-flight activities or children, failing with a clear error if any exist; (2) records WorkflowContinuedAsNew with the current RunId as parent_run_id; (3) notifies completion; (4) deregisters the old run; (5) starts a new run with start_workflow using the carried input and the same WorkflowId. Add Engine::continue_as_new as the public API method. The new run's WorkflowStarted must carry parent_run_id.

## Requirements

### R1: Pending-work guard rejects continue-as-new with in-flight activities

THE SYSTEM SHALL reject continue_as_new with EngineError when the workflow has pending in-flight activities or child workflows. THE SYSTEM SHALL NOT auto-cancel pending work — the workflow author must handle it explicitly.

**Acceptance:**
- Calling continue_as_new on a workflow with a dispatched-but-not-completed activity returns an error.
- Calling continue_as_new on a workflow with a started-but-not-completed child workflow returns an error.
- The error message clearly identifies that pending work exists.
- Calling continue_as_new on a workflow with no pending work succeeds.

### R2: Record terminal event and deregister old run

THE SYSTEM SHALL record a WorkflowContinuedAsNew event on the old run with the carried input payload, optional workflow type override, and the current RunId as parent_run_id. THE SYSTEM SHALL then deregister the old run from the registry, following the same pattern as cancel/complete/fail.

**Acceptance:**
- After continue_as_new, the old run's history ends with a WorkflowContinuedAsNew event.
- The old run is no longer in the registry.
- The old run's completion notifier is notified with a ContinuedAsNew outcome.
- The event's parent_run_id matches the old run's RunId.

### R3: Start new run under same WorkflowId

THE SYSTEM SHALL start a new workflow run under the same WorkflowId with a fresh RunId, using the carried input payload as the new run's input. If a workflow_type override is specified, the new run SHALL use that type; otherwise the same workflow type at its latest version. The new run's WorkflowStarted event SHALL carry parent_run_id linking to the old run.

**Acceptance:**
- After continue_as_new, a new run exists under the same WorkflowId.
- The new run has a different RunId from the old run.
- The new run's WorkflowStarted event carries the input from the continue_as_new call.
- The new run's WorkflowStarted event carries parent_run_id matching the old run's RunId.
- If workflow_type was specified, the new run uses that type.

### R4: Engine::continue_as_new public API

THE SYSTEM SHALL expose continue_as_new as a public async method on Engine, taking WorkflowId, RunId, input Payload, and optional workflow_type. THE SYSTEM SHALL gate the call through the shutdown gate, consistent with other lifecycle methods.

**Acceptance:**
- Engine::continue_as_new is a public async method.
- The method is gated through shutdown_gate.begin_operation().
- The method delegates to lifecycle::continue_as_new.
- Unit test: continue_as_new on an unknown workflow returns WorkflowNotFound.

## Boundaries

- SHALL NOT implement SDK bindings — those are a separate brief under the aion_flow cluster.
- SHALL NOT implement history chain navigation queries — that is AH-004.
- SHALL NOT modify the WorkflowStarted event structure if parent_run_id can be added as an optional field; if it requires a breaking change, coordinate with AC.
- SHALL NOT implement NIF-level continue-as-new dispatch — the engine method is called from the host side.
