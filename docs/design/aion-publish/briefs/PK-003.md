# PK-003: Publish order validation and automation script

> **Cluster:** aion-publish
> **Depends on:** PK-001, PK-002
> **Checklist:**
> - C12 — Publish order documented and validated against dependency graph
> - C13 — cargo publish --dry-run succeeds for every crate in order
> - C14 — Publish script or CI job automates the ordered publish
> **Stories:**
> - S4 (Rust Developer, External consumer who wants to embed Aion or build on its primitives) — I can run cargo publish --dry-run on any crate to verify it's publishable.
> - S5 (Aion Maintainer, Team member who publishes new versions to crates.io) — I can run a single script that publishes all crates in the correct dependency order.

## Purpose

Document the correct crates.io publish order, validate it with dry-runs, and provide an automation script so the team can publish reliably.

## Task

Derive the publish order from the dependency graph. Write a publish script (scripts/publish-crates.sh) that iterates crates in order, runs cargo publish --dry-run by default, and accepts a --live flag for actual publication. Validate the order with a full dry-run pass. Document the publish process in the repo.

## Requirements

### R1: Publish order derivation and documentation

THE SYSTEM SHALL document the leaf-first publish order based on the workspace dependency graph. The order SHALL be: aion-core, aion-store, aion-store-libsql, aion-package, aion-nif, aion, aion-proto, aion-server, aion-worker, aion-client, aion-cli.

**Acceptance:**
- Publish order is documented in scripts/publish-crates.sh and in the repo README.
- The order matches the actual dependency graph (verified by cargo metadata).
- No crate depends on a crate that appears later in the publish order.

**Files:**
- create: scripts/publish-crates.sh

**Checklist:**
- C12 — Publish order documented and validated against dependency graph

**Stories:**
- S5 (Aion Maintainer, Team member who publishes new versions to crates.io) — I can run a single script that publishes all crates in the correct dependency order.

### R2: Full dry-run validation

THE SYSTEM SHALL pass cargo publish --dry-run for every crate when executed in the documented publish order.

**Acceptance:**
- Running scripts/publish-crates.sh with no flags performs a dry-run of all crates.
- All 11 crates pass dry-run without errors.
- The script exits non-zero on the first failure.

**Checklist:**
- C13 — cargo publish --dry-run succeeds for every crate in order

**Stories:**
- S4 (Rust Developer, External consumer who wants to embed Aion or build on its primitives) — I can run cargo publish --dry-run on any crate to verify it's publishable.

### R3: Publish automation script

THE SYSTEM SHALL provide scripts/publish-crates.sh that iterates all workspace crates in dependency order and runs cargo publish for each. Default mode is --dry-run. A --live flag publishes for real.

**Acceptance:**
- scripts/publish-crates.sh is executable.
- Default invocation runs cargo publish --dry-run for each crate.
- scripts/publish-crates.sh --live runs cargo publish (no --dry-run) for each crate.
- The script logs each crate name and result as it proceeds.
- The script stops on the first failure.

**Files:**
- create: scripts/publish-crates.sh

**Checklist:**
- C14 — Publish script or CI job automates the ordered publish

**Stories:**
- S5 (Aion Maintainer, Team member who publishes new versions to crates.io) — I can run a single script that publishes all crates in the correct dependency order.

## Boundaries

- SHALL create a publish script that handles the full ordered publish.
- SHALL validate the dependency graph matches the publish order.
- SHALL NOT actually publish to crates.io — this brief validates with --dry-run only.
- SHALL NOT set up CI/CD publishing (that is an operational concern beyond this brief).
- SHALL NOT change any crate code or dependencies (PK-001 handles that).

- scripts/publish-crates.sh runs successfully in dry-run mode.
- All 11 crates report success from cargo publish --dry-run.
- shellcheck scripts/publish-crates.sh passes (if shellcheck is available).
