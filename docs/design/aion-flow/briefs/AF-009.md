# AF-009: Child workflows: ChildHandle, spawn, await

> **Cluster:** aion-flow
> **Depends on:** AF-008
> **Checklist:**
> - C30 — workflow.spawn(name, workflow_fn, input, codecs) -> ChildHandle(o, e) and workflow.spawn_and_wait(...) -> Result(o, e) carry the child output and error types.
> - C31 — child.await(ChildHandle(o, e)) -> Result(o, e) collects the recorded child result.
> **Stories:**
> - S8 (Workflow Author, Writing Durable Workflows in Gleam) — As a workflow author, I want to spawn child workflows and fan activities out with all/race/map, with the result types preserved, so that concurrency stays type-safe.

## Purpose

Let a workflow spawn other workflows as typed children. workflow.spawn yields a ChildHandle(output, error); spawn_and_wait blocks for the result; child.await collects a handle's recorded result. The child's output and error types are carried through.

## Task

Implement aion/child: ChildHandle(o, e); workflow.spawn(name, workflow_fn, input, codecs) -> ChildHandle(o, e) and workflow.spawn_and_wait(...) -> Result(o, e) wrappers; child.await(ChildHandle(o, e)) -> Result(o, e). Bind to AT's child-workflow spawner via the ffi layer, encoding the input and decoding the recorded child result.

## Requirements

### R1: ChildHandle and spawn

THE SYSTEM SHALL define ChildHandle(o, e) and workflow.spawn(name, workflow_fn, input, codecs) -> ChildHandle(o, e) starting a child workflow as a linked execution, carrying the child output and error types.

**Acceptance:**
- child.gleam defines an opaque ChildHandle(o, e) carrying the child's correlation/handle and the output/error Codecs.
- workflow.spawn(name, workflow_fn: fn(i) -> Result(o, e), input: i, codecs) encodes the input and binds to AT's child-spawn mechanism (linked process, ChildWorkflowStarted recorded), returning a ChildHandle(o, e).
- The workflow_fn's o and e types flow into the ChildHandle so await/spawn_and_wait return Result(o, e).
- A harness test spawns a typed child and asserts the handle carries the expected types (compiles) and a started observation is captured.

### R2: await and spawn_and_wait

THE SYSTEM SHALL define child.await(ChildHandle(o, e)) -> Result(o, e) collecting the recorded child result, and workflow.spawn_and_wait(...) -> Result(o, e) as the spawn-then-await convenience.

**Acceptance:**
- child.await(ChildHandle(o, e)) blocks for the child result via AT's await mechanism, decoding the recorded ChildWorkflowCompleted result with the output Codec or returning the decoded child error on ChildWorkflowFailed.
- workflow.spawn_and_wait(name, workflow_fn, input, codecs) -> Result(o, e) is spawn followed by await.
- On replay the recorded child result is returned without re-spawning (AT D8/AD); the SDK relies on AT/AD via ffi.
- A harness test (mocking the child result) asserts spawn_and_wait returns the decoded typed Ok and that a failing child returns the decoded typed Err.

## Boundaries

- SHALL NOT implement child-workflow lifecycle, linking, supervision, or result recording — those are AT-008 and AE. This brief binds to them.
- SHALL NOT implement all/race/map — that is AF-010.
- SHALL NOT let child input/output cross the boundary without Codecs (CO9).
