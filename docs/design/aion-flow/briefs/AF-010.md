# AF-010: Concurrency: all, race, map typed combinators

> **Cluster:** aion-flow
> **Depends on:** AF-009
> **Checklist:**
> - C32 — workflow.all(List(Activity(i, o))) -> Result(List(o), ActivityError) returns results in input order and fails fast on any child failure.
> - C33 — workflow.race(List(Activity(i, o))) -> Result(o, ActivityError) returns the first result and cancels the rest.
> - C34 — workflow.map(List(a), fn(a) -> Activity(i, o)) -> Result(List(o), ActivityError) performs dynamic fan-out and collects like all.
> **Stories:**
> - S8 (Workflow Author, Writing Durable Workflows in Gleam) — As a workflow author, I want to spawn child workflows and fan activities out with all/race/map, with the result types preserved, so that concurrency stays type-safe.

## Purpose

Give the author BEAM-native fan-out with the result types preserved. all collects N activities in input order and fails fast; race returns the first and cancels the rest; map produces activities dynamically from a list and collects like all.

## Task

Implement workflow/concurrency.gleam: all(List(Activity(i, o))) -> Result(List(o), ActivityError), race(List(Activity(i, o))) -> Result(o, ActivityError), and map(List(a), fn(a) -> Activity(i, o)) -> Result(List(o), ActivityError). Bind to AT's selective-receive collectors via the ffi layer, encoding inputs and decoding the typed results, carrying per-spawn correlation as AT requires.

## Requirements

### R1: all — ordered fan-out, fail-fast

THE SYSTEM SHALL define workflow.all(List(Activity(i, o))) -> Result(List(o), ActivityError) spawning all activities concurrently, returning results in input order, and failing fast on any activity failure.

**Acceptance:**
- all spawns the activities via AT's collector, decodes each result with its output Codec, and returns them in input order as Result(List(o), ActivityError).
- Any single activity failure fails the whole all (returning that ActivityError) and the remaining activities are cancelled (AT D9); the SDK relies on AT for cancellation via ffi.
- A harness test mocks three activities and asserts ordered results on success; a second test asserts a single failure fails the all.

### R2: race — first wins, cancel the rest

THE SYSTEM SHALL define workflow.race(List(Activity(i, o))) -> Result(o, ActivityError) returning the first result to arrive and cancelling the remaining activities.

**Acceptance:**
- race spawns the activities, returns the first arriving result decoded with its output Codec, and binds to AT's cancellation of the losers (recorded per AT D10).
- A harness test with two activities resolving at different simulated times asserts the first result wins and the loser is cancelled.
- The result type o is preserved (the list is homogeneous in output type).

### R3: map — dynamic fan-out

THE SYSTEM SHALL define workflow.map(List(a), fn(a) -> Activity(i, o)) -> Result(List(o), ActivityError) producing an activity per element and collecting like all.

**Acceptance:**
- map applies the element->Activity function to each item of the runtime list, then collects like all (ordered, fail-fast).
- A harness test maps over a list of inputs, mocks the produced activities, and asserts ordered typed results.
- A doc comment notes that homogeneous-output combinators cover the design's stated fan-out use; typed-tuple variants (all2/all3) are a later addition (per D11).
