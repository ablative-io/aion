# AF-003: Error taxonomy: ActivityError and engine-originated failures

> **Cluster:** aion-flow
> **Depends on:** AF-002
> **Checklist:**
> - C10 — ActivityError has Retryable and Terminal constructors carrying a message and structured details, mapping aion-core's retryable/terminal classification.
> - C11 — Engine-originated failures (timeout, cancellation, non-determinism, decode) are distinct typed variants, separate from author-returned ActivityError.
> - C12 — No public function panics, calls todo, or uses assert as control flow; every fallible operation returns a typed Result.
> **Stories:**
> - S4 (Workflow Author, Writing Durable Workflows in Gleam) — As a workflow author, I want to classify an activity failure as retryable or terminal once, in the type system, so that the engine applies my retry policy correctly without me re-stating the intent.

## Purpose

Express the retryable-vs-terminal classification in Gleam's type system, mapping aion-core's ActivityError, and define the distinct typed failures the engine originates (timeout, cancellation, non-determinism, decode) so an author never confuses an author-decided failure with an engine-decided one.

## Task

Define aion/error: ActivityError with Retryable and Terminal constructors carrying a message and structured details, mapping aion-core's classification. Define the engine-originated failure types (timeout, cancellation, non-determinism, decode) as distinct typed variants. Establish the no-panic / typed-Result discipline for the whole package.

## Requirements

### R1: ActivityError retryable/terminal split

THE SYSTEM SHALL define ActivityError with Retryable and Terminal constructors carrying a message and structured details, expressing aion-core's retryable/terminal classification.

**Acceptance:**
- error.gleam defines ActivityError with constructors Retryable(message, details) and Terminal(message, details).
- A doc comment states the engine retries on Retryable until the RetryPolicy's attempts are exhausted and fails immediately on Terminal, mapping aion-core's classification (aion-core C21).
- Helper constructors (e.g. retryable(message), terminal(message)) make the common case ergonomic.
- A unit test pattern-matches both constructors and asserts the classification is preserved.

### R2: Engine-originated failure types

THE SYSTEM SHALL define the engine-originated failures (timeout, cancellation, non-determinism, decode) as distinct typed variants, separate from author-returned ActivityError.

**Acceptance:**
- Distinct typed variants exist for timeout, cancellation, non-determinism violation, and decode failure (the latter wrapping the DecodeError from AF-002), used by the workflow primitives' return types (e.g. ReceiveError, the with_timeout result).
- These are not ActivityError constructors — an author cannot construct an engine-originated failure as if it were an activity decision, and a workflow handling code can distinguish them by type.
- A doc comment relates non-determinism to AD's NonDeterminismError and timeout/cancellation to the engine lifecycle, so the provenance is clear.

### R3: No-panic, typed-Result discipline

THE SYSTEM SHALL establish that no public function panics, calls todo, or uses assert as control flow; every fallible operation returns a typed Result.

**Acceptance:**
- error.gleam documents the package-wide discipline: failure is always a typed Result error, never a panic or todo.
- The error types defined here are sufficient to express every documented failure path of the later primitives (activity failure, decode failure, timeout, cancellation, non-determinism, unknown query/signal).
- Review confirms no panic / todo / assert-as-control-flow is introduced by this brief.

## Boundaries

- SHALL NOT define RetryPolicy or Backoff — those are AF-004 (they reference the classification defined here).
- SHALL NOT define the activity or workflow primitives — only the error types they return.
- SHALL NOT re-implement aion-core's Rust ActivityError — this is the Gleam mirror that maps to it across the ffi boundary.
