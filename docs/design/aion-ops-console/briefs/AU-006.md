# AU-006: Namespace selection and query scoping

> **Cluster:** aion-dashboard
> **Depends on:** AU-002
> **Checklist:**
> - C33 — A namespace context holds the selected namespace and a selector lets the operator switch it.
> - C34 — Every query and every subscription is scoped by the selected namespace; switching namespace re-scopes both the active query and the live subscription.
> **Stories:**
> - S10 (Platform Operator, Operating a Multi-Tenant Server) — As a platform operator, I want to select a namespace so that the list, history, and live stream are all scoped to the tenant I am investigating.

## Purpose

Make the dashboard multi-tenant-aware: a selected namespace scopes every query and every subscription so an operator investigates one tenant cleanly.

## Task

Build the namespace feature: a context holding the selected namespace (loaded from listNamespaces), a selector to switch it, and the scoping wiring so every query key and every subscription filter carries the selected namespace and re-scopes on switch.

## Requirements

### R1: Namespace context and selector

THE SYSTEM SHALL provide a namespace context holding the selected namespace (populated from listNamespaces) and a selector component letting the operator switch namespaces.

**Acceptance:**
- src/features/namespace/context/NamespaceContext.tsx exposes the selected namespace and a setter, with the available namespaces loaded via the api client's listNamespaces.
- NamespaceSelector renders the available namespaces (shadcn select) and updates the context on change.
- The selector renders loading and error states for the namespace list.
- A test asserts selecting a namespace updates the context value.

**Files:**
- create: apps/aion-dashboard/src/features/namespace/context/NamespaceContext.tsx
- create: apps/aion-dashboard/src/features/namespace/components/NamespaceSelector.tsx
- create: apps/aion-dashboard/src/features/namespace/index.ts

**Checklist:**
- C33 — A namespace context holds the selected namespace and a selector lets the operator switch it.

**Stories:**
- S10 (Platform Operator, Operating a Multi-Tenant Server) — As a platform operator, I want to select a namespace so that the list, history, and live stream are all scoped to the tenant I am investigating.

### R2: Scoping every query and subscription

THE SYSTEM SHALL scope every query and every subscription by the selected namespace, and WHEN the namespace changes THE SYSTEM SHALL re-scope both the active query and the live subscription. No view SHALL issue an unscoped query when a namespace is selected.

**Acceptance:**
- The list query key, the history query key, and the subscription filter all include the selected namespace.
- Changing the namespace invalidates/re-issues the active query and re-establishes the active subscription under the new namespace.
- A test switching namespace asserts the query is re-issued and the subscription re-established with the new namespace, and that no query is issued without a namespace when one is selected.

**Files:**

## Boundaries

- SHALL NOT define the REST client or WebSocket manager — AU-002, AU-003 (this brief threads the namespace through them).
- SHALL NOT build the list or detail views — AU-004, AU-005 (this brief provides the namespace they scope by).
- SHALL NOT define server namespace semantics — that is cluster AW.
