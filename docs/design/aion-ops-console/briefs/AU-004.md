## Purpose

Give the operator the entry point of the dashboard: a filterable, paginated list of workflows backed by the server's query API, with the production states a real console needs.

## Task

Build the workflow-list feature: a TanStack Query hook over the query API, a filter bar (type, status, time range), pagination against the server's contract, and a list of rows that link to the detail view — with explicit loading, empty, and error states.

## Requirements

### R1: Query hook and list rendering

THE SYSTEM SHALL load workflow summaries via TanStack Query against the query API and render them as rows showing id, type, status, and start/end time, each linking to the detail view.

**Acceptance:**
- src/features/workflow-list/hooks/useWorkflowQuery.ts wraps the api client's queryWorkflows in a TanStack Query keyed by filter + page + namespace.
- WorkflowList renders a WorkflowRow per WorkflowSummary showing id, type, a StatusBadge, and start/end time.
- Each row links to /workflows/:id.
- The hook returns query state (loading/error/data) for the view to render states from.
- A render test with a mocked client shows rows for a returned page.

### R2: Filter bar and pagination

THE SYSTEM SHALL provide a filter bar filtering by workflow type, status, and time range that drives the query, and pagination against the server's pagination contract that never loads full histories.

**Acceptance:**
- FilterBar exposes type (text/select), status (WorkflowStatus select), and a time-range control; changing any updates the WorkflowFilter driving the query.
- Pagination renders next/previous (or load-more) using the client's pagination contract and resets to the first page when the filter changes.
- The list requests only summaries (query API), never per-workflow histories.
- A test changing a filter asserts the query is re-issued with the new WorkflowFilter and pagination resets.

### R3: Loading, empty, and error states

THE SYSTEM SHALL render explicit loading (skeleton rows), empty (no workflows match this filter), and error (cause plus retry) states for the list, distinct from one another.

**Acceptance:**
- While loading, the list shows layout-reserving skeleton rows (the shared LoadingSkeleton), not a bare spinner.
- When the query returns no rows, the shared EmptyState renders 'no workflows match this filter'.
- On query error, the shared ErrorState renders the cause and a retry affordance that re-runs the query.
- A test asserts each of the three states renders for the corresponding query state.
