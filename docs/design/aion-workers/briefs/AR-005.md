# AR-005: aion-worker (Rust) reconnect, re-registration, and un-acked result re-reporting

> **Cluster:** aion-workers
> **Depends on:** AR-003
> **Checklist:**
> - C15 — On session drop the SDK reconnects with bounded exponential backoff and re-registers its activity types before serving new tasks.
> - C16 — The SDK tracks results computed locally but not yet acknowledged, and on reconnect re-reports those un-acked results before serving any new task.
> **Stories:**
> - S9 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want a worker to reconnect automatically after a network drop and re-report any work it already finished so that a disconnect never loses a completed activity or double-runs a side effect.

## Purpose

Make a worker survive a disconnect without losing completed work. On a dropped session the SDK reconnects with backoff, re-registers its activity types, and — before serving any new task — re-reports results it computed but the engine never acknowledged. This is the guarantee that a network blip never drops a finished activity or re-runs a side effect.

## Task

Implement protocol/reconnect.rs: a bounded exponential-backoff reconnect that re-establishes the session (handshake + register, reusing AR-001), and an un-acked result tracker that records every result/failure reported until the engine acknowledges it. On reconnect, re-report all un-acked results before the loop resumes serving. Rely on the engine de-duplicating by ActivityId (AW owns idempotent ingest).

## Requirements

### R1: Un-acked result tracking

WHEN the SDK reports a completion or failure, THE SYSTEM SHALL retain that report keyed by ActivityId until the engine acknowledges it, and SHALL drop it once acknowledged.

**Acceptance:**
- reconnect.rs (or a tracker it owns) records each reported result/failure keyed by ActivityId on report and removes it on the engine's acknowledgement.
- The tracker is the single source of un-acked reports; report paths from AR-003 funnel through it.
- A unit test reports two completions, acknowledges one, and asserts exactly one un-acked report remains, keyed by the un-acked ActivityId.

### R2: Backoff reconnect and re-registration

WHEN the session drops, THE SYSTEM SHALL reconnect using bounded exponential backoff and SHALL re-perform the handshake and activity-type registration before serving new tasks.

**Acceptance:**
- On a session error/disconnect, reconnect.rs retries the connection with exponential backoff and a configured cap (cap drawn from WorkerConfig, not hardcoded, per CO10).
- Each reconnect re-runs the AR-001 handshake (task queue + identity) and re-registers the worker's activity types.
- Reconnect backoff and failure are logged via tracing; a permanently-unreachable endpoint surfaces WorkerError without busy-looping at zero delay.
- A test using a fake session that fails once then succeeds asserts the worker reconnects, re-registers, and resumes.

### R3: Re-report un-acked results before serving

WHEN a session is re-established, THE SYSTEM SHALL re-report all un-acked results before pulling any new task from the receive stream.

**Acceptance:**
- After re-registration and before the first new-task pull, the worker re-reports every un-acked result/failure from the tracker (R1).
- Re-reporting relies on engine-side de-duplication by ActivityId (AW idempotent ingest); the SDK re-reports unconditionally rather than guessing what the engine retained (per D7).
- A test: report a completion, simulate a disconnect before acknowledgement, reconnect, and assert the completion is re-reported before any new task is dispatched.
- Ordering: no new task outcome is reported until the un-acked backlog has been re-sent.

## Boundaries

- SHALL NOT implement engine-side de-duplication or idempotent ingest — that is AW.
- SHALL NOT persist un-acked results to disk — in-memory tracking only this round (a crashed worker process loses its in-memory backlog; the engine retries via the activity policy).
- SHALL NOT define the Worker builder / shutdown — AR-006.
