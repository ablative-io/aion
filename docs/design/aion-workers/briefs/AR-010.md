# AR-010: aion-worker-typescript: defineActivity, context, error classification, and Worker

> **Cluster:** aion-workers
> **Depends on:** AR-009

> - C34 — A Worker class registers activities and await worker.run() connects, serves, and shuts down gracefully draining in-flight activities.
> **Stories:**
> - S4 (TypeScript Activity Author, Writing Node-Ecosystem Activities) — As a TypeScript activity author, I want defineActivity<I, O> with generic input/output types so that my handler is type-checked and serialisation is handled at the boundary.
> - S10 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want a worker to shut down gracefully, draining in-flight activities before exiting so that a deploy does not abandon running work.

## Purpose

Give the TypeScript author the native surface: defineActivity<I, O> with generic typed I/O, a ctx for heartbeat and cancellation, RetryableError/TerminalError classification, and a Worker that serves until graceful shutdown. This completes the TypeScript SDK.

## Task

Implement activity.ts (defineActivity<I, O>, JSON Payload codec, registry), context.ts (heartbeat + cancellation observation), errors.ts (RetryableError / TerminalError and unclassified-defaults-to-retryable), and worker.ts (Worker class + run with graceful drain). Fill the dispatch seam AR-009's loop calls.

## Requirements

### R1: defineActivity and JSON codec

THE SYSTEM SHALL provide defineActivity<I, O>(name, handler) registering a typed activity whose generic types drive JSON Payload encode/decode at the boundary. THE SYSTEM SHALL NOT change the content type on round-trip.

**Acceptance:**
- activity.ts defines defineActivity<I, O>(name: string, handler: (input: I, ctx: ActivityContext) => Promise<O>) and registers it in a registry keyed by name.
- Input Payload decodes to I and output O encodes to a Payload via JSON, preserving the content-type tag (CO7).
- Registering two activities under the same name throws a clear error at registration.
- The dispatch seam (called by AR-009's loop) decodes input, invokes the handler with a ctx, and encodes the result.
- A test defines an activity, dispatches an encoded input through the registry, and asserts the output Payload decodes to the expected value with matching types.

### R2: Activity context: heartbeat and cancellation

THE SYSTEM SHALL pass a ctx to each handler exposing heartbeat(detail) and cancellation observation (isCancelled() and an awaitable cancelled()). THE SYSTEM SHALL NOT heartbeat on the handler's behalf and SHALL NOT forcibly cancel the handler.

**Acceptance:**
- context.ts defines ActivityContext with activityId, attempt, heartbeat(detail?) routing to the session, isCancelled(): boolean, and cancelled(): Promise<void>.
- An engine cancellation flips the ctx cancellation flag; the handler promise is not rejected/aborted by the SDK (cooperative only, per D5 / CO9).
- heartbeat sends only on explicit call (per D4); no timer auto-heartbeats.
- A test signals cancellation mid-handler and asserts isCancelled() becomes true while the handler is allowed to resolve on its own.

### R3: Error classification

THE SYSTEM SHALL provide RetryableError and TerminalError classes for handlers to throw. IF an unclassified value is thrown from a handler, THEN THE SYSTEM SHALL classify the failure as retryable and SHALL log it as unclassified.

**Acceptance:**
- errors.ts defines RetryableError and TerminalError; throwing one yields the corresponding wire classification (aligned with aion-core ActivityError per CO8).
- An unclassified throw is reported as a retryable failure and logged at warn level as unclassified (never silent, per D6 / CO8).
- The reported failure carries the error message and, where available, a detail Payload.
- Tests assert: RetryableError → retryable wire failure; TerminalError → terminal wire failure; a thrown Error → retryable wire failure plus an emitted unclassified warning.

### R4: Worker class, run, and graceful drain

THE SYSTEM SHALL provide a Worker class that registers activities and, on await worker.run(), connects and serves until graceful shutdown, draining in-flight activities before returning.

**Acceptance:**
- worker.ts defines a Worker taking a WorkerConfig and registered activities; await worker.run() connects (AR-009), serves, and on a shutdown signal (e.g. an AbortSignal) stops pulling new tasks and drains in-flight ones before returning.
- Cooperative cancellation of in-flight activities on shutdown is offered via ctx (per R2) but not forced.
- A test runs a Worker against the fake session, feeds a slow task, triggers shutdown mid-task, and asserts run() resolves only after the in-flight task completes.
- An end-to-end test against the fake session defines an activity, serves a task, and asserts the recorded completion — proving the full TS SDK path.
