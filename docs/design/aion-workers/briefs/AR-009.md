# AR-009: aion-worker-typescript: packaging, protocol session, loop, and reconnect

> **Cluster:** aion-workers
> **Depends on:** AR-006
> **Blocked by:** aion-proto gRPC/protobuf definitions (cluster AW) must be available to generate TypeScript proto types from
> **Checklist:**
> - C27 — aion-worker-typescript is an npm package (package.json with ESM+CJS builds and bundled type declarations) depending on @grpc/grpc-js, with index.ts exposing only re-exports and strict tsconfig.
> - C28 — A session module performs connect+handshake (task queue + identity) and activity-type registration over @grpc/grpc-js, behind a session abstraction the loop depends on.
> - C29 — A worker loop receives tasks from the server stream, dispatches up to the operator-configured concurrency as concurrent promises, and reports completion or failure; backpressure is applied when the pool is full.
> - C30 — The TypeScript SDK reconnects with bounded backoff, re-registers, and re-reports un-acked results before serving new tasks.
> **Stories:**
> - S5 (TypeScript Activity Author, Writing Node-Ecosystem Activities) — As a TypeScript activity author, I want to install the SDK from npm with bundled type declarations so that it works out of the box in an ESM or CJS project.
> - S8 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want to configure the engine endpoint, task queue, identity, and max concurrency so that I can point a worker at the right engine and scale it without editing code.
> - S9 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want a worker to reconnect automatically after a network drop and re-report any work it already finished so that a disconnect never loses a completed activity or double-runs a side effect.

## Purpose

Bring the worker protocol to the Node ecosystem. Port the reference state machine (AR-001..AR-006) to TypeScript: an npm package with a session over @grpc/grpc-js, a concurrent worker loop, and reconnect-with-re-report. The author-facing typed-activity surface lands in AR-010; this brief is the protocol substrate.

## Task

Create the aion-worker-typescript package (package.json with ESM+CJS builds and bundled type declarations, strict tsconfig) depending on @grpc/grpc-js, with proto types generated from aion-proto. Implement session.ts (connect+handshake+register behind a session interface), loop.ts (receive→dispatch→report with operator-configured bounded concurrency and backpressure), and reconnect.ts (backoff reconnect, re-register, re-report un-acked results). Mirror the Rust/Python semantics exactly. The dispatch seam calls into the typed-activity layer from AR-010.

### R2: Protocol session over @grpc/grpc-js

THE SYSTEM SHALL implement a session that connects to the engine endpoint, performs the handshake (task queue + identity), and registers activity-type names, behind a session interface the loop depends on.

**Acceptance:**
- session.ts defines a session interface and a @grpc/grpc-js-backed implementation with connect+handshake (taskQueue, identity), register(activityTypes), a receive-task async iterable, reportResult, reportFailure, and sendHeartbeat.
- The loop and reconnect modules depend on the session interface (per D3) so a fake in-memory session can substitute in tests (CO6).
- Config (endpoint, taskQueue, identity, maxConcurrency, credentials passthrough) is read from an explicit WorkerConfig type with no hardcoded concurrency default (CO10).
- A test substitutes a fake session and asserts handshake and registration carry the configured task queue, identity, and activity names.

### R3: Worker loop with bounded concurrency

WHILE connected, THE SYSTEM SHALL consume the receive stream, dispatch each task as a concurrent promise up to the configured max concurrency, and report completion or failure. THE SYSTEM SHALL apply backpressure when saturated and SHALL NOT dispatch unbounded.

**Acceptance:**
- loop.ts drives the session's receive async iterable, dispatching each task to a seam (filled by AR-010) and reporting outcomes via the session.
- Concurrency is bounded by maxConcurrency (a counting limiter); when saturated the loop stops pulling new tasks (backpressure), not buffering unboundedly.
- A test sets maxConcurrency=2, feeds 5 slow tasks via the fake session, and asserts peak concurrent dispatch is 2.
- Each received/reported task is logged via the package's logger without logging payload contents.

### R4: Reconnect and un-acked re-report

WHEN the session drops, THE SYSTEM SHALL reconnect with bounded backoff, re-register, and re-report un-acked results before serving new tasks.

**Acceptance:**
- reconnect.ts tracks reported-but-unacknowledged results keyed by activityId and reconnects with bounded exponential backoff (cap from config, not hardcoded).
- On reconnect it re-runs handshake + registration and re-reports the un-acked backlog before the loop pulls any new task (per D7 / CO11).
- A test reports a completion, simulates a disconnect before acknowledgement, reconnects, and asserts the completion is re-reported before any new task is dispatched.
- The full reconnect path runs against the fake session with no live engine (CO6).
