# AR-006: aion-worker (Rust) typed Activity registration and the Worker builder

> **Cluster:** aion-workers
> **Depends on:** AR-002, AR-003, AR-004, AR-005
> **Checklist:**
> - C17 — An Activity is definable as an async handler taking a typed input (Serialize/DeserializeOwned) and an &ActivityContext, returning Result<Output, ActivityFailure>, and registered by activity-type name.
> - C18 — A Worker builder accepts a WorkerConfig and registered activities; Worker::run() connects, serves, and returns on a graceful shutdown signal after draining in-flight activities.
> **Stories:**
> - S6 (Rust Activity Author, Writing Isolated / Independently-Scaled Rust Activities) — As a Rust activity author, I want to register an async handler returning Result<Output, ActivityFailure> on a Worker builder and call Worker::run() so that out-of-process Rust activities serve a running engine with full durability semantics.
> - S10 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want a worker to shut down gracefully, draining in-flight activities before exiting so that a deploy does not abandon running work.

## Purpose

Give the Rust author the public surface: define a typed async activity, register it by name on a Worker builder, and run the worker until graceful shutdown. This ties the session, loop, dispatch, heartbeat, and reconnect machinery into one ergonomic API and is the brief that completes the reference SDK.

## Task

Implement the Activity registration trait/closure surface in activity.rs and the Worker builder + run() in worker.rs. An activity is an async handler over a typed Serialize/DeserializeOwned input, an &ActivityContext, returning Result<Output, ActivityFailure>. The Worker builder takes a WorkerConfig and registered activities, validates that every registered name has a handler (feeding AR-001's registration check), connects, runs the loop, and returns on a graceful shutdown signal after draining in-flight activities.

## Requirements

### R1: Typed activity registration

THE SYSTEM SHALL allow registering an activity as an async handler over a typed input (Serialize + DeserializeOwned) and an &ActivityContext, returning Result<Output, ActivityFailure> where Output is Serialize, keyed by an activity-type name.

**Acceptance:**
- activity.rs defines the registration surface (an Activity trait with a blanket impl for suitable async closures/fns, or an equivalent register_activity generic) binding Input: DeserializeOwned, Output: Serialize.
- Registration stores an erased handler invocable by the dispatch path (AR-003) given an input Payload and an &ActivityContext, producing a result Payload or ActivityFailure.
- Registering two handlers under the same activity-type name is rejected at build time with a clear error.
- A test registers a typed activity, invokes it through the registry with an encoded input, and asserts the decoded output round-trips.

### R2: Worker builder

THE SYSTEM SHALL provide a Worker builder taking a WorkerConfig and a set of registered activities, and SHALL validate before connecting that the activity-type names it will register all have handlers.

**Acceptance:**
- worker.rs defines a Worker builder accepting a WorkerConfig and accumulating registered activities (R1).
- build() collects the registered activity-type names and supplies them to the session's registration (AR-001 R5), so a name without a handler cannot be registered.
- The builder rejects an empty activity set with a clear error (a worker that serves nothing is a misconfiguration).
- A test builds a Worker with two activities and asserts the registration name set equals the two registered names.

### R3: Worker::run and graceful shutdown

WHEN Worker::run() is called, THE SYSTEM SHALL connect, register, and serve tasks until a shutdown signal is received, after which it SHALL stop pulling new tasks and drain in-flight activities before returning.

**Acceptance:**
- Worker::run() connects (AR-001), serves via the loop (AR-002) with dispatch (AR-003), heartbeat (AR-004), and reconnect (AR-005) all active.
- run() accepts or exposes a shutdown signal (e.g. a CancellationToken / future); on signal it stops pulling new tasks and awaits in-flight activities to completion before returning (the drain hook from AR-002 R3).
- Cooperative cancellation of in-flight activities on shutdown is offered to handlers via their ActivityContext (per AR-004) but not forced.
- A test runs a worker against the fake session, feeds one slow task, triggers shutdown mid-task, and asserts run() returns only after the in-flight task completes.

## Boundaries

- SHALL NOT implement the Python or TS SDKs — AR-007..AR-010.
