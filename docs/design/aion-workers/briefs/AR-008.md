# AR-008: aion-worker-python: @activity decorator, context, error classification, and Worker

> **Cluster:** aion-workers
> **Depends on:** AR-007
> **Checklist:**
> - C23 — An @activity(name=...) decorator registers an async (or pool-run sync) function as an activity; type hints on its arguments and return drive Payload encode/decode (JSON baseline).
> - C24 — A context object passed to the handler exposes heartbeat(detail) and cancellation observation (is_cancelled() and an awaitable cancelled()).
> - C25 — RetryableError and TerminalError base classes classify failures; an unclassified exception escaping a handler defaults to retryable and is logged as unclassified (never silent).
> - C26 — A Worker object registers decorated activities and await worker.run() connects, serves, and shuts down gracefully draining in-flight activities; a thread/process-pool escape hatch handles blocking/CPU-bound handlers.
> **Stories:**

> - S10 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want a worker to shut down gracefully, draining in-flight activities before exiting so that a deploy does not abandon running work.

## Purpose

Give the Python author the native surface: decorate a function with @activity, let type hints drive serialisation, heartbeat and observe cancellation through a context, classify failures as retryable or terminal, and run a Worker that serves until graceful shutdown. This completes the Python SDK and is the brief a data scientist actually touches.

## Task

Implement activity.py (the @activity decorator, type-hint-driven Payload codec, and the activity registry), context.py (heartbeat + cancellation observation), errors.py (RetryableError / TerminalError and the unclassified-defaults-to-retryable rule), and worker.py (the Worker object + async run with graceful drain and a pool escape hatch for blocking/CPU-bound handlers). Fill the dispatch seam AR-007's loop calls.

## Requirements

### R1: @activity decorator and type-hint codec

THE SYSTEM SHALL provide an @activity(name=...) decorator registering an async (or pool-run sync) function as an activity, using the function's type hints to encode/decode the input and output Payload (JSON baseline). THE SYSTEM SHALL NOT change the content type on round-trip.

**Acceptance:**
- activity.py defines @activity(name=...) that registers the function in a registry keyed by the activity-type name.
- Argument and return type hints drive JSON Payload decode/encode; the Payload content-type tag is preserved (CO7).
- Registering two activities under the same name raises a clear error at decoration/registration time.
- The dispatch seam (called by AR-007's loop) decodes the input Payload, invokes the handler with a context, and encodes the result Payload.
- A test registers an activity with typed hints, dispatches an encoded input through the registry, and asserts the output Payload decodes to the expected value.

### R2: Activity context: heartbeat and cancellation

THE SYSTEM SHALL pass a context to each handler exposing heartbeat(detail) and cancellation observation (is_cancelled() and an awaitable cancelled()). THE SYSTEM SHALL NOT heartbeat on the handler's behalf and SHALL NOT forcibly cancel the handler.

**Acceptance:**
- context.py defines a context object with activity_id, attempt, heartbeat(detail=None) routing to the session's heartbeat, is_cancelled() and an awaitable cancelled().
- An engine cancellation flips the context's cancellation flag; the handler coroutine is not cancelled out from under it (cooperative only, per D5 / CO9).
- heartbeat sends only on explicit call (per D4); no background heartbeat task exists.
- A test signals cancellation mid-handler and asserts is_cancelled() becomes true while the handler is allowed to return on its own.

### R3: Error classification

THE SYSTEM SHALL provide RetryableError and TerminalError base classes for handlers to raise. IF an unclassified exception escapes a handler, THEN THE SYSTEM SHALL classify the failure as retryable and SHALL log it as unclassified.

**Acceptance:**
- errors.py defines RetryableError and TerminalError; a handler raising one yields the corresponding wire classification (aligned with aion-core ActivityError per CO8).
- An unclassified exception escaping a handler is reported as a retryable failure and logged at warning level as unclassified (never silent, per D6 / CO8).
- The reported failure carries the exception message and, where available, a detail Payload.
- Tests assert: RetryableError → retryable wire failure; TerminalError → terminal wire failure; a bare ValueError → retryable wire failure plus an emitted unclassified warning.

### R4: Worker object, run, and pool escape hatch

THE SYSTEM SHALL provide a Worker object that registers decorated activities and, on await worker.run(), connects and serves until graceful shutdown, draining in-flight activities. THE SYSTEM SHALL run blocking/CPU-bound handlers on a thread/process pool so they do not block the event loop.

**Acceptance:**
- worker.py defines a Worker taking a WorkerConfig and the registered activities; await worker.run() connects (AR-007), serves, and on a shutdown signal stops pulling new tasks and drains in-flight ones before returning.
- A handler declared/synchronous-and-blocking is dispatched on a thread or process pool (configured) so the asyncio loop is not blocked; an async handler runs directly on the loop.
- A test runs a Worker against the fake session, feeds a slow task, triggers shutdown mid-task, and asserts run() returns only after the in-flight task completes.
- A test asserts a blocking sync handler does not stall concurrent async handlers (the pool offload works).
