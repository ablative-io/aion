# AR-001: aion-worker (Rust) scaffold, config, error, and protocol session

> **Cluster:** aion-workers
> **Blocked by:** aion-proto wire types and worker-endpoint RPC names (cluster AW) must be available to depend on; this brief targets that contract
> **Checklist:**
> - C1 — aion-worker is a workspace member with unsafe_code = deny and a lib.rs containing only declarations and re-exports; it depends on aion-proto and aion-core but not on aion, aion-server, or aion-store.
> - C2 — WorkerConfig carries the engine endpoint, task-queue name, worker identity, max concurrency, and a TLS/credentials passthrough; no field has a hardcoded default concurrency or tunable.
> - C3 — WorkerError is a closed thiserror enum covering connection, handshake, registration, decode/encode, and transport failures, and is Send + Sync.
> - C4 — A WorkerSession trait abstracts the transport (connect+handshake, register activity types, receive-task stream, report result/failure, send heartbeat) so execution machinery never touches generated gRPC stubs directly.
> - C5 — A gRPC-backed WorkerSession implementation connects to the engine's worker endpoint, performs the handshake declaring task queue and identity, and registers a set of activity-type names per the aion-proto contract.
> - C6 — Registering an activity-type name for which no handler exists is rejected by the SDK before serving begins, not at dispatch time.
> **Stories:**
> - S7 (Rust Activity Author, Writing Isolated / Independently-Scaled Rust Activities) — As a Rust activity author, I want the SDK to depend on the wire contract (aion-proto) but not on the engine internals so that my worker binary stays small and decoupled from the engine.
> - S8 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want to configure the engine endpoint, task queue, identity, and max concurrency so that I can point a worker at the right engine and scale it without editing code.

## Purpose

Stand up the Rust worker SDK crate and its protocol session — the reference implementation of the worker-side protocol client that the Python and TS SDKs port. Establish connect, handshake, and activity-type registration against the aion-proto contract, isolated behind a session abstraction so the rest of the SDK is testable without a network.

## Task

Create the aion-worker crate (crates.io target) depending on aion-proto and aion-core but not on the engine. Define WorkerConfig, WorkerError, the WorkerSession trait (the transport abstraction per D3), and a gRPC-backed implementation that connects to the engine's worker endpoint, performs the handshake (task queue + identity), and registers activity-type names. Reject registering a name with no handler before serving. Do not implement the task loop, dispatch, heartbeat, or reconnect here.

### R5: gRPC-backed session: connect, handshake, register

THE SYSTEM SHALL implement WorkerSession over the aion-proto gRPC stubs: connect to the configured endpoint, perform the handshake declaring task queue and identity, and register the worker's activity-type names. IF a requested activity-type name has no registered handler, THEN THE SYSTEM SHALL reject it before serving begins.

**Acceptance:**
- The gRPC-backed WorkerSession connects to WorkerConfig.endpoint (forwarding transport_credentials) and performs the handshake RPC with task_queue and identity.
- register sends the set of activity-type names to the engine per the aion-proto registration RPC.
- Registering an activity-type name for which no handler is present returns WorkerError::Registration before any task is served (validated against the handler set the Worker supplies in AR-006).
- A handshake or registration RPC error maps to WorkerError::Handshake / WorkerError::Registration with the source preserved.
- An integration-style test against the in-memory fake session asserts handshake and registration are issued with the configured task queue, identity, and activity names.
