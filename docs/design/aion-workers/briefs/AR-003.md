# AR-003: aion-worker (Rust) dispatch, payload codec, failure classification, and ActivityContext

> **Cluster:** aion-workers
> **Depends on:** AR-002

> - C11 — ActivityFailure carries an explicit retryable-vs-terminal classification (not a string or bool); a handler returns Result<Output, ActivityFailure>.

## Purpose

Fill the dispatch seam the loop calls: decode the input Payload to the handler's type, run the handler with an ActivityContext, encode its output back to a Payload, and report completion or a classified failure. This is where a task becomes a result and where the retryable/terminal contract is enforced.

## Task

Implement runtime/dispatch.rs and the base ActivityContext in context.rs. Dispatch decodes the input Payload (JSON baseline, honouring the content-type tag), invokes the registered handler with an &ActivityContext, encodes the Ok output to a Payload, and reports completion; on Err it reports a failure carrying the ActivityFailure's retryable/terminal classification. Define ActivityFailure here. The context's heartbeat send and cancellation-flag wiring are stubbed to the seams AR-004 fills; this brief establishes the context's identity/attempt fields and the cancellation-observation surface.

## Requirements

### R1: ActivityContext base

THE SYSTEM SHALL define ActivityContext exposing the ActivityId, the attempt number, a heartbeat method, and cancellation observation (is_cancelled and an awaitable cancelled signal).

**Acceptance:**
- context.rs defines ActivityContext with activity_id(), attempt(), heartbeat(detail: Option<Payload>), is_cancelled() -> bool, and cancelled() returning an awaitable that resolves when cancellation is signalled.
- The cancellation flag is backed by a shared atomic/notify the session can set (the setter wiring is completed in AR-004; this brief defines the observable surface and the internal handle).
- heartbeat() in this brief routes to a seam (a no-op or channel send) that AR-004 connects to WorkerSession::send_heartbeat; the signature is final here.
- A unit test constructs an ActivityContext, asserts activity_id/attempt, and asserts is_cancelled() flips to true after the internal cancellation handle is triggered.

### R2: ActivityFailure with retryable/terminal classification

THE SYSTEM SHALL define ActivityFailure carrying an explicit retryable-vs-terminal classification (not a string or bool) plus a message and optional detail Payload. THE SYSTEM SHALL NOT infer classification from a Rust error type.

**Acceptance:**
- activity.rs (or a dedicated section thereof) defines ActivityFailure with a Classification enum { Retryable, Terminal }, a message, and an optional detail: Option<Payload>.
- Constructors ActivityFailure::retryable(..) and ActivityFailure::terminal(..) exist; there is no path that guesses classification from a std::error::Error variant.
- ActivityFailure maps onto the aion-proto failure message such that Retryable and Terminal are distinguishable on the wire (aligned with aion-core ActivityError per CO8).
- A unit test builds a retryable and a terminal failure and asserts each maps to the expected wire classification.

### R3: Payload decode/encode at the dispatch boundary

THE SYSTEM SHALL decode a task's input Payload to the handler's input type and encode the handler's output to a Payload using the content-type tag (JSON baseline). THE SYSTEM SHALL NOT change the content type on round-trip.

**Acceptance:**
- dispatch.rs decodes input: Payload to the handler's Input via the content-type tag (JSON baseline), returning WorkerError::Decode on failure.
- The Ok output is encoded to a Payload carrying the same content-type tag class as the baseline codec (JSON), per CO7.
- A decode failure of the input is reported as a terminal ActivityFailure (a payload the handler cannot parse will not parse on retry), with the cause logged.
- A round-trip test: a typed input encodes, decodes to an equal value, the handler returns a typed output, and that output encodes to a Payload that decodes back equal.

### R4: Dispatch invocation and completion/failure reporting

WHEN dispatch runs a handler, THE SYSTEM SHALL report completion with the encoded output Payload for the task's ActivityId on Ok, and SHALL report a failure carrying the retryable/terminal classification on Err.

**Acceptance:**
- dispatch invokes the registered handler for the task's activity type with the decoded input and an &ActivityContext (per R1).
- On Ok(output), the encoded output Payload is reported via WorkerSession::report_result for the task's ActivityId.
- On Err(ActivityFailure), the failure with its classification is reported via WorkerSession::report_failure for the task's ActivityId.
- A panic inside a handler is caught and reported as a retryable ActivityFailure (a panic is treated as a transient fault, logged loudly), never crashing the worker loop.
- Tests against the fake session assert: an Ok handler yields a completion with the expected Payload; a retryable Err yields a retryable failure; a terminal Err yields a terminal failure; a panicking handler yields a retryable failure and the loop survives.
