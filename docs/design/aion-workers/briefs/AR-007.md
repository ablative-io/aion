# AR-007: aion-worker-python: packaging, protocol session, asyncio loop, and reconnect

> **Cluster:** aion-workers
> **Depends on:** AR-006
> **Blocked by:** aion-proto gRPC/protobuf definitions (cluster AW) must be available to generate Python stubs from
> **Checklist:**
> - C19 — aion-worker-python is a PyPI-packaged project (pyproject.toml) depending on grpcio and protobuf, with __init__.py exposing only re-exports.
> - C20 — A session module performs connect+handshake (task queue + identity) and activity-type registration over generated gRPC stubs, behind a session abstraction the loop depends on.
> - C21 — An asyncio worker loop receives tasks from the server stream, dispatches up to the operator-configured concurrency, and reports completion or failure; backpressure is applied when the pool is full.
> - C22 — The Python SDK reconnects with bounded backoff, re-registers, and re-reports un-acked results before serving new tasks.
> **Stories:**
> - S8 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want to configure the engine endpoint, task queue, identity, and max concurrency so that I can point a worker at the right engine and scale it without editing code.
> - S9 (Worker Operator, Deploying and Scaling Workers) — As a worker operator, I want a worker to reconnect automatically after a network drop and re-report any work it already finished so that a disconnect never loses a completed activity or double-runs a side effect.

## Purpose

Bring the worker protocol to Python — the home for ML and data-science activities. Port the reference state machine (AR-001..AR-006) to an asyncio implementation: a PyPI-packaged project with a session over generated gRPC stubs, a concurrent worker loop, and reconnect-with-re-report. The author-facing typed-activity surface lands in AR-008; this brief is the protocol substrate.

## Task

Create the aion-worker-python project (pyproject.toml, PyPI-ready) depending on grpcio and protobuf, with generated stubs from the aion-proto definitions. Implement session.py (connect+handshake+register over the stubs, behind a session abstraction), loop.py (asyncio receive→dispatch→report with operator-configured bounded concurrency and backpressure), and reconnect.py (backoff reconnect, re-register, re-report un-acked results). Mirror the Rust semantics exactly. The dispatch seam calls into the typed-activity layer from AR-008.

### R2: Protocol session over gRPC stubs

THE SYSTEM SHALL implement a session that connects to the engine endpoint, performs the handshake (task queue + identity), and registers activity-type names, behind a session abstraction the loop depends on.

**Acceptance:**
- session.py defines a session class with async connect+handshake (task_queue, identity), register(activity_types), a receive-task async iterator, report_result, report_failure, and send_heartbeat — over the generated stubs.
- The loop and reconnect modules depend on the session abstraction, not on raw stubs directly (per D3), so a fake in-memory session can substitute in tests (CO6).
- Configuration (endpoint, task queue, identity, max concurrency, credentials passthrough) is read from an explicit WorkerConfig dataclass with no hardcoded concurrency default (CO10).
- A test substitutes a fake session and asserts handshake and registration carry the configured task queue, identity, and activity names.

### R3: Asyncio worker loop with bounded concurrency

WHILE connected, THE SYSTEM SHALL consume the receive stream, dispatch each task as an asyncio task up to the configured max concurrency, and report completion or failure. THE SYSTEM SHALL apply backpressure when the pool is saturated and SHALL NOT spawn unbounded tasks.

**Acceptance:**
- loop.py drives the session's receive async iterator, dispatching each task to a seam (filled by AR-008) and reporting outcomes via the session.
- Concurrency is bounded by an asyncio.Semaphore (or equivalent) sized by WorkerConfig.max_concurrency; when saturated, the loop stops pulling new tasks (backpressure).
- A test sets max_concurrency=2, feeds 5 slow tasks via the fake session, and asserts peak concurrent dispatch is 2.
- Each received/reported task is logged via the standard logging module without logging payload contents.

### R4: Reconnect and un-acked re-report

WHEN the session drops, THE SYSTEM SHALL reconnect with bounded backoff, re-register, and re-report un-acked results before serving new tasks.

**Acceptance:**
- reconnect.py tracks reported-but-unacknowledged results keyed by ActivityId and reconnects with bounded exponential backoff (cap from config, not hardcoded).
- On reconnect it re-runs handshake + registration and re-reports the un-acked backlog before the loop pulls any new task (per D7 / CO11).
- A test reports a completion, simulates a disconnect before acknowledgement, reconnects, and asserts the completion is re-reported before any new task is dispatched.
- The full reconnect path runs against the fake session with no live engine (CO6).
