# AV-006: Crash-consistency recovery for visibility projection

> **Cluster:** aion-visibility
> **Depends on:** AV-001

## Purpose

Events are appended atomically but the visibility projection update happens after the append returns. A crash between the two leaves a workflow that exists in event history but is invisible to queries until the next engine restart -- and even then, only if a reconciliation pass runs. This brief closes that crash-consistency window with startup recovery, periodic background reconciliation, and warning-level logging on upsert failures.

## Task

Add a startup reconciliation pass that scans event history for workflows missing from the visibility index and backfills them. Add a periodic background reconciliation task that detects and repairs history/visibility divergence. Ensure the Recorder logs a warning (but does not fail the append) when a visibility upsert fails. Write an integration test that simulates the crash-consistency gap and verifies recovery.

## Requirements

### R1: Startup recovery reconciles visibility store against event history

WHEN the engine starts, THE SYSTEM SHALL scan event history for workflows that have events in the store but are missing from (or stale in) the visibility index, and SHALL upsert their current projected state into the visibility store before accepting new workflow starts.

**Acceptance:**
- A reconcile_visibility function (or method on the engine/visibility module) reads all distinct workflow IDs from event history and compares them against the visibility index.
- For each workflow present in history but missing from or stale in the visibility index, the function projects the current status from history and upserts it.
- Reconciliation runs during engine startup, after the event store and visibility store are opened but before the engine begins accepting workflow starts.
- A test seeds event history with a workflow that has no visibility entry, starts the engine, and asserts the workflow appears in visibility queries.

**Files:**
- modify: crates/aion/src/engine/builder.rs
- modify: crates/aion-store/src/visibility.rs

### R2: Periodic background reconciliation task

WHILE the engine is running, THE SYSTEM SHALL periodically execute a background reconciliation pass that detects and repairs divergence between event history and the visibility index. The interval SHALL be configurable via the engine builder.

**Acceptance:**
- A background task (spawned during engine startup) runs the same reconciliation logic as R1 on a configurable interval.
- The interval is set via the engine builder; if not configured, no periodic reconciliation runs (no arbitrary default per CLAUDE.md).
- The background task is cancelled on engine shutdown.
- The reconciliation pass does not block in-flight workflow operations; it reads a snapshot and upserts asynchronously.
- A test configures a short reconciliation interval, introduces a visibility gap after startup, and asserts the gap is repaired within the next reconciliation cycle.

**Files:**
- modify: crates/aion/src/engine/builder.rs
- modify: crates/aion-store/src/visibility.rs

### R3: Recorder logs warning on visibility upsert failure

WHEN the Recorder appends events successfully but the subsequent visibility upsert fails, THE SYSTEM SHALL log a warning-level message identifying the workflow and the upsert error. THE SYSTEM SHALL NOT fail the append or return an error to the caller.

**Acceptance:**
- After a successful append, if the visibility upsert returns an error, the Recorder emits a tracing::warn! with the workflow ID, run ID, and the upsert error.
- The append result returned to the caller is Ok, not Err -- the visibility failure is non-fatal.
- The warning message explicitly names the crash-consistency window so operators know to investigate.
- A test mocks a visibility store that returns an error on upsert, appends through the Recorder, and asserts the append succeeds and the warning is logged.

**Files:**
- modify: crates/aion/src/durability/recorder.rs

### R4: Integration test simulating crash-consistency gap and recovery

THE SYSTEM SHALL include an integration test that simulates the crash-consistency window (events appended, visibility not updated) and verifies that startup reconciliation and/or background reconciliation repair the gap.

**Acceptance:**
- The test appends events directly to the event store for a workflow, intentionally skipping the visibility upsert, creating the crash-consistency gap.
- The test then triggers startup reconciliation (or starts the engine) and asserts the workflow appears in visibility queries with the correct projected status.
- The test covers at least: a Running workflow missing from visibility, and a Completed workflow missing from visibility.
- The test runs against InMemoryStore and does not require external infrastructure.

**Files:**
- modify: crates/aion-store/src/visibility.rs
