# AV-003: libSQL VisibilityStore implementation

> **Cluster:** aion-visibility
> **Depends on:** AV-001, AV-002
> **Checklist:** C6
> **Stories:** S1, S2, S5, S6

## Purpose

Implement the VisibilityStore trait for the libSQL backend. Standard fields (workflow_type, status, start_time, close_time) are indexed columns for efficient filtering. Custom search attributes are stored as a typed JSON column. This enables the embedded deployment to support visibility queries without external search infrastructure.

## Task

Create aion_store_libsql::visibility module implementing VisibilityStore. Create the visibility table with indexed columns for standard fields and a JSON column for custom attributes. Implement record_visibility as an upsert, list_workflows as a parameterized query supporting all filter predicates, and count_workflows as the corresponding COUNT query. Use libSQL JSON functions for custom attribute filtering.

## Requirements

### R1: Visibility table schema with indexed standard fields

THE SYSTEM SHALL create a visibility table with columns: workflow_id (TEXT, PRIMARY KEY), run_id (TEXT), workflow_type (TEXT, indexed), status (TEXT, indexed), start_time (TEXT, indexed), close_time (TEXT, indexed), search_attributes (TEXT, JSON). Indexes SHALL exist on workflow_type, status, start_time, and close_time.

**Acceptance:**
- The visibility table is created during store initialization.
- Indexes exist on workflow_type, status, start_time, close_time.
- The schema supports the full WorkflowSummary projection.

### R2: record_visibility upserts workflow summary

THE SYSTEM SHALL implement record_visibility as an INSERT OR REPLACE that upserts the full visibility record for a workflow, including current status, times, and search attributes.

**Acceptance:**
- Recording a new workflow inserts a row.
- Recording an existing workflow updates the row.
- Search attributes are stored as valid JSON.
- The upserted row is immediately queryable.

### R3: list_workflows with filter predicates

THE SYSTEM SHALL implement list_workflows to query the visibility table with WHERE clauses derived from ListWorkflowsFilter. Standard field filters use indexed column comparisons. Custom attribute filters use libSQL JSON extraction (json_extract). Results are ordered by start_time descending, with limit/offset pagination.

**Acceptance:**
- Filtering by workflow_type returns only matching workflows.
- Filtering by status returns only matching workflows.
- Filtering by time range (started_after/before) returns workflows in range.
- Filtering by custom attribute value returns matching workflows.
- An empty filter returns all workflows (paginated).
- Limit and offset pagination works correctly.

**Files:**
- modify: crates/aion-store-libsql/src/visibility.rs

**Checklist:**

**Stories:**

### R4: count_workflows

THE SYSTEM SHALL implement count_workflows as a COUNT query using the same filter logic as list_workflows.

**Acceptance:**
- count_workflows with no filter returns the total workflow count.
- count_workflows with a status filter returns the count of matching workflows.
- count_workflows matches the length of list_workflows with the same filter.
