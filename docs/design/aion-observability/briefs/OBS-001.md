# OBS-001: Loud server/worker logging of workflow-process exits

> **Cluster:** aion-observability
> **Design anchor:**
> - ADR-003 — No default timeouts anywhere in the engine — The engine imposes no activity time bound of its own. Activity waits are unbounded, terminated only by completion, worker loss, server shutdown, or a workflow-level timeout the author explicitly chose. Rejected: a bigger default — agentic activities legitimately run for over an hour, and any number we picked would be ADR-001 violated.
> - ADR-005 — Failed runs are terminal; recovery resumes Running runs only — A failed run is a terminal, immutable record. Retry means a fresh `aion start` with a new run identity; recovery after restart resumes Running runs only. Rejected: in-place retry of failed runs — it would rewrite history that event-sourcing exists to preserve, and 'which attempt was this event from' becomes unanswerable.
> **Checklist:**
> - C1 — handle_process_exit_async in crates/aion/src/lifecycle/completion.rs emits an error-level tracing call on the abnormal-exit branch (WorkflowProcessOutcome::Failed) carrying the workflow id, run id, workflow type, and the WorkflowError.message text, before the existing notify/reconcile steps.
> - C2 — handle_process_exit_async emits an error-level tracing call on the monitor-failure branch (the Err(error) arm that builds the 'workflow process monitor failed' WorkflowError) carrying the same fields.
> - C3 — handle_process_exit_async emits an info-level tracing call on the clean-completion branch (WorkflowProcessOutcome::Completed) carrying the workflow id, run id, and workflow type.
> - C4 — The idempotent-replay branch (a terminal already present in history) emits no failure log: a recovered run whose terminal was already recorded and already logged does not re-emit the error line.
> - C5 — Both monitor wiring sites — install_completion_monitor in lifecycle/start.rs and install_recovered_completion_monitor in engine/startup.rs — route their callback through handle_process_exit, so the single funnel covers freshly-started and recovered runs alike (verified by inspection; no new log site is added at either wiring site).
> - C6 — The added tracing calls are pure side effects: no recorded event, completion notification, registry reconciliation, or return value of handle_process_exit_async changes, and the existing completion.rs tests still pass unchanged in their assertions on recorded history and notified outcome.
> - C7 — InstrumentedEventStore::record_events in crates/aion-server/src/observability/instrumented_store.rs emits an error-level tracing call on Event::WorkflowFailed carrying the namespace, workflow id, and the recorded WorkflowError.message, alongside the existing metrics counter.
> - C8 — record_events emits an error-level tracing call on Event::WorkflowTimedOut carrying the namespace and workflow id (a timeout is a terminal failure an operator must see), alongside the existing metrics counter.
> - C9 — record_events emits an error-level tracing call on Event::ChildWorkflowFailed carrying the namespace, parent workflow id, child workflow id, and the recorded error message.
> - C10 — The server log line uses the codebase tracing-field conventions (workflow_id rendered via Display, error message via Display) consistent with crates/aion-server/src/observability/tracing.rs, and the metrics counters that record_events already increments are unchanged.
> - C11 — crates/aion-worker activity error logging is confirmed to already cover the failures the worker is the first to observe (activity input-decode failure and handler panic at error level with structured fields), and OBS-001 documents that the worker hosts no workflow process and therefore adds no workflow-process-exit log it cannot truthfully emit.
> - C16 — No new Event variant, no new Payload type, and no mutable status field are introduced anywhere in the cluster; status remains derived solely by status_from_events (invariant 4), and no brief calls EventStore::append directly (invariant 3).
> - C17 — No #[allow], #[expect], unwrap, or expect appears in the non-test library code added by either brief; every touched source file stays under 500 LOC excluding tests.
> - C18 — cargo fmt --check, cargo clippy --workspace --all-targets -- -D warnings, and the full workspace test suite pass clean at the cluster's landing commit.
> **Stories:**

> - S2 (Operator, Watching a running aion server) — As an operator, I want a workflow that times out to log at error like a crash does so that a terminal failure is loud regardless of which terminal it reached, given the engine imposes no default timeout of its own.
> - S3 (Operator, Watching a running aion server) — As an operator, I want the failure logged even when the failing process ran on a remote worker the server never hosted so that I can rely on the one log surface I already watch rather than chasing per-worker logs.

## Purpose

Closes the half of the dogfood observability gap where a workflow-process crash recorded a terminal WorkflowFailed but logged nothing — the silent path that made a crash look like a hang (see design problem; RM-024 provenance). It makes every workflow-process exit loud at the moment it happens: at the engine's single exit funnel (an info line on clean completion, an error line on failure carrying the VM message) and on the server log surface (an error line whenever a terminal-failure event is appended, including timeouts and failed children). All logging reads the already-recorded terminal — status stays a projection (CN1), no new event type is added (CN3), and the log is a pure side effect that never alters control flow (CN5/P5). OBS-002 builds on the same recorded terminals to surface a failed child to the parent and watchers; this brief owns the exit-logging itself and the cluster's shared invariant/gate items (C16-C18).

## Task

Make workflow-process exits loud in the engine and server logs by adding tracing calls that read the existing recorded terminal outcomes — no new events, no status field, no control-flow change. Three surfaces: (1) The engine exit funnel crates/aion/src/lifecycle/completion.rs::handle_process_exit_async — add the exit log on the three live branches (Completed → info; Failed → error; the monitor-failure Err arm → error), each carrying workflow id, run id, workflow type, and (on failure) the WorkflowError.message. The replay branch (terminal already in history) must NOT log a failure — recovery reconciling an already-announced terminal stays quiet (C4/P4). (2) The server surface crates/aion-server/src/observability/instrumented_store.rs::record_events — add an error-level log next to the existing metrics counters on WorkflowFailed, WorkflowTimedOut, and ChildWorkflowFailed, carrying namespace + workflow id + (where present) the recorded error message. (3) The worker: confirm crates/aion-worker already logs at error the failures it is first to observe (activity input-decode failure, handler panic) and record in this brief's verification that the worker hosts no workflow process, so no untruthful workflow-process-exit log is added there. Scope boundary — in: the tracing calls on the three surfaces above plus tests asserting them, and the cluster's shared invariant/gate items C16-C18. Out: the child-failure propagation log in nif_child_watch.rs and the describe-timeline/watcher-stream verification (OBS-002); any change to the WorkflowError message format or runtime/outcome.rs; any new event, status field, or push path. Split-item note: C16, C17, C18 are cluster-wide invariant/gate items shared with OBS-002 — both briefs must satisfy the invariants and pass the gate; this brief verifies them across its own diff and OBS-002 re-verifies across its diff at the cluster landing commit.

## Requirements

### R1: Loud exit log at the engine's process-exit funnel

WHEN handle_process_exit_async observes a workflow-process exit whose terminal is not already present in history, THE SYSTEM SHALL emit one tracing call before the existing completion notification: at error level for WorkflowProcessOutcome::Failed and for the Err(error) monitor-failure arm, at info level for WorkflowProcessOutcome::Completed. The error-level calls SHALL include the workflow id, run id, workflow type, and the WorkflowError.message (the VM/exit-reason text from runtime/outcome.rs) verbatim. IF the run's terminal is already present in history (the idempotent-replay branch that returns early), THEN THE SYSTEM SHALL NOT emit a failure log. THE SYSTEM SHALL NOT change any recorded event, completion notification, registry reconciliation, or return value, and SHALL NOT reformat or truncate the WorkflowError.message (CN5, CN6).

**Acceptance:**
- On the WorkflowProcessOutcome::Failed branch, a tracing event at ERROR level is emitted that includes the workflow_id, the run_id, the workflow_type, and the WorkflowError.message string — asserted by a test using tracing-test (or an equivalent capturing subscriber) that drives handle_process_exit_async with Ok(WorkflowProcessOutcome::Failed(WorkflowError{message: "process crashed: heap full", ..})) and finds an ERROR record containing "process crashed: heap full".
- On the Err(error) monitor-failure arm, an ERROR-level tracing event is emitted whose message text contains the 'workflow process monitor failed' WorkflowError.message.
- On the WorkflowProcessOutcome::Completed branch, an INFO-level (not ERROR) tracing event is emitted carrying workflow_id, run_id, and workflow_type, and no ERROR record is produced.
- When handle_process_exit_async is driven with a Failed outcome against a history that already holds a WorkflowFailed for the run, no ERROR-level tracing record is emitted (the replay branch is silent) while the existing notify-from-history behaviour is unchanged.
- The existing tests normal_exit_records_completed_reconciles_and_notifies and abnormal_exit_records_failed_reconciles_and_notifies still pass with their recorded-history and notified-outcome assertions unchanged.

### R2: Confirm both monitor wirings route through the one funnel

THE SYSTEM SHALL route every workflow-process-exit observation — for freshly started runs via install_completion_monitor in crates/aion/src/lifecycle/start.rs and for recovered runs via install_recovered_completion_monitor in crates/aion/src/engine/startup.rs — through handle_process_exit, so the R1 log at the funnel covers both paths. THE SYSTEM SHALL NOT add a second exit-log site at either wiring location (that would risk drift and double-logging); the wiring sites SHALL retain only their existing 'monitor completion failed' error log for the case where handle_process_exit itself returns Err. This is a structural/verification requirement (no behavioural code change): the deliverable is the confirmation, captured in the brief's verification, that both callbacks call handle_process_exit and add no competing exit log.

**Acceptance:**

- install_recovered_completion_monitor in crates/aion/src/engine/startup.rs calls handle_process_exit(completion_context, completion_handle, outcome) inside its monitor callback and contains no other workflow-exit tracing call beyond the existing 'recovered workflow process monitor completion failed' error log.

### R3: Error-level server log on terminal-failure events

WHEN InstrumentedEventStore::record_events in crates/aion-server/src/observability/instrumented_store.rs inspects an appended Event that is a terminal failure — Event::WorkflowFailed, Event::WorkflowTimedOut, or Event::ChildWorkflowFailed — THE SYSTEM SHALL emit an error-level tracing call carrying the namespace and the workflow id, including the recorded WorkflowError.message for WorkflowFailed and ChildWorkflowFailed and the child workflow id for ChildWorkflowFailed, in ADDITION to the existing metrics counter for that event. THE SYSTEM SHALL NOT remove, reorder, or alter any existing metrics counter, and SHALL NOT emit an error log for non-failure terminals (WorkflowCompleted, WorkflowCancelled, WorkflowContinuedAsNew). The log SHALL use the codebase field conventions (workflow id and error rendered via Display).

**Acceptance:**
- record_events emits an ERROR-level tracing event on Event::WorkflowFailed that includes the namespace, the workflow_id, and the recorded WorkflowError.message — asserted by a capturing-subscriber test that appends a WorkflowFailed event through the wrapper and finds the ERROR record with the message text.
- record_events emits an ERROR-level tracing event on Event::WorkflowTimedOut carrying namespace and workflow_id.
- record_events emits an ERROR-level tracing event on Event::ChildWorkflowFailed carrying namespace, the parent workflow_id, the child workflow id, and the recorded error message.
- A WorkflowCompleted event produces no ERROR-level tracing record from record_events, and the workflow_completed("completed") / workflow_completed("failed") / workflow_completed("timed_out") metric counters are still incremented exactly as before (existing metrics behaviour unchanged).
- cargo test -p aion-server passes including the new record_events logging test.

### R4: Confirm and document the worker failure surface
