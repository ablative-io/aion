# OBS-002: Propagate a child's terminal failure visibly to parent and watchers

> **Cluster:** aion-observability
> **Depends on:** OBS-001
> **Design anchor:**
> - ADR-003 — No default timeouts anywhere in the engine — The engine imposes no activity time bound of its own. Activity waits are unbounded, terminated only by completion, worker loss, server shutdown, or a workflow-level timeout the author explicitly chose. Rejected: a bigger default — agentic activities legitimately run for over an hour, and any number we picked would be ADR-001 violated.
> - ADR-005 — Failed runs are terminal; recovery resumes Running runs only — A failed run is a terminal, immutable record. Retry means a fresh `aion start` with a new run identity; recovery after restart resumes Running runs only. Rejected: in-place retry of failed runs — it would rewrite history that event-sourcing exists to preserve, and 'which attempt was this event from' becomes unanswerable.
> **Checklist:**
> - C12 — record_parent_child_terminal in crates/aion/src/runtime/nif_child_watch.rs emits an error-level tracing call when it records a ChildWorkflowFailed terminal, carrying the parent workflow id, parent run id, child workflow id, and the child's WorkflowError.message — matching the existing warn/error lines the watcher emits for transient and invariant record failures.
> - C13 — The added child-failure log is emitted on the TerminalOutcome::Failed arm only (and the Cancelled/TimedOut arms that map to ChildWorkflowFailed), not on the Completed arm, and does not change the RecordDisposition returned or the events recorded.
> - C14 — A failed child's terminal surfaces on the parent run's describe timeline: the recorded ChildWorkflowFailed carries the child workflow id and the child's reason, and is encoded into the describe history by describe.rs and payload.rs::encode_history (verified end to end against a parent history containing a ChildWorkflowFailed).
> - C15 — A watcher subscribed to the event stream receives the parent-side ChildWorkflowFailed event when a child fails: the existing subscription/push surface forwards it because it is a normal appended event, with no new push path added.
> - C16 — No new Event variant, no new Payload type, and no mutable status field are introduced anywhere in the cluster; status remains derived solely by status_from_events (invariant 4), and no brief calls EventStore::append directly (invariant 3).
> - C17 — No #[allow], #[expect], unwrap, or expect appears in the non-test library code added by either brief; every touched source file stays under 500 LOC excluding tests.
> - C18 — cargo fmt --check, cargo clippy --workspace --all-targets -- -D warnings, and the full workspace test suite pass clean at the cluster's landing commit.
> **Stories:**
> - S4 (AI Team Member, Dispatching and monitoring a wave of workflows) — As an AI team member who dispatched a parent run with children, I want a child's failure to show up loudly — on the parent's timeline and in the log — so that I notice a failed child fast instead of waiting on a parent that will never converge.
> - S5 (AI Team Member, Dispatching and monitoring a wave of workflows) — As an AI orchestrator watching the event stream, I want the parent-side child-failure event delivered to my subscription so that I can react to a failed child without polling describe.
> - S6 (AI Lead, Reviewing a run after a reported failure) — As an AI lead reviewing why a run died, I want the parent's describe timeline to carry the child's id and its failure reason so that I can see which child failed and why without reconstructing it from separate logs.
> - S7 (AI Lead, Reviewing a run after a reported failure) — As an AI lead, I want the failure announcement to be derived from the recorded terminal event rather than a separately stored status so that the log I read and the durable history I audit can never disagree.

## Purpose

Closes the other half of the dogfood observability gap: a child workflow's terminal failure must surface to the parent run and to watchers without hand-investigation (RM-024). The recorded ChildWorkflowFailed terminal — written onto the parent's history by the child-terminal watcher through the parent's Recorder (invariant 3, CN4) — already drives the parent's describe timeline and is already forwarded by the existing event-stream subscription; this brief adds the loud edge (an error log when the watcher records a failed-child terminal) and verifies the two visibility paths end to end (describe timeline carries the child id + reason; a subscribed watcher receives the parent-side event). It introduces no new event type and no new push path (CN3, design non-goals): propagation is expressed entirely over the existing recorded ChildWorkflowFailed. Depends on OBS-001 so the engine/server exit logging lands first; this brief layers the child-specific edge on top and re-verifies the shared invariant/gate items C16-C18 across its own diff.

## Task

Make a failed child loud and verifiably visible using the ChildWorkflowFailed event the watcher already records — no new event, no new push channel, no status field. Two parts: (1) Add an error-level tracing call in crates/aion/src/runtime/nif_child_watch.rs::record_parent_child_terminal at the point it records a ChildWorkflowFailed terminal (the TerminalOutcome::Failed arm, and the Cancelled/TimedOut arms that map to ChildWorkflowFailed), carrying parent workflow id, parent run id, child workflow id, and the child's WorkflowError.message — matching the warn/error lines run_watch already emits for transient and invariant record failures. The Completed arm must not log at error. The log is a pure side effect: it changes no recorded event and no returned RecordDisposition (CN5). (2) Verify the two existing visibility paths carry the failure: (a) a parent history containing a ChildWorkflowFailed is encoded into the describe response timeline by crates/aion-server/src/api/handlers/describe.rs and crates/aion-server/src/api/handlers/payload.rs::encode_history, with the child id and reason intact; (b) a watcher subscribed through the existing crates/aion-server/src/stream/ subscription receives the parent-side ChildWorkflowFailed event because it is a normal appended event. Scope boundary — in: the child-failure log in nif_child_watch.rs, tests asserting it, and tests verifying the describe-timeline and watcher-stream paths carry the failure; re-verification of shared invariant/gate items C16-C18 across this diff. Out: the engine exit funnel log and the server record_events terminal-failure logs (OBS-001); any new event type, status field, or push path; any change to how the watcher records (record-before-wake discipline, retry/backoff) — only a log is added. Split-item note: C16/C17/C18 are cluster-wide invariant/gate items shared with OBS-001; this brief re-verifies them across its own diff at the cluster landing commit.

## Requirements

### R1: Loud error log when the watcher records a failed-child terminal

WHEN record_parent_child_terminal in crates/aion/src/runtime/nif_child_watch.rs records a ChildWorkflowFailed terminal for the parent — the TerminalOutcome::Failed arm, and the TerminalOutcome::Cancelled and TerminalOutcome::TimedOut arms that map to record_child_workflow_failed — THE SYSTEM SHALL emit an error-level tracing call carrying the parent workflow id, parent run id, child workflow id, and the recorded WorkflowError.message, consistent with the existing run_watch warn/error field conventions (parent_workflow_id, parent_run_id, child_workflow_id, reason/error via Display). THE SYSTEM SHALL NOT emit the error log on the TerminalOutcome::Completed arm, SHALL NOT emit it on the early-return ParentTerminal or AlreadyRecorded dispositions (no new terminal was recorded there), and SHALL NOT change the RecordDisposition returned or any event recorded (CN5).

**Acceptance:**
- On the TerminalOutcome::Failed arm, after record_child_workflow_failed succeeds, an ERROR-level tracing event is emitted carrying parent_workflow_id, parent_run_id, child_workflow_id, and the child's WorkflowError.message — asserted by a capturing-subscriber test driving record_parent_child_terminal with a Failed outcome and finding the ERROR record with the message text.
- On the TerminalOutcome::Completed arm, no ERROR-level child-failure record is emitted by record_parent_child_terminal.
- When record_parent_child_terminal returns AlreadyRecorded (the child terminal was already on the parent segment) or ParentTerminal, no new ERROR-level child-failure record is emitted (the failure is logged once, at the recording, mirroring P4).
- The Cancelled and TimedOut arms, which map to record_child_workflow_failed with the 'cancelled:'/'timed_out:' prefixed message, also emit the ERROR record carrying that message.
- cargo test -p aion passes including the new nif_child_watch logging test; the RecordDisposition returned by each arm is unchanged from before.

### R2: Failed child surfaces on the parent's describe timeline

THE SYSTEM SHALL surface a failed child on the parent run's describe output: WHEN a parent history contains a ChildWorkflowFailed event and describe is requested with history included, THE SYSTEM SHALL return a timeline that includes the encoded ChildWorkflowFailed carrying the child workflow id and the child's failure reason, via crates/aion-server/src/api/handlers/describe.rs and crates/aion-server/src/api/handlers/payload.rs::encode_history. This is a verification requirement over the existing describe path (status remains projected by status_from_events; CN1) — THE SYSTEM SHALL NOT add a new field to the describe response or a new event to carry the child failure; the existing ChildWorkflowFailed encoding is what is verified.

**Acceptance:**
- A test builds a parent history of WorkflowStarted, ChildWorkflowStarted, ChildWorkflowFailed (with a known child workflow id and a WorkflowError message), runs it through the describe handler with include_history true, and asserts the returned timeline contains the encoded ChildWorkflowFailed with the child workflow id and the reason text intact.
- The same describe response's summary status is derived by status_from_events (the parent is still Running if it has not itself terminated), confirming the child failure is visible in the timeline without being conflated into the parent's projected status.
- No new field is added to the describe response type and no new Event variant is introduced (diff review confirms the encoding of the existing ChildWorkflowFailed is what carries it).
- cargo test -p aion-server passes including the describe-timeline child-failure test.

**Checklist:**

THE SYSTEM SHALL deliver the parent-side ChildWorkflowFailed event to a watcher subscribed through the existing crates/aion-server/src/stream/ subscription surface, because the event is a normal appended event the stream already forwards through its namespace gate. This is a verification requirement over the existing push path — THE SYSTEM SHALL NOT add a new push channel, SSE endpoint, or notification integration (design non-goal): the deliverable is a test proving the existing stream carries the failed-child event to a subscriber.

**Acceptance:**
- A test subscribes through the existing event-subscription path, appends a ChildWorkflowFailed event for a workflow within the subscribed namespace, and asserts the subscriber's stream yields the encoded ChildWorkflowFailed event.
- The test asserts no new push path was needed: the event arrives through the same subscription/forward mechanism used for other appended events (the test imports only the existing stream API).
- An event for a different namespace is not delivered to the subscriber (the existing namespace gate still applies; the new visibility does not bypass authorization).

## Boundaries

- SHALL NOT add the engine exit-funnel log (handle_process_exit_async) or the server record_events terminal-failure logs — those are OBS-001.
- SHALL NOT introduce a new Event variant, a new Payload type, a mutable status field, or any second source of truth for the child's status (CN1, CN3).
- SHALL NOT add a new push channel, SSE endpoint, dashboard integration, or notification path — the existing event-stream subscription carries the ChildWorkflowFailed (design non-goal).
- SHALL NOT call EventStore::append directly; the ChildWorkflowFailed surfaced is the one already recorded through the parent's Recorder under its lock (CN4).
- SHALL NOT change the child-terminal watcher's record-before-wake discipline, retry/backoff, or run-chain following — this brief adds a log and verifies existing paths, nothing more.
- SHALL NOT add a new field to the describe response or change how the timeline is encoded — the existing ChildWorkflowFailed encoding is verified, not extended.
- SHALL NOT add any timeout, deadline, rate limit, or cap (ADR-003, CN8).
